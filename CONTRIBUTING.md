# 贡献指南

## 核心约束

### 1. 开源合规约束

**强制执行**：

- ❌ 禁止引入任何需要付费授权的商业代码、SDK、组件库
- ❌ 禁止调用核心付费 SaaS API（仅可选的增值服务除外）
- ❌ 禁止引入 GPL/LGPL 传染性协议的依赖（避免开源传染风险）
- ✅ 仅允许使用 Apache/MIT/BSD/PostgreSQL 等宽松开源协议的依赖
- ✅ 新依赖引入必须提交 PR 并注明开源协议类型，经过合规审核

**依赖审核流程**：

1. 检查 dependency 的 LICENSE 文件
2. 确认协议与 Apache 2.0 兼容
3. 在 PR 中注明依赖名称、版本、协议类型
4. 维护者审核通过后方可合并

**特殊处理**：

- Ghostscript (AGPL v3)：仅通过命令行独立进程调用，不直接链接，不修改源码，符合 AGPL 边界

---

### 2. 多租户架构约束

**数据库层**：

- 所有业务表必须包含 `tenant_id BIGINT NOT NULL` 字段
- 所有索引必须以 `tenant_id` 作为最左前缀
- DDL 变更必须同步更新所有租户的表结构

**代码层**：

- DAO层SQL必须经过 `TenantSqlInterceptor` 自动注入租户过滤条件
- 禁止在代码中硬编码绕过租户过滤
- 跨租户操作必须使用 `@SkipTenantFilter` 注解，并经过权限校验

**服务层**：

- MQ消息头必须携带 `tenant_id`
- 缓存Key必须包含 `tenant_id` 前缀
- 日志MDC必须包含 `tenant_id`，便于按租户排查问题

---

### 3. 审计与安全约束

**审计日志**：

- 所有用户操作（登录、预览、下载、导出、修改、删除）必须记录审计日志
- 审计日志表通过数据库触发器禁止 UPDATE/DELETE 操作
- 每条日志必须包含：用户ID、租户ID、操作类型、IP地址、时间戳、操作结果

**数据安全**：

- 敏感字段（数据库密码、API密钥）必须加密存储，使用 `EncryptTypeHandler`
- 导出PDF必须添加隐形水印，溯源泄露渠道
- 前端预览必须添加全屏浮水印
- 文件上传必须经过：Magic Number校验 → 病毒扫描 → 大小限制

**权限控制**：

- 所有API必须配置 Spring Security 权限校验
- 三权分立：系统管理员、安全管理员、审计管理员
- 敏感操作（DDL、数据源配置）必须经过审批流程

---

### 4. 性能与可靠性约束

**数据处理**：

- 超过 1 万行的数据操作必须使用异步处理
- 数据库查询必须使用游标（Cursor）流式读取，禁止一次性加载全量数据
- Excel导入导出必须使用 EasyExcel 流式API，禁止使用 Apache POI 内存模式
- 单文件大小限制：500MB

**任务调度**：

- 所有耗时任务必须经过公平调度器，禁止直接占用线程
- 大任务必须自动分片，单分片不超过 10 万行
- 任务必须支持幂等执行，重复执行不产生脏数据
- 任务失败必须支持自动重试（最多3次）+ 手动重试

**内存保护**：

- Chrome Headless 渲染容器必须配置内存 Limit（2GB）
- JVM 堆内存配置必须预留 30% 缓冲区
- 大对象处理后必须手动释放引用， assist GC

---

### 5. 代码质量约束

**编码规范**：

- 遵循 Alibaba Java Coding Guidelines
- 使用 `@Slf4j` 记录日志，禁止 `System.out.println`
- 异常必须包含上下文信息（tenantId, taskId, userId），便于排查
- 禁止捕获异常后吞掉不记录日志

**单元测试**：

- 核心业务模块（报表导入、导出、PDF生成）测试覆盖率 ≥ 80%
- 边界场景必须测试（空数据、超大文件、异常输入）
- 集成测试必须覆盖完整链路

**代码评审**：

- 所有代码必须经过至少1人评审后方可合并
- 评审重点：多租户隔离、安全合规、性能风险、异常处理

## 开发流程

### 设计先行原则

**任何代码编写前必须先完成设计文档并通过评审。**

```
1. 需求确认
    ↓
2. 编写设计文档（使用 docs/02-设计文档模板.md）
    ↓
3. 提交设计评审 PR
    ↓
4. 至少1人评审通过
    ↓
5. 开始编码实现
    ↓
6. 提交代码 PR
    ↓
7. 代码评审通过
    ↓
8. 合并
```

**不需要设计文档的例外情况**

- 简单 Bug 修复（不改变核心逻辑）
- 文档修正
- 代码格式调整

但即使是这些情况，也建议在 Issue 或 PR 中说明思路。

## Git 提交规范

### Commit Message 格式

```
<type>(<scope>): <subject>

<body>
```

### Type 类型

- `feat`: 新功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具/依赖更新

### 示例

```
feat(report): 支持Excel模板驱动导出模式

- 新增模板上传与管理
- 新增占位符解析引擎
- 支持合并单元格、字体样式保留

Closes #123
```

---

## 分支管理

- `main`: 主分支，稳定可发布版本
- `develop`: 开发分支
- `feature/xxx`: 功能分支，从 develop 检出
- `hotfix/xxx`: 紧急修复分支，从 main 检出

---

## 发布流程

1. 功能开发完成，合并到 develop 分支
2. 执行全量回归测试
3. 更新版本号与CHANGELOG
4. 合并到 main 分支，打 Tag
5. 执行发布脚本

---

## 问题反馈

- Bug 报告：提交 Issue，附带复现步骤、日志、环境信息
- 功能建议：提交 Issue，说明使用场景与预期效果
- 安全漏洞：请直接邮件联系维护者，不要公开提交 Issue
