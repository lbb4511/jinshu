# 开放约束规范

## 1. 核心原则

本项目采用"完全开放"模式，对所有参与者（人类开发者、AI助手、外部贡献者）一视同仁，但所有参与者都必须遵守相应的约束。

| 角色 | 许可范围 | 核心约束 |
|-----|---------|---------|
| **核心开发者** | 全功能开发、代码合并 | 遵守所有约束 + 架构评审 |
| **外部贡献者** | 提交PR、修复Bug | 遵守开源合规 + 代码规范 |
| **AI 助手** | 辅助编码、文档编写 | 禁止凭空臆造、必须基于事实 |

---

## 2. 针对所有参与者的通用约束

### 2.1 开源合规约束（强制执行）

**❌ 绝对禁止**

- 禁止引入任何需要付费授权的商业代码、SDK、组件库
- 禁止调用核心付费 SaaS API（仅可选的增值服务除外）
- 禁止引入 GPL/LGPL 等传染性协议的依赖（避免开源传染风险）
- 禁止复制未注明来源的第三方代码

**✅ 必须做到**

- 仅允许使用 Apache/MIT/BSD/PostgreSQL 等宽松开源协议的依赖
- 新依赖引入必须提交 PR 并注明开源协议类型，经过合规审核
- 复制第三方代码片段必须注明来源与协议

**依赖审核流程**

1. 检查 dependency 的 LICENSE 文件
2. 确认协议与 Apache 2.0 兼容
3. 在 PR 中注明依赖名称、版本、协议类型
4. 维护者审核通过后方可合并

**特殊处理说明**：

- Ghostscript (AGPL v3)：仅通过命令行独立进程调用，不直接链接，不修改源码，符合 AGPL 边界

---

### 2.2 架构约束

**多租户架构**

- 所有业务表必须包含 `tenant_id BIGINT NOT NULL` 字段
- 所有索引必须以 `tenant_id` 作为最左前缀
- DAO层SQL必须经过租户拦截器自动注入过滤条件
- 禁止在代码中硬编码绕过租户过滤
- 跨租户操作必须使用 `@SkipTenantFilter` 注解，并经过权限校验

**审计与安全**

- 所有用户操作必须记录审计日志
- 审计日志表通过数据库触发器禁止 UPDATE/DELETE 操作
- 敏感字段（数据库密码、API密钥）必须加密存储
- 所有API必须配置 Spring Security 权限校验
- 三权分立：系统管理员、安全管理员、审计管理员

---

### 2.3 设计先行约束（强制执行）

**🚫 绝对禁止：先写代码，后补设计**

任何功能开发必须严格遵循以下流程：

```
需求确认 → 编写设计文档 → 设计评审通过 → 开始编码
```

**设计文档要求**

1. **所有新功能必须有设计文档**
   - 新增接口、新增表、新增模块、重构核心逻辑，都必须先写设计文档
   - 设计文档使用统一模板：`docs/02-设计文档模板.md`
   - 简单 Bug 修复不需要设计文档，但需要在 PR 中说明原因

2. **设计文档的最低标准**
   - 一个合格的开发者拿到设计文档就能开始写代码
   - 不需要再问"这个地方怎么做"这类问题
   - 所有关键决策点都已经明确

3. **设计评审流程**
   - 提交 PR 时必须附带设计文档链接
   - 设计文档必须至少 1 人评审通过才能开始编码
   - 代码实现与设计文档不一致时，必须先更新设计文档

4. **设计文档的存放位置**

   ```
   docs/
   ├── 01-立项与设计/          # 系统级设计
   ├── 02-模块设计/            # 模块级设计
   │   ├── 报表导入模块设计.md
   │   └── PDF生成模块设计.md
   └── 03-功能设计/            # 单功能设计
       └── XXX功能设计.md
   ```

**违反后果**

- 没有设计文档的 PR 一律打回，不予评审
- 先写代码后补设计的，代码全部作废，从头开始
- 屡次违反者撤销提交权限

---

### 2.4 代码质量约束

**编码规范**

- 遵循 Alibaba Java Coding Guidelines
- 使用 `@Slf4j` 记录日志，禁止 `System.out.println`
- 异常必须包含上下文信息（tenantId, taskId, userId），便于排查
- 禁止捕获异常后吞掉不记录日志

**单元测试**

- 核心业务模块（报表导入、导出、PDF生成）测试覆盖率 ≥ 80%
- 边界场景必须测试（空数据、超大文件、异常输入）
- 集成测试必须覆盖完整链路

**代码评审**

- 所有代码必须经过至少1人评审后方可合并
- 评审重点：多租户隔离、安全合规、性能风险、异常处理

---

## 3. 针对人类开发者的专项约束

### 3.1 行为约束

- ✅ 提交代码前必须本地测试通过
- ✅ 新功能必须包含单元测试（核心模块≥80%覆盖率）
- ✅ 代码必须经过至少1人评审方可合并
- ❌ 禁止提交未完成的代码到主干分支
- ❌ 禁止擅自修改核心架构（需经过架构评审）
- ❌ 禁止 force push 主干分支

### 3.2 Git 提交规范

**Commit Message 格式**

```
<type>(<scope>): <subject>

<body>
```

**Type 类型**

- `feat`: 新功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具/依赖更新

### 3.3 分支管理

- `main`: 主分支，稳定可发布版本
- `develop`: 开发分支
- `feature/xxx`: 功能分支，从 develop 检出
- `hotfix/xxx`: 紧急修复分支，从 main 检出

---

## 4. 违约处理

| 违约类型 | 处理方式 |
|---------|---------|
| 轻度（代码风格、注释问题） | 评审打回，要求修改 |
| 中度（架构违规、缺少测试） | 代码回滚，通报批评 |
| 重度（开源合规事故、安全漏洞） | 撤销提交权限，追究责任 |
| 屡次严重违约 | 永久驱逐出项目 |

---

## 5. 本文件的约束力

本文件是项目开发的**强制性规范**：

- 所有人类开发者必须阅读并同意
- 所有参与本项目的 AI 助手，其系统提示词必须引用本文件
- 任何违反本文件约定的代码提交与文档变更，一律打回重写

> 本规范的核心精神：**开放但不纵容**。自由来自于对规则的尊重。
