# review-design - 设计文档评审技能

按照项目标准自动评审设计文档的完整性、规范性和安全性，确保开发前的设计质量。

## 评审范围

### ✅ 结构完整性检查
- 检查 11 个必填章节是否完整
- 验证每个章节的内容深度是否达标

### ✅ 多租户架构检查
- 数据库表是否包含 `tenant_id` 字段
- 索引是否以 `tenant_id` 作为最左前缀
- SQL 查询是否考虑租户过滤
- 是否有绕过租户拦截器的风险说明

### ✅ 安全合规检查
- 是否包含权限控制设计
- 敏感数据是否有加密方案
- 是否有审计日志设计
- 是否符合等保三级要求

### ✅ API 设计检查
- 是否符合统一返回格式规范
- 是否定义了错误码
- 是否有参数校验设计
- 是否考虑了幂等性

### ✅ 性能设计检查
- 是否有性能目标指标
- 是否考虑了大数据量场景
- 是否有缓存方案设计
- 是否有压测方案

### ✅ 测试方案检查
- 是否包含单元测试计划
- 是否包含集成测试场景
- 是否包含边界值测试用例
- 是否包含并发测试设计

## 使用方法

```bash
# 评审单个设计文档
/review-design docs/04-模块设计/01.报表导入模块设计.md

# 评审整个目录
/review-design docs/04-模块设计/

# 严格模式（不通过则返回错误）
/review-design docs/05-功能设计/ --strict

# 仅输出评审摘要
/review-design docs/04-模块设计/ --summary

# 生成评审报告文件
/review-design docs/04-模块设计/ --report
```

## 评审等级

| 等级 | 颜色 | 说明 | 处理方式 |
|------|------|------|---------|
| P0 | 🔴 严重 | 缺少必填章节或有严重安全漏洞 | 必须修复，不允许进入开发 |
| P1 | 🟡 重要 | 设计不完整或有明显缺陷 | 建议修复后再开始开发 |
| P2 | 🟢 建议 | 可以优化的点 | 可在开发过程中逐步完善 |

## 标准通过条件

设计文档评审通过必须满足：
1. ✅ 11 个必填章节完整且内容充实
2. ✅ 数据库设计符合多租户隔离规范
3. ✅ API 设计符合统一返回规范
4. ✅ 包含明确的安全考量
5. ✅ 包含可执行的测试方案
6. ❌ 无 P0 级别问题
7. ❌ P1 级别问题不超过 2 个

## 评审报告示例

```
📋 设计文档评审报告
════════════════════════════
文档: 01.报表导入模块设计.md
评审时间: 2026-05-12

🔴 P0 问题 (1个):
  - 缺少「异常处理设计」章节

🟡 P1 问题 (2个):
  - 数据库设计未明确说明 tenant_id 索引策略
  - 性能目标缺少具体的 TPS/响应时间指标

🟢 P2 建议 (1个):
  - 建议增加断点续传场景的测试用例

════════════════════════════
❌ 评审不通过 - 请修复 P0 问题后重新评审
```
