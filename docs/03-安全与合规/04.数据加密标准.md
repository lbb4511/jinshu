## 1. 背景与目标

### 1.1 问题背景

锦书系统涉及以下敏感数据，当前均以明文存储/传输：

| 敏感数据类别         | 示例                           | 风险                   |
| -------------------- | ------------------------------ | ---------------------- |
| 数据源连接密码       | PostgreSQL/MySQL 密码          | 数据库被拖库后直接泄露 |
| API Key / Token      | 企业微信 Webhook Key           | 攻击者可调用外部接口   |
| 租户个人身份信息     | 手机号、身份证号、银行卡号     | 违反个保法             |
| 报表业务敏感数据     | 财务报表金额、人力报表薪资     | 商业机密泄露           |
| 水印追溯信息         | PDF 隐式水印编码的用户 ID      | 泄露追踪链断裂         |

### 1.2 设计目标

| 需求               | 方案                                          |
| ------------------ | --------------------------------------------- |
| 字段级加密         | AES-256-GCM 认证加密 + MyBatis TypeHandler 透明加解密 |
| 密钥管理           | K8s Secret 注入 + 版本化密钥 + 在线轮换       |
| 敏感字段脱敏       | 角色级动态脱敏 + MyBatis TypeHandler + 可配置规则表 |
| 传输加密           | TLS 1.3，内部服务走 K8s 内部网络（不对外暴露） |
| 合规基线           | 满足等保三级"数据安全"检查项的加密和脱敏要求  |

### 1.3 非目标

- 不实现全盘/全表加密（如 PostgreSQL TDE）—— 依赖云服务商提供
- 不实现硬件加密（HSM）—— 初期使用软件 AES
- 不实现同态加密/可搜索加密 —— 超出业务需求
- 不在 MVP 阶段实现文件级透明加密（共享存储 PVC 的快照备份已满足 RPO 要求）

## 3. 字段级加密设计

### 3.1 加密算法

采用 **AES-256-GCM**（Galois/Counter Mode）：

- GCM 模式提供 **认证加密**（AEAD），自动校验密文完整性
- 每次加密生成随机 12 字节 IV，与密文拼接存储
- 存储格式：`{version}:{base64(iv)}:{base64(ciphertext)}`

```
加密流程：
  plaintext → AES-GCM(key, random_iv) → iv + ciphertext + auth_tag
  → Base64(version || iv || ciphertext+tag)
  → 写入 DB

解密流程：
  DB 读取 → Base64 解码 → 提取 version, iv, ciphertext
  → 根据 version 获取对应 key
  → AES-GCM(key, iv, ciphertext) → plaintext
```

### 3.2 需要加密的字段

| 表名            | 字段                   | 加密原因                     | 加密后长度 |
| --------------- | ---------------------- | ---------------------------- | ---------- |
| `data_source`   | `connection_config`    | 包含数据库密码（JSONB 中密码字段加密） | TEXT 列，约 +30% |
| `report_subscriptions` | `webhook_url`   | 含企业微信/钉钉 Webhook Key  | VARCHAR(512) → VARCHAR(1024) |
| `tenant`         | `smtp_password`        | 邮件推送 SMTP 密码           | VARCHAR(128) → VARCHAR(256) |
| `tenant`         | `api_key`              | 第三方 API Key               | VARCHAR(256) → VARCHAR(512) |
| `data_source`    | `password` (JDBC)      | 数据源连接密码               | VARCHAR(128) → VARCHAR(256) |

> `connection_config` 为 JSONB 类型，加密策略：仅加密子字段 `password`，其他字段保持明文以便于查询。

### 3.4 EncryptTypeHandler 实现

```java
// common/src/main/java/com/jinshu/common/security/EncryptTypeHandler.java
@MappedJdbcTypes(JdbcType.VARCHAR)
@MappedTypes(String.class)
public class EncryptTypeHandler extends BaseTypeHandler<String> {

    private final KeyManager keyManager;

    public EncryptTypeHandler(KeyManager keyManager) {
        this.keyManager = keyManager;
    }

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i,
                                     String parameter, JdbcType jdbcType) throws SQLException {
        String encrypted = keyManager.encrypt(parameter);
        ps.setString(i, encrypted);
    }

    @Override
    public String getNullableResult(ResultSet rs, String columnName) throws SQLException {
        String encrypted = rs.getString(columnName);
        return (encrypted != null) ? keyManager.decrypt(encrypted) : null;
    }

    @Override
    public String getNullableResult(ResultSet rs, int columnIndex) throws SQLException {
        String encrypted = rs.getString(columnIndex);
        return (encrypted != null) ? keyManager.decrypt(encrypted) : null;
    }

    @Override
    public String getNullableResult(CallableStatement cs, int columnIndex) throws SQLException {
        String encrypted = cs.getString(columnIndex);
        return (encrypted != null) ? keyManager.decrypt(encrypted) : null;
    }
}
```

### 3.5 AES-256-GCM 工具类

```java
// common/src/main/java/com/jinshu/common/security/CryptoUtils.java
public class CryptoUtils {

    private static final String ALGORITHM = "AES/GCM/NoPadding";
    private static final int GCM_IV_LENGTH = 12;      // 96 bit
    private static final int GCM_TAG_LENGTH = 128;    // bit
    private static final SecureRandom SECURE_RANDOM = new SecureRandom();

    /**
     * 加密
     * @return "{keyVersion}:{base64(iv)}:{base64(ciphertext)}"
     */
    public static String encrypt(byte[] key, int keyVersion, byte[] plaintext) {
        try {
            byte[] iv = new byte[GCM_IV_LENGTH];
            SECURE_RANDOM.nextBytes(iv);

            Cipher cipher = Cipher.getInstance(ALGORITHM);
            GCMParameterSpec spec = new GCMParameterSpec(GCM_TAG_LENGTH, iv);
            SecretKeySpec keySpec = new SecretKeySpec(key, "AES");
            cipher.init(Cipher.ENCRYPT_MODE, keySpec, spec);

            byte[] encrypted = cipher.doFinal(plaintext);

            return keyVersion + ":"
                + Base64.getEncoder().encodeToString(iv) + ":"
                + Base64.getEncoder().encodeToString(encrypted);

        } catch (Exception e) {
            throw new RuntimeException("AES-GCM 加密失败", e);
        }
    }

    /**
     * 解密
     * @param encryptedText "{keyVersion}:{base64(iv)}:{base64(ciphertext)}"
     */
    public static byte[] decrypt(byte[] key, String encryptedText) {
        try {
            String[] parts = encryptedText.split(":", 3);
            if (parts.length != 3) {
                throw new IllegalArgumentException("密文格式错误");
            }

            int version = Integer.parseInt(parts[0]);
            byte[] iv = Base64.getDecoder().decode(parts[1]);
            byte[] ciphertext = Base64.getDecoder().decode(parts[2]);

            Cipher cipher = Cipher.getInstance(ALGORITHM);
            GCMParameterSpec spec = new GCMParameterSpec(GCM_TAG_LENGTH, iv);
            SecretKeySpec keySpec = new SecretKeySpec(key, "AES");
            cipher.init(Cipher.DECRYPT_MODE, keySpec, spec);

            return cipher.doFinal(ciphertext);

        } catch (AEADBadTagException e) {
            throw new SecurityException("密文完整性校验失败（可能被篡改）", e);
        } catch (Exception e) {
            throw new RuntimeException("AES-GCM 解密失败", e);
        }
    }
}
```

## 4. 密钥管理

### 4.1 密钥存储

```
密钥生命周期：
  K8s Secret (Base64)
       │
       ▼
  Pod 环境变量 (JINSHU_ENCRYPTION_KEY_V1, JINSHU_ENCRYPTION_KEY_V2, ...)
       │
       ▼
  Spring @ConfigurationProperties 读取
       │
       ▼
  KeyManager 初始化（多版本密钥 Map）
       │
       ▼
  EncryptTypeHandler / DesensitizeTypeHandler 调用
```

**K8s Secret 定义**：

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: jinshu-encryption-keys
type: Opaque
data:
  key-v1: "base64encoded-256bit-key-v1"
  key-v2: "base64encoded-256bit-key-v2"
  latest-version: "Mg=="  # base64 of "2"
```

**Spring 配置读取**：

```java
// common/src/main/java/com/jinshu/common/security/KeyManager.java
@Component
public class KeyManager {

    /** 当前加密密钥版本 → 密钥字节 */
    private final Map<Integer, byte[]> keys = new ConcurrentHashMap<>();
    /** 最新版本号（新数据加密时使用） */
    private int latestVersion;

    public KeyManager(@Value("${jinshu.encryption.key-v1}") String keyV1,
                      @Value("${jinshu.encryption.key-v2:#{null}}") String keyV2) {
        addKey(1, keyV1);
        if (keyV2 != null) {
            addKey(2, keyV2);
            this.latestVersion = 2;
        } else {
            this.latestVersion = 1;
        }
    }

    private void addKey(int version, String base64Key) {
        keys.put(version, Base64.getDecoder().decode(base64Key));
    }

    public String encrypt(String plaintext) {
        byte[] key = keys.get(latestVersion);
        return CryptoUtils.encrypt(key, latestVersion, plaintext.getBytes(StandardCharsets.UTF_8));
    }

    public String decrypt(String encryptedText) {
        // 从密文中提取版本号
        int version = extractVersion(encryptedText);
        byte[] key = keys.get(version);
        if (key == null) {
            throw new SecurityException("未知密钥版本: " + version);
        }
        byte[] plainBytes = CryptoUtils.decrypt(key, encryptedText);
        return new String(plainBytes, StandardCharsets.UTF_8);
    }

    private int extractVersion(String encryptedText) {
        return Integer.parseInt(encryptedText.split(":")[0]);
    }
}
```

### 4.2 密钥轮换机制

**触发条件**：

- 定期轮换：每 90 天
- 事件触发：疑似密钥泄露（安全响应流程 §3）

**轮换流程**：

```
第 1 步：生成新密钥（v3），加入 K8s Secret
         → 重启所有 Pod（新密钥读入 KeyManager）
第 2 步：新写入数据自动使用 v3 加密（latestVersion = 3）
         旧数据仍用 v1/v2 解密（多版本共存）
第 3 步（后台）：异步任务批量重新加密历史数据
         UPDATE data_source SET password = RE_ENCRYPT(password)
         替换为 v3 密文
第 4 步：确认无 v1 密文后，从 K8s Secret 删除 key-v1
         再次重启 Pod
```

### 4.3 密钥版本迁移流程

```java
// 后台任务：数据重加密
@Component
public class KeyRotationTask {

    @Scheduled(cron = "0 0 3 * * ?")  // 每天凌晨 3 点
    public void reEncryptStaleData() {
        List<Long> staleIds = dataSourceMapper
            .selectIdsByOldKeyVersion(latestVersion - 1);

        for (Long id : staleIds) {
            DataSourceEntity ds = dataSourceMapper.selectById(id);
            // password 为旧版密文，setPassword() 触发 EncryptTypeHandler 自动用新密钥重加密
            dataSourceMapper.updateById(ds);
        }
    }
}
```

## 5. 数据传输加密

| 通信路径                     | 加密方式   | 说明                                     |
| ---------------------------- | ---------- | ---------------------------------------- |
| 用户浏览器 ↔ Nginx Ingress  | TLS 1.3    | Let's Encrypt / 自签证书（内网）         |
| Nginx ↔ API (K8s 内部)      | HTTP       | K8s 内部网络隔离，NetworkPolicy 控制     |
| API → Worker（MQ）          | TLS 1.2    | RabbitMQ TLS，消息体不额外加密           |
| Worker → Renderer (HTTP)    | HTTP       | K8s 内部网络隔离                         |
| API/Worker → PostgreSQL     | TLS 1.3    | PostgreSQL SSL 连接                      |
| 应用 → Redis                | TLS        | Redis TLS（生产环境）                    |

> MVP 阶段：K8s 内部网络默认不加密（集群内网隔离足够）。生产环境启用 PostgreSQL SSL + Redis TLS。

## 6. 敏感字段脱敏设计

### 6.1 脱敏规则定义

| 规则类型        | 原始值              | 脱敏后                | 适用字段         |
| --------------- | ------------------- | --------------------- | ---------------- |
| `PHONE`         | `13812345678`       | `138****5678`         | 手机号           |
| `ID_CARD`       | `110101199001011234`| `110101********1234`  | 身份证号         |
| `BANK_CARD`     | `6222021234567890`  | `6222******7890`      | 银行卡号         |
| `EMAIL`         | `zhangsan@abc.com`  | `z***@abc.com`        | 邮箱             |
| `NAME`          | `张三`              | `张*`                 | 姓名             |
| `AMOUNT`        | `125680.50`         | `***` / `125***`      | 金额（分级脱敏） |
| `FULL_MASK`     | `any value`         | `****`                | 任意敏感字段     |

### 6.2 分级脱敏策略

| 角色            | 手机号        | 身份证号      | 金额         | 说明                       |
| --------------- | ------------ | ------------- | ------------ | -------------------------- |
| `ADMIN`           | 明文         | 明文          | 明文         | 平台管理员，全量数据可见   |
| `SECURITY_ADMIN`  | 明文         | 明文          | 明文         | 安全管理员，审计需要       |
| `USER`（制表人）  | `138****5678`| `110101******1234` | 明文  | 可看报表数据，个人信息脱敏 |
| `VIEWER`（访客）  | `138****5678`| `110101******1234` | `***`  | 仅看汇总，明细金额脱敏     |
| `AUDITOR`（审计员）| 明文         | 明文          | 明文         | 审计需要全量数据           |

> 脱敏策略默认 USER 级。VIEWER 级的金额脱敏由报表数据查询层实现（见 §7）。

### 6.4 DesensitizeTypeHandler 实现

```java
// common/src/main/java/com/jinshu/common/security/DesensitizeTypeHandler.java
@Component
public class DesensitizeService {

    /**
     * 根据当前用户角色和脱敏规则，对字段值进行脱敏
     */
    public String desensitize(String value, DesensitizeType type) {
        if (value == null || value.isEmpty()) return value;

        // 获取当前用户角色
        String role = UserContext.getRole();
        if (role == null || "ADMIN".equals(role) || "SECURITY_ADMIN".equals(role)
            || "AUDITOR".equals(role)) {
            return value; // 管理员/安全管理员/审计员 不脱敏
        }

        // VISITOR 角色额外增加金额脱敏（在规则表查询）
        // 基础规则执行脱敏
        return switch (type) {
            case PHONE    -> maskPhone(value);
            case ID_CARD  -> maskIdCard(value);
            case BANK_CARD -> maskBankCard(value);
            case EMAIL    -> maskEmail(value);
            case NAME     -> maskName(value);
            case AMOUNT   -> maskAmount(value, role);
            case FULL_MASK -> "****";
        };
    }

    private String maskPhone(String value) {
        return value.replaceAll("(\\d{3})\\d{4}(\\d{4})", "$1****$2");
    }

    private String maskIdCard(String value) {
        return value.replaceAll("(\\d{6})\\d{8}(\\d{4})", "$1********$2");
    }

    private String maskBankCard(String value) {
        return value.replaceAll("(\\d{4})\\d{8,12}(\\d{4})", "$1******$2");
    }

    private String maskEmail(String value) {
        return value.replaceAll("(.).*@(.*)", "$1***@$2");
    }

    private String maskName(String value) {
        if (value.length() <= 2) return value.charAt(0) + "*";
        return value.charAt(0) + "*".repeat(value.length() - 1);
    }

    private String maskAmount(String value, String role) {
        if ("VIEWER".equals(role)) return "***";
        return value; // USER 角色可见金额明文
    }
}
```

### 6.5 脱敏规则配置表

## 8. 异常处理

| 异常场景                 | 错误码  | 处理方式                                     |
| ------------------------ | ------- | -------------------------------------------- |
| 密文格式错误（无法解析） |  500    | 记录 ERROR 日志，返回"数据解密失败"          |
| 密文完整性校验失败       |  500    | AEADBadTagException → 安全告警（可能被篡改） |
| 密钥版本不存在           |  500    | KeyManager 缺少对应版本密钥                  |
| 加密操作失败             |  500    | 记录 ERROR 日志 + 告警                       |
| K8s Secret 未挂载        | 启动失败 | 应用启动时校验必填密钥                        |

**启动时密钥校验**：

```java
@Configuration
public class KeyManagerHealthIndicator implements HealthIndicator {

    private final KeyManager keyManager;

    @Override
    public Health health() {
        try {
            String test = keyManager.encrypt("health_check");
            String decrypted = keyManager.decrypt(test);
            if ("health_check".equals(decrypted)) {
                return Health.up().build();
            }
            return Health.down().withDetail("error", "加解密自检失败").build();
        } catch (Exception e) {
            return Health.down().withDetail("error", e.getMessage()).build();
        }
    }
}
```

## 9. 性能考量

| 操作             | 耗时         | 优化措施                           |
| ---------------- | ------------ | ---------------------------------- |
| AES-256-GCM 加密 | < 0.1ms/次   | JDK 内置硬件加速（AES-NI）         |
| AES-256-GCM 解密 | < 0.1ms/次   | 同上                               |
| 单行加解密       | < 0.2ms      | TypeHandler 透明调用，无感知       |
| 批量重加密（万行）| ~2 秒        | 后台异步任务，不影响在线业务       |
| 脱敏（正则替换） | < 0.01ms/次  | 简单正则，可忽略                   |
| 脱敏规则查询     | < 1ms        | Redis 缓存（TTL 5 分钟）           |

> 加解密对单次 API 请求的影响 < 1ms，可忽略不计。

## 10. 安全考量

- K8s Secret 必须启用 `encryption-at-rest`（etcd 加密），防止 Secret 明文存储在 etcd
- 密钥绝不硬编码在代码或配置文件中（仅通过环境变量注入）
- 加密所用的 `SecureRandom` 使用默认强随机算法（`NativePRNGNonBlocking`）
- 脱敏规则表仅 `ADMIN` 和 `SECURITY_ADMIN` 角色可修改
- 日志中禁止输出解密后的明文敏感字段（Logback 自定 `replace` 规则或 MDC 过滤）
- TypeHandler 异常不得将明文泄露到日志

## 12. 上线方案

### 12.1 上线步骤

1. 生成 AES-256 密钥（`openssl rand -base64 32`），写入 K8s Secret
2. 部署 `EncryptTypeHandler` + `KeyManager` + `CryptoUtils`
3. 为 `data_source.password` 等字段添加 `@EncryptField` 注解
4. 修改 MyBatis XML 中相关列的 `typeHandler`
5. 后台任务对存量明文数据执行加密（首次上线，全部重加密）
6. 部署 `DesensitizeService` + `DesensitizeTypeHandler`
7. 初始化 `desensitize_rule` 表（Flyway V4）
8. 验证：新建数据源 → 查询 password 正常 → 直接查 DB 为密文

### 12.2 灰度策略

- 加密：先对 `data_source` 表启用，验证 1 周无异常后逐步扩展到其他表
- 脱敏：先对 `users.phone` 字段启用 USER/VIEWER 脱敏，验证后扩展到全字段
- 密钥轮换后台任务先禁用，验证稳定后启用

### 12.3 回滚方案

- 加密回滚：`@EncryptField(enabled = false)` → 新读写走明文，存量密文不变
- 脱敏回滚：`desensitize_rule.enabled = false` → 全局停用脱敏
- 密钥异常：回滚 K8s Secret 到上一版本，重启 Pod
- 存量密文回退明文：执行一次性 SQL 用 `KeyManager` 批量解密后写回
