## 1. 背景与目标

### 1.1 问题背景

当前系统缺乏用户认证与授权机制，存在以下问题：

| 问题                   | 影响                                                                            |
| ---------------------- | ------------------------------------------------------------------------------- |
| 无登录认证             | 任何请求理论上都可访问受保护资源                                                |
| 无 Token 签发/校验     | `SecurityConfig` 标记了 `anyRequest().authenticated()` 但无实际认证逻辑         |
| 多租户上下文注入链断裂 | `TenantContext` 和 `UserContext` 已就绪，但缺少从 Token 提取租户/用户信息的环节 |
| 无权限控制实施载体     | `@PreAuthorize` 可用，但没有 Token 中的角色信息作为判定依据                     |
| 无登出/Token 吊销机制  | 用户退出后 Token 仍可继续使用，存在安全隐患                                     |

### 1.2 设计目标

| 指标                 | 要求                              |
| -------------------- | --------------------------------- |
| 认证方式             | JWT（HS512 签名），无状态         |
| Access Token 有效期  | 30 分钟                           |
| Refresh Token 有效期 | 7 天                              |
| Token 吊销           | 支持（Redis 黑名单 + 版本号机制） |
| 登录失败锁定         | 5 次连续失败锁定账号 15 分钟      |
| Token 解析开销       | < 1ms（本地验证，不走网络）       |
| 并发支持             | ≥ 1000 登录请求/秒                |

### 1.3 非目标

- 不支持 OAuth 2.0 / OpenID Connect 第三方登录（预留扩展接口）
- 不支持双因素认证（2FA）—— 开发环境跳过，生产环境通过网关层实现
- 不支持社交账号登录（微信/钉钉）
- 不支持 SSO 单点登录（多系统共享 Token）

## 2. 需求分析

### 2.1 功能需求

- [x] **登录**：用户名 + 密码验证，返回 Access Token + Refresh Token
- [x] **Token 刷新**：使用 Refresh Token 换发新的 Access Token（Refresh Token 不续期）
- [x] **登出**：将当前 Access Token 加入黑名单，标记 Refresh Token 版本号失效
- [x] **Token 校验**：每个请求的 Filter 自动校验 Token 有效性
- [x] **上下文注入**：从 Token 提取 `userId`、`tenantId`、`role` 注入 `UserContext` 和 `TenantContext`
- [x] **角色鉴权**：基于 `@PreAuthorize` 注解实现方法级权限控制
- [x] **登录失败锁定**：5 次失败后锁定 15 分钟（Redis 计数）
- [x] **Token 版本号吊销**：管理员可强制某用户的所有 Token 失效

### 2.2 用户故事

```
作为 系统用户，我希望 使用用户名和密码登录系统，
以便 获取一个有时效性的 Token，安全访问受保护的 API 资源。

作为 前端开发者，我希望 在 Token 即将过期时自动使用 Refresh Token 刷新，
以便 用户不会在使用过程中被强制退出。

作为 安全管理员，我希望 能够在发现安全威胁时强制吊销某个用户的全部 Token，
以便 快速阻断未授权访问。

作为 被禁用的用户，我希望 系统立即拒绝我的 Token（即使尚未过期），
以便 权限变更能够即时生效。
```

### 2.3 边界场景

| 场景                         | 处理方式                                                            |
| ---------------------------- | ------------------------------------------------------------------- |
| Access Token 过期            | 返回 401，前端使用 Refresh Token 自动刷新                           |
| Refresh Token 过期           | 返回 401，前端跳转登录页                                            |
| Token 签名无效（伪造）       | 直接拒绝，记录 WARN 日志（含来源 IP），不暴露签名细节               |
| 用户被禁用后 Token 仍有效    | Filter 每次校验时查询 Redis 版本号，不匹配则拒绝                    |
| 登录时租户不存在/已禁用      | 拒绝登录，返回 `TENANT_NOT_FOUND`                                   |
| Redis 不可用（Token 黑名单） | 降级为仅验签 + 验过期，允许所有未过期 Token（可用性优先）           |
| 并发登录（同一账号多设备）   | 允许，各自持有独立的 Access Token，Refresh Token 各自独立           |
| Token 提前解析（预热攻击）   | Token 解析不依赖外部服务，无此风险                                  |
| 超大 Token（Payload 过重）   | JWT Body 仅含必要字段（userId + tenantId + role），不含用户详细信息 |

## 3. 架构设计

### 3.1 整体流程

**登录流程**：

```
用户提交 POST /api/auth/login { username, password }
        ↓
AuthController.login()
        ↓
AuthService.authenticate()
  ├── 1. 查询 Redis 锁定计数，超限直接拒绝
  ├── 2. UserMapper.selectByUsername() 查用户
  ├── 3. 用户不存在 → 返回 USERNAME_PASSWORD_ERROR（不区分用户/密码错误）
  ├── 4. 用户被禁用 → 返回 USER_DISABLED
  ├── 5. BCryptPasswordEncoder.matches() 验密
  ├── 6. 密码不匹配 → Redis 失败计数 +1，达到 5 次锁定 15 分钟
  ├── 7. 密码匹配 → 清除失败计数
  └── 8. 验证租户状态（不存在或禁用则拒绝）
        ↓
JwtTokenProvider.createAccessToken(userId, tenantId, role)
JwtTokenProvider.createRefreshToken(userId, tenantId)
        ↓
返回 { accessToken, refreshToken, expiresIn }
        ↓
记录审计日志 AuditLogger.log(LOGIN)
```

**请求认证流程**：

```
客户端请求（Header: Authorization: Bearer <accessToken>）
        ↓
JwtAuthenticationFilter.doFilterInternal()
  ├── 1. 提取 Authorization Header
  ├── 2. 无 Header → 放行（由 SecurityConfig 决定是否需要认证）
  ├── 3. 解析 Token（JwtTokenProvider.parseToken()）
  ├── 4. 验证签名、过期时间
  ├── 5. 检查 Redis 黑名单（如果 Redis 可用）
  ├── 6. 检查用户版本号（Redis user:{userId}:token_version）
  ├── 7. 版本号不匹配 → 拒绝，返回 401
  ├── 8. 提取 Claims → 设置 UserContext + TenantContext
  └── 9. 构建 UsernamePasswordAuthenticationToken 注入 SecurityContext
        ↓
请求结束 → Filter 的 finally 块清除 UserContext + TenantContext
```

**Token 刷新流程**：

```
客户端 POST /api/auth/refresh { refreshToken }
        ↓
AuthController.refresh()
        ↓
AuthService.refreshToken()
  ├── 1. 解析 Refresh Token，验证签名/过期
  ├── 2. 查询 Redis refresh:{userId}:{tokenId} 是否存在
  ├── 3. 不存在 → 拒绝（已登出或已过期）
  ├── 4. 查询用户状态（是否被禁用）
  ├── 5. 生成新的 Access Token（Refresh Token 本身不刷新）
  └── 6. 返回 { accessToken, expiresIn }
```

**登出流程**：

```
客户端 POST /api/auth/logout
Authorization: Bearer <accessToken>
        ↓
AuthController.logout()
        ↓
AuthService.logout()
  ├── 1. 将当前 Access Token 加入 Redis 黑名单（TTL = Token 剩余有效期）
  ├── 2. 从 Token 中提取 userId + tokenId
  ├── 3. 删除 Redis refresh:{userId}:{tokenId}
  ├── 4. 自增 Redis user:{userId}:token_version（吊销该用户所有 Token）
  └── 5. 清除 UserContext + TenantContext
        ↓
返回 200 OK
        ↓
记录审计日志 AuditLogger.log(LOGOUT)
```

### 3.3 关键算法与策略

**JWT Payload 结构**：

```json
// Access Token
{
  "sub": "1001",              // userId
  "tenant_id": "1",           // 租户ID
  "role": "ADMIN",            // 角色
  "iat": 1715937600,          // 签发时间（秒）
  "exp": 1715939400,          // 过期时间（签发 + 30分钟）
  "jti": "uuid-xxx"           // Token 唯一ID（用于黑名单）
}

// Refresh Token
{
  "sub": "1001",              // userId
  "tenant_id": "1",           // 租户ID
  "type": "refresh",          // Token 类型区分
  "iat": 1715937600,
  "exp": 1716542400,          // 签发 + 7天
  "jti": "uuid-yyy"           // Token 唯一ID
}
```

**Token 版本号吊销机制**：

```
Redis Key: user:{userId}:token_version  (String, 自增整数)

生成 Token 时：将当前版本号写入 Claims.version
校验 Token 时：
  1. 取 Token 中的 version
  2. 查询 Redis 当前版本号
  3. 不一致 → 拒绝（该用户所有 Token 已失效）

触发场景：
  - 用户登出（可选——仅删除 Refresh Token 即可，版本号不变）
  - 管理员强制吊销
  - 用户密码被修改
  - 用户角色被修改
  - 用户被禁用
```

**登录失败锁定策略**：

```
Redis Key: login_fail:{username}  (String, 自增计数器，TTL = 15分钟)

计数 ≥ 5 → 拒绝登录，返回 "账号已锁定，请15分钟后重试"
登录成功 → 删除计数器
每次失败 → 自增 + 重置 TTL
```

**Token 黑名单策略**：

```
Redis Key: token_blacklist:{jti}  (String, TTL = Token剩余过期时间)

登出时：将当前 Access Token 的 jti 写入黑名单，TTL 等于 Token 剩余有效期
验证时：检查黑名单是否存在
Redis 不可用时：跳过黑名单检查，仅验证签名 + 过期（可用性优先）
```

## 4. 数据库设计

### 4.1 表结构

**复用现有 `users` 表**（DDL 已就绪，无需新建）：

```sql
-- 已存在的 users 表，无需变更
CREATE TABLE users (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       BIGINT NOT NULL,
    username        VARCHAR(64) NOT NULL,
    password_hash   VARCHAR(255) NOT NULL,   -- BCrypt 哈希
    email           VARCHAR(128),
    role            VARCHAR(32) NOT NULL,    -- ADMIN / USER / VIEWER
    status          VARCHAR(16) NOT NULL DEFAULT 'ACTIVE',  -- ACTIVE / DISABLED
    created_at      TIMESTAMP DEFAULT NOW(),
    updated_at      TIMESTAMP DEFAULT NOW(),
    UNIQUE(tenant_id, username)
);
```

**复用现有 `tenant` 表**（DDL 已就绪，无需新建）：

```sql
-- 已存在的 tenant 表，无需变更
CREATE TABLE tenant (
    id              BIGSERIAL PRIMARY KEY,
    name            VARCHAR(128) NOT NULL,
    code            VARCHAR(64) NOT NULL UNIQUE,
    status          VARCHAR(16) NOT NULL DEFAULT 'ACTIVE',  -- ACTIVE / DISABLED
    quota_config    JSONB,
    created_at      TIMESTAMP DEFAULT NOW(),
    updated_at      TIMESTAMP DEFAULT NOW()
);
```

> 本模块不新增表。Token 黑名单、版本号、锁定计数全部存储在 Redis 中，不写入 PostgreSQL。

### 4.2 索引设计

`users` 表已有索引：

```sql
-- 已存在
CREATE INDEX idx_users_tenant ON users(tenant_id);
-- UNIQUE(tenant_id, username) 约束自带索引
```

**补充索引建议**（非强制）：

```sql
-- 按用户名全局搜索（管理员场景）
CREATE INDEX idx_users_username ON users(username);
```

### 4.3 数据迁移方案

- 本模块不涉及 DDL 变更，无需数据迁移
- 初始化用户数据：通过 Flyway 迁移脚本 `V2__seed_admin_user.sql` 插入默认管理员账号：

```sql
-- 默认管理员：admin / admin123（首次登录后强制修改）
INSERT INTO users (tenant_id, username, password_hash, email, role, status)
VALUES (0, 'admin', '$2a$10$...', 'admin@example.com', 'ADMIN', 'ACTIVE')
ON CONFLICT (tenant_id, username) DO NOTHING;
```

> BCrypt 哈希值手工生成或由应用启动时 `CommandLineRunner` 生成。

## 5. API 接口设计

### 5.1 接口列表

| 接口       | Method | 路径              | 权限要求 | 说明                          |
| ---------- | ------ | ----------------- | -------- | ----------------------------- |
| 登录       | POST   | /api/auth/login   | 无       | 用户名密码登录                |
| Token 刷新 | POST   | /api/auth/refresh | 无       | Refresh Token 换 Access Token |
| 登出       | POST   | /api/auth/logout  | 登录用户 | 吊销当前 Token                |

> 路径前缀 `/api` 由 `server.servlet.context-path=/api` 配置。

### 5.2 请求/响应示例

**登录**

```http
POST /api/auth/login
Content-Type: application/json

{
  "username": "admin",
  "password": "admin123"
}

Response (200):
{
  "code": 0,
  "data": {
    "accessToken": "eyJhbGciOiJIUzUxMiJ9...",
    "refreshToken": "eyJhbGciOiJIUzUxMiJ9...",
    "tokenType": "Bearer",
    "expiresIn": 1800,
    "user": {
      "id": 1001,
      "username": "admin",
      "tenantId": 1,
      "role": "ADMIN"
    }
  }
}
```

**登录失败（密码错误）**

```http
Response (401):
{
  "code": 1003,
  "message": "用户名或密码错误",
  "data": null
}
```

**登录失败（账号锁定）**

```http
Response (423):
{
  "code": 1004,
  "message": "账号已锁定，请15分钟后重试",
  "data": {
    "lockedUntil": "2026-05-17T12:15:00Z"
  }
}
```

**Token 刷新**

```http
POST /api/auth/refresh
Content-Type: application/json

{
  "refreshToken": "eyJhbGciOiJIUzUxMiJ9..."

{
  "code": 0,
  "data": {
    "accessToken": "eyJhbGciOiJIUzUxMiJ9...",

}
```

**Token 刷新失败（Refresh Token 过期/已登出）**

```http
}
```

**登出**

```http

{
  "code": 0,
  "message": "success",
  "data": null
}
```

**Token 校验失败（通用）**

```http
Response (401):
{
  "code": 401,
  "message": "Token 无效或已过期",
  "data": null
}
```

## 6. 异常处理

| 异常场景                      | HTTP 码 | 错误码 | 提示信息                    | 处理方式                                       |
| ----------------------------- | ------- | ------ | --------------------------- | ---------------------------------------------- |
| 用户名或密码错误              | 401     | 1003   | 用户名或密码错误            | Redis 失败计数 +1，记录审计日志                |
| 账号已锁定                    | 423     | 1004   | 账号已锁定，请 N 分钟后重试 | 返回锁定剩余时间                               |
| 用户不存在（验证阶段）        | 401     | 1003   | 用户名或密码错误            | 不区分用户不存在 vs 密码错误（防用户枚举）     |
| 用户已禁用                    | 403     | 1005   | 用户已被禁用，请联系管理员  | 记录审计日志（含操作者 IP）                    |
| 租户不存在或已禁用            | 401     | 1001   | 租户不存在或已禁用          | 记录 WARN 日志                                 |
| Access Token 过期             | 401     | 401    | Token 已过期                | Filter 返回 401，前端自动刷新                  |
| Access Token 签名错误         | 401     | 401    | Token 无效                  | Filter 返回 401，不暴露签名细节                |
| Refresh Token 过期/无效       | 401     | 401    | Refresh Token 已失效        | 前端跳转登录页                                 |
| Token 被吊销（版本号不匹配）  | 401     | 401    | Token 已被吊销，请重新登录  | Filter 返回 401                                |
| Token 在黑名单中              | 401     | 401    | Token 已被吊销              | Filter 返回 401                                |
| 请求缺少 Authorization Header | 401     | 401    | 未提供认证凭证              | Filter 返回 401（仅对受保护路径）              |
| Redis 不可用                  | —       | —      | —                           | 降级：跳过黑名单 + 版本号检查，仅验签 + 验过期 |
| 新增错误码：TENANT_DISABLED   | 401     | 1006   | 租户已被禁用                | 记录审计日志                                   |
| 新增错误码：USER_DISABLED     | 403     | 1005   | 用户已被禁用，请联系管理员  | 记录审计日志                                   |
| 新增错误码：ACCOUNT_LOCKED    | 423     | 1004   | 账号已锁定，请 N 分钟后重试 | 返回锁定剩余时间                               |

> 需在 `ErrorCode.java` 中新增 `ACCOUNT_LOCKED(1004, "账号已锁定")`、`USER_DISABLED(1005, "用户已被禁用")`、`TENANT_DISABLED(1006, "租户已被禁用")`。

## 7. 性能考量

### 7.1 性能目标

| 场景                        | 指标                           |
| --------------------------- | ------------------------------ |
| Token 解析（本地）          | < 1ms                          |
| 登录接口响应                | < 200ms（含 DB 查询 + BCrypt） |
| 带 Token 验证的请求增加延迟 | < 3ms（含 Redis 查询）         |
| 并发登录                    | ≥ 1000 req/s                   |
| Redis 降级后延迟            | < 1ms（仅本地验签）            |

### 7.2 性能优化点

1. **本地验签**：Token 合法性（签名 + 过期）在应用本地完成，不依赖任何外部服务
2. **Redis 批量操作**：黑名单检查 + 版本号检查合并为一次 Redis Pipeline 调用
3. **Redis 本地缓存兜底**：用户版本号使用 Caffeine 本地缓存（TTL = 30 秒），减少 Redis 查询
4. **BCrypt cost 权衡**：使用 cost = 10（默认），在安全性与登录延迟之间平衡（约 100ms/次）
5. **Filter 路径白名单**：对 `/auth/login`、`/auth/refresh`、`/health`、`/ready`、`/live` 直接放行，不执行 Token 解析
6. **避免重复解析**：Request 级别的 Token Claims 存储在 `request.setAttribute()`，避免同一次请求内多次解析

**延迟估算（带 Token 的请求）**：

```
Token 解码（Base64）          ≈ 0.01ms
签名验证（HMAC-SHA512）       ≈ 0.05ms
过期时间检查                  ≈ 0.001ms
Redis 检查（网络 + 执行）     ≈ 1-2ms（本地 Redis）
                                 ↓
Redis 不可用时降级            ≈ 0ms
----------------------------------------
总计 ≈ 0.1ms（无 Redis）～ 2ms（有 Redis）✅
```

## 8. 安全考量

### 8.1 权限控制

- 登录、Token 刷新接口无需认证（`permitAll()`）
- 其他所有 API 要求有效 Token
- 角色定义：

| 角色       | 编码   | 说明                     |
| ---------- | ------ | ------------------------ |
| 系统管理员 | ADMIN  | 管理租户、用户、系统配置 |
| 普通用户   | USER   | 创建/编辑/导出自有报表   |
| 访客       | VIEWER | 仅可查看已公开的报表     |

> 三权分立（系统管理员/安全管理员/审计员）在后续 `权限控制矩阵` 文档中详细设计，本模块仅实现基础角色框架。

- 方法级权限控制示例：

```java
@PreAuthorize("hasRole('ADMIN')")
@GetMapping("/tenant/{id}")
public Result<Tenant> getTenant(@PathVariable Long id) { ... }
```

### 8.2 数据安全

- 密码存储：BCrypt（cost = 10），永不存储明文
- Token 不包含敏感信息（仅 userId + tenantId + role），防止 JWT Payload 泄露用户隐私
- JWT Signing Key 通过环境变量 `JINSHU_JWT_SECRET` 注入，不写入配置文件
- 审计日志：登录、登出、Token 刷新全部记录审计日志（`operation = LOGIN/LOGOUT/REFRESH_TOKEN`）
- Refresh Token 存储在客户端（localStorage / httpOnly cookie），不存储在服务端数据库

### 8.3 防攻击

- **暴力破解**：5 次失败锁定 15 分钟（Redis 计数，按用户名维度）
- **用户枚举**：登录失败不区分"用户不存在"和"密码错误"，统一返回 `USERNAME_PASSWORD_ERROR`
- **Token 重放**：Access Token 的 jti 写入黑名单后（登出/吊销），即使 Token 未过期也无法使用
- **时序攻击**：BCrypt 算法内部已防时序攻击；手动比较密码的步骤不存在
- **密钥泄露**：密钥仅存在于环境变量，不写入日志/代码/配置文件。泄露后重新生成，所有 Token 立即失效（需配合版本号递增）
- **XSS 窃取 Token**：前端 Token 存储策略见 §12.2；后端设置 `HttpOnly` Cookie 可进一步降低风险
- **CSRF**：已禁用 CSRF 保护（REST API 无 Cookie 会话），Token 通过 Header 传递

### 9.2 集成测试

## 11. 回滚方案

### 11.1 回滚触发条件

- 登录成功率 < 90%
- Token 解析大面积失败（可能是密钥配置错误）
- JWT Filter 阻塞健康检查端点（导致 K8s 探针失败，Pod 被 Kill）
- 出现 Token 越权漏洞（如 Token 可访问其他租户数据）

### 11.2 回滚步骤

1. 设置 `jinshu.jwt.enabled = false`，重启 `api` 服务（JWT Filter 跳过所有请求）
2. 同时将 `SecurityConfig` 中 `.anyRequest().authenticated()` 临时改为 `.anyRequest().permitAll()`（如有必要）
3. 回滚 `api` 模块到上一版本
4. 清空 Redis 中所有 JWT 相关 Key（`token_blacklist:*`、`refresh:*`、`user:*:token_version`）
5. 通知用户：认证服务临时维护，无需登录

### 11.3 数据补救

- 回滚期间已登录用户的 Token 仍然有效（黑名单清空后无影响）
- 登录失败计数器在 Redis TTL 到期后自动清除，无需手动处理
- `users` 表不受影响，无脏数据产生
- 审计日志中可能遗留部分 `LOGIN` 记录，不影响数据完整性

## 12. 前后端约定

### 12.1 Token 传递方式

所有需要认证的 API 请求必须携带：

```
Authorization: Bearer <accessToken>
```

前端 Axios 请求拦截器统一注入：

```typescript
// request.ts
request.interceptors.request.use((config) => {
  const token = getAccessToken();
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});
```

### 12.2 前端 Token 管理

| 存储位置        | Access Token      | Refresh Token | 说明                   |
| --------------- | ----------------- | ------------- | ---------------------- |
| localStorage    | ✓（可选）         | ✓（可选）     | 简单但有 XSS 风险      |
| httpOnly Cookie | 由后端 Set-Cookie | 推荐          | 防止 JS 读取 Token     |
| 内存变量        | ✓（推荐）         | 存储中        | 最安全，但刷新页面丢失 |

> 推荐方案：Access Token 存内存变量，Refresh Token 存 httpOnly Cookie（需后端配合 Set-Cookie）。初期可先用 localStorage 快速实现，安全评审时升级。

**Token 刷新时机**：

```
前端检测 Access Token 将在 5 分钟内过期
        ↓
发起 POST /api/auth/refresh（携带 Refresh Token）
        ↓
成功 → 更新 Access Token，重试原始请求
失败 → 清除本地 Token，跳转登录页
```

### 12.3 401 响应处理

前端 Axios 响应拦截器统一处理：

```typescript
request.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401) {
      // 尝试刷新 Token
      const newToken = await refreshAccessToken();
      if (newToken) {
        // 重试原始请求
        error.config.headers.Authorization = `Bearer ${newToken}`;
        return request(error.config);
      }
      // 刷新失败 → 跳转登录
      clearTokens();
      window.location.href = "/login";
    }
    return Promise.reject(error);
  },
);
```
