## 1. 总则

### 1.1 目的

定义审计日志的记录标准、存储策略和查询规范，确保：

1. 所有用户操作可追溯、可审计
2. 审计日志防篡改，满足等保三级合规要求
3. 审计日志写入不影响业务系统性能
4. 三权分立原则在审计体系中得到贯彻

### 1.2 适用范围

本规范适用于锦书企业级报表系统中所有后端模块的审计日志记录：

- `backend/api/` — API 服务
- `backend/batch/` — 批处理服务
- `backend/worker/` — MQ 消费者 + PDF 生成
- `backend/data-sync/` — 数据同步服务

### 1.3 术语定义

| 术语              | 定义                                                           |
| ----------------- | -------------------------------------------------------------- |
| **审计日志**      | 记录用户操作行为的不可篡改日志                                 |
| **哈希链**        | 每条日志的哈希值包含前一条日志哈希，形成链式依赖               |
| **Merkle 根哈希** | 每小时所有日志组成的 Merkle 树的根节点哈希值                   |
| **三权分立**      | 系统管理员、安全管理员、审计员三权分离，互相制约               |
| **AOP 自动采集**  | 通过 `@AuditLog` 注解自动拦截 Controller 方法并记录审计日志    |
| **手动埋点**      | 在无法通过 AOP 覆盖的场景中，调用 `AuditLogger.log()` 手动记录 |

## 2. 审计事件分类

### 2.1 操作类型

| 编码   | 操作类型 | 说明                           | 示例场景                       |
| ------ | -------- | ------------------------------ | ------------------------------ |
| CREATE | 新增     | 创建新资源                     | 创建报表、创建数据源、注册用户 |
| UPDATE | 修改     | 修改已有资源                   | 编辑报表、修改配置、更新任务   |
| DELETE | 删除     | 删除资源（逻辑或物理删除）     | 删除报表、删除数据源           |
| IMPORT | 导入     | 从外部导入数据                 | Excel 导入报表数据             |
| EXPORT | 导出     | 将数据导出到外部               | 导出报表为 CSV/Excel           |
| AUTH   | 认证     | 登录/登出/Token 刷新等认证操作 | 用户登录、Token 刷新           |
| GRANT  | 授权     | 权限授予或变更                 | 分配角色、修改权限             |
| CONFIG | 配置     | 系统级配置变更                 | 修改存储路径、修改限流策略     |

### 2.2 目标对象

| 编码       | 目标对象   | 说明                  |
| ---------- | ---------- | --------------------- |
| REPORT     | 报表       | 报表元数据 + 报表数据 |
| USER       | 用户       | 用户账户              |
| TENANT     | 租户       | 租户配置              |
| DATASOURCE | 数据源     | 外部数据源配置        |
| TASK       | 任务       | 导入/导出/PDF 任务    |
| WORKFLOW   | 工作流     | 报表审批流程          |
| ROLE       | 角色与权限 | RBAC 角色定义与分配   |
| SYSTEM     | 系统配置   | 全局配置项            |

### 2.3 事件分级

| 级别 | 标识 | 说明                             | 处理要求                    |
| ---- | ---- | -------------------------------- | --------------------------- |
| L1   | 普通 | 日常查询类操作（R）              | 不强制记录（可选）          |
| L2   | 重要 | 所有 CUD 操作、导入导出、审批    | **必须记录**                |
| L3   | 严重 | 权限变更、系统配置变更、数据删除 | 必须记录 + 记录变更前后内容 |
| L4   | 致命 | 安全事件、绕过安全策略的操作     | 必须记录 + 安全告警         |

> **强制规则**：所有 CUD 操作（CREATE/UPDATE/DELETE）以及 IMPORT/EXPORT/AUTH/GRANT/CONFIG 类型操作均属于 L2 级别以上，必须 100% 记录审计日志。

## 3. 审计字段标准

### 3.1 必填字段

| 字段           | 类型         | 说明                                       | 示例                          |
| -------------- | ------------ | ------------------------------------------ | ----------------------------- |
| `id`           | BIGINT       | 自增主键                                   | 1                             |
| `tenant_id`    | BIGINT       | 租户 ID                                    | 1001                          |
| `user_id`      | BIGINT       | 操作用户 ID                                | 2001                          |
| `username`     | VARCHAR(100) | 操作用户名（冗余，防用户被删后无法追溯）   | `zhangsan`                    |
| `operation`    | VARCHAR(32)  | 操作类型编码（见 2.1 节）                  | `CREATE`                      |
| `target_type`  | VARCHAR(32)  | 目标对象编码（见 2.2 节）                  | `REPORT`                      |
| `target_id`    | VARCHAR(128) | 目标对象 ID                                | `550e8400-e29b-41d4-a716-...` |
| `target_name`  | VARCHAR(255) | 目标对象名称（冗余，防对象被删后无法追溯） | `2024Q4财务季报`              |
| `action_desc`  | VARCHAR(512) | 操作描述（人类可读）                       | `创建报表：2024Q4财务季报`    |
| `request_ip`   | VARCHAR(64)  | 请求来源 IP                                | `192.168.1.100`               |
| `request_time` | TIMESTAMP    | 请求时间                                   | `2026-05-18 14:30:00`         |
| `log_level`    | VARCHAR(8)   | 事件级别（L1/L2/L3/L4）                    | `L2`                          |
| `log_hash`     | VARCHAR(128) | 日志哈希值（SHA-256）                      | `a1b2c3...`                   |
| `prev_hash`    | VARCHAR(128) | 前一条日志哈希值（哈希链）                 | `d4e5f6...`                   |
| `created_at`   | TIMESTAMP    | 记录创建时间（数据库自动填充）             | `2026-05-18 14:30:00.123456`  |

### 3.2 扩展字段

根据操作类型补充，存储在 `extra_info` JSONB 字段中：

| 场景      | 扩展字段内容                                                                          |
| --------- | ------------------------------------------------------------------------------------- |
| 数据修改  | `{"before": {...}, "after": {...}}` — L3 级别事件必填                                 |
| 导入操作  | `{"file_name": "...", "row_count": 10000, "success_count": 9800, "error_count": 200}` |
| 导出操作  | `{"file_name": "...", "row_count": 50000, "format": "CSV"}`                           |
| 权限变更  | `{"role": "ReportDesigner", "granted_by": "admin", "reason": "..."}`                  |
| 登录/登出 | `{"login_status": "success", "user_agent": "..."}`                                    |

### 3.3 敏感字段处理

审计日志中**禁止**记录以下内容：

| 禁止记录的内容 | 替代方案                        |
| -------------- | ------------------------------- |
| 明文密码       | 不记录，仅记录认证结果          |
| Token 完整内容 | 仅记录 Token 前 8 位 + 签发时间 |
| 身份证号       | 脱敏 `320***********1234`       |
| 手机号         | 脱敏 `138****5678`              |
| 数据库连接密码 | 不记录，记录"密码已更新"标识    |

## 4. 审计记录规范

### 4.1 自动采集（AOP）

使用 `@AuditLog` 注解标记 Controller 方法，拦截器自动采集：

```java
@AuditLog(
    operation = AuditOperation.CREATE,
    targetType = AuditTargetType.REPORT
)
@PostMapping("/reports")
public Result<Report> createReport(@RequestBody ReportCreateReq req) {
    // 业务逻辑
}
```

**注解参数规范**：

| 参数         | 说明                         | 是否必填 |
| ------------ | ---------------------------- | -------- |
| `operation`  | 操作类型                     | 是       |
| `targetType` | 目标对象类型                 | 是       |
| `actionDesc` | 操作描述（支持 SpEL 表达式） | 否       |

**AOP 自动填充字段**：

- `tenant_id` → `TenantContext.getTenantId()`
- `user_id` → `UserContext.getUserId()`
- `username` → `UserContext.getUsername()`
- `request_ip` → `HttpServletRequest.getRemoteAddr()`
- `request_time` → `System.currentTimeMillis()`
- `target_id` → 从方法参数或返回值提取
- `log_level` → 根据操作类型自动判定（CUD = L2，GRANT = L3）

### 4.2 手动埋点

对于 AOP 无法覆盖的场景（如登录、MQ 消费、批处理），使用手动埋点：

```java
AuditLogger.log(AuditLogEntry.builder()
    .tenantId(tenantId)
    .userId(userId)
    .username(username)
    .operation(AuditOperation.AUTH)
    .targetType(AuditTargetType.USER)
    .targetId(userId.toString())
    .targetName(username)
    .actionDesc("用户登录：%s".formatted(username))
    .requestIp(ip)
    .logLevel(LogLevel.L2)
    .build());
```

**手动埋点适用场景**：

1. 登录/登出操作（不经过 Controller 的 `/api/login`）
2. JWT Token 刷新
3. MQ 消费者的批量处理操作
4. Spring Batch 的 Job 启动/完成
5. 定时任务的执行
6. 数据库直连操作（如数据迁移脚本）

### 4.3 全量覆盖清单

| 模块       | 操作                          | 采集方式 | 优先级 |
| ---------- | ----------------------------- | -------- | ------ |
| 用户管理   | 创建/编辑/删除/启用/禁用用户  | AOP      | P0     |
| 租户管理   | 创建/编辑/启用/暂停/删除租户  | AOP      | P0     |
| 报表管理   | 创建/编辑/删除/复制/归档报表  | AOP      | P0     |
| 报表审批   | 提交/通过/驳回/撤回审批       | AOP      | P0     |
| 数据源管理 | 创建/编辑/删除/测试连接       | AOP      | P0     |
| 报表导入   | 发起导入/导入完成             | AOP      | P0     |
| 报表导出   | 发起导出/导出完成             | AOP      | P0     |
| PDF 生成   | 发起渲染/渲染完成             | AOP      | P0     |
| 角色权限   | 创建/编辑/删除角色、分配权限  | AOP      | P0     |
| 系统配置   | 修改任意系统级配置项          | AOP      | P0     |
| 用户认证   | 登录/登出/Token 刷新/强制下线 | 手动埋点 | P0     |
| MQ 消费    | 消息成功处理/死信转发         | 手动埋点 | P0     |
| Batch Job  | Job 启动/完成/失败            | 手动埋点 | P0     |
| 数据同步   | 同步任务启动/完成/失败        | 手动埋点 | P1     |
| 定时任务   | 定时清理/归档任务执行         | 手动埋点 | P1     |

### 4.4 不记录的操作

以下操作明确不记录审计日志，避免日志污染：

| 操作                                     | 原因                              |
| ---------------------------------------- | --------------------------------- |
| GET 请求的查询操作（只读）               | 性能考虑，必要时可查询 Nginx 日志 |
| 健康检查（`/health`、`/ready`、`/live`） | 非业务操作，K8s 频繁调用          |
| 静态资源请求                             | 非业务操作                        |
| 系统内部 RPC 调用                        | 不涉及用户操作                    |
| 失败的参数校验请求（返回阶段）           | 未进入业务逻辑，无需记录          |

## 5. 存储与保留

### 5.1 保留期限

| 数据状态 | 保留位置             | 保留期限      |
| -------- | -------------------- | ------------- |
| 热数据   | `audit_log` 表       | 6 个月        |
| 温数据   | 归档表或对象存储     | 6 个月 ~ 3 年 |
| 冷数据   | 对象存储（S3/MinIO） | 3 年以上      |
| 哈希根   | `audit_root_hash` 表 | 永久保留      |

### 5.2 归档策略

- **归档触发**：每月 1 日凌晨 3:00 自动执行归档任务
- **归档条件**：`request_time < now() - INTERVAL '6 months'`
- **归档方式**：导出为 Parquet 格式 → 上传至对象存储 → 删除原表记录
- **不可归档条件**：涉及未结案安全事件的日志标记为 `do_not_archive=true`
- **归档验证**：归档完成后，从对象存储抽样 1% 比对数据完整性

### 5.3 容量规划

| 指标             | 估算值                   |
| ---------------- | ------------------------ |
| 日均操作量       | ~50,000 条               |
| 单条日志大小     | ~2 KB                    |
| 日均磁盘占用     | ~100 MB                  |
| 6 个月磁盘占用   | ~18 GB                   |
| 归档存储（3 年） | ~110 GB（压缩后 ~20 GB） |

## 6. 查询与导出规范

### 6.1 查询权限

| 角色       | 查询范围               | 限制                 |
| ---------- | ---------------------- | -------------------- |
| 审计员     | 全部租户的全部审计日志 | 无限制               |
| 超管       | 全部租户的全部审计日志 | 查询操作本身也被审计 |
| 租户管理员 | 仅本租户的审计日志     | 时间跨度 ≤ 90 天     |
| 其他角色   | 仅本人的审计日志       | 时间跨度 ≤ 30 天     |

### 6.2 查询限制

| 限制项         | 限制值                 | 说明                         |
| -------------- | ---------------------- | ---------------------------- |
| 最大时间跨度   | 90 天                  | 审计员可调整，其他角色硬限制 |
| 单页最大记录数 | 200 条                 | 超出提示缩小范围             |
| 排序字段       | 仅 `request_time DESC` | 防止全表扫描                 |
| 导出最大记录数 | 100,000 条             | 超出需分批导出               |
| 查询频率       | 10 次/分钟             | 超过触发限流，返回 HTTP 429  |

### 6.3 导出规范

- 导出格式：CSV（使用 UTF-8 BOM，兼容 Excel 打开）
- 导出文件名：`audit_log_{tenant_id}_{start_date}_{end_date}.csv`
- 导出字段：全量必填字段 + `extra_info` JSONB 展平展开
- 敏感字段：导出时自动脱敏（密码字段置空、Token 仅保留前 8 位）
- 审计记录：导出操作本身也记录一条审计日志

## 7. 防篡改验证规范

### 7.1 哈希链验证流程

```
输入：起始 ID、结束 ID
输出：验证通过/验证失败（失败的具体位置）

步骤：
1. 从数据库读取结束 ID 记录的 prev_hash
2. 计算上一条记录 (id-1) 的 log_hash
3. 比较 prev_hash 与上一条的 log_hash 是否一致
4. 递归向前验证，直到起始 ID
5. 任意一条不匹配 → 验证失败，标记断裂位置
```

### 7.2 Merkle 树验证流程

```
输入：小时时间范围 [start_hour, end_hour]
输出：验证通过/验证失败（失败的时段）

步骤：
1. 从 audit_root_hash 表读取目标时段的 root_hash
2. 从 audit_log 表读取该时段的所有日志
3. 重新计算 Merkle 树的根哈希
4. 比较计算的 root_hash 与存储的 root_hash 是否一致
5. 不一致 → 该时段存在篡改，触发告警
```

### 7.3 发现篡改的响应

| 阶段      | 动作                                                       |
| --------- | ---------------------------------------------------------- |
| 检测到    | 立即告警通知安全管理员（邮件 + 即时消息）                  |
| 5 分钟内  | 安全管理员确认事件，标记受影响的时间段                     |
| 30 分钟内 | 从备份恢复影响时段的审计日志，重新计算哈希链和 Merkle 树   |
| 1 小时内  | 排查篡改来源（数据库操作日志、DBA 操作记录），评估影响范围 |
| 24 小时内 | 输出安全事件报告，记录到 `security_incident` 表            |

## 8. 三权分立

### 8.1 角色定义

| 角色       | 职责                               |
| ---------- | ---------------------------------- |
| 系统管理员 | 用户管理、系统配置、运维部署       |
| 安全管理员 | 安全策略配置、密钥管理、安全响应   |
| 审计员     | 审计日志查询、完整性验证、合规报告 |

### 8.2 权限边界

| 操作               | 系统管理员 | 安全管理员 | 审计员 |
| ------------------ | ---------- | ---------- | ------ |

## 9. 合规要求

### 9.1 等保三级要求

等保三级对审计日志的核心要求及本规范对应实现：

| 等保要求                                        | 本规范实现                                   |
| ----------------------------------------------- | -------------------------------------------- |
| 审计范围覆盖所有用户和安全管理员的操作          | 4.3 节全量覆盖清单，AOP + 手动埋点           |
| 审计记录包含事件时间、用户、类型、结果、来源 IP | 3.1 节必填字段完整覆盖                       |
| 审计记录受保护，无法修改/删除                   | 7.1 节哈希链 + 7.2 节 Merkle 树 + DDL 触发器 |
| 提供审计记录的查询、统计、分析功能              | 6 节查询与导出规范                           |
| 审计记录保存时间不少于 6 个月                   | 5.1 节保留期限（6 个月热数据 + 3 年归档）    |
| 系统管理员、安全管理员、审计员权限分离          | 8 节三权分立                                 |

### 9.2 审计记录内容要求

根据 GB/T 22239-2019 8.1.4.3 条要求，每条审计记录必须包含：

1. ✅ **事件发生的时间** — `request_time`
2. ✅ **用户标识** — `user_id` + `username`
3. ✅ **事件类型** — `operation` + `target_type`
4. ✅ **事件结果** — `log_level` + `action_desc`
5. ✅ **事件来源** — `request_ip`
