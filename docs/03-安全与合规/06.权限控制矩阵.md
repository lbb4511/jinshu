## 1. 背景与目标

### 1.1 问题背景

锦书系统涉及多角色协作和多模块操作。当前 `SecurityConfig` 仅区分"是否登录"（`.anyRequest().authenticated()`），未实现角色粒度的权限控制。已有设计（报表全生命周期 §4）定义了 USER/ADMIN/VIEWER 三种角色，但缺少系统级权限矩阵。

### 1.2 设计目标

- 所有角色、权限、资源有一份唯一权威的矩阵定义
- 覆盖全部 12 个业务模块 × 5 种角色
- 权限粒度到操作级（CRUD + 模块特定动作）
- 支持所有权规则（资源创建者获得额外权限）
- 支持三权分立（管理员 / 安全管理员 / 审计员 互不越权）

### 1.3 权限模型

采用 **RBAC（基于角色的访问控制）**：

```
用户 (User)
  │
  ├── 角色 (Role): ADMIN / USER / VIEWER / SECURITY_ADMIN / AUDITOR
  │
  ├── 静态权限：角色 → 模块操作（见 §3 矩阵）
  │
  └── 动态权限：
       ├── 所有权：资源的 created_by = 当前用户 ID
       └── 审查指派：资源的 reviewed_by = 当前用户 ID
```

## 2. 角色定义

### 2.1 五角色体系

| 角色             | 标识       | 说明                                           | 典型用户          |
| ---------------- | ---------- | ---------------------------------------------- | ----------------- |
| `ADMIN`          | 平台管理员 | 全局超级权限，可管理租户、用户、所有报表       | 平台运维人员      |
| `SECURITY_ADMIN` | 安全管理员 | 负责安全策略配置：加密密钥、脱敏规则、权限分配 | 安全负责人        |
| `AUDITOR`        | 审计员     | 只读所有审计日志，不可操作业务数据             | 合规审计人员      |
| `USER`           | 普通用户   | 创建/编辑自己的报表，提交审查，导出            | 制表人/数据分析师 |
| `VIEWER`         | 访客       | 仅查看已公开的报表，不可编辑、导出             | 外部合作方        |

> `*` = 额外受所有权/审查指派规则限制

### 2.3 角色授予规则

- 每个用户有且仅有一个角色
- `ADMIN` 角色仅可由现有 `ADMIN` 授予（首个 ADMIN 通过 DB 初始化脚本创建）
- `SECURITY_ADMIN` 仅可由 `ADMIN` 授予
- `AUDITOR` 仅可由 `ADMIN` 或 `SECURITY_ADMIN` 授予
- `USER` 和 `VIEWER` 可由 `ADMIN` 或租户管理员授予
- 角色变更必须记录审计日志（目标用户、操作人、变更前角色、变更后角色）

## 4. 特殊权限规则

### 4.1 所有权限制

对于报表、数据源等资源，在角色权限基础上叠加所有权规则：

| 资源     | 所有者判定            | 额外权限                                    |
| -------- | --------------------- | ------------------------------------------- |
| 报表     | `created_by` = userId | 编辑、删除、提交审查（DRAFT/REJECTED 状态） |
| 数据源   | `created_by` = userId | 编辑、删除、测试连接                        |
| 导出任务 | `created_by` = userId | 查看状态、下载、取消                        |
| PDF 任务 | `created_by` = userId | 查看状态、下载                              |

**跨所有者权限**：仅 `ADMIN` 可操作他人的资源。

### 4.2 审查人限制

- 审查操作仅允许 `reviewed_by` 字段指定的用户执行（或 `ADMIN`）
- 审查人不能是报表创建者本人（自审禁止）
- 审查人角色必须为 `USER` 或 `ADMIN`（`VIEWER`、`AUDITOR`、`SECURITY_ADMIN` 不可审查报表）

### 4.3 租户隔离与豁免

| 角色             | 跨租户数据可见性           |
| ---------------- | -------------------------- |
| `ADMIN`          | 可查看所有租户数据         |
| `SECURITY_ADMIN` | 同租户（仅脱敏规则为全局） |
| `AUDITOR`        | 可查看所有租户审计日志     |
| `USER`           | 仅本租户                   |
| `VIEWER`         | 仅本租户（已公开报表）     |

实现方式：

- `TenantInterceptor` 自动注入 `tenant_id` 过滤条件
- `ADMIN` 和 `AUDITOR` 通过 `@SkipTenantFilter` 注解豁免

### 4.4 三权分立

等保三级核心要求：系统管理员、安全管理员、审计员三权分立。

| 权限                     | ADMIN | SECURITY_ADMIN | AUDITOR |
| ------------------------ | :---: | :------------: | :-----: |
| 管理用户/租户/业务资源   |   ✓   |       ✗        |    ✗    |
| 管理安全策略/加密/脱敏   |   ✗   |       ✓        |    ✗    |
| 查看/导出审计日志        |   ✗   |       ✗        |    ✓    |
| 操作业务数据             |   ✓   |       ✗        |    ✗    |
| 查看业务数据（脱敏豁免） |   ✓   |       ✓        |    ✓    |

> `ADMIN` 可创建/管理账户，但无权修改安全策略和查看审计日志（防止管理员操作后删除审计痕迹）。`AUDITOR` 可查看全部审计日志但无权操作任何业务数据。`SECURITY_ADMIN` 管理加密和脱敏但无权管理用户。

## 5. 权限检查实现

### 5.1 注解驱动权限

```java
// common/src/main/java/com/jinshu/common/security/RequireRole.java
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireRole {
    String[] value(); // 允许的角色列表，如 {"ADMIN", "USER"}
}

// common/src/main/java/com/jinshu/common/security/RequireOwner.java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireOwner {
    String resourceIdParam() default "id"; // 方法参数名
}
```

**Controller 使用示例**：

```java
@RestController
@RequestMapping("/api/v1/reports")
public class ReportController {

    @GetMapping
    @RequireRole({"ADMIN", "USER", "VIEWER", "SECURITY_ADMIN", "AUDITOR"})
    public Result<PageResult<Report>> list(/* ... */) { }

    @PostMapping
    @RequireRole({"ADMIN", "USER"})
    public Result<Report> create(@Valid @RequestBody CreateReportRequest request) { }

    @PutMapping("/{id}")
    @RequireRole({"ADMIN", "USER"})
    @RequireOwner(resourceIdParam = "id")  // 仅所有者或 ADMIN 可编辑
    public Result<Report> update(@PathVariable Long id, @RequestBody UpdateReportRequest req) { }

    @PostMapping("/{id}/submit")
    @RequireRole({"ADMIN", "USER"})
    @RequireOwner(resourceIdParam = "id")
    public Result<Report> submitForReview(@PathVariable Long id, @RequestBody SubmitRequest req) { }

    @PostMapping("/{id}/review")
    @RequireRole({"ADMIN", "USER"})
    // 不需要 @RequireOwner，审查权限由 reviewed_by 字段决定
    public Result<Report> review(@PathVariable Long id, @RequestBody ReviewRequest req) { }
}
```

**AOP 切面实现**：

```java
// common/src/main/java/com/jinshu/common/security/RoleAspect.java
@Aspect
@Component
public class RoleAspect {

    @Around("@annotation(requireRole)")
    public Object checkRole(ProceedingJoinPoint joinPoint, RequireRole requireRole) throws Throwable {
        String currentRole = UserContext.getRole();
        if (currentRole == null || !Arrays.asList(requireRole.value()).contains(currentRole)) {
            throw new BusinessException(ErrorCode.FORBIDDEN, "无权限执行此操作");
        }
        return joinPoint.proceed();
    }

    @Around("@annotation(requireOwner)")
    public Object checkOwner(ProceedingJoinPoint joinPoint, RequireOwner requireOwner) throws Throwable {
        String role = UserContext.getRole();
        if ("ADMIN".equals(role)) return joinPoint.proceed(); // ADMIN 豁免

        Long resourceId = extractParam(joinPoint, requireOwner.resourceIdParam());
        Long currentUserId = UserContext.getUserId();
        // 由 Service 层校验所有权
        // (AOP 仅做角色过滤，所有权校验在 Service 层完成)
        return joinPoint.proceed();
    }
}
```

### 5.2 编程式权限

```java
// common/src/main/java/com/jinshu/common/security/PermissionUtils.java
public class PermissionUtils {

    public static void checkRole(String... allowedRoles) {
        String role = UserContext.getRole();
        if (role == null || !Set.of(allowedRoles).contains(role)) {
            throw new BusinessException(ErrorCode.FORBIDDEN, "无权限执行此操作");
        }
    }

    public static void checkOwner(Long resourceOwnerId) {
        if ("ADMIN".equals(UserContext.getRole())) return;
        if (!Objects.equals(UserContext.getUserId(), resourceOwnerId)) {
            throw new BusinessException(ErrorCode.FORBIDDEN, "仅资源所有者可执行此操作");
        }
    }

    public static void checkReviewer(Long reviewerId) {
        if ("ADMIN".equals(UserContext.getRole())) return;
        if (!Objects.equals(UserContext.getUserId(), reviewerId)) {
            throw new BusinessException(ErrorCode.FORBIDDEN, "仅指派的审查人可执行此操作");
        }
    }
}
```

### 5.3 前端权限控制

```typescript
// utils/permission.ts
export type Role = "ADMIN" | "SECURITY_ADMIN" | "AUDITOR" | "USER" | "VIEWER";

export function hasRole(
  userRole: string | undefined,
  allowedRoles: Role[],
): boolean {
  if (!userRole) return false;
  return allowedRoles.includes(userRole as Role);
}

export function isOwner(
  resourceCreatorId: number,
  currentUserId: number,
): boolean {
  return resourceCreatorId === currentUserId;
}
```

**前端按钮控制**：

```tsx
const { user } = useAuth();

// 仅制表人和管理员可见"提交审查"按钮
{
  hasRole(user?.role, ["ADMIN", "USER"]) && report.createdBy === user?.id && (
    <Button label="提交审查" onClick={handleSubmit} />
  );
}

// 仅 AUDITOR 可见"审计日志"菜单
{
  hasRole(user?.role, ["AUDITOR"]) && (
    <SidebarMenuItem label="审计日志" to="/audit" />
  );
}
```

> 前端权限仅做 UX 优化（按钮显隐），真正的权限校验在后端。禁止依赖前端隐藏作为安全措施。

## 6. 权限变更审计

所有权限相关操作必须记录审计日志：

| 操作           | 审计 `operation`                   | 记录内容                           |
| -------------- | ---------------------------------- | ---------------------------------- |
| 角色变更       | `CHANGE_ROLE`                      | 目标用户、操作人、旧角色、新角色   |
| 用户启用/禁用  | `ENABLE_USER` / `DISABLE_USER`     | 目标用户、操作人                   |
| 租户启用/禁用  | `ENABLE_TENANT` / `DISABLE_TENANT` | 目标租户、操作人                   |
| 权限拒绝 (403) | `ACCESS_DENIED`                    | 尝试 URL、用户、所需角色、实际角色 |
