## 1. 总则

### 1.1 目的

1. 建立个人信息（PII）的识别、分类和保护标准
2. 确保系统在处理报表数据时满足隐私法规要求
3. 为数据导出、删除等用户权利提供技术实现依据
4. 为未来 GDPR 等国际隐私法规合规预留扩展点

### 1.2 适用范围

本规范适用于系统中所有涉及个人信息处理的活动：

- 用户账户信息（用户名、邮箱、手机号等）
- 报表数据中可能包含的个人身份信息
- 审计日志中可能含有的个人信息
- 导入导出文件中的个人信息

### 1.3 参考法规

| 法规                       | 适用场景                   |
| -------------------------- | -------------------------- |
| 《个人信息保护法》（PIPL） | 中国境内运营               |
| GDPR                       | 处理欧盟公民数据时（预留） |
| 《数据安全法》             | 数据分类分级               |

## 2. PII 识别与分类

### 2.1 个人信息定义

按照 PIPL 定义：以电子或其他方式记录的与已识别或可识别的自然人有关的各种信息。

### 2.2 本项目中的 PII 清单

| PII 类别 | 具体字段                   | 存储位置               |
| -------- | -------------------------- | ---------------------- |
| 身份标识 | 用户名、邮箱、手机号       | `users` 表             |
| 身份标识 | 用户 IP 地址               | `audit_log` 表         |
| 身份标识 | 用户姓名（display_name）   | `users` 表             |
| 身份标识 | 身份证号（如有白名单录入） | 可能存在于报表数据中   |
| 身份标识 | 工号、员工编号             | 可能存在于报表数据中   |
| 金融信息 | 薪资数据（如人力报表）     | `report_data` 表       |
| 金融信息 | 银行卡号                   | 可能存在于报表数据中   |
| 联系方式 | 手机号、座机号             | `users` 表、报表数据中 |
| 联系方式 | 电子邮箱                   | `users` 表、报表数据中 |
| 行为数据 | 操作记录（审计日志）       | `audit_log` 表         |
| 行为数据 | 登录时间/IP                | `users` 表             |

### 2.3 敏感程度分级

| 级别      | 定义                   | 示例                                 | 保护要求                              |
| --------- | ---------------------- | ------------------------------------ | ------------------------------------- |
| L1 — 一般 | 基本信息               | 用户名、邮箱、IP 地址                | 加密传输、访问控制                    |
| L2 — 敏感 | 个人敏感信息           | 手机号、身份证号、薪资数据、银行卡号 | 加密存储 + 加密传输 + 脱敏展示 + 审计 |
| L3 — 高敏 | 生物识别、金融账户密码 | 用户密码、财务密钥                   | L2 全部 + 独立加密密钥 + 禁止日志记录 |

## 3. 数据最小化原则

### 3.1 采集最小化

- 用户注册仅要求：用户名、密码（必需），邮箱、手机号为可选项
- 报表导入时提示用户移除不必要的 PII 列
- 不在用户无感知的情况下采集设备指纹、位置信息

### 3.2 存储最小化

- 密码使用 BCrypt 哈希，不存储明文
- 身份证号仅在入库报表数据时存储，系统中不设独立 PII 字段
- IP 地址在审计日志中保留 6 个月，归档后可转为 IP 段 `192.168.*.*`

### 3.3 访问最小化

- 手机号、邮箱等信息默认仅租户管理员和用户本人可见
- 审计日志仅审计员角色可查阅
- 报表数据中的 PII 按租户隔离，不可跨租户查询

## 4. PII 处理规范

### 4.1 存储规范

| 数据级别 | 加密方式                    | 密钥管理             |
| -------- | --------------------------- | -------------------- |
| L1 一般  | 数据库 TLS + 行级权限       | 无需字段级加密       |
| L2 敏感  | AES-256-GCM 字段级加密      | 每个租户独立加密密钥 |
| L3 高敏  | AES-256-GCM + 独立 HSM 密钥 | HSM 或 KMS 管理      |

**加密实现**：参考 `03-安全与合规/04.数据加密标准.md`。

### 4.2 传输规范

- 所有数据传输使用 TLS 1.2+
- PII 数据不在 URL Query String 中传递
- 导出 PII 数据的文件使用密码加密（ZIP + AES）

### 4.3 展示与脱敏规范

| 数据类型 | 脱敏规则               | 示例                  |
| -------- | ---------------------- | --------------------- |
| 手机号   | 中间 4 位替换为 `****` | `138****5678`         |
| 身份证号 | 中间 11 位替换         | `320***********1234`  |
| 邮箱     | 用户名部分脱敏         | `z***@example.com`    |
| 银行卡号 | 仅显示后 4 位          | `**** **** **** 6789` |
| IP 地址  | 最后一段替换           | `192.168.1.*`         |
| 薪资数据 | 仅显示范围或隐藏       | `<10,000` 或 `****`   |

**脱敏实现**：参考 `03-安全与合规/04.数据加密标准.md` 第 6 节 `@DesensitizeField`。

### 4.4 日志规范

参考 `02-流程与规范/00-基础规范/07.结构化日志规范.md` 第 4 节：

- ❌ 禁止在日志中输出明文 PII
- ❌ 禁止在日志中输出完整身份证号、手机号、银行卡号
- ✅ 必须脱敏后输出

## 5. 用户权利实现

### 5.1 数据导出

用户有权获取系统存储的其个人数据副本：

| 步骤 | 操作                                               |
| ---- | -------------------------------------------------- |
| 1    | 用户在个人设置中提交"数据导出"请求                 |
| 2    | 系统异步生成包含用户 PII 的压缩文件                |
| 3    | 生成完成后通过通知提醒用户下载                     |
| 4    | 导出文件包含：用户基本信息、操作记录、关联报表列表 |
| 5    | 导出文件使用 AES 加密，密码通过独立渠道发送        |

**API**：`POST /api/users/me/export-data`（响应为异步任务 ID）

### 5.2 数据删除

用户有权要求删除其个人数据：

| 数据                     | 处理方式                                                       |
| ------------------------ | -------------------------------------------------------------- |
| 用户账户                 | 标记为 DISABLED，30 天后物理删除                               |
| 用户个人信息（手机号等） | 置为 NULL 或脱敏值                                             |
| 用户创建的报表           | 转移给租户管理员或归档（不直接删除）                           |
| 审计日志                 | **不删除**。以匿名化代替：user_id→0，username→deleted_user_xxx |
| 文件                     | 从文件存储中物理删除                                           |

**API**：`DELETE /api/users/me `（标记删除，30 天宽限期）

**限制**：

- 租户管理员不可被删除（先转移管理权限）
- 等保合规要求的审计日志匿名化而非删除
- 从提交请求到完成删除，SLA ≤ 30 天

### 5.3 数据更正

- 用户可在个人设置中修改 display_name、邮箱、手机号
- 修改敏感信息需二次密码验证
- 修改操作记录到审计日志

## 6. 数据生命周期

### 6.1 采集阶段

- 明确告知数据用途（隐私政策页面）
- 获取用户同意（注册时勾选）
- 记录同意时间和版本

### 6.2 使用阶段

- 仅用于声明的用途（报表生成与管理）
- 不用于用户画像、自动化决策
- 不向第三方出售或共享 PII

### 6.3 存储阶段

| 数据类型     | 最长保留时间                |
| ------------ | --------------------------- |
| 用户账户信息 | 账户存续期 + 30 天          |
| 审计日志     | 6 个月（热） + 3 年（归档） |
| 报表数据     | 由租户配置决定              |
| 导出数据文件 | 下载后 7 天自动清理         |

### 6.4 销毁阶段

| 数据位置     | 销毁方式                       |
| ------------ | ------------------------------ |
| 数据库记录   | DELETE（物理删除）             |
| 文件存储     | 物理删除文件                   |
| 备份数据     | 随备份过期自动清理             |
| 对象存储归档 | 到期后自动删除（S3 Lifecycle） |

## 7. 跨境数据传输

当前系统部署于中国境内，报表数据不主动跨境传输。如未来涉及：

- 需完成个人信息保护影响评估
- 通过国家网信部门安全评估
- 与境外接收方签订标准合同
- 获取用户的单独同意

## 8. 隐私影响评估

在新功能上线前，如涉及 PII 处理，需完成隐私影响评估（PIA）：

| 评估项                     | 是/否/不适用 | 说明 |
| -------------------------- | ------------ | ---- |
| 是否采集新的 PII 类型？    |              |      |
| 是否变更 PII 使用目的？    |              |      |
| 是否新增 PII 共享方？      |              |      |
| 是否变更数据保留期限？     |              |      |
| 是否涉及跨境传输？         |              |      |
| 是否涉及自动化决策？       |              |      |
| 用户是否有撤回同意的途径？ |              |      |
| 是否有数据泄露通知机制？   |              |      |

## 9. 违规响应

参考 `02.安全漏洞响应流程.md`，以下为 PII 泄露的额外响应要求：

| 时间线    | 动作                                      |
| --------- | ----------------------------------------- |
| 发现后 1h | 通知安全负责人和 DPO（如有）              |
| 4h 内     | 评估受影响的 PII 类型和数量               |
| 24h 内    | 评估是否达到监管报告门槛（PIPL 第 57 条） |
| 72h 内    | 如需报告监管部门，按时提交                |
| 之后      | 通知受影响用户、发布公告、完成整改        |
