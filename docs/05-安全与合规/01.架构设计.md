## 1. 多租户隔离详细设计

### 1.1 租户上下文传递链

```
前端请求（Header: X-Tenant-Id）
    ↓
Spring MVC 拦截器（TenantInterceptor）
    ↓
ThreadLocal<TenantContext> 存储
    ↓
MyBatis 拦截器（TenantSqlInterceptor）
    ↓
自动追加 tenant_id 条件到 SQL
    ↓
请求结束 → 清除 ThreadLocal（避免内存泄漏）
```

### 1.4 任务公平调度与防抢占

#### 1.4.1 问题分析

**反例（无调度优化）：**

- 租户A提交 100万行导出任务 → 占用 Worker 全部 10 个线程 5 分钟
- 租户B/C/D/E 同时提交预览请求 → 全部排队等待 5 分钟
- 系统看起来"卡住"，用户体验极差

**正例（优化后）：**

- 租户A大任务拆分为 10 个小任务，每个 10 万行
- 调度器按租户轮询：A → B → A → C → A → D → A → E → ...
- 租户B的预览请求 3秒内开始执行，总耗时 5秒完成
- 租户A的大任务总耗时从 5分钟 变为 6分钟（可接受的牺牲）

#### 1.4.2 四层保障体系

```
┌─────────────────────────────────────────────────┐
│  1. 优先级队列                                  │
│  ┌───────────────────────────────────────────┐  │
│  │ P0: 预览、小导出(<1万行)  → 立即执行       │  │
│  │ P1: 普通导出(1-10万行)                     │  │
│  │ P2: 大导出(>10万行)                        │  │
│  │ P3: 数据同步、定时任务                      │  │
│  └───────────────────────────────────────────┘  │
├─────────────────────────────────────────────────┤
│  2. 租户并发限制                                │
│     单个租户并发上限 = min(租户配额, 总并发/活跃租户数)  │
│     例：总并发20, 活跃租户5个 → 单租户最多4个并发      │
├─────────────────────────────────────────────────┤
│  3. 时间片轮转（核心）                          │
│     T1→T2→T3→T1→T2→T3→... 每个租户每次只取1个任务     │
├─────────────────────────────────────────────────┤
│  4. 大任务自动分片                              │
│     >10万行自动拆分为多个子任务，分片间让出资源          │
└─────────────────────────────────────────────────┘
```

#### 1.4.3 公平调度器实现

```java
// 租户任务队列 - 每个租户独立的小队列
public class TenantTaskQueue {
    private final Map<Long, LinkedBlockingQueue<Task>> tenantQueues = new ConcurrentHashMap<>();
    private final Map<Long, Integer> tenantConcurrency = new ConcurrentHashMap<>();
    private final AtomicInteger roundRobinIndex = new AtomicInteger(0);

    // 获取下一个要执行的任务（公平调度）
    public Optional<Task> pollNextTask() {
        List<Long> activeTenants = new ArrayList<>(tenantQueues.keySet());
        if (activeTenants.isEmpty()) {
            return Optional.empty();
        }

        // 最多尝试一轮，找有可用配额且有待执行任务的租户
        for (int i = 0; i < activeTenants.size(); i++) {
            int idx = (roundRobinIndex.incrementAndGet()) % activeTenants.size();
            Long tenantId = activeTenants.get(idx);

            // 检查该租户是否达到并发上限
            if (tenantConcurrency.getOrDefault(tenantId, 0) >= getTenantMaxConcurrency(tenantId)) {
                continue;
            }

            Task task = tenantQueues.get(tenantId).poll();
            if (task != null) {
                tenantConcurrency.merge(tenantId, 1, Integer::sum);
                return Optional.of(task);
            }
        }

        return Optional.empty();
    }

    // 任务完成时释放配额
    public void onTaskComplete(Long tenantId) {
        tenantConcurrency.merge(tenantId, -1, Integer::sum);
    }

    // 计算单租户最大并发（动态调整）
    private int getTenantMaxConcurrency(Long tenantId) {
        int totalConcurrency = runtimeConfig.getTotalWorkerConcurrency();
        int activeTenantCount = Math.max(1, tenantQueues.size());
        int baseQuota = totalConcurrency / activeTenantCount;

        // VIP租户配额 x2
        if (isVipTenant(tenantId)) {
            return Math.min(totalConcurrency, baseQuota * 2);
        }
        return Math.max(1, baseQuota);
    }
}
```

#### 1.4.4 任务自动分片策略

```java
// 大导出任务分片器
public class ExportTaskSplitter {

    public List<SubTask> split(ExportTask task) {
        long totalRows = task.getEstimatedRows();

        // 小于10万行，不分片
        if (totalRows < 100000) {
            return List.of(new SubTask(task, 0, totalRows));
        }

        // 每片 10万行，最多拆分为 50 片（防止过度分片）
        int shardSize = 100000;
        int shardCount = Math.min(50, (int) ((totalRows + shardSize - 1) / shardSize));

        List<SubTask> shards = new ArrayList<>();
        for (int i = 0; i < shardCount; i++) {
            long start = (long) i * shardSize;
            long end = Math.min(totalRows, (long) (i + 1) * shardSize);
            shards.add(new SubTask(task, start, end));
        }

        return shards;
    }
}
```

## 2. 审计日志详细设计

### 2.1 核心表结构

```sql
CREATE TABLE audit_log (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       BIGINT NOT NULL,
    user_id         BIGINT NOT NULL,
    username        VARCHAR(64) NOT NULL,
    operation       VARCHAR(32) NOT NULL,   -- LOGIN, VIEW, DOWNLOAD, EXPORT, UPDATE, DELETE, DDL
    target_type     VARCHAR(32),            -- REPORT, USER, DATASOURCE
    target_id       BIGINT,
    target_name     VARCHAR(255),
    ip_address      VARCHAR(45),
    user_agent      VARCHAR(512),
    request_params  JSONB,                   -- 请求参数快照
    response_data   JSONB,                   -- 响应摘要
    status          VARCHAR(16) NOT NULL,   -- SUCCESS, FAILURE
    error_message   TEXT,
    created_at      TIMESTAMP DEFAULT NOW(),
    log_hash        VARCHAR(64) NOT NULL    -- 日志记录哈希，防篡改
);

CREATE INDEX idx_audit_tenant_time ON audit_log(tenant_id, created_at DESC);
CREATE INDEX idx_audit_user ON audit_log(user_id);
```

### 2.2 防篡改机制

1. **单条记录哈希**：`SHA256(id + tenant_id + operation + created_at + secret_salt)`
2. **批量 Merkel 树**：每小时对该小时的所有日志计算 Merkel 树根哈希，存入独立的 `audit_root_hash` 表
3. **数据库层面禁止修改**：通过触发器实现：

   ```sql
   CREATE OR REPLACE FUNCTION prevent_audit_modify() RETURNS TRIGGER AS $$
   BEGIN
       RAISE EXCEPTION '审计日志不可修改或删除';
   END;
   $$ LANGUAGE plpgsql;

   CREATE TRIGGER audit_log_no_update BEFORE UPDATE OR DELETE ON audit_log
       FOR EACH ROW EXECUTE FUNCTION prevent_audit_modify();
   ```

## 3. 数据加密方案

### 3.1 加密分层

| 层级         | 加密方式   | 算法    | 适用对象                      |
| ------------ | ---------- | ------- | ----------------------------- |
| **传输加密** | HTTPS      | TLS 1.3 | 所有网络传输                  |
| **存储加密** | 字段级加密 | AES-256 | 数据源密码、API Key、敏感配置 |
| **文件加密** | 透明加密   | AES-256 | 共享存储中的报表文件          |

### 3.2 字段级加密实现

```java
// 敏感字段注解
@EncryptField
private String datasourcePassword;

// MyBatis TypeHandler 自动加解密
public class EncryptTypeHandler extends BaseTypeHandler<String> {
    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, String parameter, JdbcType jdbcType) {
        ps.setString(i, AESUtils.encrypt(parameter));
    }

    @Override
    public String getNullableResult(ResultSet rs, String columnName) {
        return AESUtils.decrypt(rs.getString(columnName));
    }
}
```

### 3.3 密钥管理

- 主密钥存储于 K8s Secret，启动时注入环境变量
- 支持密钥轮换：新数据用新密钥加密，旧数据后台任务异步重加密
- 密钥版本号与加密数据一同存储，解密时自动匹配对应版本密钥

## 4. 灾备与恢复

### 4.1 RPO/RTO 目标

| 故障级别    | RPO（数据丢失） | RTO（恢复时间）        |
| ----------- | --------------- | ---------------------- |
| 单 Pod 故障 | 0（无数据丢失） | < 30秒（K8s 自动重建） |
| 单节点故障  | 0               | < 5分钟                |
| 单机房故障  | ≤ 15分钟        | < 1小时                |
| 数据误删除  | ≤ 24小时        | < 4小时                |

### 4.2 备份策略

```
PostgreSQL 备份：
├── 全量备份：每日凌晨 2 点，保留 30 天
├── 增量备份：WAL 归档，每 5 分钟一次
└── 备份存储：对象存储（MinIO / S3），跨机房复制

文件备份：
├── PVC 快照：每小时快照，保留 24 小时
└── 全量备份：每日同步至对象存储
```

### 4.3 恢复演练

- 每季度执行一次恢复演练
- 演练内容：数据库恢复、文件恢复、全环境恢复
- 演练报告存档，作为合规审计材料

### 5.2 PDF 隐形水印

**微小白点编码方案**：

- 在 PDF 每页角落，用 8x8 点阵编码 64 位信息
- 白点大小 0.1pt，肉眼不可见
- 编码内容：`userId (32bit) + timestamp (32bit)`
- 可通过专门工具提取溯源

## 6. 等保三级合规清单

### 6.1 物理安全（依赖云服务商）

- ✅ 机房门禁管理
- ✅ 防火、防水、防雷、防静电
- ✅ 电力冗余（双路供电 + UPS）

### 6.2 网络安全

- ✅ 防火墙规则配置，最小权限原则
- ✅ 网络分区：DMZ 区、业务区、数据库区
- ✅ 入侵检测：WAF 防护 Web 攻击
- ✅ HTTPS 传输加密（TLS 1.2+）

### 6.3 主机安全

- ✅ 操作系统加固（关闭无用端口、最小化安装）
- ✅ 主机入侵检测 HIDS
- ✅ 定期漏洞扫描与补丁更新
- ✅ 弱口令检测与强制密码策略

### 6.4 应用安全

- ✅ 用户身份认证（用户名/密码 + 验证码）
- ✅ 登录失败处理：5次失败锁定 15 分钟
- ✅ 会话超时：30分钟无操作自动登出
- ✅ 权限分离：三权分立（系统管理员、安全管理员、审计管理员）
- ✅ 输入校验：防 SQL 注入、XSS、CSRF
- ✅ 审计日志：全量操作记录，不可删除

### 6.5 数据安全

- ✅ 敏感数据加密存储
- ✅ 数据备份与恢复机制
- ✅ 数据脱敏展示
- ✅ 残余数据清除：删除数据时真正擦除
