> 安全无小事。安全漏洞响应的核心原则是：快速响应、透明沟通、彻底修复。
>
> **最重要的原则：永远不要试图隐瞒安全漏洞。**

---

## 1. 漏洞分级

| 级别              | 定义                                               | 响应时间要求                  | 修复时限         |
| ----------------- | -------------------------------------------------- | ----------------------------- | ---------------- |
| **Critical 危急** | 可导致大规模数据泄露、系统被完全控制、影响所有用户 | 立即响应，1小时内必须有人介入 | 24小时内必须修复 |
| **High 高危**     | 可导致普通用户数据泄露、权限绕过、影响大量用户     | 4小时内响应                   | 72小时内必须修复 |
| **Medium 中危**   | 普通的安全漏洞，影响有限，需要特定条件才能利用     | 24小时内响应                  | 1周内修复        |
| **Low 低危**      | 轻微的安全问题，几乎没有实际危害                   | 下个工作日响应                | 下个版本修复     |

### Critical 示例

- 任意SQL注入，可拖库
- 身份认证完全绕过，不需要密码就能登录
- 远程代码执行漏洞
- 数据库备份公开可下载

### High 示例

- 越权访问，可以查看其他租户的数据
- 认证绕过，可以登录任意普通用户账号
- 敏感信息明文存储在前端

## 2. 漏洞接收渠道

### 2.1 官方接收渠道

- 安全专用邮箱：security@jinshu.com（示例）
- GitHub Security Advisories（推荐）
- 核心维护者的私人联系方式

### 2.2 重要提醒

**请不要在公开的Issue、讨论组里披露安全漏洞详情。**

公开披露漏洞会给攻击者可乘之机，在我们修复之前就利用漏洞发动攻击。

---

## 3. 漏洞处理标准流程

### 3.1 第一步：接收与确认（1小时内）

**收到漏洞报告后：**

1. **第一时间回复报告者**，确认我们已经收到，正在处理
2. **指定安全专员**负责本次漏洞处理
3. **验证漏洞真实性**：按照报告者提供的步骤，复现漏洞
4. **评估漏洞级别**：按照上面的标准定级
5. **登记漏洞信息**：漏洞内容、报告者信息、接收时间、级别

**给报告者的第一封回复模板：**

```
您好，

感谢您的漏洞报告。我们已经收到，正在验证和处理。
我们会在XX小时内给您进一步的反馈。

再次感谢您对项目安全的贡献！
锦书安全团队
```

---

### 3.2 第二步：遏制与缓解（Critical/High级）

对于Critical和High级漏洞，在正式修复之前，先采取临时缓解措施：

| 漏洞类型 | 临时缓解措施                     |
| -------- | -------------------------------- |
| SQL注入  | 暂时关闭相关接口，或加WAF规则    |
| 越权访问 | 暂时关闭相关功能                 |
| 认证绕过 | 加额外的验证逻辑，或暂时关闭登录 |
| 数据泄露 | 撤回公开访问权限，修改密钥       |

**目标：在修复完成前，先堵住漏洞，防止被大规模利用。**

---

### 3.3 第三步：修复与验证

**修复过程要求：**

1. **在私有分支开发修复**，不要在公开的PR里修（避免泄露漏洞）
2. **编写完整的测试用例**，覆盖所有可能的利用场景
3. **做回归测试**，确保修复不会引入新的问题
4. **多人Review**：安全相关的修复，至少2个核心维护者Review
5. **让报告者验证**：修复后，让漏洞报告者验证是否真的修好了

---

### 3.4 第四步：发布与公告

**修复发布：**

1. 发布包含安全修复的新版本
2. 明确标注这是安全更新，建议所有用户尽快升级
3. 提供升级指导

**漏洞公告：**

修复发布后，发布正式的安全公告，包含：

- 漏洞编号（如：JINSHU-2026-0001）
- 漏洞影响版本
- 漏洞严重级别
- 漏洞描述（不要太细节，避免攻击者利用）
- 修复方法（升级到哪个版本）
- 受影响的用户
- 漏洞报告者（如果报告者同意署名）
- 修复时间线

**公告发布渠道：**

- GitHub Release Notes
- 项目官网/博客
- 社区讨论组
- 邮件通知重要用户

---

### 3.5 第五步：事后复盘

安全漏洞修复后，做复盘：

1. **这个漏洞为什么会出现？**
   - 是代码Review没看出来？
   - 是测试没覆盖到？
   - 是没有安全扫描？

2. **我们的流程哪里有漏洞？**
   - 为什么没能在上线前发现？
   - 以后怎么避免同类漏洞？

3. **需要加什么防御措施？**
   - 是不是要加自动化安全扫描？
   - 是不是要加安全培训？
   - 是不是要做渗透测试？

## 4. 漏洞披露政策

### 4.1 披露时间

我们遵循"负责任披露"原则：

1. 报告者私下向我们报告漏洞
2. 我们在承诺的时间内修复并发布公告
3. 公告发布后，报告者可以公开讨论漏洞细节

**典型的时间线：**

- Critical：24小时内修复，修复后立即公告
- High：72小时内修复，修复后24小时内公告
- Medium：1周内修复，随版本公告
- Low：随下一个版本公告

### 4.2 与报告者的沟通

- ✅ 全程保持沟通，及时同步进展
- ✅ 让报告者参与验证修复
- ✅ 如果报告者同意，在公告中署名感谢
- ✅ 如果有漏洞奖励计划，按规则发奖励
- ❌ 不要在没有报告者同意的情况下公开报告者的身份

## 5. 禁止事项

### ❌ 绝对禁止

1. **隐瞒漏洞** - 知道有漏洞但不修复，也不告诉用户
2. **悄悄修复不公告** - 偷偷在某个版本里修了，但不告诉用户有安全问题
3. **甩锅给报告者** - "这不叫漏洞，是你不会用"
4. **拖延时间** - 明明是Critical漏洞，拖好几天才修

### ✅ 应该做

1. **诚实面对** - 有漏洞很正常，关键是怎么处理
2. **快速响应** - 越快修复，影响越小
3. **透明沟通** - 告诉用户真实情况，以及他们应该怎么做
4. **感谢报告者** - 人家是在帮我们改进

## 6. 漏洞编号与存档

### 6.1 编号规则

`JINSHU-YYYY-XXX`

- `YYYY`：年份
- `XXX`：当年的第几个漏洞，从001开始

示例：`JINSHU-2026-001`

### 6.2 存档要求

每个漏洞都必须存档记录，包含：

- 漏洞编号、标题、级别
- 漏洞报告者、报告时间
- 漏洞详细描述、利用方式
- 影响版本、影响范围
- 修复方案、修复时间
- 复盘结论、改进措施

存档仅在核心维护者内部可见，不对外公开。

## 7. 安全开发生命周期

为了从源头上减少安全漏洞，我们要求：

### 开发阶段

- 所有涉及权限、多租户、加密的代码，必须由安全专员Review
- 不要自己发明加密算法，用成熟的标准库
- 所有用户输入都必须做校验，永远不要信任用户输入

### Code Review阶段

- 安全是Code Review的必查项
- 任何PR都要问：这个改动有没有安全风险？

### 测试阶段

- 安全测试是发布前的必过项
- 核心模块必须做渗透测试

### 发布后

- 定期做依赖安全扫描
- 定期做完整的安全审计
- 关注业界的安全动态，及时修补相关漏洞

## 8. 给漏洞报告者的承诺

如果您向我们报告安全漏洞：

1. ✅ 我们会在24小时内确认收到您的报告
2. ✅ 我们会及时跟您同步处理进展
3. ✅ 我们会认真对待您的报告，不会说"这不是漏洞"
4. ✅ 如果您同意，我们会在公告中公开感谢您
5. ✅ 我们不会追究您善意的安全研究行为
6. ✅ 我们不会公开您的身份，除非您同意
