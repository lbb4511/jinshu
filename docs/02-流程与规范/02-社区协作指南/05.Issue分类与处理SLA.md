> 本文档定义：Issue如何分类、谁负责处理、处理时限是多少。
>
> **核心目标：** 让用户的每一个反馈都有回音，不会石沉大海。

---

## 1. Issue 类型分类

所有Issue必须打上对应的类型标签：

| 标签            | 类型        | 说明                 | 处理时限                            |
| --------------- | ----------- | -------------------- | ----------------------------------- |
| `bug`           | 🐛 Bug报告  | 现有功能不正常工作   | 24小时内首次响应                    |
| `feature`       | ✨ 功能需求 | 新功能建议或增强     | 3个工作日内首次响应                 |
| `question`      | ❓ 问题     | 使用问题、咨询       | 24小时内首次响应                    |
| `discussion`    | 💬 讨论     | 架构、设计、方向讨论 | 3个工作日内首次响应                 |
| `documentation` | 📖 文档     | 文档改进建议         | 3个工作日内首次响应                 |
| `performance`   | ⚡ 性能     | 性能相关问题         | 24小时内首次响应                    |
| `security`      | 🔒 安全     | 安全漏洞报告         | 1小时内响应（_见安全漏洞响应流程_） |

## 2. Bug 分级与处理流程

### 2.1 Bug 严重程度分级

| 级别 | 标签                | 定义                                         | 修复时限     |
| ---- | ------------------- | -------------------------------------------- | ------------ |
| P0   | `priority/critical` | 核心功能完全不可用，没有Workaround           | 24小时内修复 |
| P1   | `priority/high`     | 核心功能部分不可用，或有Workaround但体验很差 | 72小时内修复 |
| P2   | `priority/medium`   | 非核心功能问题，或有明显的Workaround         | 下个版本修复 |
| P3   | `priority/low`      | 轻微问题、UI问题、不影响使用                 | 有空再修     |

### 2.2 Bug 处理标准流程

```
用户提交Bug
    ↓
1小时内：维护者确认收到，打上标签
    ↓
24小时内：确认是否真的是Bug，能否复现
    ↓
    ├─ 不是Bug → 说明原因，关闭Issue
    ├─ 重复Bug → 标注重复，链接到原Issue，关闭
    └─ 确实是Bug → 确认严重级别，指定负责人
        ↓
负责人评估，给出预计修复时间
    ↓
修复，提交PR
    ↓
Review通过，合并
    ↓
通知Issue提交者，验证修复
    ↓
关闭Issue
```

### 2.3 一个好的Bug报告应该包含

请用户提交Bug时提供以下信息：

1. **版本号**：你用的是哪个版本？
2. **环境**：什么操作系统？什么浏览器？
3. **复现步骤**：一步一步告诉我怎么操作能重现这个问题
4. **期望结果**：你觉得应该是什么样？
5. **实际结果**：实际上是什么样？
6. **截图/日志**：有截图或错误日志吗？

## 3. 功能需求处理流程

### 3.1 功能需求评估维度

收到功能需求后，从以下维度评估：

| 维度       | 评估要点                                 |
| ---------- | ---------------------------------------- |
| **价值**   | 有多少用户需要这个功能？能解决什么痛点？ |
| **成本**   | 开发需要多少人天？维护成本高吗？         |
| **风险**   | 有没有技术风险？会不会影响现有功能？     |
| **优先级** | 跟现有Roadmap比，优先级高吗？            |
| **契合度** | 跟项目的定位和方向一致吗？               |

### 3.2 功能需求处理流程

```
用户提交功能需求
    ↓
3个工作日内：维护者确认收到
    ↓
维护者初步评估：
  ├─ 明显不适合 → 说明原因，关闭
  ├─ 已经有类似需求 → 标注重复，链接，关闭
  └─ 值得讨论 → 打上 `need-discussion` 标签
        ↓
公开讨论，收集更多意见和使用场景
    ↓
维护者评估，给出结论：
  ├─ 接收，加入Roadmap → 说明计划的版本和时间
  ├─ 接收，但优先级不高 → 放进Backlog，以后再做
  └─ 不接收 → 详细说明原因，关闭Issue
```

### 3.3 怎么提高功能需求被接受的概率

✅ **说清楚"为什么"**：比"我要XX功能"更重要的是"我遇到了XX问题，需要XX功能来解决"
✅ **提供使用场景**：你在什么情况下会用这个功能？解决什么具体问题？
✅ **考虑通用性**：这个功能是只有你需要，还是很多人都会需要？
✅ **给出方案建议**：如果你有具体的实现想法，也可以说出来

## 4. Issue 生命周期管理

### 4.1 状态标签

| 标签                 | 说明                           |
| -------------------- | ------------------------------ |
| `needs-info`         | 需要更多信息才能继续处理       |
| `needs-reproduction` | 需要有人帮忙复现问题           |
| `needs-discussion`   | 需要讨论才能决定怎么做         |
| `good-first-issue`   | 适合新人上手的Issue            |
| `help-wanted`        | 欢迎社区贡献者来做             |
| `in-progress`        | 有人正在做了                   |
| `blocked`            | 被其他事情卡住了               |
| `wontfix`            | 这个问题我们不会修（说明原因） |
| `duplicate`          | 重复的Issue                    |

### 4.2 Stale Issue 处理

超过30天没有活动的Issue：

1. 自动打上 `stale` 标签
2. 留言提醒："这个Issue已经30天没有活动了，如果没有进一步的信息，7天后会自动关闭"
3. 再过7天还是没有活动，自动关闭

**例外情况：**

- 已经放进Roadmap的功能需求
- 确认会修但优先级不高的Bug
- 正在讨论中的重要话题

## 5. Issue 负责人分配规则

### 5.1 自动分配

1. **安全相关** → 自动分配给安全专员
2. **性能相关** → 自动分配给性能负责人
3. **特定模块** → 自动分配给对应模块的Maintainer
4. **其他** → 轮流分配，或谁有空谁接

### 5.2 负责人职责

- 负责整个Issue的处理全程
- 及时同步状态和进展
- 给出最终的处理结论
- 确保Issue不会被遗忘

## 6. 回复用户的基本原则

### ✅ 应该做到

1. **及时回复**：哪怕只是说"我们收到了，正在看"
2. **诚实告知**：做就是做，不做就是不做，不要画饼
3. **解释原因**：不做的话，一定要说明为什么不做
4. **礼貌友好**：感谢用户花时间反馈
5. **给出预期**：大概什么时候能有结果

### ❌ 不应该做

1. **不回复**：用户提了Issue，石沉大海，再也没人理
2. **甩锅**："这不是Bug，是你不会用"
3. **画饼**："我们会做的"，然后再也没下文
4. **态度恶劣**："这么简单的问题都不会？"
5. **自动回复机器人来回好几次，始终没有真人出现**

## 7. 常见问题

### Q: 用户提的需求很离谱怎么办？

**A:** 礼貌地说明为什么不做，不要嘲讽。

**好的回复：**

> "感谢你的建议！我们评估了一下，这个功能跟项目当前的定位不太一致，而且只有极少数用户会用到，所以暂时不打算做。不过还是非常感谢你的反馈！"

**坏的回复：**

> "这需求太奇葩了，不做。"

---

### Q: 同一个Bug被反复提怎么办？

**A:** 写一个FAQ或者常见问题汇总，每次有人提就把链接发给他。同时考虑是不是错误提示不够友好，导致用户不知道这是正常的。

---

### Q: 提Issue的人再也不回复了怎么办？

**A:** 打上 `needs-info` 标签，等7天。还是不回复的话，留言说明，关闭Issue。等他什么时候回来了，有了新的信息再重新打开。

---

### Q: 很多人催一个功能怎么办？

**A:** 诚实告知当前的优先级和排期。如果确实很多人需要，可以调整优先级。不要为了安抚用户就随便承诺做不到的事情。

---

> 📌 最后一句话：
>
> **Issue是用户给你的礼物。**
>
> 有人愿意花时间给你反馈，不管反馈的内容是什么，这本身就是一件值得感谢的事情。好好对待每一个Issue，就是好好对待你的用户。
