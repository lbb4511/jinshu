> 本文档定义项目中的决策机制：谁、在什么时间、如何做决策。
>
> **核心原则：** 小决策快，大决策慎，所有决策留痕。

---

## 1. 决策分级

### 1.1 第1类：重大决策（不可逆）

**定义：** 一旦决策错误，回滚成本极高，会影响整个项目的发展方向。

**包括：**

- 技术栈重大变更（如：MyBatis → JPA，PostgreSQL → MySQL）
- 架构模式重大变更（如：单体 → 微服务）
- 核心约束变更（如：放宽多租户隔离要求）
- 开源协议变更
- 项目方向重大调整

**审批人：** 所有核心维护者一致同意

**决策时限：** 至少72小时讨论期，不急着拍板

---

### 1.2 第2类：重要决策（部分可逆）

**定义：** 影响范围较大，有一定回滚成本，但不会致命。

**包括：**

- 新增核心模块的架构设计
- 数据库表结构的重大变更
- 引入新的重量级依赖（如：规则引擎、工作流引擎）
- 流程规范的重大变更
- 安全策略的调整

**审批人：** 至少2名核心维护者同意

**决策时限：** 24-48小时讨论期

---

### 1.3 第3类：常规决策（可逆）

**定义：** 影响范围有限，容易回滚，即使决策错了成本也不高。

**包括：**

- 单个功能的实现方案选择
- 代码层面的重构方案
- 非核心依赖的升级
- 文档结构调整
- 工具链的小调整

**审批人：** 对应模块的负责人即可

**决策时限：** 快速决策，几小时到1天

---

### 1.4 第4类：日常决策（完全可逆）

**定义：** 几乎没有负面影响，随时可以改。

**包括：**

- 变量命名、代码格式
- 文档措辞调整
- 简单的Bug修复方案
- 开发工具的个人选择

**审批人：** 开发者自行决定，不需要审批

---

## 2. 决策流程

### 2.1 标准决策流程

```
提出问题
    ↓
收集信息，列出可选方案
    ↓
评估每个方案的 Pros / Cons
    ↓
公开讨论（Issue / 讨论组）
    ↓
根据决策级别，由相应审批人决策
    ↓
记录决策结果与理由
    ↓
执行决策
```

### 2.2 紧急决策流程

**适用场景：** 线上故障、安全漏洞等需要立即处理的情况

```
发现紧急问题
    ↓
直接联系对应模块负责人或核心维护者
    ↓
口头/即时通讯快速决策
    ↓
立即执行
    ↓
事后24小时内补决策记录
```

**注意：** 紧急决策只能由核心维护者做出，事后必须补记录。

## 3. 异议升级机制

### 3.1 升级触发条件

当出现以下情况时，可以启动异议升级：

1. 讨论超过48小时仍无法达成共识
2. 双方各执一词，都有充分理由
3. 决策可能影响到你负责的模块，但你没有被充分征求意见

### 3.2 升级路径

```
第一级：当事人之间直接沟通
    ↓ 无法解决
第二级：拉上模块负责人一起讨论
    ↓ 仍无法解决
第三级：提交所有核心维护者，投票表决
```

### 3.3 投票规则

- 核心维护者一人一票
- 简单多数（>50%）通过
- 若平票，由项目创始人做最终决定
- 投票结果必须公开，每个人的投票和理由都要记录

### 3.4 决策后的要求

一旦做出最终决策：

- ✅ 所有人必须执行，即使你不同意
- ✅ 可以保留个人意见，但不能阻碍执行
- ✅ 如果事后证明决策错了，可以重新启动决策流程，但不能在执行阶段"阳奉阴违"
- ❌ 不允许"私下改回去"
- ❌ 不允许在其他地方吐槽、抱怨

## 4. 决策记录要求

### 4.1 所有第1、2类决策必须记录

**记录模板：**

| 项         | 内容                                    |
| ---------- | --------------------------------------- |
| 决策主题   | 一句话概括决策内容                      |
| 决策日期   | YYYY-MM-DD                              |
| 决策人     | 谁拍板的                                |
| 参与讨论者 | 哪些人参与了讨论                        |
| 背景       | 为什么需要做这个决策                    |
| 可选方案   | 当时考虑了哪几个方案                    |
| 最终选择   | 选了哪个方案                            |
| 选择理由   | 为什么选这个，主要考虑了哪些因素        |
| 反对意见   | 有没有人反对，反对的理由是什么          |
| 后续评估点 | 什么时间/什么指标下需要重新评估这个决策 |

### 4.2 记录存档位置

- 第1类决策：存入 `docs/99-附录/重大决策记录.md`
- 第2类决策：存入 `docs/99-附录/重要决策记录.md`
- 第3、4类决策：可以不专门记录，但在PR/Issue中要体现讨论过程

## 5. 决策的重新评估

### 5.1 什么情况下可以重新讨论已做出的决策

1. 出现了决策时没有考虑到的重大新信息
2. 决策执行后，实际结果明显不符合预期
3. 外部环境发生了重大变化（如：某个依赖停止维护了）

### 5.2 什么情况下**不应该**重新讨论

1. 仅仅是"我还是觉得当初那个方案好"，没有新信息
2. 决策刚做，还没看到结果就想推翻
3. 换了新人，"新官不理旧账"

**原则：** 不要反复折腾。决策做出后，至少给它3个月的验证期，除非出现了明显的问题。

## 6. 常见误区

### ❌ 反模式

1. **决策瘫痪**：收集了无穷多的信息，永远在讨论，永远不拍板
2. **拍脑袋决策**：既不收集信息，也不征求意见，一个人说了算
3. **人人都要同意**：追求100%共识，导致决策无限拖延
4. **朝令夕改**：今天做的决策，明天就推翻
5. **不记录**：决策做了，但没人记为什么，后人无法理解

### ✅ 好模式

1. **可逆决策尽快做，不可逆决策慎重做**
2. **谁执行，谁主导决策**（让干活的人有话语权）
3. **用数据说话，而不是凭感觉**
4. **异议当场说，会后不嘀咕**
5. **对事不对人，我们争论的是方案，不是人品**

## 7. 权责对照表

| 决策类型      | 谁可以提议     | 谁可以否决     | 最终拍板               |
| ------------- | -------------- | -------------- | ---------------------- |
| 第1类重大决策 | 任何人         | 任何核心维护者 | 所有核心维护者一致同意 |
| 第2类重要决策 | 任何人         | 模块负责人     | 至少2名核心维护者同意  |
| 第3类常规决策 | 对应模块开发者 | 模块负责人     | 模块负责人             |
| 第4类日常决策 | 开发者本人     | -              | 开发者本人             |

---

> 📌 最后提醒：
>
> 1. **没有完美的决策**：任何决策都有风险，关键是"足够好 + 可调整"
> 2. **决策质量 ≠ 共识程度**：不是同意的人越多，决策就越好
> 3. **勇于承认错误**：如果发现决策错了，承认并改正，这不丢人
