## 核心原则

Code Review 的目的：**提升代码质量，发现潜在问题，统一团队风格，知识共享**。

不是为了"挑错"，也不是为了"证明谁更厉害"。

---

## Review 流程

```
提交 PR
    ↓
自动检查（编译、测试、代码规范）
    ↓
至少 1 人 Review 通过
    ↓
合并
```

### Review 时限

- 普通 PR：24 小时内完成 Review
- 紧急 Bug 修复：2 小时内完成 Review
- 大型重构/新模块：提前预约时间

## Review 标准

### 🟥 必须修改（不修改不能合并）

#### 1. 功能正确性

- [ ] 代码逻辑有明显错误
- [ ] 边界场景没有处理（空值、0值、最大值、并发）
- [ ] 与设计文档不一致（必须先更新设计文档）

#### 2. 安全问题

- [ ] 存在 SQL 注入、XSS、CSRF 等安全漏洞
- [ ] 没有做权限校验
- [ ] 绕过了多租户隔离
- [ ] 敏感信息明文存储或打印到日志
- [ ] 审计日志缺失

#### 3. 可靠性问题

- [ ] 异常被吞掉（catch 后不记录日志）
- [ ] 资源没有释放（数据库连接、文件句柄）
- [ ] 死锁风险
- [ ] 没有幂等处理（重复提交会出问题）

#### 4. 合规问题

- [ ] 引入了商业代码或传染性协议依赖
- [ ] 复制了第三方代码但没有注明来源

---

### 🟧 建议修改（不阻塞合并，但最好改）

#### 1. 性能问题

- [ ] N+1 查询（循环里查数据库）
- [ ] 全表扫描，没有索引
- [ ] 大对象没有及时释放，有内存溢出风险
- [ ] 并发场景没有考虑线程安全

#### 2. 可维护性

- [ ] 命名不清晰（看不懂变量/函数是做什么的）
- [ ] 函数太长（超过 50 行）
- [ ] 嵌套太深（超过 3 层）
- [ ] 重复代码（同样的逻辑写了 3 遍以上）
- [ ] 魔数（没有解释的数字常量）

#### 3. 可测试性

- [ ] 核心逻辑没有单元测试
- [ ] 代码写死了，无法 Mock 测试
- [ ] 测试只测了正常流程，没测异常流程

---

### 🟩 风格问题（不阻塞合并，建议统一）

- 代码格式（缩进、空行、括号风格）
- 命名风格（小驼峰、大驼峰）
- 注释风格

这类问题建议通过自动化工具（CheckStyle、ESLint）统一，不要在 Review 里纠结。

## Review 技巧

### 给 Reviewer 的建议

1. **理解背景**：先看设计文档和 Issue，理解这个 PR 要解决什么问题
2. **关注重点**：不要纠结代码格式，重点看逻辑、安全、性能
3. **具体明确**：指出问题时要说"第 XX 行，XXX 情况会出问题"，而不是"这里写得不好"
4. **提供方案**：发现问题时，最好同时给出改进建议
5. **区分严重程度**：用 emoji 或标签明确哪些是必须改的，哪些是建议
6. **及时响应**：收到 Review 请求尽快处理，不要让别人等太久

### 给 Author 的建议

1. **小步提交**：一个 PR 只做一件事，最好不要超过 500 行
2. **写好描述**：PR 描述要写清楚背景、变更内容、测试结果
3. **主动解释**：复杂逻辑可以在代码里加注释，或者在 PR 描述里说明
4. **虚心接受**：Review 是为了代码更好，不是针对个人
5. **不要争论**：有不同意见时，线下沟通，不要在评论里拉锯

## 常见误区

### ❌ 反模式

1. **只看不思考**：随便看两眼就点通过
2. **吹毛求疵**：纠结缩进、空行这类格式问题，忽略真正的风险
3. **过度设计**：要求作者为"未来可能的需求"做大量抽象
4. **人身攻击**："你怎么又写出这种代码"
5. ** rubber stamp**：不管内容直接点通过

### ✅ 好模式

1. **学习心态**：Review 也是学习别人代码的机会
2. **关注结果**：代码好不好，关键是能不能解决问题，而不是写得"炫"
3. **灵活处理**：有时候"能用的代码"比"完美的代码"更重要
4. **肯定优点**：看到写得好的地方，不要吝啬赞美

## 否决权

以下情况任何核心开发者都可以行使否决权，要求重写：

1. 存在严重安全漏洞
2. 破坏多租户隔离
3. 引入商业代码或违反开源合规
4. 设计文档都没有，直接写了几千行代码

## 附录：Review CheckList

提交 PR 前自查：

- [ ] 设计文档已评审通过
- [ ] 本地编译和测试全部通过
- [ ] 所有 SQL 都有 tenant_id 过滤
- [ ] 敏感操作都有审计日志
- [ ] 异常都有记录日志，没有吞掉
- [ ] 没有硬编码的密码或密钥
- [ ] 核心逻辑有单元测试
- [ ] 边界场景都考虑到了
