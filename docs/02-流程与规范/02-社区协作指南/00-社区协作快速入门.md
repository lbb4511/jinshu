> 本指南专为：有外部贡献者参与的开源项目场景优化。
>
> **核心理念：降低参与门槛，提高协作效率，保证项目健康可持续发展。**

---

## 🎯 社区协作的四大核心目标

| 目标         | 具体表现                           |
| ------------ | ---------------------------------- |
| **降低门槛** | 新人来了知道怎么参与，不会无从下手 |
| **提高效率** | Issue/PR 快速处理，不会石沉大海    |
| **保证质量** | 多人协作代码质量不会下降           |
| **留住人才** | 贡献者有成长路径，愿意持续参与     |

---

## 📊 社区成长的三个阶段

### 阶段一：种子阶段（≤5 个贡献者）

**特点：** 大家都认识，沟通成本低。

**推荐流程简化：**

- ✅ Issue 不用严格分类，有人理就行
- ✅ PR Review 不用太严格，能跑就行
- ✅ 决策大家群里聊两句就定了
- ❌ 不要搞太复杂的流程，反而影响效率

---

### 阶段二：成长阶段（5-20 个贡献者）

**特点：** 有陌生人加入，沟通成本上升。

**推荐启用完整流程：**

- ✅ Issue 按模板提交，分类处理
- ✅ PR 严格按 Code Review 标准
- ✅ 正式的决策流程，不再拍脑袋
- ✅ 贡献者晋升路径开始运行

---

### 阶段三：成熟阶段（≥20 个贡献者）

**特点：** 有明确的模块负责人，分层管理。

**推荐加强：**

- ✅ 模块 OWNER 制度，每个模块有专门负责人
- ✅ 正式的投票决策机制
- ✅ 定期的社区会议和公开路线图

---

## 👥 社区角色与分工

### 角色一：Issue 处理员（可以让 AI 帮忙）

**职责：**

1. 新 Issue 24 小时内必须有回复
2. 给 Issue 打标签（bug/feature/question...）
3. 重复的 Issue 标注重复，链接到原 Issue
4. 需要更多信息的，@ 提交者要信息

**标准回复模板库：**

- **收到 Bug：**

  > "感谢反馈！我们已经收到，正在确认问题。会尽快给你回复。"

- **需要更多信息：**

  > "麻烦提供一下具体的复现步骤、版本号、错误日志，这样我们能更快定位问题。"

- **重复 Issue：**
  > "这个问题之前已经有人提过了，参见 #123。我们会在那边统一跟进，这个先关闭了。"

---

### 角色二：PR 审核员

**职责：**

1. PR 提交后 3 个工作日内必须有第一次 Review
2. 按 Code Review 标准审核，区分"必须改"和"建议改"
3. 审核通过后合并

**给新贡献者的 PR 审核原则：**

- ✅ 多鼓励，少批评
- ✅ 不仅说"不对"，还要说"怎么改才对"
- ✅ 第一个 PR，哪怕写得不好，也要耐心指导
- ✅ 合并后一定要说一句"感谢贡献！欢迎继续参与！"

---

### 角色三：模块负责人

**职责：**

1. 负责这个模块的架构和质量
2. 这个模块的所有 PR 都需要他审核
3. 这个模块的相关 Issue 由他处理
4. 带领这个模块的发展方向

---

### 角色四：项目维护者

**职责：**

1. 项目整体方向和路线图
2. 重大决策和冲突仲裁
3. 社区氛围和文化建设
4. 贡献者晋升审核

---

## 🔄 社区协作标准流程

### 流程一：用户提 Issue → 处理

```
用户提交 Issue
    ↓
24小时内：Issue 处理员确认收到，打标签
    ↓
    ├─ 重复 → 标注，链接，关闭
    ├─ 需要更多信息 → @用户要信息，等7天不回复就关闭
    ├─ 明显不会做 → 礼貌说明原因，关闭
    └─ 值得做 → 分配给对应模块负责人
        ↓
处理完成 → 通知用户，关闭 Issue
```

---

### 流程二：贡献者提 PR → 合并

```
贡献者提交 PR
    ↓
自动检查（CI跑测试、代码规范检查）
    ↓
3个工作日内：第一次 Review
    ↓
    ├─ 需要修改 → 给出具体修改意见
    └─ 没问题 → 合并，感谢贡献
        ↓
如果是新人贡献，欢迎加入社区，邀请继续参与
```

**PR 审核的黄金 24 小时原则：**

> 不管 PR 写得好不好，24 小时内必须有人看一眼，哪怕只是说一句"收到了，我会看的"。
>
> 最打击贡献者积极性的，是提了 PR 之后石沉大海，再也没人理。

---

### 流程三：功能需求讨论 → 决策

```
有人提功能需求
    ↓
3个工作日内：维护者确认收到
    ↓
公开讨论，收集意见和使用场景
    ↓
维护者评估，给出结论：
  ├─ 接收，加入 Roadmap → 说明计划版本
  ├─ 接收，但优先级不高 → 放进 Backlog
  └─ 不接收 → 详细说明为什么，不要只说"不做"
```

**拒绝需求的正确姿势：**

> ❌ 错误："不做。"
>
> ✅ 正确："感谢你的建议！我们评估了一下，这个功能跟项目当前的定位不太一致，而且目前只有你一个人提出这个需求，所以暂时不打算做。不过还是非常感谢你的反馈！"

## 🎁 激励贡献者的 5 个低成本方法

贡献者不只是为了代码而来，更是为了认可和成长。

### 1. 公开感谢

- 每个 Release Note 里列出所有贡献者
- 重要的贡献，在社区里公开表扬
- 第一次贡献的新人，专门欢迎一下

### 2. 给权限

- 贡献过 3 个 PR 以上，就可以给 Committer 权限
- 让他可以 Review 别人的代码，可以自己合并小的 PR
- 被信任的感觉，比什么都强

### 3. 给责任

- 让贡献者负责一个小模块
- 让他做这个模块的负责人，自己做决策
- 人有了责任感，就会更投入

### 4. 给成长

- 给新人指导，教他怎么写更好的代码
- 让他参与设计讨论，听听他的想法
- 帮助他成长，他会用贡献来回报

### 5. 给荣誉

- 每年评个"最佳贡献者"
- 给个虚拟的头衔或者小奖品
- 花不了多少钱，但大家会很有成就感

## 🚫 社区协作的 7 条禁忌

1. ❌ **不要不理人** - 用户提了 Issue/PR，哪怕说一句"收到了"也行，不要石沉大海
2. ❌ **不要态度恶劣** - "这么简单的问题都不会？" "写的什么垃圾代码" - 说一次就会永远赶走一个贡献者
3. ❌ **不要只提问题不给方案** - 不要只说"这里写得不好"，要说"你可以改成 xxx，这样更好"
4. ❌ **不要画饼** - 做就是做，不做就是不做，不要说"我们以后会做"然后再也没下文
5. ❌ **不要搞特权** - 维护者的代码也要接受 Review，不能自己直接合并
6. ❌ **不要搞小圈子** - 公开讨论，公开决策，不要私下拉小群搞暗箱操作
7. ❌ **不要忘了新人** - 新人的第一个 PR 是最重要的，一定要认真对待，耐心指导

## 💡 常见问题

### Q: 没人愿意贡献怎么办？

**A:**

1. 先降低门槛，标一些 `good-first-issue`，让新人容易上手
2. 有人贡献了，好好对待，他会带来更多人
3. 社区是慢慢长起来的，不要急，先把基础做好

### Q: 有人提的需求很离谱怎么办？

**A:** 礼貌说明为什么不做，不要嘲讽。哪怕需求再离谱，人家也是花了时间给你提的，值得尊重。

### Q: PR 写得太烂，怎么办？

**A:**

- 不要直接拒绝，更不要嘲讽
- 耐心告诉你应该怎么改
- 如果实在差太远，可以说"这个改动比较大，建议我们先在 Issue 里讨论一下方案再写代码"
- 只要人家愿意学，就多给点耐心

### Q: 贡献者之间吵架怎么办？

**A:**

- 维护者要及时出来拉架
- 对事不对人，只讨论技术，不讨论人品
- 再吵就先锁 Issue，让大家冷静一下
- 屡教不改的，只能请他离开

## 🎯 最后一句话

**开源项目不是比谁代码写得好，是比谁能聚集更多的人一起把事情做好。**

技术很重要，但社区更重要。
代码可以重写，但人心散了就再也聚不起来了。

对每一个贡献者好一点，耐心一点，你的项目会越来越好的。
