## 1. 背景与目标

### 1.1 问题背景

当前项目处于骨架阶段，仅实现了健康检查接口。随着核心模块设计文档就绪、编码工作即将展开，必须在团队开始写 Controller 之前统一 API 规范：

| 问题                     | 影响                                                 |
| ------------------------ | ---------------------------------------------------- |
| 无统一的路径前缀和版本号 | 路由混乱、API 升级时无法灰度                         |
| 无参数校验规范           | 跨团队开发时校验方式不一致（有的用注解、有的手动写） |
| 无文件上传/下载规范      | 大文件场景缺少分片、断点续传约定                     |
| 无接口废弃流程           | 旧接口下线可能造成前端/第三方兼容性问题              |
| Swagger 未配置           | 前后端联调依赖口头约定，效率低                       |
| 内部服务调用无规范       | Worker/DataSync 服务间接口认证方式未约定             |

### 1.2 设计目标

| 指标       | 要求                                           |
| ---------- | ---------------------------------------------- |
| 一致性     | 所有 Controller 遵循相同的路径、参数、响应格式 |
| 可版本化   | 支持 API 版本号，破坏性变更新版本灰度          |
| 可自文档化 | Swagger/OpenAPI 自动生成，接口即文档           |
| 前后端对齐 | 前端 TypeScript 类型与后端 Swagger Schema 一致 |
| 安全       | 统一认证方式、输入校验、防攻击                 |

### 1.3 适用范围

- `backend/api` 模块的所有 REST Controller
- `backend/data-sync` 模块对外暴露的 REST API
- `backend/batch`、`backend/worker` 的对内管理端口（如就绪探针）
- 前端 `services/` 层的 API 调用封装

## 2. URL 路径规范

### 2.1 路径前缀

| 前缀         | 用途                | 说明                                        |
| ------------ | ------------------- | ------------------------------------------- |
| `/api/`      | 前端调用的业务 API  | 所有用户可感知的业务接口                    |
| `/internal/` | 内部服务间调用      | Worker → API、Worker → DataSync             |
| 无前缀       | K8s 探针 + 管理端点 | `/health`, `/ready`, `/live`, `/actuator/*` |

**示例**：

```
GET  /health                        # K8s 探针（公开）
GET  /ready                         # K8s 就绪探针（公开）
GET  /live                          # K8s 存活探针（公开）
POST /api/auth/login                # 用户登录
POST /api/auth/refresh              # Token 刷新
GET  /api/reports                   # 报表列表
GET  /api/reports/{id}              # 报表详情
PUT  /api/reports/{id}              # 更新报表
POST /api/reports                   # 创建报表
POST /api/import/upload             # 文件上传
GET  /api/export/estimate            # 导出预估
POST /api/export/submit             # 提交导出任务
POST /internal/reports/{id}/render-data   # 内部：获取渲染数据
GET  /internal/health               # 内部健康检查
```

### 2.2 路径命名

| 规则              | 正确示例                       | 错误示例                                        |
| ----------------- | ------------------------------ | ----------------------------------------------- |
| 全部小写          | `/api/reports`                 | `/api/Reports`                                  |
| 连字符分隔多词    | `/api/report-data`             | `/api/report_data`、`/api/reportdata`           |
| 复数资源名        | `/api/reports`、`/api/tenants` | `/api/getReport`、`/api/createTenant`           |
| 层级不超过 3 层   | `/api/reports/{id}/preview`    | `/api/reports/{id}/details/columns/data`        |
| 动词仅用于非 CRUD | `/api/reports/{id}/submit`     | `/api/reports/create`（应用 POST /api/reports） |

### 2.3 版本管理

- 版本号位于路径中：`/api/v1/reports`、`/api/v2/reports`
- 初期默认版本为 `v1`，容许不带版本号访问（兼容期 3 个月后下线）
- 破坏性变更（删除字段、修改字段类型、修改必填/可选性）必须发布新版本
- 非破坏性变更（新增可选字段、新增接口）无需升级版本号
- Swagger 按版本分组：`@Tag(name = "v1 - Reports")`

```java
// 推荐：显式版本号
@RestController
@RequestMapping("/api/v1/reports")
@Tag(name = "v1 - Reports")
public class ReportController { }

// 兼容期（不推荐新接口使用）：
@RestController
@RequestMapping({"/api/v1/reports", "/api/reports"})
@Tag(name = "v1 - Reports")
public class ReportController { }
```

## 3. HTTP 方法规范

### 3.1 方法映射

| HTTP 方法 | 语义     | 示例                       | 幂等性 | 请求体 |
| --------- | -------- | -------------------------- | :----: | :----: |
| GET       | 查询     | `GET /api/reports/{id}`    |   ✓    |   ✗    |
| POST      | 创建     | `POST /api/reports`        |   ✗    |   ✓    |
| PUT       | 全量更新 | `PUT /api/reports/{id}`    |   ✓    |   ✓    |
| PATCH     | 部分更新 | `PATCH /api/reports/{id}`  |   ✗    |   ✓    |
| DELETE    | 删除     | `DELETE /api/reports/{id}` |   ✓    |   ✗    |

> 本项目暂不使用 PATCH，部分更新统一用 PUT 全量传递字段（字段缺失视为置 null 或保持原值，具体由模块设计决定）。

**POST 的特殊用例（非创建语义）**：

| 场景         | 方法 | 路径示例                    | 说明                  |
| ------------ | ---- | --------------------------- | --------------------- |
| 提交审查     | POST | `/api/reports/{id}/submit`  | 触发状态流转          |
| 审批通过     | POST | `/api/reports/{id}/approve` |                       |
| 驳回         | POST | `/api/reports/{id}/reject`  |                       |
| 发布         | POST | `/api/reports/{id}/publish` |                       |
| Token 刷新   | POST | `/api/auth/refresh`         |                       |
| 文件上传     | POST | `/api/import/upload`        | `multipart/form-data` |
| 提交导出任务 | POST | `/api/export/submit`        |                       |

### 3.2 幂等性要求

- GET、PUT、DELETE 必须是幂等的（重复调用结果一致）
- POST 原则上非幂等，但关键操作（如提交审查）必须防重复提交：
  - 前端提交后按钮禁用，直到收到响应
  - 后端通过业务状态机拒绝重复操作（如已通过的报表再次审批返回"审批已完成"）

## 4. 请求规范

### 4.1 请求头

| Header            |   要求   | 说明                                                           |
| ----------------- | :------: | -------------------------------------------------------------- |
| `Authorization`   | 条件必须 | `Bearer <accessToken>`，认证接口外必填                         |
| `Content-Type`    | 条件必须 | `application/json`（默认），文件上传时为 `multipart/form-data` |
| `X-Trace-Id`      |   可选   | 全链路追踪 ID（前端生成或 Consumer 传递）                      |
| `Accept`          |   推荐   | `application/json`                                             |
| `Accept-Language` |   可选   | 国际化预留（二期）                                             |

### 4.2 请求体

- 统一使用 JSON 格式
- 日期时间格式：ISO 8601（`2026-05-18T10:30:00Z` 或 `2026-05-18`）
- 字段命名：camelCase（与 Java 字段名保持一致，Jackson 自动映射）

```json
// 正确
{
  "reportName": "销售月报",
  "tenantId": 1,
  "createdAt": "2026-05-18T10:00:00Z"
}

// 错误
{
  "report_name": "销售月报",    // 应用 camelCase
  "tenant_id": 1,               // 应用 camelCase
  "created_at": "2026-05-18"   // 日期时间应用 ISO 8601
}
```

### 4.3 路径参数与查询参数

| 参数类型 | 使用场景           | 示例                                               |
| -------- | ------------------ | -------------------------------------------------- |
| 路径参数 | 资源标识（ID）     | `/api/reports/{id}`                                |
| 查询参数 | 筛选、排序、分页   | `/api/reports?status=DRAFT&page=1&size=20`         |
| 查询参数 | 可选过滤条件       | `/api/reports?createdBy=张三&startDate=2026-05-01` |
| 请求体   | 创建/更新/复杂查询 | `POST /api/reports`                                |

**查询参数命名**：camelCase，与请求体字段保持一致。

### 4.4 参数校验

使用 Spring Validation 注解约束，不手写 `if` 校验：

```java
@PostMapping
public Result<Report> create(@Valid @RequestBody CreateReportRequest request) {
    // Controller 层只做校验注解和调用 Service
    return Result.success(reportService.create(request));
}
```

**请求 DTO 校验注解**：

```java
public class CreateReportRequest {

    @NotBlank(message = "报表名称不能为空")
    @Size(max = 100, message = "报表名称最长 100 字符")
    private String reportName;

    @NotNull(message = "租户 ID 不能为空")
    @Positive(message = "租户 ID 必须为正数")
    private Long tenantId;

    @Size(max = 500, message = "描述最长 500 字符")
    private String description;
}
```

## 5. 响应规范

### 5.1 统一返回格式

所有业务接口必须通过 `Result<T>` 包装返回。详见 `03.统一返回与异常处理设计.md`。

```json
// 成功
{
  "code": 0,
  "message": "success",
  "data": { "id": 1001, "name": "销售月报" }
}

// 失败
{
  "code": 10001,
  "message": "报表不存在",
  "data": null
}
```

> 例外：K8s 探针接口（`/health`、`/ready`、`/live`）和文件下载接口不走 `Result<T>` 包装，直接返回 `ResponseEntity`。

### 5.2 分页返回格式

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [
      { "id": 1001, "name": "报表 A" },
      { "id": 1002, "name": "报表 B" }
    ],
    "total": 142,
    "page": 1,
    "size": 20,
    "totalPages": 8
  }
}
```

### 5.3 HTTP 状态码

| 状态码 | 场景                             | 说明                    |
| :----: | -------------------------------- | ----------------------- |
|  200   | 请求成功                         | 含空列表、空数据        |
|  201   | 创建资源成功                     | 返回新资源的 URI + 数据 |
|  204   | 删除成功（无返回内容）           | 不返回 body             |
|  400   | 参数校验失败                     | `code` 在 40000-40999   |
|  401   | 未认证（Token 无效/过期）        | `code` 在 20000-20099   |
|  403   | 无权限（Token 有效但角色不匹配） | `code` 在 20000-20099   |
|  404   | 资源不存在                       | `code` 在 10000-19999   |
|  409   | 资源冲突（如状态不允许当前操作） | `code` 在 10000-19999   |
|  413   | 上传文件超限                     | `code` 在 40000-40999   |
|  415   | 不支持的文件类型                 | `code` 在 40000-40999   |
|  423   | 账号/资源被锁定                  | `code` 在 20000-20099   |
|  429   | 请求过于频繁（限流）             | `code` 在 50000-50999   |
|  500   | 服务器内部错误                   | `code` 在 50000-50999   |
|  503   | 服务暂时不可用                   | 探针接口 / 降级响应     |

### 5.4 业务错误码

详见 `ErrorCode.java` 枚举，以下为分段速查：

| 段    | 范围        | 说明                               |
| ----- | ----------- | ---------------------------------- |
| 0     | 0           | 成功                               |
| 1xxxx | 10000-19999 | 资源错误（不存在、冲突、状态异常） |
| 2xxxx | 20000-29999 | 认证授权错误                       |
| 4xxxx | 40000-49999 | 参数校验错误                       |
| 5xxxx | 50000-59999 | 系统错误（限流、超时、IO）         |

## 6. 认证与鉴权

### 6.1 Token 传递

所有 `/api/` 前缀的业务接口必须通过 JWT Token 认证（除 `/api/auth/login` 和 `/api/auth/refresh`）：

```
Authorization: Bearer eyJhbGciOiJIUzUxMiJ9...
```

Token 含 `userId`、`tenantId`、`role` 等 Claims，后端通过 `JwtAuthFilter` 解析后注入 `UserContext` 和 `TenantContext`。

### 6.2 内部服务调用

Worker、DataSync 等内部服务调用走 `/internal/` 前缀，使用独立的内部 JWT Token：

- 签发方：调用方（如 `PdfRenderService`）
- 签名密钥：与用户 JWT 分离的 `jinshu.internal.jwt.secret`
- 有效期：≤ 5 分钟（短时效）
- Claims：仅含 `sub`（调用方服务名）和 `reportId`（资源 ID）

```http
GET /internal/reports/1001/render-data
Authorization: Bearer <internalJwt>
```

内部接口必须通过 K8s NetworkPolicy 限制仅允许 Worker/DataSync Pod 访问，不对外暴露。

### 6.3 公开接口

| 接口                | 认证要求 | 说明              |
| ------------------- | :------: | ----------------- |
| `/health`           |    无    | K8s 探针          |
| `/ready`            |    无    | K8s 探针          |
| `/live`             |    无    | K8s 探针          |
| `/api/auth/login`   |    无    | 登录              |
| `/api/auth/refresh` |    无    | Token 刷新        |
| `/internal/**`      | 内部 JWT | 仅 K8s 内部服务间 |
| `/api/**`（其他）   | 用户 JWT | 所有业务接口      |

## 7. 分页规范

**请求参数**：

| 参数   | 类型    | 默认值 | 范围              | 说明     |
| ------ | ------- | :----: | ----------------- | -------- |
| `page` | Integer |   1    | ≥ 1               | 页码     |
| `size` | Integer |   20   | 1-100             | 每页条数 |
| `sort` | String  |   —    | "field,asc\|desc" | 排序字段 |

**响应**：使用 `PageResult<T>`，见 5.2。

**后端实现**：

```java
// Controller
@GetMapping
public Result<PageResult<Report>> list(
    @RequestParam(defaultValue = "1") int page,
    @RequestParam(defaultValue = "20") int size,
    @RequestParam(required = false) String sort) {

    return Result.success(reportService.list(page, size, sort));
}

// Service
public PageResult<Report> list(int page, int size, String sort) {
    // 使用 PageHelper 或手写 LIMIT/OFFSET
    PageHelper.startPage(page, Math.min(size, 100));
    List<Report> list = reportMapper.selectAll();
    return PageResult.of(list);
}
```

## 8. 文件上传与下载

### 8.1 文件上传

```http
POST /api/import/upload
Content-Type: multipart/form-data

file: (binary)
```

**Controller 签名**：

```java
@PostMapping("/upload")
public Result<UploadResult> upload(
    @RequestParam("file") MultipartFile file) {

    // 1. 校验文件类型（Magic Number）
    // 2. 校验文件大小
    // 3. 保存到 PVC 共享存储
    // 4. 返回 fileKey + 元信息
    return Result.success(fileService.upload(file));
}
```

**上传约束**：

| 约束项         | 值                                                |
| -------------- | ------------------------------------------------- |
| 允许的文件类型 | `.xlsx`, `.xls`, `.csv`                           |
| 单文件最大     | 500 MB                                            |
| 校验方式       | Magic Number（文件头字节），不信任扩展名          |
| 超时时间       | 300 秒                                            |
| 返回内容       | `{ fileKey, originalName, fileSize, uploadedAt }` |

**大文件分片上传**（> 100 MB，二期实现）：

```http
POST /api/import/upload/chunk
Content-Type: multipart/form-data

file: (binary chunk)
chunkIndex: 0
totalChunks: 5
fileKey: "xxx"
```

### 8.2 文件下载

```http
GET /api/export/download/{taskId}
```

**Controller 签名**：

```java
@GetMapping("/download/{taskId}")
public ResponseEntity<Resource> download(@PathVariable Long taskId) {
    ExportFile file = exportService.getExportFile(taskId);
    return ResponseEntity.ok()
        .header(HttpHeaders.CONTENT_DISPOSITION,
            "attachment; filename=\"" + URLEncoder.encode(file.getName(), StandardCharsets.UTF_8) + "\"")
        .header(HttpHeaders.CONTENT_TYPE, file.getContentType())
        .body(new FileSystemResource(file.getPath()));
}
```

> 下载接口不走 `Result<T>` 包装，直接返回 `ResponseEntity<Resource>`。

## 9. 接口文档（OpenAPI）

### 9.1 Swagger 配置

```java
// config/OpenApiConfig.java
@Configuration
public class OpenApiConfig {

    @Bean
    public OpenAPI jinshuOpenAPI() {
        return new OpenAPI()
            .info(new Info()
                .title("锦书报表系统 API")
                .version("v1.0")
                .description("企业级报表系统 REST API 文档")
                .license(new License().name("Apache 2.0")))
            .addSecurityItem(new SecurityRequirement().addList("Bearer"))
            .components(new Components()
                .addSecuritySchemes("Bearer", new SecurityScheme()
                    .type(SecurityScheme.Type.HTTP)
                    .scheme("bearer")
                    .bearerFormat("JWT")));
    }
}
```

**访问地址**：`http://localhost:8080/swagger-ui.html`

### 9.2 注解规范

每个 Controller 方法必须包含：

```java
@RestController
@RequestMapping("/api/v1/reports")
@Tag(name = "v1 - Reports", description = "报表管理接口")
public class ReportController {

    @Operation(summary = "查询报表列表", description = "分页查询当前租户的报表列表，支持按状态、创建时间筛选")
    @ApiResponses({
        @ApiResponse(responseCode = "200", description = "查询成功"),
        @ApiResponse(responseCode = "401", description = "未登录"),
        @ApiResponse(responseCode = "403", description = "无权限")
    })
    @GetMapping
    public Result<PageResult<Report>> list(
        @Parameter(description = "页码", example = "1")
        @RequestParam(defaultValue = "1") int page,

        @Parameter(description = "每页条数", example = "20")
        @RequestParam(defaultValue = "20") int size,

        @Parameter(description = "报表状态", example = "DRAFT")
        @RequestParam(required = false) String status) {

        return Result.success(reportService.list(page, size, status));
    }

    @Operation(summary = "创建报表")
    @PostMapping
    public Result<Report> create(@Valid @RequestBody CreateReportRequest request) {
        return Result.success(reportService.create(request));
    }
}
```

**要求**：

- 每个接口必须有 `@Operation(summary = "...")`
- 修改/删除/状态变更接口必须有 `@ApiResponse(responseCode = "403")`
- 路径参数必须用 `@Parameter(description = "...")`
- 请求体 DTO 类使用 `@Schema(description = "...")` 标注字段

### 10.2 接口废弃流程

1. **标记废弃**：在 `@Operation` 中标注 `deprecated = true`
2. **通知期**：至少保留 2 个版本（~4 周），在 Changelog 和 Swagger 中公示
3. **监控调用量**：通过 Prometheus 指标观察旧接口调用量，确认无调用后下线
4. **下线**：删除 Controller 方法，更新 Swagger 文档

```java
@Operation(summary = "查询报表列表（已废弃）", deprecated = true)
@Deprecated
@GetMapping("/list")
public Result<List<Report>> listOld() {
    // 转发到新接口
    return Result.success(reportService.paginatedList(1, 100).getList());
}
```

### 10.3 破坏性变更处理

| 变更类型          | 是否破坏性 | 处理方式                     |
| ----------------- | :--------: | ---------------------------- |
| 新增可选字段      |     否     | 正常迭代                     |
| 新增必填字段      |     是     | 发布新版本 `/api/v2/reports` |
| 删除字段          |     是     | 发布新版本                   |
| 修改字段类型      |     是     | 发布新版本                   |
| 修改字段含义/语义 |     是     | 发布新版本                   |
| 修改 HTTP 方法    |     是     | 发布新版本                   |
| 新增接口          |     否     | 正常迭代                     |
| 删除接口          |     是     | 走废弃流程，至少保留 4 周    |

### 11.3 禁止事项

- ❌ Controller 方法中不得出现任何业务逻辑（仅做参数校验 + 调用 Service + 返回 Result）
- ❌ 不得在 Controller 中手动 `try-catch` 包装异常（由 `GlobalExceptionHandler` 统一处理）
- ❌ 不得在 Controller 中直接注入 DAO/Mapper
- ❌ 不得返回 `Result<String>` 拼接的 JSON 字符串
- ❌ 不得在 Controller 中操作 `ThreadLocal`（TenantContext/UserContext 由拦截器管理）

## 12. 安全考量

- 所有 `/api/` 接口强制 JWT 认证（除 login/refresh），由 `SecurityConfig` 统一配置
- 所有请求参数由 Spring Validation 校验，防止非法输入
- SQL 注入防护：MyBatis `#{}` 参数化查询（禁止 `${}`）
- XSS 防护：Controller 返回的 HTML 内容由前端渲染前 DOMPurify 清洗
- CSRF 防护：SPA + JWT Bearer Token 模式天然免疫 CSRF（不依赖 Cookie 存储 Token）
- Rate Limit：通过 `TenantRateLimiter` 限制单租户 QPS，超限返回 429
- 敏感字段（密码、密钥）不得出现在 Swagger 文档中（`@Schema(hidden = true)` 或 DTO 不包含）
- 文件上传必须校验 Magic Number（防止后缀伪造）

## 13. 跨模块调用规范

跨子模块（`api` → `data-sync`、`worker` → `api`）走 HTTP REST 或 MQ，不走 Java 方法调用。

**REST 调用方（如 Worker）**：

```java
@Service
public class ReportDataClient {

    private final RestTemplate restTemplate;

    public ReportRenderData getRenderData(Long reportId) {
        String url = "http://api-service:8080/internal/reports/" + reportId + "/render-data";
        HttpHeaders headers = new HttpHeaders();
        headers.setBearerAuth(generateInternalToken(reportId));
        HttpEntity<Void> request = new HttpEntity<>(headers);

        ResponseEntity<RenderDataResponse> response = restTemplate.exchange(
            url, HttpMethod.GET, request, RenderDataResponse.class);
        return response.getBody().getData();
    }
}
```

**约束**：

- 内部 API URL 使用 K8s Service 名称作为 Host（`http://api-service:8080`），不走 Ingress
- 所有内部调用必须携带内部 JWT Token
- 调用方必须处理超时（`RestTemplate.setReadTimeout(30_000)`）和熔断（Resilience4j CircuitBreaker）

## 14. 附录：接口检查清单

每个新接口上线前必须逐项确认：

```
□ URL 路径是否符合规范（前缀、命名、层级）
□ HTTP 方法是否正确映射 CRUD/动作
□ 请求参数是否有 @Valid 校验注解
□ 是否返回 Result<T> 或 PageResult<T>（下载/探针除外）
□ 是否有 @Operation(summary) 和 @ApiResponse
□ 是否有 JWT 认证（或已标记 @SkipAuth 并说明理由）
□ 租户隔离：Controller 是否依赖 TenantContext（由拦截器注入）
□ 审计日志：状态变更/删除操作是否标注 @AuditLog
□ 分页接口是否限制最大 pageSize（≤ 100）
□ 文件上传是否有 Magic Number 校验
□ 破坏性变更是否考虑了版本号或废弃流程
```
