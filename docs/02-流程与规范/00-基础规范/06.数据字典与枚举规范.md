## 1. 总则

### 1.1 目的

1. 统一管理和登记系统中的所有枚举值，避免硬编码魔法值
2. 定义数据字典的存储和访问方式，支持动态配置的业务字典
3. 确保 Java 枚举、数据库值、前端展示三端一致

### 1.2 适用范围

本规范适用于锦书企业级报表系统中所有涉及固定枚举值和动态数据字典的场景：

- 后端 Java 枚举类定义
- 数据库 VARCHAR 字段的可选值范围
- 前端下拉框/单选框的选项配置
- 动态业务字典（如行业分类、报表类型）的管理

## 2. 数据字典管理

### 2.1 设计原则

| 原则         | 说明                                                                                             |
| ------------ | ------------------------------------------------------------------------------------------------ |
| 固定 vs 动态 | 系统级枚举（角色、状态）→ 硬编码在 Java 枚举中；业务级字典（行业、地区）→ 存储在 `sys_dict` 表中 |
| 租户隔离     | `sys_dict` 支持租户级自定义，系统级字典 tenant_id = 0                                            |
| 不可删除     | 字典项被引用后不可物理删除，使用 `is_enabled` 标记停用                                           |
| 缓存优先     | 字典数据通过 Redis + Caffeine 缓存，避免频繁查库                                                 |

### 2.2 sys_dict 表设计

| 字段          | 类型         | 说明                             |
| ------------- | ------------ | -------------------------------- |
| `id`          | BIGSERIAL PK | 主键                             |
| `tenant_id`   | BIGINT       | 租户 ID（0 = 系统级字典）        |
| `dict_code`   | VARCHAR(100) | 字典编码，如 `REPORT_TYPE`       |
| `dict_name`   | VARCHAR(200) | 字典名称，如"报表类型"           |
| `description` | TEXT         | 描述                             |
| `is_system`   | BOOLEAN      | 是否系统级（系统级不可租户修改） |
| `created_at`  | TIMESTAMP    | 创建时间                         |
| `updated_at`  | TIMESTAMP    | 更新时间                         |

### 2.3 sys_dict_item 表设计

| 字段         | 类型         | 说明                   |
| ------------ | ------------ | ---------------------- |
| `id`         | BIGSERIAL PK | 主键                   |
| `dict_id`    | BIGINT       | 关联 sys_dict.id       |
| `item_value` | VARCHAR(100) | 字典项值（存储值）     |
| `item_label` | VARCHAR(200) | 字典项标签（展示文字） |
| `sort_order` | INTEGER      | 排序                   |
| `is_enabled` | BOOLEAN      | 是否启用               |
| `extra_attr` | JSONB        | 扩展属性               |
| `created_at` | TIMESTAMP    | 创建时间               |
| `updated_at` | TIMESTAMP    | 更新时间               |

### 2.4 字典缓存策略

| 层级       | 实现                         | TTL     | 说明                                |
| ---------- | ---------------------------- | ------- | ----------------------------------- |
| 本地缓存   | Caffeine                     | 10 分钟 | 减少 Redis 网络开销                 |
| 分布式缓存 | Redis                        | 1 小时  | Key: `dict:{tenant_id}:{dict_code}` |
| 失效机制   | 字典变更时发布 Redis Pub/Sub | 实时    | 通知所有实例刷新本地缓存            |

## 3. 系统枚举规范

### 3.1 Java 枚举命名规范

```java
// 包路径：com.jinshu.common.enums
// 枚举类名：业务名词 + Enum
// 枚举值：大写字母 + 下划线

public enum ReportStatusEnum {
    DRAFT("DRAFT", "草稿"),
    PENDING_REVIEW("PENDING_REVIEW", "待审核"),
    APPROVED("APPROVED", "已通过"),
    REJECTED("REJECTED", "已驳回"),
    PUBLISHED("PUBLISHED", "已发布"),
    ARCHIVED("ARCHIVED", "已归档");

    private final String code;
    private final String label;

    // 静态方法
    public static ReportStatusEnum fromCode(String code) { ... }
    public static boolean isValid(String code) { ... }
}
```

### 3.2 枚举值编码规范

| 规则           | 示例                                   |
| -------------- | -------------------------------------- |
| 全大写         | `ACTIVE`（非 `Active`）                |
| 下划线分隔     | `PENDING_REVIEW`（非 `pendingReview`） |
| 语义明确       | `TENANT_ADMIN`（非 `ROLE_2`）          |
| 不超过 50 字符 |                                        |
| 使用英文       | 值为英文字符串，中文仅用于 `label`     |

### 3.3 枚举与字典的映射

- **系统级枚举**：在 `common` 模块的 `enums/` 包中定义，数据库只存储 `code` 值
- **业务级字典**：通过 `sys_dict` + `sys_dict_item` 表管理，不创建 Java 枚举类
- 前端统一从 API 获取：`GET /api/dict/{dictCode}` 返回当前租户可用的字典项

## 4. 全局枚举清单

### 4.1 租户状态

| 代码        | 标签   | 说明               |
| ----------- | ------ | ------------------ |
| `ACTIVE`    | 正常   | 可正常使用         |
| `SUSPENDED` | 已停用 | 所有用户无法登录   |
| `ARCHIVED`  | 已归档 | 数据保留，不可操作 |

### 4.2 用户状态

| 代码       | 标签   | 说明                     |
| ---------- | ------ | ------------------------ |
| `ACTIVE`   | 正常   | 可正常登录使用           |
| `DISABLED` | 已禁用 | 管理员禁用，无法登录     |
| `LOCKED`   | 已锁定 | 登录失败超次数，自动锁定 |

### 4.3 用户角色

| 代码              | 标签       | 说明             |
| ----------------- | ---------- | ---------------- |
| `SUPER_ADMIN`     | 超级管理员 | 跨租户管理权限   |
| `TENANT_ADMIN`    | 租户管理员 | 本租户内最高权限 |
| `REPORT_DESIGNER` | 报表设计者 | 创建/编辑报表    |
| `REVIEWER`        | 审核者     | 审核报表         |
| `VIEWER`          | 查看者     | 仅查看           |

### 4.4 报表状态

| 代码             | 标签   | 说明       |
| ---------------- | ------ | ---------- |
| `DRAFT`          | 草稿   | 初始状态   |
| `PENDING_REVIEW` | 待审核 | 已提交审批 |
| `APPROVED`       | 已通过 | 审核通过   |
| `REJECTED`       | 已驳回 | 审核驳回   |
| `PUBLISHED`      | 已发布 | 可对外使用 |
| `ARCHIVED`       | 已归档 | 只读保留   |

| 代码        | 标签   | 说明         |

| `PENDING`   | 待处理 | 已创建未执行 |

**任务类型**：

| 代码         | 标签       |

**数据源状态**：

| 代码       | 标签   | 说明         |

## 5. 数据库存储约定

### 5.1 枚举字段存储格式

- 所有枚举字段使用 `VARCHAR` 类型存储 `code` 值（非中文 label）
- 字段名以状态类字段为例：`status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE'`
- 不使用 ENUM 类型（扩展性差，ALTER 成本高）

### 5.2 CHECK 约束

关键枚举字段建议添加 CHECK 约束：

```sql
-- 租户状态
ALTER TABLE tenant ADD CONSTRAINT chk_tenant_status
  CHECK (status IN ('ACTIVE', 'SUSPENDED', 'ARCHIVED'));

-- 用户状态
ALTER TABLE users ADD CONSTRAINT chk_user_status
  CHECK (status IN ('ACTIVE', 'DISABLED', 'LOCKED'));

-- 用户角色
ALTER TABLE users ADD CONSTRAINT chk_user_role
  CHECK (role IN ('SUPER_ADMIN', 'TENANT_ADMIN', 'REPORT_DESIGNER', 'REVIEWER', 'VIEWER'));

-- 报表状态
ALTER TABLE report_metadata ADD CONSTRAINT chk_report_status
  CHECK (status IN ('DRAFT', 'PENDING_REVIEW', 'APPROVED', 'REJECTED', 'PUBLISHED', 'ARCHIVED'));

-- 任务状态
ALTER TABLE task ADD CONSTRAINT chk_task_status
  CHECK (status IN ('PENDING', 'RUNNING', 'SUCCESS', 'FAILED', 'CANCELLED'));
```

### 5.3 前端展示映射

前端通过字典 API 获取 label 映射：

```typescript
// GET /api/dict/REPORT_STATUS → 返回
[
  { "value": "DRAFT", "label": "草稿" },
  { "value": "PENDING_REVIEW", "label": "待审核" },
  ...
]

// 前端展示
const statusLabel = dictMap['REPORT_STATUS'][record.status];
// 或使用 PrimeReact Tag 组件 + 颜色映射
<StatusTag status={record.status} dict="REPORT_STATUS" />
```

## 6. 变更管理

### 6.1 新增枚举值流程

```
1. 在对应 Java 枚举类中添加新值
2. 修改数据库 CHECK 约束（ALTER TABLE ... DROP CONSTRAINT + ADD CONSTRAINT）
3. 如果是业务字典，通过 API 或 SQL 插入 sys_dict_item
4. 更新前端字典缓存
5. 更新本文档第 4 节"全局枚举清单"
6. 发布缓存失效通知
```

### 6.2 废弃枚举值流程

```
1. 确定该枚举值无历史数据引用（或确认历史数据保留该值）
2. 在 Java 枚举中标记 @Deprecated
3. 数据库不删除 CHECK 约束中的旧值（保留历史数据兼容）
4. 字典表中设置 is_enabled = false
5. 前端下拉框过滤掉 is_enabled = false 的项
6. 更新本文档第 4 节
```

> **静态枚举不删除原则**：已被历史数据引用的枚举值不删除 CHECK 约束，确保数据兼容性。
