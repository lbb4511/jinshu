### 1.1 目的

1. 统一事务隔离级别和传播行为的使用规范

所有后端模块中涉及数据库操作的 Service 层代码。

## 2. 事务隔离级别

### 2.1 默认隔离级别

> **默认使用 READ COMMITTED**（PostgreSQL 默认级别）。

| 隔离级别           | 脏读 | 不可重复读 | 幻读 | PostgreSQL 实现                |
| ------------------ | ---- | ---------- | ---- | ------------------------------ |
| READ UNCOMMITTED   | 可能 | 可能       | 可能 | 等同 READ COMMITTED            |
| **READ COMMITTED** | 不可 | 可能       | 可能 | **默认**                       |
| REPEATABLE READ    | 不可 | 不可       | 可能 | 可能（但 PG 实现中幻读极罕见） |
| SERIALIZABLE       | 不可 | 不可       | 不可 | 可能降低并发性能               |

### 2.2 各场景推荐级别

| 场景                 | 隔离级别        | 理由                           |
| -------------------- | --------------- | ------------------------------ |
| 通用 CRUD            | READ COMMITTED  | 平衡并发性能与数据一致性       |
| 审批状态变更         | REPEATABLE READ | 防止审批重复提交（乐观锁冗余） |
| 库存/配额扣减        | REPEATABLE READ | 防止超卖                       |
| 审计日志写入         | READ COMMITTED  | 仅追加，不修改，无需更高隔离   |
| 哈希链/Merkle 树计算 | SERIALIZABLE    | 需要严格顺序一致性             |
| 跨表复杂统计         | READ COMMITTED  | 可通过时间戳控制数据范围       |

## 3. 事务管理规范

### 3.1 声明式事务

**Service 层统一使用 `@Transactional`**：

```java
@Service
public class ReportService {

    // ✅ 默认配置：READ COMMITTED + REQUIRED
    @Transactional
    public Report createReport(ReportCreateReq req) {
        Report report = reportMapper.insert(buildReport(req));
        workflowService.initWorkflow(report.getId());
        auditLogger.log(/* ... */);
        return report;
    }

    // ✅ 需要更高隔离级别时显式指定
    @Transactional(isolation = Isolation.REPEATABLE_READ)
    public void approveReport(Long reportId, Long reviewerId) {
        Report report = reportMapper.selectByIdForUpdate(reportId);
        if (report.getStatus() != ReportStatus.PENDING_REVIEW) {
            throw new BusinessException(ErrorCode.INVALID_STATUS_TRANSITION);
        }
        reportMapper.updateStatus(reportId, ReportStatus.APPROVED);
    }
}
```

### 3.2 编程式事务

仅在需要精细控制事务边界的场景使用：

```java
// 场景：仅部分操作失败需要回滚
transactionTemplate.execute(status -> {
    try {
        taskMapper.updateStatus(taskId, TaskStatus.RUNNING);
        processImport(task);
        taskMapper.updateStatus(taskId, TaskStatus.SUCCESS);
    } catch (Exception e) {
        status.setRollbackOnly();
        taskMapper.updateStatus(taskId, TaskStatus.FAILED);
    }
    return null;
});
```

### 3.3 事务边界

| 规则                | 说明                                          |
| ------------------- | --------------------------------------------- |
| Controller 禁止事务 | 事务必须在 Service 层                         |
| 事务内禁止 RPC      | 事务内不可调用远程服务（HTTP/MQ），防止长事务 |
| 事务内禁止异步      | 事务内不可 `@Async`，异步方法事务独立         |
| 事务内禁止文件 I/O  | 文件操作不属于事务范围，抽取到事务外          |
| 一个事务一次提交    | 避免在循环中多次提交小事务                    |

**错误示例**：

```java
// ❌ 事务内发送 MQ 消息——如果 MQ 发送成功但事务回滚，数据不一致
@Transactional
public void createAndNotify(Report report) {
    reportMapper.insert(report);
    rabbitTemplate.convertAndSend("q.report", report);  // 移出事务！
}

// ✅ 先在事务中创建，提交后再发 MQ
@Transactional
public Report create(ReportCreateReq req) {
    return reportMapper.insert(buildReport(req));
}

// 在调用方（Controller）或事件监听中发送 MQ
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handleReportCreated(ReportCreatedEvent event) {
    rabbitTemplate.convertAndSend("q.report", event.getReport());
}
```

### 3.4 事务超时

| 场景       | 超时时间 | 说明              |
| ---------- | -------- | ----------------- |
| 通用 CRUD  | 10 秒    | 默认值            |
| 批量写入   | 30 秒    | 1,000 行以内批次  |
| 大数据导入 | 不设全局 | 见第 5 节分片策略 |

```java
@Transactional(timeout = 30)  // 30 秒超时
public void batchInsert(List<ReportData> rows) { ... }
```

## 4. 并发控制策略

### 4.1 乐观锁

**适用场景**：读多写少、冲突概率低的场景。

**实现方式**：版本号字段

```sql
-- 表添加 version 字段
ALTER TABLE report_metadata ADD COLUMN version INTEGER NOT NULL DEFAULT 0;
```

```java
// MyBatis Mapper
@Update("UPDATE report_metadata SET name = #{name}, version = version + 1 " +
        "WHERE id = #{id} AND version = #{version}")
int updateWithVersion(Report report);

// Service
@Transactional
public void updateReport(ReportUpdateReq req) {
    Report current = reportMapper.selectById(req.getId());
    current.setName(req.getName());
    current.setVersion(req.getVersion());  // 传入旧版本号

    int affected = reportMapper.updateWithVersion(current);
    if (affected == 0) {
        throw new BusinessException(ErrorCode.CONCURRENT_UPDATE_CONFLICT,
            "数据已被他人修改，请刷新后重试");
    }
}
```

**使用乐观锁的场景**：

- 报表编辑（同一时间一般只有一人在编辑）
- 数据源配置修改
- 用户信息编辑

### 4.2 悲观锁

**适用场景**：写冲突频率高、或必须保证顺序的场景。

```java
// MyBatis 使用 SELECT ... FOR UPDATE
@Select("SELECT * FROM report_metadata WHERE id = #{id} FOR UPDATE")
Report selectByIdForUpdate(Long id);
```

**使用悲观锁的场景**：

- 审批状态变更（短时间内可能多人操作）
- 配额扣减
- 任务队列的抢占消费

### 4.3 分布式锁

**适用场景**：跨服务实例的并发控制。

**实现方式**：Redis `SETNX` + Lua 脚本

```java
public class RedisDistributedLock {
    private final StringRedisTemplate redisTemplate;

    public boolean tryLock(String key, long timeoutSeconds) {
        String lockKey = "lock:" + key;
        String lockValue = UUID.randomUUID().toString();
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, lockValue, timeoutSeconds, TimeUnit.SECONDS);
        if (Boolean.TRUE.equals(success)) {
            // 保存 lockValue 到 ThreadLocal，释放时校验
            LockContext.set(lockKey, lockValue);
            return true;
        }
        return false;
    }

    public void unlock() {
        LockContext.Entry entry = LockContext.get();
        if (entry != null) {
            // Lua 原子释放：校验 value 后删除
            String script = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                           "return redis.call('del', KEYS[1]) else return 0 end";
            redisTemplate.execute(new DefaultRedisScript<>(script, Long.class),
                List.of(entry.key), entry.value);
            LockContext.clear();
        }
    }
}
```

**使用分布式锁的场景**：

- 同一报表的 PDF 渲染（防止重复生成）
- Merkle 哈希树计算（每小时仅一个实例执行）
- 定时任务执行（防止多实例重复执行）

### 4.4 选型决策树

```
并发控制选型
    │
    ├── 操作跨实例？ ─── 是 ───► 分布式锁（Redis）
    │
    ├── 写冲突频率高？ ─── 是 ───► 悲观锁（FOR UPDATE）
    │
    ├── 读多写少？ ─── 是 ───► 乐观锁（version 字段）
    │
    └── 纯追加操作（如审计日志）──► 不加锁，依赖数据库唯一约束
```

## 5. 长事务处理

### 5.1 百万行导入事务策略

**核心原则：分批提交，每 1,000 行一个事务**。

```java
@Service
public class ImportService {

    @Transactional(propagation = Propagation.REQUIRES_NEW, timeout = 30)
    public void processBatch(List<ReportData> batch, Long taskId) {
        // 每批 1,000 行
        reportDataMapper.batchInsert(batch);
        taskMapper.updateProgress(taskId, batch.size());
    }

    // 调用方
    public void importAll(InputStream input, Long taskId) {
        List<ReportData> batch = new ArrayList<>(1000);
        // EasyExcel 流式读取
        excelReader.read(input, row -> {
            batch.add(buildReportData(row));
            if (batch.size() >= 1000) {
                importService.processBatch(new ArrayList<>(batch), taskId);
                batch.clear();
            }
        });
        // 最后一批
        if (!batch.isEmpty()) {
            importService.processBatch(batch, taskId);
        }
    }
}
```

**关键设计**：

- `Propagation.REQUIRES_NEW`：每批独立事务，一批失败不影响其他批
- 批次大小 1,000 行，平衡事务开销和回滚代价
- 失败批次记录到 `import_error_log`，用户可单独重试

### 5.2 大事务拆分

| 原操作                      | 拆分策略                                                |
| --------------------------- | ------------------------------------------------------- |
| 删除报表（含数据 + 工作流） | 先更新状态为 DELETING → 分表分批删除 → 最终标记 DELETED |
| 归档审计日志（百万行）      | 按天分批归档，每批 10,000 行                            |
| PDF 多页渲染                | 分段渲染，最后合并（各段独立，无事务依赖）              |

## 6. 死锁预防与处理

### 6.1 预防策略

| 策略                      | 说明                                              |
| ------------------------- | ------------------------------------------------- |
| 统一资源访问顺序          | 多表操作时按固定顺序（如：先 report 后 workflow） |
| 使用索引                  | 避免全表扫描导致的表锁升级                        |
| 减少事务大小              | 锁持有时间越短，死锁概率越低                      |
| 避免在事务中交互等待      | 不在事务内等待用户输入或外部服务响应              |
| 使用 NOWAIT / SKIP LOCKED | 获取锁失败立即返回，而非阻塞等待                  |

```sql
-- 获取任务时使用 SKIP LOCKED 跳过已被锁定的行
SELECT * FROM task
WHERE status = 'PENDING' AND tenant_id = #{tenantId}
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

### 6.2 检测与恢复

| 环节       | 策略                                                                 |
| ---------- | -------------------------------------------------------------------- |
| 检测       | PostgreSQL 自动检测死锁并选择一个事务回滚（`deadlock_timeout` = 1s） |
| 应用层恢复 | 捕获死锁异常 → 等待随机 100-500ms → 最多重试 3 次                    |
| 记录       | 死锁发生记录 WARN 日志，含参与的 SQL 和锁信息                        |

```java
@Retryable(
    retryFor = {CannotAcquireLockException.class, DeadlockLoserDataAccessException.class},
    maxAttempts = 3,
    backoff = @Backoff(delay = 100, maxDelay = 500, multiplier = 2)
)
@Transactional
public void approve(Long reportId) {
    // ...
}
```

## 8. 禁止事项

| 禁止                                          | 理由                           |
| --------------------------------------------- | ------------------------------ |
| ❌ Controller 层使用 `@Transactional`         | 违反分层架构，事务边界模糊     |
| ❌ 事务内调用 HTTP/RPC/MQ                     | 长事务 + 分布式事务不一致      |
| ❌ 事务内进行文件操作                         | 文件操作无法回滚               |
| ❌ 使用 `Propagation.REQUIRES_NEW` 过多嵌套   | 事务嵌套复杂，排查困难         |
| ❌ `try-catch` 吞掉异常后不 `setRollbackOnly` | 事务不会回滚                   |
| ❌ 数据库默认 `autocommit=true`               | 依赖 Spring 管理事务           |
| ❌ 在一行数据上长时间持有排他锁               | 阻塞其他操作，可能导致连锁阻塞 |
| ❌ 在事务中 `Thread.sleep()`                  | 人为延长锁持有时间             |
