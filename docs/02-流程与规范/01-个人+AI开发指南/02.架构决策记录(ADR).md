> 架构决策记录（Architecture Decision Record，简称ADR）是一种记录软件架构重要决策的方式。它记录了：什么决策、为什么做这个决策、以及考虑了哪些选项。
>
> **核心价值：** 让后来的人能够理解"当初为什么这么设计"，避免反复踩同样的坑。

---

## 1. 什么决策需要记录

### ✅ 需要记录的决策

1. **技术选型决策**
   - 为什么选MyBatis不选JPA？
   - 为什么用PostgreSQL不用MySQL？
   - 为什么选RabbitMQ不用Kafka？

2. **架构模式决策**
   - 为什么用单体而不是微服务？
   - 为什么用分层架构？
   - 为什么选择这种多租户实现方案？

3. **核心设计原则决策**
   - 为什么要求所有SQL必须走租户拦截器？
   - 为什么审计日志必须不可修改？
   - 为什么要求"设计先行，禁止先写代码后补设计"？

4. **流程规范决策**
   - 为什么一个PR不能超过500行？
   - 为什么必须写单元测试？
   - 为什么引入新依赖需要合规审核？

5. **重要的"不做什么"的决策**
   - 为什么现在不做微服务拆分？
   - 为什么不引入缓存层？
   - 为什么不支持XX功能？

### ❌ 不需要记录的决策

- 代码实现细节
- 变量命名、代码格式
- 普通的Bug修复方案
- 日常开发的小决策

## 2. ADR 模板

```markdown
# ADR-XXX: [决策标题]

| 项       | 内容                      |
| -------- | ------------------------- |
| 编号     | ADR-XXX                   |
| 创建日期 | YYYY-MM-DD                |
| 状态     | 提议/已通过/已废弃/已替代 |
| 决策人   | 谁拍板的                  |

---

## 1. 背景

为什么需要做这个决策？当时面临什么问题或挑战？

（描述清楚决策的上下文，让后人能理解当时的情况）

## 2. 考虑过的选项

### 选项一：[方案名称]

#### 优点

- ...
- ...

#### 缺点

- ...
- ...

### 选项二：[方案名称]

#### 优点

- ...
- ...

#### 缺点

- ...
- ...

### 选项三：[方案名称]（如果有）

...

## 3. 最终选择

我们选择了：[方案名称]

## 4. 选择理由

主要考虑了以下因素：

1. ...
2. ...
3. ...

（尽量具体，不要只写"这个方案更好"，要写出决策时的权衡和考量）

## 5. 已知的缺点和风险

选择这个方案，我们知道有以下缺点和风险：

1. ...
2. ...

我们计划这样应对这些风险：

- ...

## 6. 后续评估

我们将在以下情况下重新评估这个决策：

- 时间：[比如：6个月后]
- 条件：[比如：当租户数量超过100个时]

## 7. 参考资料

- [链接到相关文档、讨论、文章]
- ...
```

## 3. ADR 编号与命名规则

- **编号规则**：`ADR-001`, `ADR-002`, `ADR-003`... 按创建顺序递增
- **命名规则**：文件名 = `ADR-XXX-简短描述.md`
  - 示例：`ADR-001-选择MyBatis而非JPA.md`
  - 示例：`ADR-002-采用行级多租户方案.md`
- **存放位置**：`docs/99-附录/adr/` 目录下

---

## 4. ADR 生命周期

### 状态说明

| 状态     | 说明                           |
| -------- | ------------------------------ |
| `提议`   | 有人提出了这个决策，正在讨论中 |
| `已通过` | 决策已经通过评审，开始执行     |
| `已废弃` | 这个决策还没执行就被废弃了     |
| `已替代` | 这个决策已经被新的决策替代了   |

### 状态变更

- ADR一旦创建，**永远不要删除**
- 如果决策变更了，新建一个ADR，并在旧的ADR里标注"已替代，参见ADR-XXX"
- 如果发现当初的决策错了，不要修改历史记录，新建一个ADR说明"我们改变主意了，原因是..."

## 5. 如何创建新的 ADR

### 流程

```
发现需要记录的重要决策
    ↓
复制模板，填写内容
    ↓
提交PR
    ↓
核心维护者评审
    ↓
合并，ADR正式生效
```

### 谁可以提议创建ADR

- 任何人都可以提议创建新的ADR
- 核心维护者负责最终评审和合并

### 好的ADR的标准

1. ✅ 清楚描述了当时的背景和约束条件
2. ✅ 列出了所有认真考虑过的选项
3. ✅ 诚实地说明了每个选项的优缺点
4. ✅ 清楚解释了为什么选择这个，为什么不选那个
5. ✅ 承认了所选方案的缺点和风险
6. ✅ 不是"事后诸葛亮"，不美化历史

**最重要的：真实。** 即使后来证明决策错了，也要如实记录当时为什么那么选。错误的决策也是宝贵的经验。

## 7. 常见问题

### Q: 写ADR会不会太费时间？

**A:** 写的时候确实花1-2小时，但长远看节省的时间远不止这点。

试想：半年后新人问"当初为什么这么设计？"，你得花半天时间回忆、解释，还不一定记得全。如果有ADR，新人花10分钟就能理解。

而且写的过程本身就是梳理思路的过程，很多时候写着写着就发现考虑不周的地方。

### Q: 如果后来证明决策错了怎么办？

**A:** 很正常。如实记录就好。

错误的决策也是宝贵的经验。ADR不是为了证明"我们当初多么英明"，而是为了让后人理解"当初我们是怎么想的"。

如果证明错了，新建一个ADR说明"我们改变主意了，原因是XXX"，旧的ADR保留，标注"已替代"。

### Q: 谁负责写ADR？

**A:** 谁提议的决策谁写，或者谁主导讨论谁写。核心维护者也可以指定人写。

```markdown

| 项       | 内容         |
| -------- | ------------ |
| 编号     | ADR-000      |
| 创建日期 | 2026-05-12   |
| 状态     | 已通过       |
| 决策人   | 核心维护团队 |

## 1. 背景

项目初期，很多重要的架构决策都在口头讨论中做出了，没有留下书面记录。

随着项目发展，新人加入时经常会问：

- 当初为什么这么设计？
- 为什么不那样做？
- 这个方案有什么考虑？

老成员需要反复解释，而且时间长了，很多当时的考量也记不清了。大家反复讨论同样的问题，浪费时间。

我们需要一种方式，把重要的架构决策记录下来。

## 2. 考虑过的选项

### 选项一：使用ADR格式记录

ADR（Architecture Decision Record）是业界广泛使用的架构决策记录格式，简单、轻量、容易上手。

#### 优点

- 格式标准化，大家容易理解
- 轻量，写一个只需要1-2小时
- 跟代码一起存在Git里，版本化管理
- 很多开源项目都在用，有成熟的最佳实践

#### 缺点

- 需要养成写ADR的习惯
- 多了一点流程 overhead

### 选项二：在设计文档里顺便记录

每个模块的设计文档里，顺便记录一下决策背景和考虑过的选项。

#### 优点

- 不需要额外的文档格式
- 跟设计文档在一起，容易找到

#### 缺点

- 分散在各个文档里，没有全局视图
- 很多跨模块的决策，不属于任何一个模块的设计文档
- 没有统一的格式，质量参差不齐

### 选项三：不专门记录，靠口口相传

#### 优点

- 零成本，什么都不用做

#### 缺点

- 老人走了，知识就丢了
- 新人反复问同样的问题
- 容易反复讨论同样的问题，浪费时间
- 时间长了，大家就忘了当初为什么这么设计，然后有人会说"我们重新设计吧"，结果重新踩一遍同样的坑

## 3. 最终选择

我们选择了：选项一 - 使用ADR格式记录架构决策。

## 4. 选择理由

1. ADR是业界事实标准，简单成熟，学习成本低
2. 轻量，不会带来太重的流程负担
3. 跟代码一起存在Git里，版本化管理，谁改了什么一目了然
4. 可以积累成项目的知识库，对新人和老人都有价值

## 5. 已知的缺点和风险

1. 可能有人觉得写ADR麻烦，不愿意写
   - 应对：核心维护者带头写，做出榜样
   - 应对：Code Review时，如果发现PR涉及重要决策但没有ADR，要求先补ADR

2. 可能写出来的ADR质量不高，太简单，没有价值
   - 应对：提供模板和示例
   - 应对：ADR提交后，核心维护者认真Review，要求补充必要的信息

## 6. 后续评估

我们将在6个月后（2026-11-12）重新评估这个决策：

- ADR有没有真的帮到大家？
- 新人是不是更容易上手了？
- 有没有减少重复讨论？

如果效果好，继续坚持；如果效果不好，再考虑调整。
```

> 📌 最后提醒：

> **ADR不是为了记录"正确的决策"，而是为了记录"真实的决策过程"。**
>
> 即使后来证明决策错了，只要当时诚实地记录了考虑过程，它就是一份有价值的ADR。
