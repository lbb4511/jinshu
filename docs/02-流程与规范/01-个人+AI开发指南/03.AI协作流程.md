> 本文档定义：人类开发者和AI助手如何高效协作。
>
> **核心原则：** AI是助手，不是替代品；人类做决策，AI做执行；所有AI生成内容必须经过人类审核。
>
> 📌 **个人开发场景特别说明：**
>
> - 你既是开发者，也是审核者，还是架构师
> - 可以省去所有流程性的东西，只保留最核心的：设计 → 生成 → 审核 → 测试
> - 重点关注：多租户、安全、异常处理这三块，AI最容易在这些地方出问题

---

## 1. 人与AI的职责边界

### 人类负责

✅ **需求理解与决策**：这个功能要不要做？做成什么样？
✅ **架构设计**：整体怎么拆分？模块怎么划分？
✅ **方案评审**：AI给出的方案好不好？有没有问题？
✅ **最终把关**：代码能不能合并？能不能上线？
✅ **异常处理**：遇到AI搞不定的复杂情况，兜底处理
✅ **AI训练与反馈**：告诉AI哪里做得好，哪里需要改进

### AI负责

✅ **信息整理**：把人类的想法整理成结构化的文档
✅ **代码生成**：根据设计文档生成对应的代码
✅ **测试辅助**：帮助写测试用例、生成测试数据
✅ **Code Review 辅助**：帮助检查代码规范、常见问题
✅ **文档撰写**：帮助写API文档、用户手册
✅ **重复性工作**：大量相似的、有规律的机械工作

### 绝对不可以让AI独自做

❌ **直接合并代码**：AI生成的代码必须经过人类审核才能合并
❌ **直接上线**：AI生成的内容不能直接上线
❌ **架构决策**：AI可以提建议，但最终架构决策必须由人拍板
❌ **安全相关**：涉及权限、加密、多租户隔离的核心逻辑，人类必须逐行检查
❌ **回答用户问题**：AI不能以项目官方身份回答用户问题，所有官方回复必须经过人类审核

## 2. 标准AI开发流程

### 2.1 需求分析阶段

```
人类提出需求（一句话描述）
    ↓
人类补充：背景、目标、不做什么
    ↓
AI帮助整理成结构化的需求文档
    ↓
人类审核、修改需求文档
    ↓
需求确认
```

**AI在这个阶段的输出：**

- 帮助把模糊的想法整理成清晰的需求列表
- 帮助识别可能遗漏的边界场景
- 帮助提出潜在的风险和问题

**人类必须做的：**

- 确认需求的优先级
- 明确"做什么"和"不做什么"的边界
- 拍板所有的取舍和权衡

---

### 2.2 设计阶段

```
人类：给出设计的核心思路（画个草图也行）
    ↓
AI：按照设计文档模板，生成完整的设计文档初稿
    ↓
人类：审核、修改、补充设计文档
    ↓
（如果需要）AI：生成几种可选方案，列出各自优缺点
    ↓
人类：决策最终采用哪个方案
    ↓
设计评审通过
```

**AI在这个阶段的输出：**

- 填充设计文档模板
- 生成API接口定义
- 生成数据库表结构定义
- 列出可能的技术选型和优缺点

**人类必须做的：**

- 确保设计文档的正确性、完整性
- 拍板方案选型
- 检查有没有遗漏安全、性能、多租户等关键考量

---

### 2.3 编码阶段

```
人类：确认设计文档已经评审通过
    ↓
AI：按照设计文档，生成代码
    ↓
人类：审核AI生成的代码（必须！）
    ↓
    ├─ 代码没问题 → 进入测试阶段
    └─ 代码有问题 → 告诉AI哪里有问题，AI修改
    ↓
人类：做最终的代码Review
```

**AI代码审核的必查项：**

| 检查项       | 谁负责             | 要求                               |
| ------------ | ------------------ | ---------------------------------- |
| 多租户隔离   | 人类必须逐行检查   | 所有SQL都有tenant_id过滤           |
| 安全漏洞     | 人类必须逐行检查   | 有没有SQL注入、权限绕过            |
| 审计日志     | 人类检查           | 敏感操作有没有记录审计日志         |
| 异常处理     | AI先检查，人类确认 | 有没有吞异常？异常有没有记录日志？ |
| 代码规范     | AI检查             | 命名、注释、代码风格               |
| 是否符合设计 | 人类检查           | 代码逻辑跟设计文档一致吗？         |

**重要提示：** AI生成的代码，**人类必须至少读一遍**，绝对不能看都不看直接合并。

---

### 2.4 测试阶段

```
人类：定义测试范围和重点
    ↓
AI：生成单元测试代码
    ↓
AI：生成测试数据
    ↓
人类：审核测试用例的完整性和正确性
    ↓
运行测试，根据失败情况迭代
```

**AI在这个阶段的输出：**

- 单元测试代码
- 边界场景测试用例
- 测试数据生成脚本
- 性能测试脚本

---

### 2.5 文档阶段

```
人类：给出文档要点
    ↓
AI：生成文档初稿
    ↓
人类：审核、修改、补充
    ↓
文档发布
```

## 3. AI Code Review 流程

### 3.1 AI先做第一遍检查

```
提交PR
    ↓
AI自动扫描，检查：
  - 代码规范问题
  - 常见Bug模式
  - 有没有SQL注入风险
  - 有没有遗漏tenant_id过滤
  - 有没有吞异常
    ↓
AI在PR里评论，标出问题
    ↓
人类：判断AI说的对不对，决定要不要改
```

### 3.2 AI Review 的可信度分级

| 类型               | 可信度 | 人类需要做什么                       |
| ------------------ | ------ | ------------------------------------ |
| 代码格式、命名风格 | 95%    | 大概率是对的，扫一眼就行             |
| 明显的语法错误     | 90%    | 大概率是对的，确认一下               |
| 异常处理问题       | 80%    | 需要仔细确认                         |
| SQL注入风险        | 70%    | 必须仔细检查，AI可能误报也可能漏报   |
| 多租户隔离问题     | 60%    | 人类必须逐行检查，不能完全相信AI     |
| 业务逻辑正确性     | <50%   | AI很可能理解错需求，人类必须仔细审核 |
| 架构设计合理性     | <30%   | 几乎不能信，人类必须自己判断         |

**重要原则：** 越接近业务逻辑、越接近核心架构，AI的可信度越低。

## 4. AI失败的识别与处理

### 4.1 什么情况下AI大概率会出错

出现以下情况时，提高警惕，不要相信AI：

1. 🚨 **需求非常模糊**：你自己都没想清楚，AI大概率会瞎编
2. 🚨 **涉及复杂的业务逻辑**：尤其是有很多特殊规则、历史包袱的业务
3. 🚨 **涉及并发、分布式事务**：AI很容易遗漏边界场景
4. 🚨 **涉及安全、多租户隔离**：错了就是严重事故
5. 🚨 **需要理解整个项目上下文**：AI容易"只见树木不见森林"
6. 🚨 **AI连续3次修改都没改对**：说明AI根本没理解问题，不要再让它改了，人类自己上

### 4.2 发现AI在瞎编怎么办

1. **立刻停止**：不要继续跟AI绕圈子，浪费时间
2. **人类接手**：这种情况通常意味着问题太复杂，或者AI理解错了
3. **拆解问题**：把大问题拆成更小的、AI能理解的小问题
4. **提供更多上下文**：给AI更多的背景信息、代码示例
5. **记录下来**：把这次失败记录到"AI失败案例库"，帮助以后避免同样的问题

## 5. Prompt 最佳实践

### 5.1 好的Prompt vs 坏的Prompt

❌ **坏Prompt：** "帮我写个报表导入功能"

✅ **好Prompt：**

```
任务：根据设计文档，实现报表导入功能的Service层

上下文：
1. 设计文档见：docs/04-模块设计/01.报表导入模块设计.md
2. 已有的实体类：Report.java
3. 已有的Mapper：ReportMapper.java
4. 需要使用EasyExcel进行导入

要求：
1. 所有SQL必须通过租户拦截器，不能硬编码tenant_id
2. 所有异常必须记录日志，不能吞掉
3. 导入超过1万行时自动转为异步处理
4. 所有敏感操作必须记录审计日志

请生成对应的Service代码。
```

### 5.2 Prompt 黄金法则

1. **明确范围**：告诉AI具体做什么，不要做什么
2. **提供上下文**：给AI看相关的代码、文档、已有实现
3. **列出约束条件**：多租户、安全、性能、规范要求
4. **要求AI解释**：让AI先说出它的理解，确认它理解对了再开始写代码
5. **分步来**：不要一次让AI写整个模块，拆成Controller、Service、DAO，一步步来

## 6. AI 开发的自查清单

每次让AI生成代码前，确认以下几点：

- [ ] 我已经清楚地理解了需求
- [ ] 设计文档已经存在并通过评审
- [ ] 我给AI的Prompt清晰、明确、有足够的上下文
- [ ] 我知道AI生成的代码哪里需要重点检查
- [ ] 我有时间和能力审核AI生成的所有代码
- [ ] 如果AI搞砸了，我有能力兜底修复

## 7. AI 知识管理

### 7.1 AI成功案例库

每次AI出色地完成了任务，记录下来：

- 任务是什么
- Prompt是怎么写的
- AI的输出好在哪里
- 可以复用的经验

### 7.2 AI失败案例库

每次AI失败了，更要记录下来：

- 任务是什么
- AI是怎么失败的
- 为什么失败了（理解错了？知识不足？幻觉？）
- 以后怎么避免类似的失败
- 这种任务以后应该怎么给AI提需求

### 7.3 Prompt 模板库

把经过验证的好Prompt整理成模板，大家可以复用：

- 生成Controller的Prompt模板
- 生成Service的Prompt模板
- 生成单元测试的Prompt模板
- Code Review的Prompt模板

## 8. 人类与AI协作的心态

### 8.1 不要高估AI

❌ 错误心态："AI什么都能做，我等着就行"
✅ 正确心态："AI是个能干但粗心的实习生，它能帮你干很多活，但你必须盯着它，检查它的工作"

### 8.2 不要低估AI

❌ 错误心态："AI写的代码都是垃圾，我自己写更快"
✅ 正确心态："AI写的代码可能有问题，但它能帮我节省80%的机械劳动，我只需要花20%的时间审核和修改"

### 8.3 把AI当作"思考伙伴"

AI最好的用法不是"帮你写代码"，而是"跟你一起思考"：

- 你有个想法，让AI帮你梳理清楚
- 你有几个方案，让AI帮你列出各自的优缺点
- 你写了代码，让AI帮你看看有没有明显的问题
- 让AI当你的"杠精"，帮你发现你没考虑到的地方

## 9. 常见问题

### Q: AI生成的代码有Bug，谁负责？

**A:** 人类负责。最终是人类审核后合并的代码，责任在人类，不在AI。

AI只是工具，就像你用编译器编译出来的程序有Bug，你不会怪编译器，你会怪自己没写对。

### Q: 应该让AI一次做多大的任务？

**A:** 建议一次不超过300行代码的任务。任务越小，AI的正确率越高，审核成本越低。

如果是大功能，拆成多个小任务，逐个完成、逐个审核。

### Q: 发现AI总是在同一个地方犯错怎么办？

**A:** 两种方式：

1. 把这个错误加到Prompt里，明确告诉AI"不要犯这个错误"
2. 如果AI总是在某类任务上犯错，以后这类任务就不要让AI做了，人类自己来

### Q: AI生成的代码跟项目风格不一致怎么办？

**A:** 在Prompt里明确告诉AI要参考的代码示例，让AI照着那个风格写。或者，AI写完后，人类做一次格式调整。

> 📌 最后一句话：
>
> **AI是放大器。好的开发者用AI，效率提升10倍；差的开发者用AI，灾难也放大10倍。**
>
> 用AI的正确姿势不是"让AI替你干活"，而是"让AI帮你干那些你不想干的机械劳动，把你的时间留给思考和决策"。
