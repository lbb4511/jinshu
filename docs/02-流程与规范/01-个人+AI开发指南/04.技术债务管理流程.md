> 技术债务就像信用卡欠债：适当借一点可以加速发展，但欠太多利息会把你压垮。
>
> **核心原则：** 主动管理，定期偿还，不要累积到失控。

## 1. 什么是技术债务

### ✅ 算技术债务

1. **代码质量问题**
   - 命名混乱，看不懂
   - 函数太长，嵌套太深
   - 重复代码，同样的逻辑抄了N遍
   - 没有注释，复杂逻辑没人看得懂

2. **测试缺失**
   - 核心模块没有单元测试
   - 测试覆盖率低
   - 重要的边界场景没有测试

3. **架构债务**
   - 模块边界不清晰，耦合严重
   - 依赖混乱，循环依赖
   - 该抽象的地方没抽象，不该抽象的地方过度抽象

4. **技术栈债务**
   - 使用的库版本太老，有安全漏洞
   - 用了已经不再维护的库
   - 同样的功能用了好几个不同的库，没有统一

5. **文档债务**
   - 代码改了，文档没更新
   - 没有API文档
   - 设计文档缺失，新人看不懂架构

### ❌ 不算技术债务

- "我觉得写得不好看" - 纯粹的风格问题
- "应该用最新的技术重写一遍" - 为了用新技术而用新技术
- "如果是我写我就不会这么写" - 个人偏好，没有实际问题

## 2. 技术债务分级

| 级别               | 定义                           | 影响                         | 偿还时限    |
| ------------------ | ------------------------------ | ---------------------------- | ----------- |
| **A 类：危险债务** | 随时可能出问题，出了就是大事故 | 影响稳定性、安全性、性能     | 立即偿还    |
| **B 类：严重债务** | 已经影响开发效率，改代码很痛苦 | 开发速度下降50%以上          | 1个月内偿还 |
| **C 类：一般债务** | 有影响，但还能忍               | 偶尔踩坑，开发速度下降10-20% | 3个月内偿还 |
| **D 类：轻微债务** | 影响很小，几乎感觉不到         | 几乎不影响开发               | 有空再还    |

### A类债务示例

- 核心模块完全没有测试，一改就崩
- 用了有严重安全漏洞的依赖版本
- 代码逻辑混乱，没人看得懂，一改就出Bug
- 严重的性能问题，随时可能挂

### B类债务示例

- 一个函数写了500行，没人敢改
- 同样的逻辑在5个地方重复，改一次要改5个地方
- 依赖版本落后3个大版本以上
- 没有设计文档，新人要花2周才能看懂

## 3. 技术债务发现流程

### 3.1 定期扫描（每月一次）

每个月第一个周五，花1小时做技术债务盘点：

1. **代码层面**
   - 看看哪些地方大家都"不敢碰"
   - 看看哪些地方改Bug的时候经常改出其他Bug
   - 看看重复代码最多的地方

2. **依赖层面**
   - 跑一下依赖安全扫描，看看有没有有漏洞的依赖
   - 看看哪些依赖版本太老了该升级了

3. **流程层面**
   - 大家抱怨最多的地方是什么？
   - 哪些地方开发效率最低？

4. **测试层面**
   - 哪些模块测试覆盖率最低？
   - 哪些地方出Bug最多？

### 3.2 随时登记

任何时候发现技术债务，都可以：

1. 提一个Issue，打上 `tech-debt` 标签
2. 在代码里加 `// TODO(tech-debt): xxx` 注释
3. 在代码Review的时候提出来

## 4. 技术债务登记与跟踪

### 4.1 登记模板

每个技术债务都要登记以下信息：

| 项             | 内容                                       |
| -------------- | ------------------------------------------ |
| 债务描述       | 一句话描述是什么问题                       |
| 级别           | A/B/C/D类                                  |
| 位置           | 哪个模块/哪个文件                          |
| 影响           | 现在造成了什么影响？继续累积会有什么后果？ |
| 偿还成本       | 修复需要多少人天？                         |
| 推荐的偿还方案 | 应该怎么修复？                             |
| 登记人         | 谁发现的                                   |
| 登记日期       | YYYY-MM-DD                                 |
| 计划偿还日期   | 计划什么时候修                             |
| 负责人         | 谁来修                                     |

### 4.2 跟踪看板

用一个专门的看板跟踪所有技术债务：

- `待评估` → 刚提上来，还没分级
- `待处理` → 已经分级，排好了优先级
- `处理中` → 有人正在修
- `已完成` → 已经修完了

## 5. 技术债务偿还流程

### 5.1 偿还节奏

**每个版本至少偿还20%的时间。**

比如一个迭代是2周，那么至少要有2天专门用来还技术债务。

**比例原则：**

- 新功能开发：60%
- Bug修复：20%
- 技术债务偿还：20%

### 5.2 偿还优先级排序

按"影响 ÷ 成本"排序：

|            | 修复成本低            | 修复成本高                |
| ---------- | --------------------- | ------------------------- |
| **影响大** | ✅ 最高优先级，立即修 | ⚠️ 制定计划，分阶段修     |
| **影响小** | ✅ 有空就修           | ❌ 先不修，除非有其他理由 |

### 5.3 什么情况下必须先还债再开发

出现以下情况时，停下新功能开发，先还债：

1. 🚨 出现A类危险债务
2. 🚨 同一个地方连续出3个Bug
3. 🚨 开发速度下降到无法接受的程度
4. 🚨 新人加入后，因为代码太烂，1个月都没法上手

### 5.4 偿还验收标准

技术债务还完了，怎么验证真的还完了？

1. **代码质量债务**：有人Code Review通过，确认问题解决了
2. **测试债务**：测试覆盖率达到要求，测试用例完整
3. **架构债务**：架构评审通过，模块边界清晰了
4. **文档债务**：有人审核过文档，确认文档是对的、全的

## 6. 防止技术债务累积

### 6.1 代码Review阶段

Code Review的时候必须问：

- 这个改动会不会引入新的技术债务？
- 有没有更简洁、更清晰的写法？
- 有没有加必要的测试？
- 有没有加必要的注释？

**如果PR引入了明显的技术债务，打回去重写。**

### 6.2 设计评审阶段

设计评审的时候必须问：

- 这个设计3个月后会不会成为技术债务？
- 能不能适应未来半年的需求变化？
- 有没有过度设计？有没有欠设计？

### 6.3 定期复盘

每个季度末，花2小时做技术债务复盘：

1. 这个季度我们还了多少技术债务？
2. 这个季度我们新欠了多少技术债务？
3. 哪类债务增长最快？为什么？
4. 下个季度的偿还重点是什么？

## 7. 关于技术债务的常见误区

### ❌ 误区1：技术债务都是不好的

**正确理解：** 适当的技术债务是必要的。

为了赶上线时间，先写个简单版本快速验证，验证成功了再重构，这是合理的技术债务。

问题不在于有技术债务，而在于借了不还，利息越滚越高。

---

### ❌ 误区2："以后再重构" = 永远不会重构

**正确理解：** 如果你不在提交代码的时候写好注释和测试，"以后再补"基本上等于"永远不会补"。

能现在还的债，就不要拖到以后。

---

### ❌ 误区3：重写 = 解决技术债务

很多人一看到代码烂，第一反应就是"推倒重写"。

但历史告诉我们：

- 重写经常会引入新的Bug
- 重写会花掉比预期多好几倍的时间
- 重写过程中还会欠新的技术债务
- 很多时候重写完了，发现还不如原来的

**正确的做法：** 小步重构，一点点还。不要想着一口吃成个胖子。

---

### ❌ 误区4：技术债务是技术问题，跟产品没关系

**正确理解：** 技术债务直接影响开发速度和产品质量，最终会影响业务。

- 技术债务多 → 开发慢 → 新功能上得慢 → 业务发展慢
- 技术债务多 → Bug多 → 用户体验差 → 用户流失

技术债务不只是技术团队的事，是整个团队的事。

## 8. 技术债务的"利息"计算

怎么跟产品经理/老板解释技术债务的严重性？给你一个简单的计算方式：

假设：

- 团队有5个开发者
- 因为技术债务，每个人每天有2小时在"踩坑"、"绕路"、"改因为代码烂导致的Bug"
- 一年工作250天

那么：

```
每年浪费的时间 = 5人 × 2小时/天 × 250天 = 2500人时
= 312.5人天
= 1.25个人全年什么都不干，专门填坑
```

**技术债务不是"代码写得不好看"，是真金白银的浪费。**

>
> 技术债务不可怕，可怕的是借了不还，利滚利，最后压垮整个项目。
