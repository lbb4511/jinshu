## 1. 背景与目标

### 1.1 问题背景

系统多份设计文档已各自规划 Redis 缓存用途，但存在以下问题：

| 问题                       | 影响                                                                        |
| -------------------------- | --------------------------------------------------------------------------- |
| Redis 依赖未统一引入       | `api` 模块缺少 `spring-boot-starter-data-redis`，但 JWT 设计依赖 Redis      |
| Key 命名无规范             | 各模块自行起 Key（`token_blacklist:*` vs `jinshu:mq:consumed:*`），可能冲突 |
| 无缓存抽象层               | 设计中既有 Redis 又有 Caffeine，但无线程安全的统一管理                      |
| 无防穿透/防击穿/防雪崩措施 | 高并发下可能对数据库造成瞬时压力                                            |
| 进度更新策略分散           | 导入/导出/PDF 三模块各自设计进度 Key，无统一模式                            |
| Redis 容器无持久化配置     | Pod 重启后所有缓存数据丢失，JWT 黑名单/进度全部清空                         |
| 进度 Key 无自动清理        | 已完成 Task 的进度 Key 永久残留，浪费内存                                   |

### 1.2 设计目标

| 指标             | 要求                                   |
| ---------------- | -------------------------------------- |
| Key 命名规范     | 统一前缀 `jinshu:`，按域分级           |
| 缓存架构         | Redis（分布式） + Caffeine（本地）双层 |
| 防穿透           | 缓存空值（TTL 5 分钟）                 |
| 防雪崩           | TTL 加随机偏移（±10%）                 |
| 热点数据本地缓存 | Caffeine TTL 30 秒，maxSize 1000       |
| Redis 内存上限   | 生产环境 256MB，开发环境 64MB          |
| Redis 连接池     | Lettuce 连接池，maxIdle=8, maxTotal=16 |
| 进度 Key TTL     | 任务完成后 1 小时自动清除              |

### 1.3 非目标

- 不实现 Redis Cluster / Sentinel（单实例可满足 MVP）
- 不实现 Redisson 分布式锁（无强一致性需求）
- 不使用 `@Cacheable` 拦截 MyBatis 查询（避免与 TenantInterceptor 的 SQL 改写冲突）
- 不实现 Redis Streams（使用 RabbitMQ 替代）
- 不在此文档设计 Redis 存储的持久化方案（运维文档负责）

## 2. 需求分析

### 2.1 功能需求

- [x] **连接池配置**：Lettuce 连接池统一配置
- [x] **Key 命名规范**：全系统统一的 Key 前缀与命名约定
- [x] **JWT Token 缓存**：黑名单、版本号、登录锁定计数、Refresh Token 存储
- [x] **任务进度缓存**：导入/导出/PDF 渲染的实时进度
- [x] **调度配额缓存**：租户并发计数、配额上限、Round-Robin 顺序
- [x] **消息去重缓存**：MQ 消息消费幂等记录
- [x] **业务数据缓存**：租户配置、用户权限、报表元数据
- [x] **本地缓存**：Caffeine 缓存热点数据（租户配额、用户版本号）
- [x] **防穿透/击穿/雪崩**：空值缓存 + TTL 随机化
- [x] **Key 生命周期管理**：明确的 TTL 策略和自动清理

### 2.2 用户故事

```
作为 后端开发者，我希望 有一个统一的缓存工具类来操作 Redis，
以便 各模块不需要各自配置 RedisTemplate 和想 Key 名。

作为 运维人员，我希望 Redis 中的过期 Key 能自动清理，
以便 内存不会被历史进度数据占满。

作为 前端开发者，我希望 频繁查询的配置数据（如租户配置）有缓存加速，
以便 页面加载更快。
```

### 2.3 边界场景

| 场景                     | 处理方式                                                                 |
| ------------------------ | ------------------------------------------------------------------------ |
| Redis 连接失败           | 所有读写操作 fallback 返回 null / 0 / false，业务继续（降级）            |
| Redis 内存满（OOM）      | `maxmemory-policy = allkeys-lru`，淘汰最少使用的 Key                     |
| Key 写入冲突（并发 SET） | 使用原子操作（`SETNX`、`INCR`），不使用分布式锁                          |
| 进度 Key 泄漏            | 所有进度 Key 必须设置 TTL（最长 1 小时），任务完成后主动删除             |
| 本地缓存与 Redis 不一致  | 本地缓存 TTL 30 秒，允许短暂不一致（最终一致性），写入时主动失效本地缓存 |
| Spring Cache 注解穿透    | `@Cacheable` 仅用于纯查询场景，不用于含 MyBatis 动态 SQL 的方法          |

## 3. 架构设计

### 3.1 双层缓存架构

```
┌────────────────────────────────────────────────────┐
│                    应用程序                         │
│  ┌──────────────────────────────────────────────┐  │
│  │          Caffeine 本地缓存（JVM 内）           │  │
│  │  - 用户版本号        TTL: 30s, maxSize: 1000  │  │
│  │  - 租户配额配置      TTL: 30s, maxSize: 500   │  │
│  │  - 报表元数据摘要    TTL: 5min, maxSize: 2000 │  │
│  └───────────────────┬──────────────────────────┘  │
│                      │ Cache Miss                  │
│  ┌───────────────────▼──────────────────────────┐  │
│  │            Redis 分布式缓存                    │  │
│  │  - JWT 黑名单/版本号/锁定计数                  │  │
│  │  - 任务进度追踪                                │  │
│  │  - 调度配额与轮转顺序                          │  │
│  │  - MQ 消息去重                                 │  │
│  │  - 业务配置/权限元数据                          │  │
│  └───────────────────┬──────────────────────────┘  │
│                      │ Cache Miss                  │
│  ┌───────────────────▼──────────────────────────┐  │
│  │            PostgreSQL 持久化存储               │  │
│  └──────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────┘
```

> 查询路径：Caffeine → Redis → PostgreSQL。只有多实例间需要共享状态的缓存才进入 Redis 层，仅本实例需要的热点数据仅存 Caffeine。

### 3.2 整体流程

**Redis 读写流程**：

```
Service 层调用 CacheService.get(key)
        ↓
CacheService 实现
  ├── 如果是 @Cacheable 注解的方法 → Spring Cache 接管
  │     ├── Caffeine L1 命中 → 直接返回
  │     └── Caffeine L1 未命中 → 执行方法体 → 可能查 DB → 写入 Caffeine
  │
  └── 如果是显式 Redis 操作（进度/配额等）
        ├── RedisTemplate 执行命令
        ├── Redis 异常 → 返回 null/0/false（降级）
        └── Redis 正常 → 返回结果
```

**进度缓存写入流程**：

```
Worker 执行任务中
        ↓
每 1000 行 / 每处理完一个分片
        ↓
ProgressTracker.update(taskId, processedRows)
  ├── redisTemplate.opsForHash().put("jinshu:task:progress:{taskId}", "processed", rows)
  ├── redisTemplate.opsForHash().put("jinshu:task:progress:{taskId}", "total", total)
  ├── redisTemplate.opsForHash().put("jinshu:task:progress:{taskId}", "updatedAt", now)
  └── redisTemplate.expire("jinshu:task:progress:{taskId}", 1, TimeUnit.HOURS)
        ↓
前端轮询 GET /api/task/{taskId}/progress → 读取 Redis Hash
```

### 3.4 关键策略

**防穿透策略**：

```java
// CacheService.java
public <T> T getOrLoad(String key, Class<T> type, Supplier<T> dbLoader, Duration ttl) {
    T value = redisTemplate.opsForValue().get(key);
    if (value != null) {
        // 检查是否为"空值标记"（防穿透）
        if (value instanceof NullMarker) return null;
        return value;
    }
    // 未命中 → 查 DB
    T dbValue = dbLoader.get();
    if (dbValue != null) {
        redisTemplate.opsForValue().set(key, dbValue, ttl);
    } else {
        // 缓存空值，防止穿透
        redisTemplate.opsForValue().set(key, NullMarker.INSTANCE, Duration.ofMinutes(5));
    }
    return dbValue;
}
```

**防雪崩策略**：

```java
// 所有 TTL 增加 ±10% 随机偏移
private Duration withJitter(Duration ttl) {
    double jitter = 1.0 + ThreadLocalRandom.current().nextDouble(-0.1, 0.1);
    return Duration.ofMillis((long) (ttl.toMillis() * jitter));
}
```

**租户隔离**：

```
所有业务缓存 Key 必须包含 tenantId 前缀。
例如：jinshu:report:meta:{tenantId}:{reportId}
     jinshu:task:progress:{tenantId}:{taskId}
非业务 Key（JWT、MQ 去重）不受租户约束。
```

## 4. Key 命名规范与全景清单

### 4.1 命名规范

```
格式：jinshu:{domain}:{entity}[:{tenantId}]:{id}[:{field}]

domain   = task | mq | auth | report | tenant | user | scheduler
entity   = progress | consumed | blacklist | version | meta | config | quota
tenantId = 租户 ID（非租户级数据省略）
id       = 实体主键（taskId, userId, msgId 等）
```

| 规则         | 示例                                                     |
| ------------ | -------------------------------------------------------- |
| 统一前缀     | 所有 Key 以 `jinshu:` 开头                               |
| 域隔离       | 第二段标识业务域（auth / task / report / mq / ...）      |
| 租户分层     | 业务数据 Key 必须含 `{tenantId}`                         |
| 大小写       | 全小写，下划线分隔                                       |
| Key 模板常量 | 所有 Key 模式定义在 `CacheConstants.java` 中，禁止硬编码 |

### 4.2 JWT 认证缓存 Key

| Key 模式                                 | 类型   | TTL                | 说明                 | 来源     |
| ---------------------------------------- | ------ | ------------------ | -------------------- | -------- |
| `jinshu:auth:blacklist:{jti}`            | String | Token 剩余有效期   | Access Token 黑名单  | JWT §3.3 |
| `jinshu:auth:version:{userId}`           | String | 持久（管理员删除） | Token 版本号（自增） | JWT §3.3 |
| `jinshu:auth:refresh:{userId}:{tokenId}` | String | 7 天               | Refresh Token 存储   | JWT §3.3 |
| `jinshu:auth:login_fail:{username}`      | String | 15 分钟            | 登录失败计数器       | JWT §3.3 |

### 4.3 任务进度缓存 Key

| Key 模式                                   | 类型 | TTL    | 说明            | 来源 |
| ------------------------------------------ | ---- | ------ | --------------- | ---- |
| `jinshu:task:progress:{tenantId}:{taskId}` | Hash | 1 小时 | 任务进度详情    | 通用 |
| — `processed` (field)                      | int  | —      | 已处理行数/页数 |      |
| — `total` (field)                          | int  | —      | 总行数/总页数   |      |
| — `status` (field)                         | str  | —      | 当前状态        |      |
| — `updatedAt` (field)                      | str  | —      | 最后更新时间    |      |

### 4.4 任务调度缓存 Key

| Key 模式                                           | 类型   | TTL                          | 说明                  | 来源      |
| -------------------------------------------------- | ------ | ---------------------------- | --------------------- | --------- |
| `jinshu:scheduler:tenant:{tenantId}:running_tasks` | String | 持久（任务完成/失败后 DECR） | 租户当前运行任务数    | 调度 §3.3 |
| `jinshu:scheduler:tenant:{tenantId}:quota`         | String | 持久（管理员修改）           | 租户并发配额上限      | 调度 §3.3 |
| `jinshu:scheduler:tenant_order`                    | List   | 持久                         | 租户 Round-Robin 顺序 | 调度 §3.3 |

### 4.5 消息队列缓存 Key

| Key 模式                     | 类型   | TTL     | 说明                | 来源    |
| ---------------------------- | ------ | ------- | ------------------- | ------- |
| `jinshu:mq:consumed:{msgId}` | String | 24 小时 | MQ 消息消费幂等记录 | MQ §3.6 |

### 4.6 业务数据缓存 Key

| Key 模式                                   | 类型   | TTL     | 说明             | 来源 |
| ------------------------------------------ | ------ | ------- | ---------------- | ---- |
| `jinshu:tenant:config:{tenantId}`          | String | 10 分钟 | 租户配置（JSON） | 新增 |
| `jinshu:user:roles:{tenantId}:{userId}`    | String | 5 分钟  | 用户角色         | 新增 |
| `jinshu:report:meta:{tenantId}:{reportId}` | String | 5 分钟  | 报表元数据摘要   | 新增 |

> 业务数据缓存在写入时主动失效（`DELETE`），TTL 仅作为兜底。

## 6. 配置与实现

### 6.1 Redis 连接配置

```yaml
## application.yml (api / worker / batch / data-sync 统一)
spring:
  data:
    redis:
      host: ${REDIS_HOST:localhost}
      port: ${REDIS_PORT:6379}
      password: ${REDIS_PASSWORD:}
      database: 0
      timeout: 3000ms
      lettuce:
        pool:
          max-active: 16
          max-idle: 8
          min-idle: 2
          max-wait: 3000ms
```

```java
// RedisConfig.java (common 或各模块)
@Configuration
public class RedisConfig {

    @Bean
    public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);

        // Key 序列化：String
        template.setKeySerializer(new StringRedisSerializer());
        template.setHashKeySerializer(new StringRedisSerializer());

        // Value 序列化：Jackson JSON
        Jackson2JsonRedisSerializer<Object> serializer =
                new Jackson2JsonRedisSerializer<>(Object.class);
        ObjectMapper mapper = new ObjectMapper();
        mapper.registerModule(new JavaTimeModule());
        mapper.disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);
        mapper.setVisibility(PropertyAccessor.ALL, JsonAutoDetect.Visibility.ANY);
        mapper.activateDefaultTyping(
                BasicPolymorphicTypeValidator.builder().allowIfBaseType(Object.class).build(),
                ObjectMapper.DefaultTyping.NON_FINAL);
        serializer.setObjectMapper(mapper);
        template.setValueSerializer(serializer);
        template.setHashValueSerializer(serializer);

        template.afterPropertiesSet();
        return template;
    }

    @Bean
    public CacheService cacheService(RedisTemplate<String, Object> redisTemplate) {
        return new CacheServiceImpl(redisTemplate);
    }
}
```

### 6.2 缓存管理器配置

```java
// CacheConfig.java (api 模块)
@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public CacheManager cacheManager() {
        SimpleCacheManager cacheManager = new SimpleCacheManager();
        cacheManager.setCaches(List.of(
                caffeineCache("tenant_config", 10),
                caffeineCache("user_roles", 5),
                caffeineCache("report_meta", 5),
                caffeineCache("token_version", 30),
                caffeineCache("tenant_quota", 30)
        ));
        return cacheManager;
    }

    private CaffeineCache caffeineCache(String name, int ttlSeconds) {
        return new CaffeineCache(name, Caffeine.newBuilder()
                .expireAfterWrite(Duration.ofSeconds(ttlSeconds))
                .maximumSize(1000)
                .recordStats()
                .build());
    }
}
```

**使用示例**：

```java
// 租户配置查询（L1 Caffeine 缓存）
@Cacheable(value = "tenant_config", key = "#tenantId")
public TenantConfig getTenantConfig(Long tenantId) {
    return tenantMapper.selectConfig(tenantId);
}

// 租户配置更新时清除缓存
@CacheEvict(value = "tenant_config", key = "#config.tenantId")
public void updateTenantConfig(TenantConfig config) {
    tenantMapper.updateConfig(config);
}
```

### 6.3 Caffeine 本地缓存配置

| Cache 名称      | TTL     | maxSize | 用途              |
| --------------- | ------- | ------- | ----------------- |
| `tenant_config` | 10 分钟 | 500     | 租户配置          |
| `user_roles`    | 5 分钟  | 2000    | 用户角色查询      |
| `report_meta`   | 5 分钟  | 2000    | 报表元数据摘要    |
| `token_version` | 30 秒   | 1000    | 用户 Token 版本号 |
| `tenant_quota`  | 30 秒   | 500     | 租户配额配置      |

> `token_version` 和 `tenant_quota` 设置为 30 秒短 TTL，因为安全相关变更需要尽快生效。其他配置类缓存允许几分钟延迟。

## 7. 异常处理

| 异常场景                    | 处理方式                                                            |
| --------------------------- | ------------------------------------------------------------------- |
| Redis 连接超时              | 捕获 `RedisConnectionFailureException`，返回 null/0/false，业务降级 |
| Redis 命令执行失败          | 捕获 `RedisSystemException`，记录 WARN 日志，返回默认值             |
| Redis 序列化失败            | 捕获 `SerializationException`，直接查 DB，跳过缓存                  |
| Redis OOM（内存满）         | `allkeys-lru` 自动淘汰，不抛出异常                                  |
| Caffeine 写入失败（极少见） | 降级为直接查 Redis/DB                                               |
| Redis 恢复可用              | 无需额外操作，Lettuce 连接池自动重连，后续请求正常命中              |

**降级决策矩阵**：

| 缓存用途     | Redis 不可用时             | 影响                               |
| ------------ | -------------------------- | ---------------------------------- |
| JWT 黑名单   | 跳过黑名单检查，仅验签     | 登出的 Token 仍可用（15 分钟窗口） |
| JWT 版本号   | 跳过版本号检查，仅验签     | 用户吊销不即时生效                 |
| 登录失败计数 | 跳过计数，不锁账号         | 暴力破解风险短暂增加               |
| 任务进度     | 进度查询返回 503           | 前端进度条不可用                   |
| 调度配额     | 不限配额，所有任务直接执行 | 单租户可能占用全部资源             |
| MQ 去重      | 降级为 PostgreSQL 去重表   | 性能下降，但不丢失保障             |
| 业务数据缓存 | 直接查 DB                  | 响应时间增加 5-50ms                |

## 8. 性能考量

### 8.1 性能目标

| 指标                      | 目标                 |
| ------------------------- | -------------------- |
| Redis 单次 GET（本地）    | < 1ms                |
| Redis 单次 SET（本地）    | < 1ms                |
| Caffeine 缓存命中         | < 0.01ms             |
| 缓存命中率（Caffeine L1） | ≥ 80%                |
| 缓存命中率（Redis L2）    | ≥ 95%                |
| 进度更新频率              | 每 1000 行 / 每 5 秒 |
| Redis 内存使用峰值        | < 256MB              |

### 8.2 性能优化点

1. **Pipeline 批量操作**：同一请求内多次 Redis 操作（如 JWT 校验：黑名单 + 版本号）合并为一次 Pipeline 调用
2. **Caffeine L1 挡读**：高频读取的配置数据（租户配置、用户角色）优先走 Caffeine，减少 Redis 网络开销
3. **异步写入**：非关键的缓存更新（如进度更新）使用 `CompletableFuture.runAsync()`，不阻塞主流程
4. **JSON 序列化**：使用 Jackson `JavaTimeModule` 避免 `Date` 序列化性能问题
5. **Lettuce 连接复用**：Lettuce 基于 Netty，天然非阻塞 + 多路复用，比 Jedis 更优
6. **大 Key 拆分**：进度 Hash 的 `total` 和 `processed` 用 `HINCRBY` 原子更新，避免全量 Hash 读写

## 9. 安全考量

### 9.1 数据安全

- 所有业务缓存 Key 包含 `tenantId` 前缀，天然租户隔离
- 缓存中不存储密码、密钥等敏感数据
- Token 黑名单仅存 jti（Token ID），不存储 Token 原文
- Redis 密码通过 K8s Secret 注入（`REDIS_PASSWORD`），不写死在配置文件

### 9.2 防攻击

- **Key 注入**：使用 `CacheKeyBuilder` 统一构建 Key，禁止字符串拼接（防止 Key 冲突攻击）
- **Flush 攻击**：不提供全量 Flush 接口，仅允许按租户/按任务清理
- **大 Value 攻击**：单个缓存 Value 上限 1MB，超限拒绝写入并告警
- **连接耗尽**：Lettuce 连接池 maxTotal=16，超出排队等待

## 11. 上线方案

1. **添加依赖**：所有模块的 `build.gradle.kts` 增加 `data-redis` + `cache` + `caffeine`

- 各阶段监控 Redis 内存增长曲线，确保无泄漏

## 12. 回滚方案

### 12.1 回滚触发条件

- Redis 内存增长异常（疑似缓存泄漏）
- JWT 黑名单导致大面积 401
- 缓存序列化错误导致服务异常
- Redis 频繁连接超时影响业务响应

### 12.2 回滚步骤

1. 设置 `spring.cache.type = none`（禁用 Spring Cache 注解）
2. 在 `CacheService` 实现中加入 `jinshu.cache.enabled = false` 开关，跳过所有 Redis 操作
3. 回滚 api/worker/batch/data-sync 到上一版本
4. `FLUSHDB` 清空测试期间写入的缓存数据
5. NOTICE：缓存服务降级，响应时间可能增加 5-50ms

### 12.3 数据补救

- Redis 缓存数据全量可丢弃（缓存是可重建的）
- Token 黑名单清空后：回滚期间登出的 Token 可重新使用（15 分钟时间窗口，风险低）
- 进度缓存清空后：前端进度条暂时不可用，任务完成后通过数据库查询结果
- 无需数据库数据补救

## 13. 运维考量

### 13.1 Redis 持久化策略

当前 Docker Compose 和 K8s 部署均**未配置 Redis 持久化**（无 RDB/AOF，无 Volume 挂载），Pod 重启后所有数据丢失。建议：

| 环境     | 策略                                                    |
| -------- | ------------------------------------------------------- |
| 开发环境 | 无需持久化（数据量小，重建成本低）                      |
| 测试环境 | RDB 快照（`save 900 1`），存储到 PVC                    |
| 生产环境 | RDB + AOF 双开（`appendonly yes`），PVC 挂载 + 定期备份 |

> 持久化配置属于基础设施运维范畴，在 `docs/06-运维与部署/` 中详细展开。

### 13.2 内存估算与配额

| 缓存类型        | 单条大小 | 预估数量 | 总占用（估） |
| --------------- | -------- | -------- | ------------ |
| JWT 黑名单      | 200B     | 500 条   | 100KB        |
| Token 版本号    | 50B      | 1000 条  | 50KB         |
| 登录失败计数    | 30B      | 100 条   | 3KB          |
| 任务进度 (Hash) | 500B     | 200 条   | 100KB        |
| 调度配额        | 30B      | 100 条   | 3KB          |
| MQ 去重         | 100B     | 10000 条 | 1MB          |
| 业务数据缓存    | 2KB      | 5000 条  | 10MB         |
| **合计**        |          |          | **~12MB**    |

> 256MB 内存配额绰绰有余。内存主要消耗者不是业务缓存，而是 Redis 内部数据结构开销。
