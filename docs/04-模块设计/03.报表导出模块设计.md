## 1. 背景与目标

### 1.1 问题背景

报表系统的核心价值之一是将数据库中的海量数据导出为可下载的文件格式（Excel/CSV），供用户离线分析或归档。当前系统已完成数据导入模块，但导出功能尚未设计。导出场景面临的挑战包括：

- **数据量大**：单张报表可能包含 100 万行以上的数据，传统全量加载方式必然导致 OOM
- **格式多样**：不同场景需要 Excel（带样式）或 CSV（纯数据、体积小）
- **并发压力**：多租户环境下，大量并发导出任务可能耗尽数据库连接和内存资源
- **进度感知**：导出是耗时操作，用户需要实时了解进度，不能"点了按钮就盲等"

### 1.2 设计目标

| 指标         | 要求                               |
| ------------ | ---------------------------------- |
| 最大支持行数 | 100 万行（Excel）/ 无上限（CSV）   |
| 内存占用峰值 | < 512MB（100 万行导出时）          |
| 导出速度     | 10 万行 < 30 秒；100 万行 < 5 分钟 |
| 并发控制     | 单租户最大同时导出任务数 = 2       |
| 失败重试     | 支持自动重试 3 次                  |
| 进度可查     | 实时返回导出进度                   |

### 1.3 非目标

- 不支持 `.xls` 格式（仅 `.xlsx`）
- 不支持超过 100 万行的 Excel 导出（自动提示改用 CSV）
- 不支持实时同步导出（所有导出均为异步任务）
- 首次版本不实现 Excel 模板驱动模式（预留扩展接口）

## 2. 需求分析

### 2.1 功能需求

- [ ] **数据量预估**：用户选择筛选条件后，先调用预估接口返回数据行数和预估文件大小
- [ ] **Excel 导出**：使用 EasyExcel SXSSF 流式写入，支持表头样式、列宽、数据格式化
- [ ] **CSV 导出**：Java NIO 流式写入，强制 UTF-8 with BOM 编码，自动 ZIP 压缩
- [ ] **格式自动推荐**：超过 10 万行推荐 CSV；超过 100 万行强制 CSV
- [ ] **进度实时查询**：Redis 存储进度，前端轮询或 SSE 推送
- [ ] **下载链接生成**：任务完成后生成带签名的临时下载链接（有效期 1 小时）
- [ ] **临时文件清理**：任务完成后 24 小时自动清理临时文件
- [ ] **模板驱动导出（二期）**：业务人员上传 Excel 模板，系统按模板格式填充数据

### 2.2 用户故事

```
作为 报表管理员，我希望 将百万行报表数据导出为 Excel 或 CSV，
以便 在本地用 Excel/WPS 进行离线分析和归档。

作为 普通用户，我希望 在点击导出后能看到实时进度条，
以便 了解任务还需要多久完成，而不是盲目等待。

作为 系统管理员，我希望 导出任务不会阻塞其他租户的操作，
以便 保证系统的整体可用性和公平性。
```

### 2.3 边界场景

| 场景               | 处理方式                                               |
| ------------------ | ------------------------------------------------------ |
| 0 行数据           | 生成空文件（Excel 含表头，CSV 含 BOM），标记 SUCCESS   |
| 100 万+ 行选 Excel | 预估接口返回 `suggestFormat: "CSV"`，前端弹窗确认      |
| 服务重启中断       | Worker 重启后消费 MQ，重新执行导出（幂等，覆盖原文件） |
| 并发超限           | 返回 429，提示"当前导出任务已满，请等待"               |
| 磁盘空间不足       | 写入前检查磁盘，不足则标记 FAILED 并告警               |
| 数据库查询超时     | Cursor 流式查询设置超时 10 分钟，超时分片重试          |

## 3. 架构设计

### 3.1 整体流程

```
用户选择报表 + 筛选条件
        ↓
[同步] GET /api/export/estimate
        ↓
返回: { totalRows, estimatedFileSize, suggestFormat }
        ↓
用户确认格式，点击导出
        ↓
[同步] POST /api/export
创建 task 记录，状态 = PENDING
发送消息到 RabbitMQ jinshu.export 队列
立即返回 taskId（HTTP 202 Accepted）
        ↓
[异步消费] Worker 模块启动导出任务
        ↓
┌─────────────────────────────────────────────────────────┐
│  1. 更新 task 状态 = PROCESSING                         │
│  2. MyBatis Cursor 流式查询（FORWARD_ONLY）             │
│     - fetchSize = Integer.MIN_VALUE（PostgreSQL 游标）  │
│     - 逐行读取，不占内存                                │
│  3. 边读边写入 PVC 临时文件                             │
│     - Excel: EasyExcel SXSSF 流式写入                   │
│     - CSV: BufferedWriter + NIO，每 1000 行 flush       │
│  4. 每处理 1000 行更新 Redis 进度                       │
│  5. CSV 完成后自动 ZIP 压缩                             │
└─────────────────────────────────────────────────────────┘
        ↓
更新 task 状态 = SUCCESS，记录 output_file_path
        ↓
Redis 推送完成通知 / WebSocket 通知前端
        ↓
用户调用 GET /api/export/{taskId}/download 获取签名下载链接
        ↓
24 小时后 CronJob 自动清理临时文件
```

│   ├── CsvExportHandler.java        # CSV 导出处理器

com.jinshu.api             # API 模块（HTTP 接口）

### 3.3 关键算法与策略

**格式选择策略**

| 数据行数         | 推荐格式 | 用户可选  | 说明                          |
| ---------------- | -------- | --------- | ----------------------------- |
| < 10 万行        | Excel    | Excel/CSV | 默认 Excel，用户可改 CSV      |
| 10 万 - 100 万行 | CSV      | Excel/CSV | 推荐 CSV，用户仍可强选 Excel  |
| > 100 万行       | CSV      | 仅 CSV    | Excel 单 Sheet 上限，强制 CSV |

**流式查询策略**

```java
// MyBatis Cursor 配置（PostgreSQL 游标模式）
@Select("SELECT row_no, data_json FROM report_data " +
        "WHERE tenant_id = #{tenantId} AND report_id = #{reportId} " +
        "ORDER BY row_no")
@Options(resultSetType = ResultSetType.FORWARD_ONLY, fetchSize = Integer.MIN_VALUE)
Cursor<ReportData> streamQuery(@Param("tenantId") Long tenantId,
                               @Param("reportId") Long reportId);
```

**进度计算策略**

```
progress = (已处理行数 / 预估总行数) × 100
// 预估行数来自 count(*) 查询，实际行数可能略有偏差
// 处理完成后强制置为 100%
```

## 4. 数据库设计

### 4.1 表结构变更

**复用现有 `task` 表**（已存在，无需新建），通过 `type = 'EXPORT'` 区分导出任务。

`task` 表现有结构：

```sql
CREATE TABLE task (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       BIGINT NOT NULL,
    type            VARCHAR(32) NOT NULL,    -- 'IMPORT' / 'EXPORT' / 'PDF'
    status          VARCHAR(16) NOT NULL,    -- PENDING / PROCESSING / SUCCESS / FAILED
    priority        INTEGER NOT NULL DEFAULT 0,
    config          JSONB,                   -- 导出配置：reportId, format, filters
    progress        INTEGER DEFAULT 0,
    error_message   TEXT,
    created_by      BIGINT,
    created_at      TIMESTAMP DEFAULT NOW(),
    started_at      TIMESTAMP,
    completed_at    TIMESTAMP
);
```

**config JSONB 结构示例（导出任务）**：

```json
{
  "reportId": 1001,
  "format": "EXCEL",
  "filters": {
    "dateRange": ["2026-01-01", "2026-03-31"],
    "status": ["APPROVED"]
  },
  "outputPath": "/data/output/1001_20260517120000_001.xlsx",
  "estimatedRows": 50000
}
```

### 4.2 索引设计

`task` 表已有索引满足需求：

```sql
-- 现有索引
CREATE INDEX idx_task_tenant_status ON task(tenant_id, status);
CREATE INDEX idx_task_priority ON task(priority DESC, created_at);
```

**补充索引建议（可选）**：

```sql
-- 按任务类型查询导出任务
CREATE INDEX idx_task_type ON task(type, created_at DESC);
```

### 4.3 数据迁移方案

本模块不涉及存量数据迁移，新任务直接写入 `task` 表即可。

## 6. 异常处理

| 异常场景          | 错误码 | 提示信息                          | 处理方式                                |
| ----------------- | ------ | --------------------------------- | --------------------------------------- |
| 报表不存在        | 404    | 报表不存在                        | 记录审计日志                            |
| 无导出权限        | 403    | 无权导出该报表                    | 记录审计日志                            |
| 并发超限（>2）    | 429    | 当前导出任务已满，请等待          | 拒绝，不创建任务                        |
| 数据量为 0        | —      | —                                 | 正常生成空文件，标记 SUCCESS            |
| 数据库查询超时    | 500    | 导出任务超时，已自动重试          | 重试 3 次，失败标记 FAILED              |
| 磁盘空间不足      | 500    | 存储空间不足，请联系管理员        | 标记 FAILED，触发告警                   |
| 格式不支持        | 400    | 不支持的导出格式                  | 直接返回，不创建任务                    |
| Excel 超 100 万行 | 400    | 数据量超出 Excel 上限，请改用 CSV | 预估接口已拦截，此处兜底                |
| Worker 服务重启   | —      | —                                 | MQ 消息重新消费，幂等执行（覆盖原文件） |

## 7. 性能考量

### 7.1 性能目标

| 场景          | 指标                |
| ------------- | ------------------- |
| 10 万行 Excel | < 30 秒             |
| 100 万行 CSV  | < 5 分钟            |
| 内存占用峰值  | < 512MB             |
| 单租户并发    | ≤ 2                 |
| 进度更新频率  | 每 1000 行或每 5 秒 |

### 7.2 性能优化点

1. **数据库游标**：使用 `FORWARD_ONLY` + `fetchSize = Integer.MIN_VALUE`，PostgreSQL 以服务端游标模式逐行推送，避免客户端内存堆积
2. **批量写入**：Excel 每 1000 行 flush 一次；CSV 每 1000 行 `BufferedWriter.flush()`
3. **零拷贝传输**：下载文件时使用 `Files.copy()` 或 `FileChannel.transferTo()`，减少用户态/内核态切换
4. **ZIP 压缩**：CSV 导出完成后使用 `ZipOutputStream` 流式压缩，不将全部数据加载到内存
5. **Redis 进度合并**：每 1000 行或每 5 秒更新一次 Redis，避免高频写入

**内存估算**：

```
单行数据 ≈ 1KB（JSONB 反序列化后）
批量缓冲 = 1000 行 ≈ 1MB
EasyExcel SXSSF 窗口 = 100 行 ≈ 0.1MB
CSV BufferedWriter = 8KB
+ JVM 基础开销 ≈ 100MB
------------------------
峰值预估 ≈ 120MB < 512MB ✅
```

## 8. 安全考量

### 8.1 权限控制

- 只有报表所属租户的用户可以导出该报表（通过 `tenant_id` 过滤）
- 访客角色（GUEST）只能导出"已公开"状态的报表
- 下载链接使用 JWT 签名，包含 `taskId`、`userId`、`tenantId`、`exp`，防止伪造和越权下载

### 8.2 数据安全

- 所有导出操作记录审计日志（`operation = 'EXPORT'`）
- 下载链接有效期 1 小时，过期后 token 自动失效
- 临时文件存储在 PVC 中，不直接暴露于公网，必须通过 API 获取签名链接后才能下载
- CSV 强制 UTF-8 with BOM，防止 Excel 打开中文乱码的同时避免公式注入攻击

### 8.3 防攻击

- **CSV 公式注入防护**：对以 `=`、`+`、`-`、`@` 开头的单元格值前添加单引号 `'`，防止自动执行公式
- **SQL 注入**：所有查询条件通过 MyBatis `#{}` 预编译参数传递，禁止字符串拼接 SQL
- **重放攻击**：下载链接 token 包含唯一 `jti`（JWT ID），Redis 缓存已使用的 token，禁止重复使用
- **拒绝服务**：单租户并发限制 + 数据量预估拦截，防止恶意大数据量导出拖垮系统

## 9. 测试方案

### 9.1 单元测试

| 测试用例             | 说明                                 |
| -------------------- | ------------------------------------ |
| 格式选择策略         | 验证不同行数下推荐格式是否正确       |
| Excel SXSSF 流式写入 | 验证 10 万行写入后内存 < 200MB       |
| CSV ZIP 压缩         | 验证压缩后体积减小 60%+              |
| JWT 下载链接签名     | 验证 token 解析正确，越权 token 拒绝 |
| 进度计算             | 验证进度从 0 递增到 100              |

### 9.2 集成测试

- **完整导出链路**：提交导出 → MQ 消费 → 文件生成 → 进度查询 → 下载验证文件内容
- **并发测试**：同一租户同时提交 3 个导出任务，验证第 3 个被 429 拒绝
- **大数据量测试**：100 万行 CSV 导出，验证耗时 < 5 分钟，无 OOM
- **服务重启恢复**：导出过程中 Kill Worker Pod，验证 MQ 重试后任务成功完成

### 9.3 回归测试

- 导出模块不影响导入模块的 MQ 队列消费
- 导出任务不影响报表查询 API 的响应时间
- 临时文件清理 CronJob 不误删其他模块的文件

## 10. 上线方案

### 10.1 上线步骤

1. 部署 `worker` 模块更新包（新增导出消费者）
2. 部署 `api` 模块更新包（新增导出 Controller）
3. 验证 RabbitMQ `jinshu.export` 队列已创建
4. 验证 `/data/output/` 和 `/data/temp/` 目录权限正确
5. 执行集成测试用例（预估 → 提交 → 下载）

### 10.2 灰度策略

- 首日仅对内部测试租户开放导出功能
- 观察 24 小时内的任务成功率、内存占用、磁盘增长
- 无异常后全量开放

### 10.3 监控告警

| 监控项            | 阈值      | 告警方式        |
| ----------------- | --------- | --------------- |
| 导出任务失败率    | > 5%      | 企业微信 + 邮件 |
| 单任务耗时        | > 10 分钟 | 企业微信        |
| Worker 内存使用率 | > 80%     | 企业微信        |
| 磁盘剩余空间      | < 10GB    | 企业微信 + 短信 |
| MQ 导出队列堆积   | > 100 条  | 企业微信        |

## 11. 回滚方案

### 11.1 回滚触发条件

- 导出任务失败率 > 30%
- Worker 频繁 OOM 重启
- 下载链接存在越权漏洞
- 磁盘被临时文件占满

### 11.2 回滚步骤

1. 关闭导出功能开关（`export.enabled = false`，API 层直接返回 503）
2. 回滚 `api` 和 `worker` 到上一版本
3. 清空 RabbitMQ `jinshu.export` 队列中未消费的消息（避免旧版本消费者异常）
4. 通知用户：导出功能临时维护

### 11.3 数据补救

- 回滚期间已创建但未完成的 `task` 记录：批量更新 `status = 'FAILED'`，`error_message = '系统维护，请重新提交'`
- 临时文件保留 7 天，不立即删除，防止用户正在下载
