## 1. 背景与目标

### 1.1 问题背景

系统包含多个需要异步执行的耗时操作（导入、导出、PDF 渲染、数据同步），以及模块间的解耦通信需求。当前状态：

| 问题                              | 影响                                                                                                    |
| --------------------------------- | ------------------------------------------------------------------------------------------------------- |
| 无 MQ 集成代码                    | 所有设计文档中的异步流程无法落地                                                                        |
| 队列命名不一致                    | 调度模块定义 `jinshu.task.p0~p4`，导出模块用 `jinshu.export`，PDF 模块用 `jinshu.render` — 三套命名并存 |
| 无消息格式约定                    | 各模块设计各自定义消息体，无公共字段（traceId 等）                                                      |
| 无死信/重试机制                   | 失败任务没有统一的兜底策略                                                                              |
| api 模块无 AMQP 依赖              | 生产者代码无法在 api 模块编译                                                                           |
| worker/batch 有 AMQP 依赖但零代码 | 依赖已引入但未使用                                                                                      |

### 1.2 设计目标

| 指标       | 要求                                         |
| ---------- | -------------------------------------------- |
| 队列拓扑   | 1 个 Exchange + 7 个 Queue + 2 个 DLX        |
| 命名规范   | 统一前缀 `jinshu.{domain}.{purpose}`         |
| 消息体规范 | 公共头（traceId/tenantId/msgId/ts） + 业务体 |
| 消费者幂等 | 基于 msgId 去重表，AT LEAST ONCE 语义        |
| 重试策略   | 失败重试 3 次，间隔 5s/30s/5min              |
| 死信处理   | 重试耗尽进入 DLX，人工介入                   |
| 消息持久化 | 所有消息 `delivery_mode = 2`                 |
| 优先级队列 | 5 级（P0-P4）用于任务调度                    |

### 1.3 非目标

- 不实现 RabbitMQ 集群/镜像队列（基础设施层负责）
- 不实现消息轨迹追踪（二期结合 OpenTelemetry）
- 不在此文档设计具体消费者的业务逻辑（由各模块设计文档负责）
- 不实现延迟队列插件（`rabbitmq_delayed_message_exchange`）

## 2. 需求分析

### 2.1 功能需求

- [x] **任务队列**：导入/导出/PDF/数据同步任务通过 MQ 异步投递
- [x] **优先级支持**：9 级优先级队列（P0-P4，各含高/低两个子优先级）
- [x] **死信队列**：重试耗尽的消息自动进入 DLX
- [x] **幂等消费**：基于 msgId 的消费去重
- [x] **TraceId 传递**：MQ 消息头携带 traceId，贯穿全链路
- [x] **租户隔离**：消息体含 tenantId，消费者校验
- [x] **MQ 健康检查**：`/health` 端点包含 RabbitMQ 连接状态

### 2.2 用户故事

```
作为 后端开发者，我希望 所有异步任务通过统一的 MQ 接口发送和消费，
以便 模块间解耦，各自独立部署和伸缩。

作为 运维工程师，我希望 失败的消息能自动重试并最终进入死信队列，
以便 不会静默丢失任务，同时有清晰的人工介入路径。

作为 前端开发者，我希望 提交异步任务后能通过 taskId 轮询进度，
以便 用户感知到任务的实时状态。
```

### 2.3 边界场景

| 场景                     | 处理方式                                                         |
| ------------------------ | ---------------------------------------------------------------- |
| RabbitMQ 连接断开        | 自动重连（`spring.rabbitmq` 默认开启），重连期间消息持久化不丢失 |
| 消费者处理消息时 JVM OOM | 消息未 ACK，重新入队，Worker 重启后重新消费                      |
| 同一条消息被重复投递     | 消费者基于 msgId + 去重表幂等处理                                |
| 消费者处理超时           | `prefetch = 1` + 手动 ACK，超时未 ACK 则重入队                   |
| 队列堆积超过阈值         | 触发 HPA 扩容 Worker Pod，同时告警                               |
| 消息体超大（>10MB）      | 不在消息体中传输文件内容，只传文件路径/metadata                  |
| 优先级队列不生效         | 设置 `x-max-priority = 10`，生产者设置 `priority` header         |
| DLX 也堆积               | 告警升级（短信），人工批量处理或清空                             |

## 3. 架构设计

### 3.1 拓扑总览

```
                                  ┌─────────────────────────────────────────────────────┐
                                  │                 jinshu.task.direct                    │
                                  │                  (Direct Exchange)                     │
                                  └───┬─────────┬─────────┬─────────┬─────────┬──────────┘
                                      │         │         │         │         │
              Routing Key:   task.p0/ task.p1/ task.p2/ task.p3/ task.p4/ task.dlq
                                      ↓         ↓         ↓         ↓         ↓
                              ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
                              │jinshu.   │ │jinshu.   │ │jinshu.   │ │jinshu.   │ │jinshu.   │
                              │task.p0   │ │task.p1   │ │task.p2   │ │task.p3   │ │task.p4   │
                              │(x-pri=10)│ │(x-pri=10)│ │(x-pri=10)│ │(x-pri=10)│ │(x-pri=10)│
                              └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘
                                   │            │            │            │            │
                        x-dead-letter-exchange: jinshu.task.dlx
                        x-dead-letter-routing-key: task.dlq
                                   │            │            │            │            │
                                   └────────────┴────────────┴────────────┴────────────┘
                                                            │
                                                            ↓
                                              ┌──────────────────────┐
                                              │   jinshu.task.dlx    │
                                              │  (Direct Exchange)   │
                                              └──────────┬───────────┘
                                                         │
                                             routing-key: task.dlq
                                                         ↓
                                              ┌──────────────────────┐
                                              │   jinshu.task.dlq    │
                                              │  (Dead Letter Queue) │
                                              └──────────────────────┘
```

> **统一命名**：所有业务消息都通过上述 5 个优先级队列流转。之前各模块设计文档中的 `jinshu.export`、`jinshu.render`、`jinshu.render.segment`、`import 队列` 等名称**统一迁移到此拓扑**，由 routing key 区分业务类型。消息头中携带 `messageType` 字段标识具体业务（IMPORT / EXPORT / PDF / PDF_SEGMENT / DATA_SYNC / BATCH）。

### 3.2 队列定义

| 队列名            | 类型 | 优先级 | x-max-priority | 消费者             | 说明                      |
| ----------------- | ---- | ------ | -------------- | ------------------ | ------------------------- |
| `jinshu.task.p0`  | 任务 | P0     | 10             | worker             | 预览请求、<1 万行小导出   |
| `jinshu.task.p1`  | 任务 | P1     | 10             | worker             | 1-10 万行导出、<50 页 PDF |
| `jinshu.task.p2`  | 任务 | P2     | 10             | worker             | >10 万行导出、>50 页 PDF  |
| `jinshu.task.p3`  | 任务 | P3     | 10             | worker / data-sync | 数据同步任务              |
| `jinshu.task.p4`  | 任务 | P4     | 10             | batch              | 定时批量任务、后台清理    |
| `jinshu.task.dlq` | 死信 | —      | —              | (人工)             | 重试 3 次后仍失败的消息   |

**队列属性**（所有队列）：

| 属性                                  | 值                | 说明                 |
| ------------------------------------- | ----------------- | -------------------- |
| `durable`                             | true              | 服务重启后队列不丢失 |
| `autoDelete`                          | false             | 无消费者时不自动删除 |
| `arguments.x-dead-letter-exchange`    | `jinshu.task.dlx` | 死信交换机           |
| `arguments.x-dead-letter-routing-key` | `task.dlq`        | 死信路由键           |
| `arguments.x-max-priority`            | 10                | 支持消息优先级       |

### 3.3 生产者与消费者矩阵

| 模块          | 角色            | 生产到                      | 消费自         | 需要 AMQP 依赖 |
| ------------- | --------------- | --------------------------- | -------------- | -------------- |
| **api**       | 生产者          | 所有 5 个队列（根据优先级） | —              | **需新增**     |
| **worker**    | 消费者 + 生产者 | P0-P3（子任务分发）         | P0, P1, P2, P3 | ✅ 已有        |
| **batch**     | 消费者          | —                           | P4             | ✅ 已有        |
| **data-sync** | 消费者          | —                           | P3             | **需新增**     |

> `api` 模块需要新增 `spring-boot-starter-amqp` 依赖（当前只有 web、security、mybatis、postgresql）。`data-sync` 同样需要新增。

### 3.4 消息流转

**典型流程：用户提交导出任务**：

```
api 模块（生产者）
  ├── POST /api/export → ExportController
  ├── ExportTaskService.createTask()
  │     ├── 插入 task 记录（status = PENDING）
  │     └── 构建 ExportTaskMessage
  │           ├── header: traceId, tenantId, userId
  │           └── body: taskId, reportId, format, filters
  ├── 计算优先级 → TaskPriorityCalculator
  ├── rabbitTemplate.convertAndSend(
  │       "jinshu.task.direct",
  │       "task.p1",           // 10 万行 → P1
  │       message,
  │       m → { m.getMessageProperties().setPriority(7); }
  │   )
  └── 返回 taskId（HTTP 202）
        ↓
worker 模块（消费者）
  ├── @RabbitListener(queues = "jinshu.task.p1")
  │     └── ExportTaskConsumer.handle(message)
  ├── 幂等检查（msgId 去重表）
  ├── 执行导出逻辑（MyBatis Cursor + EasyExcel 流式写入）
  ├── 更新 task 状态 + Redis 进度
  ├── 成功 → channel.basicAck()
  └── 失败 → channel.basicNack(requeue=false)
                ↓
          消息进入 DLX → jinshu.task.dlq
```

### 3.6 关键算法与策略

**优先级计算**（与任务调度模块设计一致）：

```java
public class TaskPriorityCalculator {

    public static PriorityResult calculate(TaskMessage task) {
        return switch (task.getMessageType()) {
            case "PREVIEW"            -> new PriorityResult("task.p0", 10);
            case "EXPORT"  when rows < 10_000   -> new PriorityResult("task.p0", 9);
            case "EXPORT"  when rows < 100_000  -> new PriorityResult("task.p1", 7);
            case "EXPORT"  when rows >= 100_000 -> new PriorityResult("task.p2", 5);
            case "PDF"     when pages < 50      -> new PriorityResult("task.p1", 6);
            case "PDF"     when pages >= 50     -> new PriorityResult("task.p2", 4);
            case "PDF_SEGMENT"                  -> new PriorityResult("task.p2", 4);
            case "DATA_SYNC"                    -> new PriorityResult("task.p3", 3);
            case "BATCH"                        -> new PriorityResult("task.p4", 1);
            default                             -> new PriorityResult("task.p4", 1);
        };
    }
}
```

**消费者重试策略**：

```
第 1 次失败 → 等待 5 秒 → 重试
第 2 次失败 → 等待 30 秒 → 重试
第 3 次失败 → 等待 5 分钟 → 重试
第 4 次失败 → 不再重试，消息进入 DLX
```

实现方式：使用 RabbitMQ `x-death` header 计数，消费者捕获异常后 `basicNack(requeue=false)`，利用队列的 DLX 配置自动路由。延迟通过 DLX 队列的 `x-message-ttl` 组合实现，或使用 `RabbitTemplate` 的 `retryTemplate`。

**幂等消费**：

```
Redis Key: jinshu:mq:consumed:{msgId}  (String, TTL = 24 小时)

消费者处理流程：
  1. 接收消息，提取 msgId
  2. Redis SETNX jinshu:mq:consumed:{msgId} "1" EX 86400
  3. SETNX 返回 false → 已消费，直接 ACK 丢弃
  4. SETNX 返回 true → 首次消费，执行业务逻辑
  5. 业务成功 → ACK
  6. 业务失败 → 删除 Redis Key + NACK 触发重试
```

> 使用 Redis 而非数据库去重表，因为 msgId 去重是临时性的（24 小时窗口），Redis 更轻量且不产生数据库写入压力。

## 4. 消息格式设计

### 4.1 消息基类

所有 MQ 消息遵循统一结构。**消息头**通过 `MessageProperties.headers` 传递，**消息体**通过 `MessageProperties.payload`（JSON）传递。

**消息头（Headers）**：

| Header           | 类型    | 必填 | 说明                                                              |
| ---------------- | ------- | ---- | ----------------------------------------------------------------- |
| `traceId`        | String  | 是   | 全链路追踪 ID（前端 → API → MQ → Worker）                         |
| `tenantId`       | Long    | 是   | 租户 ID（消费者校验）                                             |
| `userId`         | Long    | 是   | 操作用户 ID                                                       |
| `messageType`    | String  | 是   | 业务类型：IMPORT / EXPORT / PDF / PDF_SEGMENT / DATA_SYNC / BATCH |
| `messageVersion` | String  | 是   | 消息格式版本（默认 "1.0"）                                        |
| `priority`       | Integer | 否   | RabbitMQ 内置优先级 header                                        |

**消息体基类**：

```json
{
  "msgId": "uuid-xxxx",
  "taskId": 10086,
  "createdAt": "2026-05-17T12:00:00Z"
}
```

| 字段        | 类型   | 必填 | 说明                                 |
| ----------- | ------ | ---- | ------------------------------------ |
| `msgId`     | String | 是   | 消息唯一 ID（UUID v4），用于幂等去重 |
| `taskId`    | Long   | 否   | 关联的 task 表记录 ID                |
| `createdAt` | String | 是   | 消息创建时间（ISO 8601）             |

### 4.2 各业务消息体

**导入任务消息**：

```json
{
```

## 6. 配置与声明

### 6.1 Exchange/Queue/Binding 声明

```java
// RabbitMqInitializer.java (在 common 模块或各模块复制)
@Configuration
public class RabbitMqInitializer {

    // --- Exchanges ---
    @Bean
    public DirectExchange taskExchange() {
        return ExchangeBuilder.directExchange(MqConstants.EXCHANGE_TASK)
                .durable(true)
                .build();
    }

    @Bean
    public DirectExchange deadLetterExchange() {
        return ExchangeBuilder.directExchange(MqConstants.EXCHANGE_DLX)
                .durable(true)
                .build();
    }

    // --- Queues ---
    @Bean
    public Queue taskP0Queue() {
        return QueueBuilder.durable(MqConstants.QUEUE_TASK_P0)
                .withArgument("x-dead-letter-exchange", MqConstants.EXCHANGE_DLX)
                .withArgument("x-dead-letter-routing-key", MqConstants.ROUTING_KEY_DLQ)
                .withArgument("x-max-priority", 10)
                .build();
    }
    // ... P1-P4 同理

    @Bean
    public Queue deadLetterQueue() {
        return QueueBuilder.durable(MqConstants.QUEUE_DLQ).build();
    }

    // --- Bindings ---
    @Bean
    public Binding bindP0() {
        return BindingBuilder.bind(taskP0Queue()).to(taskExchange()).with(MqConstants.ROUTING_KEY_TASK_P0);
    }
    // ... P1-P4 同理

    @Bean
    public Binding bindDlq() {
        return BindingBuilder.bind(deadLetterQueue()).to(deadLetterExchange())
                .with(MqConstants.ROUTING_KEY_DLQ);
    }
}
```

### 6.2 application.yml 配置

**api 模块配置**（生产者端）：

```yaml
spring:
  rabbitmq:
    host: ${RABBITMQ_HOST:localhost}
    port: ${RABBITMQ_PORT:5672}
    username: ${RABBITMQ_USER:guest}
    password: ${RABBITMQ_PASSWORD:guest}

    # 生产者确认
    publisher-confirm-type: correlated
    publisher-returns: true
    # 连接重试
    connection-timeout: 5000
    template:
      retry:
        enabled: true
        initial-interval: 1000
        max-attempts: 3
```

**worker 模块配置**（消费者端）：

```yaml
spring:
  rabbitmq:
    host: ${RABBITMQ_HOST:localhost}
    port: ${RABBITMQ_PORT:5672}
    username: ${RABBITMQ_USER:guest}
    password: ${RABBITMQ_PASSWORD:guest}
    virtual-host: /
    listener:
      simple:
        acknowledge-mode: manual # 手动 ACK
        prefetch: 1 # 每次只取 1 条（公平分发）
        concurrency: 4 # 最小消费者数
        max-concurrency: 10 # 最大消费者数
        retry:
          enabled: false # 不使用 Spring 自动重试（走 DLX 机制）
```

> `prefetch = 1` 确保每个 Worker 一次只处理一条消息，避免某个 Worker 积压多条大消息。

## 7. 异常处理

| 异常场景                           | 处理方式                                                                                |
| ---------------------------------- | --------------------------------------------------------------------------------------- |
| 消息发送失败（Broker 不可达）      | Spring Retry 3 次，间隔 1s/3s/5s。全部失败抛出异常，API 返回 503                        |
| 消息路由失败（无匹配队列）         | `publisher-returns = true`，记录 ERROR 日志，触发告警                                   |
| 消费者处理抛出 `BusinessException` | catch 后记录错误到 task 表，NACK（不重回队列），进入 DLX                                |
| 消费者处理抛出 `RuntimeException`  | 重新投递 3 次（DLX 机制），3 次后进入 DLQ                                               |
| 消费者 OOM / JVM 崩溃              | 消息未 ACK，Broker 重新投递。Worker 重启后消费                                          |
| 消息体反序列化失败                 | 记录原始消息体到 `mq_consumed_message`，记录 ERROR 日志，直接 ACK（不重试，因为总失败） |
| DLQ 消息处理                       | 管理员通过管理 API 查看 DLQ 堆积，支持重新投递或丢弃                                    |
| Redis 不可用（幂等去重）           | 降级为 PostgreSQL `mq_consumed_message` 表兜底                                          |

## 8. 性能考量

### 8.1 性能目标

| 指标                        | 目标                                    |
| --------------------------- | --------------------------------------- |
| 消息发送延迟（P99）         | < 50ms                                  |
| 消息消费延迟（P99，含业务） | 按任务类型：小导出 < 30s，大导出 < 5min |
| 消息吞吐量                  | ≥ 500 msg/s                             |
| 队列堆积告警阈值            | > 1000 条                               |
| 单条消息大小上限            | 10KB（不含文件内容）                    |

### 8.2 性能优化点

1. **消息体中不含大文件**：只传文件路径（`/data/upload/xxx.xlsx`），不传文件内容
2. **Jackson 序列化**：使用 `Jackson2JsonMessageConverter`，比默认的 `SimpleMessageConverter`（JDK 序列化）更快且可读
3. **prefetch = 1**：公平分发，避免某个 Worker 积压而其他空闲
4. **消费者并发**：`concurrency = 4`，`max-concurrency = 10`，根据队列压力自动调整
5. **连接池**：Spring AMQP 默认 `CachingConnectionFactory` 使用单连接多 Channel 复用
6. **消息确认**：生产者端 `publisher-confirm-type = correlated`（非 `simple`），异步确认不阻塞发送线程
7. **持久化代价**：`delivery_mode = 2` 会写入磁盘，延迟增加约 1ms，可接受

## 9. 安全考量

### 9.1 权限控制

- RabbitMQ 使用专用账号（`jinshu`），不共用 `guest` 账号（生产环境）
- RabbitMQ 管理界面（15672）仅内网可访问
- 消费者校验消息头中的 `tenantId` 是否与 `TenantContext` 一致（防止消息伪造跨租户）

### 9.2 数据安全

- MQ 连接使用 TLS（生产环境配置 `spring.rabbitmq.ssl.enabled = true`）
- 消息体不传输敏感数据（密码、密钥等）
- `traceId` + `userId` + `tenantId` 写入消息头，用于审计追溯

### 9.3 防攻击

- **消息洪泛**：api 模块对同一用户/租户限制消息发送速率（后续结合租户配额模块）
- **恶意消息**：消息体反序列化失败直接 ACK 丢弃（防止恶意 payload 持续重试）
- **连接耗尽**：每个模块到 RabbitMQ 的连接数受 `CachingConnectionFactory` 限制
- **权限隔离**：生产者和消费者使用不同的 RabbitMQ 账号（可选，更严格）

## 12. 回滚方案

### 12.1 回滚触发条件

- 消息大量丢失（发送成功但消费者无法收到）
- 消费者频繁崩溃
- 队列声明异常导致旧消费者报错
- RabbitMQ 集群不可用且短期内无法恢复

### 12.2 回滚步骤

1. 停止 Worker 消费者（设置 `spring.rabbitmq.listener.simple.auto-startup = false`）
2. 回滚 `api`、`worker`、`batch`、`data-sync` 模块到上一版本
3. 清空所有未消费的消息（在 RabbitMQ 管理界面 Purge Queues）
4. 恢复业务：api 模块的异步任务改为同步执行或返回 503
5. NOTICE：MQ 服务临时回退，异步任务暂不可用

### 12.3 数据补救

- 回滚期间已发送但未消费的消息：清空后丢失，但不影响数据库数据（task 记录已写入）
- task 表中状态 = PENDING 的记录：批量标记为 FAILED
- Redis 进度缓存：TTL 到期后自动清除
- `mq_consumed_message` 表：无需清理（24 小时自动清理）
