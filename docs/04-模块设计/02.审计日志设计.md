## 1. 背景与目标

### 1.1 问题背景

企业级报表系统必须满足等保三级合规要求，对用户操作进行全量审计记录。当前状态：

| 问题                       | 严重度 | 影响                                       |
| -------------------------- | ------ | ------------------------------------------ |
| 审计日志 AOP 已写未启用    | 🔴 致命 | `AuditLogAspect.java` 存在但 `@Component` 被注释 |
| Merkle 哈希树未实现        | 🟡 高   | `HashUtils.java` 实现了 SHA-256 但 Merkle 树构建逻辑空白 |
| 审计日志查询/导出接口未设计 | 🟡 高  | 审计员无法查看审计日志                     |
| 归档策略未定义             | 🟡 高   | 超过 6 个月的日志如何处理无约定            |
| `audit_root_hash` 表已有但裸 | 🟡 高  | DDL 已建表但无代码写入                     |

### 1.2 设计目标

| 指标               | 要求                                       |
| ------------------ | ------------------------------------------ |
| 覆盖率             | 所有 CUD 操作 100% 记录审计日志             |
| 写入性能           | 峰值 ≥ 1000 TPS（异步批量写入）            |
| 防篡改             | 数据库触发器 + 哈希链 + Merkle 根哈希三重保障 |
| 保留期限           | ≥ 6 个月（热数据），>6 个月归档到对象存储   |
| 查询性能           | 按 tenant_id + 时间范围查询 < 200ms（100 万条内） |
| 哈希链断裂检测     | 任意一条日志被修改/删除，后续所有哈希断裂   |
| 三权分立           | 系统管理员/安全管理员/审计员三权分离，审计日志仅审计员可查 |

### 1.3 非目标

- 不在此模块实现审计报告生成（PDF 导出由 PDF 渲染模块负责）
- 不实现实时审计告警（二期，结合可观测性模块）
- 不实现用户行为分析/异常检测（二期 AI 模块）
- 不记录系统级操作审计（CronJob 执行日志等走 ELK）

## 2. 需求分析

### 2.1 功能需求

- [x] **AOP 自动采集**：`@AuditLog` 注解标记 Controller 方法，自动记录操作
- [x] **手动埋点**：特殊场景（如登录/登出）通过 `AuditLogger.log()` 手动记录
- [x] **审计查询**：按租户、用户、操作类型、目标类型、时间范围组合查询
- [x] **审计导出**：审计日志导出为 CSV（供合规审计用）
- [x] **哈希链**：每条日志含 `log_hash = SHA256(上一条 hash + 本条内容)`
- [x] **Merkle 树**：每小时生成根哈希存入 `audit_root_hash` 表
- [x] **完整性校验**：支持按小时范围验证日志未被篡改
- [x] **防删改**：数据库触发器、REVOKE UPDATE/DELETE 权限
- [x] **异步写入**：审计日志通过独立线程池异步写入，不阻塞主流程
- [x] **定期归档**：超过 6 个月的日志归档到对象存储或专用归档表

### 2.2 用户故事

```
作为 审计员，我希望 按时间段和用户查询所有操作记录，
以便 在合规检查时快速定位特定操作。

作为 安全管理员，我希望 系统能验证审计日志未被任何人篡改，
以便 满足等保三级对审计日志的防篡改要求。

作为 系统管理员，我希望 审计日志写入不影响业务接口的响应时间，
以便 合规要求不拖累系统性能。

作为 被审计的用户，我希望 我的合法操作有完整的记录和哈希链，
以便 在发生争议时有据可查。
```

### 2.3 边界场景

| 场景                       | 处理方式                                                     |
| -------------------------- | ------------------------------------------------------------ |
| 某条日志被 DBA 直接修改    | 触发器阻止（`prevent_audit_modify()`），操作失败             |
| Merkle 验证发现不一致       | 标记该时间段为"可能被篡改"，触发告警                         |
| 审计日志写入量激增         | 异步队列缓冲，RabbitMQ 解耦，消费者批量写入                  |
| audit_root_hash 表被篡改   | root_hash 与日志内容可独立校验（重新计算 Merkle 树比对）     |
| 审计员查询超大时间范围     | 强制分页 + 限制最大时间跨度 90 天，超过提示缩小范围          |
| 跨小时查询哈希链断裂       | 支持按小时分段校验，每小时的哈希链独立                       |
| 归档数据恢复查询           | 提供从对象存储恢复归档日志到热表的工具                       |
| 日志写入时服务重启         | 未到达批量阈值（1000 条或 5 秒）的日志丢失（允许少量丢失）  |

## 3. 架构设计

### 3.1 整体流程

```
用户操作（登录/创建报表/导出/审批/...）
        ↓
Controller 方法被调用（@AuditLog 标记 或 手动调用 AuditLogger）
        ↓
┌─────────────────────────────────────────────────────┐
│  AuditLogAspect（AOP 拦截器）                        │
│  ├── 1. 提取 @AuditLog 注解中的 operation/targetType │
│  ├── 2. 从 UserContext 获取 userId/username          │
│  ├── 3. 从 TenantContext 获取 tenantId               │
│  ├── 4. 从 HttpServletRequest 获取 IP/User-Agent     │
│  ├── 5. 序列化 requestParams（截断 2KB）             │
│  ├── 6. 执行目标方法                                 │
│  ├── 7. 成功 → 记录 status=SUCCESS                   │
│  ├── 8. 异常 → 记录 status=FAILED + errorMessage     │
│  └── 9. 异步投递 AuditLogEvent 到内存队列             │
└───────────────────┬─────────────────────────────────┘
                    ↓
AuditLogWriter（独立线程消费内存队列）
  ├── 1. 聚合 1000 条或等待 5 秒
  ├── 2. 批量写入 audit_log 表（JDBC batchInsert）
  └── 3. 返回（不等待 Merkle 树计算）
                    ↓
MerkleTreeBuilder（每小时 Cron 触发）
  ├── 1. 查询该小时内所有 audit_log 记录
  ├── 2. 计算 Merkle 根哈希
  ├── 3. 写入 audit_root_hash 表
  └── 4. 如果上小时根哈希 ≠ 数据库记录 → 告警
```

### 3.2 AOP 自动采集

```java
// @AuditLog 注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface AuditLog {
    String operation();    // LOGIN / CREATE_REPORT / EXPORT / SUBMIT_REVIEW / ...
    String targetType() default "";  // REPORT / USER / DATASOURCE / TASK / ...
}

// 使用示例
@AuditLog(operation = "SUBMIT_REVIEW", targetType = "REPORT")
@PostMapping("/reports/{id}/submit")
public Result<Void> submitForReview(@PathVariable Long id, @RequestBody SubmitRequest req) {
    return Result.success(reportWorkflowService.submitForReview(id, req));
}
```

**自动提取的通用字段**：

| 字段            | 来源                          |
| --------------- | ----------------------------- |
| `tenantId`      | `TenantContext.getTenantId()`  |
| `userId`        | `UserContext.getUserId()`      |
| `username`      | `UserContext.getUsername()`    |
| `operation`     | `@AuditLog.operation()`       |
| `targetType`    | `@AuditLog.targetType()`      |
| `targetId`      | 方法参数自动提取（首个 Long 参数） |
| `ipAddress`     | `HttpServletRequest.getRemoteAddr()` |
| `userAgent`     | `HttpServletRequest.getHeader("User-Agent")` |
| `requestParams` | 方法参数 JSON 序列化（截断 2KB） |
| `status`        | 方法正常返回 = SUCCESS，抛出异常 = FAILED |

### 3.3 手动埋点

部分场景不适合 AOP（如登录：用户未认证时 `UserContext` 为空），使用手动埋点：

```java
// AuditLogger.java
public final class AuditLogger {

    private static final BlockingQueue<AuditLogEvent> queue = new LinkedBlockingQueue<>(10000);

    public static void log(String operation, String targetType, Long targetId, String targetName,
                           String status, String errorMessage) {
        AuditLogEvent event = AuditLogEvent.builder()
            .tenantId(TenantContext.getTenantId())
            .userId(UserContext.getUserId())
            .username(UserContext.getUsername())
            .operation(operation)
            .targetType(targetType)
            .targetId(targetId)
            .targetName(targetName)
            .ipAddress(RequestContext.getIp())
            .userAgent(RequestContext.getUserAgent())
            .status(status)
            .errorMessage(errorMessage)
            .build();
        queue.offer(event);  // 非阻塞入队
    }
}

// 使用示例（AuthService.login 中）
AuditLogger.log("LOGIN", "USER", user.getId(), user.getUsername(), "SUCCESS", null);
```

## 4. 防篡改设计

### 4.1 数据库层防护

```sql
-- 触发器禁止 UPDATE/DELETE（已在 DDL 中实现）
CREATE OR REPLACE FUNCTION prevent_audit_modify() RETURNS TRIGGER AS $$
BEGIN
    RAISE EXCEPTION '审计日志不可修改或删除';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER audit_log_no_update
    BEFORE UPDATE OR DELETE ON audit_log
    FOR EACH ROW EXECUTE FUNCTION prevent_audit_modify();

-- 撤销 UPDATE/DELETE 权限（双重保险）
REVOKE UPDATE, DELETE ON audit_log FROM PUBLIC;
REVOKE UPDATE, DELETE ON audit_root_hash FROM PUBLIC;
```

### 4.2 哈希链

每条日志的 `log_hash` 包含前一条日志的哈希，形成单向链式结构：

```
log_hash(N) = SHA256(log_hash(N-1) + "|" + tenant_id + "|" + user_id + "|" + operation + "|" + created_at)
```

```java
// AuditLogWriter 批量写入时计算哈希链
String previousHash = auditLogMapper.getLastHash(tenantId, hourStart);

for (AuditLogEvent event : batch) {
    String raw = previousHash + "|" + event.getTenantId() + "|" + event.getUserId()
               + "|" + event.getOperation() + "|" + event.getCreatedAt();
    String hash = HashUtils.sha256(raw);
    event.setLogHash(hash);
    previousHash = hash;
}

auditLogMapper.batchInsert(batch);
```

**篡改检测**：如果任意一条日志被修改，该条及之后所有日志的 `log_hash` 都会断裂。只需重新计算哈希链即可发现断裂点。

### 4.3 Merkle 哈希树

每小时生成一棵 Merkle 树，根哈希存入 `audit_root_hash` 表：

```
每小时审计日志（如 1200 条）：
  [log-1] [log-2] [log-3] [log-4] ... [log-1200]
    ↓     ↓       ↓       ↓              ↓
  hash1  hash2  hash3  hash4  ...     hash1200
    └──┬──┘       └──┬──┘                └──┬──┘
     pair12       pair34              pair1199-1200
       └─────┬──────┘                      │
          level2-1                    level2-600
               └──────────┬───────────────┘
                         ...
                          ↓
                      ROOT HASH
```

```java
// MerkleTreeBuilder.java
@Scheduled(cron = "0 0 * * * *")  // 每小时整点
public void buildMerkleTree() {
    LocalDateTime hourStart = LocalDateTime.now().minusHours(1).truncatedTo(ChronoUnit.HOURS);
    LocalDateTime hourEnd = hourStart.plusHours(1);

    for (Long tenantId : tenantMapper.selectActiveIds()) {
        // 1. 查询该小时所有审计日志
        List<AuditLogEntry> logs = auditLogMapper.selectByHour(tenantId, hourStart, hourEnd);
        if (logs.isEmpty()) continue;

        // 2. 计算 Merkle 根哈希
        String rootHash = computeMerkleRoot(logs);

        // 3. 写入 audit_root_hash
        AuditRootHash entry = new AuditRootHash();
        entry.setTenantId(tenantId);
        entry.setHourStart(hourStart);
        entry.setRootHash(rootHash);
        entry.setLogCount(logs.size());
        auditRootHashMapper.insert(entry);
    }
}
```

### 4.4 完整性校验

```java
// AuditQueryService.java
public IntegrityCheckResult verifyIntegrity(Long tenantId, LocalDateTime hourStart) {
    // 1. 从 audit_root_hash 获取记录的根哈希
    AuditRootHash stored = auditRootHashMapper.selectByHour(tenantId, hourStart);
    if (stored == null) return new IntegrityCheckResult(false, "该小时无根哈希记录");

    // 2. 重新查询该小时日志并计算 Merkle 根
    List<AuditLogEntry> logs = auditLogMapper.selectByHour(tenantId, hourStart, hourStart.plusHours(1));
    String computedRoot = computeMerkleRoot(logs);

    // 3. 比对
    boolean consistent = stored.getRootHash().equals(computedRoot);
    return new IntegrityCheckResult(consistent,
        consistent ? "审计日志完整，未发现篡改" : "审计日志可能被篡改！根哈希不一致");
}
```

## 5. 数据库设计

### 5.1 audit_log 表

```sql
CREATE TABLE audit_log (
    id              BIGSERIAL PRIMARY KEY,

```

### 5.2 audit_root_hash 表

已在 DDL 中创建（`sql/ddl/01_init_schema.sql` §6），无需变更：

```sql
CREATE TABLE audit_root_hash (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       BIGINT NOT NULL,
    hour_start       TIMESTAMP NOT NULL,
    root_hash       VARCHAR(64) NOT NULL,
    log_count       INTEGER NOT NULL,
    created_at      TIMESTAMP DEFAULT NOW(),
    UNIQUE(tenant_id, hour_start)
);
```

### 5.3 索引设计

```sql
-- 已有索引
CREATE INDEX idx_audit_tenant_time ON audit_log(tenant_id, created_at DESC);
CREATE INDEX idx_audit_user ON audit_log(user_id);

-- 补充索引（按操作类型筛选）
CREATE INDEX idx_audit_operation ON audit_log(tenant_id, operation, created_at DESC);

-- 补充索引（按目标查所有操作）
CREATE INDEX idx_audit_target ON audit_log(target_type, target_id);

-- 按小时范围查（Merkle 树计算用）
CREATE INDEX idx_audit_hour ON audit_log(tenant_id, created_at)
    WHERE created_at >= date_trunc('hour', NOW() - INTERVAL '2 hours');
```

### 5.4 归档策略

```
audit_log 热数据：       创建日期 ≤ 6 个月  → 当前表
audit_log 归档数据：     创建日期 > 6 个月  → 迁移到 audit_log_archive 表 或 对象存储 CSV

归档步骤（CronJob 每天凌晨 3 点）：
  1. SELECT * FROM audit_log WHERE created_at < NOW() - INTERVAL '6 months'
  2. 导出为 CSV（保留哈希链完整性）
  3. 上传到对象存储（如 MinIO/S3）
  4. 验证 CSV 行数 = DB 行数
  5. DELETE FROM audit_log WHERE created_at < NOW() - INTERVAL '6 months'
     （注意：DELETE 被触发器阻止！需要用超级用户绕开触发器或使用分区表）
```

> 因触发器禁止 DELETE，建议方案是使用 PostgreSQL **表分区**（按月份分区），超过 6 个月的分区直接 `DETACH` + 归档文件 + `DROP TABLE`。

## 6. API 接口设计

### 6.1 接口列表

## 7. 异常处理

| 异常场景                     | 处理方式                                                     |
| ---------------------------- | ------------------------------------------------------------ |
| 审计日志写入失败（DB 不可用）| 内存队列满 10000 条后丢弃新事件，记录 ERROR 日志，触发告警  |
| 哈希链计算异常               | 标记该批次日志 `log_hash = 'ERROR'`，不中断主流程           |
| Merkle 树构建失败             | Cron 任务失败告警，下一小时重试（不影响当日志写入）          |
| 触发器阻止 DELETE（归档时）   | 使用超级用户绕开触发器，或使用表分区 `DETACH` 方案          |
| 审计员查询超大数据集          | 强制 `pageSize ≤ 100`，`dateRange ≤ 90 天`，超限返回 400    |
| userContext 为空（登录阶段）  | 登录阶段通过手动埋点 `AuditLogger.log()` 显式传入用户信息    |
| 内存队列积压 >80%            | 触发告警，自动增加批量写入频率（从 5s 降至 1s）             |

## 8. 性能考量

### 8.1 性能目标

| 指标                     | 目标            |
| ------------------------ | --------------- |
| 审计写入吞吐量           | ≥ 1000 TPS      |
| AOP 切入延迟             | < 1ms           |
| 审计日志查询（100 万条） | < 200ms         |
| Merkle 树构建（2000 条/小时） | < 2s        |
| 归档导出（100 万条）     | < 30 秒         |

### 8.2 性能优化点

1. **异步非阻塞写入**：AOP 切面仅将事件放入 `LinkedBlockingQueue`，不做 DB 写入，不阻塞业务请求
2. **批量插入**：每 1000 条聚合一次 `batchInsert`，单次 DB 往返
3. **Merkle 树计算延迟**：Merkle 树每小时异步计算一次，不影响日志写入延迟
4. **查询索引覆盖**：`idx_audit_tenant_time` 覆盖 90% 的查询场景，按月分区的归档表独立索引
5. **请求参数截断**：`requestParams` 和 `responseData` JSONB 字段截断到 2KB，防止单条日志过大
6. **分区表**：按月分区，查询时走分区裁剪，6 个月后直接分离分区

## 9. 安全考量

### 9.1 三权分立

| 角色           | 权限                                                       |
| -------------- | ---------------------------------------------------------- |
| **系统管理员** (ADMIN) | 管理租户/用户/系统配置，**不可**查看审计日志           |
| **安全管理员** | 配置安全策略、审计规则，**不可**查看审计日志              |
| **审计员** (AUDITOR) | **仅**查看/导出审计日志，校验完整性，无其他任何操作权限 |

> 当前角色体系中暂未实现"安全管理员"和"审计员"，使用 `ADMIN` 角色暂代。二期通过 `role` 字段扩展 `AUDITOR` 和 `SECURITY_ADMIN` 角色。

### 9.2 数据安全

- 审计日志表触发器 + REVOKE 双重禁止 UPDATE/DELETE
- `log_hash` 保证单条不可篡改，哈希链保证批量不可篡改，Merkle 根哈希保证整小时不可篡改
- 审计日志查询接口强制 `@PreAuthorize("hasRole('ADMIN')")`（过渡期），后续改为 `AUDITOR`
- 导出 CSV 时保留 `log_hash`，可在离线环境下验证完整性

### 9.3 防攻击

- **审计日志洪泛**：批量插入速率受限于 DB 连接池，内存队列满 10000 条后丢弃新事件，防止 OOM
- **查询拖库**：审计查询强制分页 + 时间范围限制，防止全表扫描
- **哈希碰撞**：SHA-256 碰撞概率可忽略不计（< 2^-256）
- **root_hash 表篡改**：root_hash 可从 `audit_log` 数据重新计算验证，与表中值比对即可

## 10. 测试方案

### 10.1 单元测试

| 测试用例                   | 说明                                         |

| 内存队列满溢出              | 验证队列满时丢弃 + 告警                     |

### 10.2 集成测试

- **完整采集链路**：调用带 `@AuditLog` 的 API → 验证 `audit_log` 表有记录 → 字段正确
- **手动埋点**：登录/登出 → 验证 LOGIN/LOGOUT 记录存在

- 审计日志表触发器不误伤 `audit_root_hash` 表的正常写入

## 11. 上线方案

### 11.1 上线步骤

1. **启用现有 AOP**：取消 `AuditLogAspect` 的 `@Component` 注释
2. **部署 common 模块**：新增 `AuditLogWriter` + `MerkleTreeBuilder` + `AuditLogEntry` 实体
3. **部署 api 模块**：新增 `AuditController` + `AuditQueryService` + `AuditLogMapper`
4. **授予审计查询权限**：为审计员角色开放 `/api/audit/**` 访问
5. **验证**：执行几个操作 → 检查 `audit_log` 有记录 → 等整点后检查 `audit_root_hash`
6. **执行完整性校验**：调用 `/api/audit/integrity` 确认通过

### 11.2 灰度策略

- 首日：AOP 仅拦截 `IMPORT_REPORT`、`EXPORT_REPORT` 两种操作
- 验证写入性能无影响后，次日扩展至全部 `@AuditLog` 标注的方法
- Merkle 树定时任务先以 WARN 日志代替 DB 写入，验证无性能问题后启用 DB 写入

### 11.3 监控告警

| 监控项                       | 阈值          | 告警方式        |
| ---------------------------- | ------------- | --------------- |
| 审计日志内存队列积压 > 80%   | > 8000 条     | 企业微信 + 邮件 |
| 审计日志写入失败率           | > 1%          | 企业微信        |
| Merkle 树构建失败             | 1 次          | 企业微信        |
| 完整性校验发现不一致          | 1 次          | 企业微信 + 短信 |
| audit_log 表行数增长速率     | > 50000/小时  | 企业微信        |

## 12. 回滚方案

### 12.1 回滚触发条件

- AOP 切入导致业务 API 响应时间异常增大
- 审计日志写入失败率高导致内存队列频繁溢出
- Merkle 树定时任务阻塞调度线程
- 审计日志查询拖慢数据库

### 12.2 回滚步骤

1. 注释 `AuditLogAspect` 的 `@Component` 注解（回退到禁用状态）
2. 停止 `MerkleTreeBuilder` 的 `@Scheduled` 定时任务
3. 回滚 `api` 模块到上一版本

### 12.3 数据补救

- 回滚期间已写入的 `audit_log` 记录：保留，不影响功能
- 已生成的 `audit_root_hash` 记录：保留，可用于后续校验
- 内存队列中未写入的审计事件：丢弃（审计日志允许少量丢失）
