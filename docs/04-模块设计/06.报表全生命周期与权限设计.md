## 1. 背景与目标

### 1.1 问题背景

当前系统存在以下结构性问题：

| 问题                                   | 严重度 | 影响                                       |
| -------------------------------------- | ------ | ------------------------------------------ |
| 报表无状态管理                         | 🔴 致命 | 无法区分草稿/审查中/已公开的报表           |
| 无工作流审批机制                       | 🔴 致命 | 报表创建后直接可见，缺少审查环节           |
| 权限模型空白                           | 🔴 致命 | `SecurityConfig` 仅区分是否登录，不区分角色 |
| `report_workflows` 表已建但无代码      | 🔴 致命 | 状态列无 CHECK 约束，无状态机实现          |
| `report_metadata` 缺少 `status` 字段   | 🟡 高   | 查询报表概览时需 JOIN workflow 表          |
| ReportMapper 使用硬删除                | 🟡 高   | 违反 DB 设计规范中的"不允许物理删除"原则   |
| 实体类 `Report.java` 无 `status` 字段  | 🟡 高   | 与 DB 设计脱节                             |

### 1.2 设计目标

| 指标             | 要求                                             |
| ---------------- | ------------------------------------------------ |
| 报表状态管理     | 5 态状态机（DRAFT → PENDING_REVIEW → APPROVED → PUBLISHED，含 REJECTED 分支） |
| 状态流转控制     | Java 代码强校验，禁止非法跳转                    |
| 权限粒度         | 操作级（CRUD + 提交/审批/驳回/公开）             |
| 元数据版本控制   | `schema_version` 递增，历史报表可正确渲染        |
| 审查人指派       | 提交审查时指定，管理员可修改                     |
| 审计全覆盖       | 所有状态变更、权限操作记录审计日志               |

### 1.3 非目标

- 不实现多级审批链（一级审批即可）
- 不实现审批超时自动驳回（二期）
- 不实现审批代理（A 不在时 B 代审）
- 不实现审批条件分支（仅支持 approve/reject 二元决策）
- 不在此模块实现动态表 DDL 变更审批（那是数据安全模块的职责）

## 2. 需求分析

### 2.1 功能需求

- [x] **报表 CRUD**：创建、查询、更新、删除（软删除）报表
- [x] **状态机引擎**：严格的状态流转控制（DRAFT → PENDING_REVIEW → APPROVED/REJECTED → PUBLISHED）
- [x] **审查流程**：提交审查 → 审查人审批/驳回 → 驳回后回退至 DRAFT
- [x] **版本控制**：元数据变更时 `schema_version` 递增
- [x] **权限控制**：基于 JWT 角色（ADMIN/USER/VIEWER）+ 报表所有权 + 审查人指派的三层权限
- [x] **公开报表**：APPROVED 状态的报表可以发布为 PUBLISHED
- [x] **操作审计**：所有状态变更、权限操作自动记录审计日志
- [x] **列表过滤**：支持按状态、创建人、审查人筛选报表列表

### 2.2 用户故事

```
作为 报表制表人，我希望 创建报表草稿并在完成后提交审查，
以便 我的报表经过规范流程后才能公开发布。

作为 报表审查人，我希望 审查分配给自己的报表并批准或驳回，
以便 确保公开的报表质量符合标准。

作为 访客，我希望 只能看到已公开的报表列表和内容，
以便 系统按权限正确隔离未完成的报表。

作为 系统管理员，我希望 可以查看和管理所有报表，包括覆盖任何不良状态，
以便 在异常情况下可以介入处理。
```

### 2.3 边界场景

| 场景                             | 处理方式                                                     |
| -------------------------------- | ------------------------------------------------------------ |
| 非 creator 编辑他人报表          | 返回 403，记录审计日志                                       |
| 审查人不是 creator               | 系统保证审查人与提交人不同（自审禁止）                       |
| PENDING_REVIEW 状态编辑报表       | 禁止编辑，必须驳回后回到 DRAFT 才能编辑                      |
| 已公开报表删除                   | 不允许物理删除，改为下架（状态 → ARCHIVED 或软删 `is_deleted = true`） |
| 同一报表重复提交审查             | 仅 DRAFT 和 REJECTED 状态可提交                              |
| 审查人不存在或已禁用             | 提交审查时校验审查人状态，无效则拒绝                         |
| CREATOR 角色的用户不存在于该租户 | 创建报表时从 `TenantContext` 和 `UserContext` 提取，不依赖前端传入 |
| 并发修改报表（A编辑同时B提交审查）| 乐观锁：`updated_at` 作为版本号，更新时校验                  |
| 报表删除后重新创建同名           | `is_deleted = true` 的记录不影响新建（UNIQUE 约束不含 deleted） |

## 3. 架构设计

### 3.1 状态机设计

```
                 ┌──────────┐
                 │   DRAFT   │  ← 新建报表 / 被驳回后
                 └─────┬─────┘
                       │ submitForReview()
                       ↓
              ┌─────────────────┐
              │  PENDING_REVIEW  │
              └────┬────────┬───┘
      approve()    │        │  reject()
                   ↓        ↓
            ┌──────────┐  ┌───────────┐
            │ APPROVED  │  │ REJECTED   │
            └─────┬─────┘  └─────┬─────┘
                  │              │
       publish()  │   edit() → back to DRAFT
                  ↓              │
            ┌──────────┐  ┌──────┘
            │ PUBLISHED │←─┘
            └──────────┘
```

**合法的状态转换**：

| 当前状态         | 允许转换为                 | 触发操作     | 谁可以操作       |
| ---------------- | -------------------------- | ------------ | ---------------- |
| (新建)           | DRAFT                      | 创建报表     | USER / ADMIN     |
| DRAFT            | PENDING_REVIEW             | 提交审查     | creator          |
| DRAFT            | (删除 → is_deleted = true) | 软删除       | creator          |
| PENDING_REVIEW   | APPROVED                   | 审批通过     | reviewer         |
| PENDING_REVIEW   | REJECTED                   | 驳回         | reviewer         |
| REJECTED         | DRAFT                      | 编辑（自动） | creator          |
| REJECTED         | PENDING_REVIEW             | 重新提交审查 | creator          |
| APPROVED         | PUBLISHED                  | 公开发布     | creator / ADMIN  |
| APPROVED         | REJECTED                   | 下架         | ADMIN            |
| PUBLISHED        | REJECTED                   | 下架         | ADMIN            |

**禁止的非法转换（示例）**：

| 非法转换             | 拒绝原因               |
| -------------------- | ---------------------- |
| DRAFT → PUBLISHED    | 必须先经过审查         |
| PENDING_REVIEW → DRAFT| 必须先驳回             |
| PUBLISHED → DRAFT    | 公开后不可直接退回草稿 |
| APPROVED → DRAFT     | 必须先驳回             |

**状态枚举**：

```java
public enum ReportStatus {
    DRAFT("草稿"),
    PENDING_REVIEW("待审查"),
    APPROVED("已通过"),
    REJECTED("已驳回"),
    PUBLISHED("已公开");

    private final String label;
}
```

### 3.2 整体流程

**创建报表**：

```
POST /api/reports
Authorization: Bearer <token>  (role = USER or ADMIN)
        ↓
ReportController.create()
        ↓
ReportService.createReport()
  ├── 1. 从 UserContext 提取 userId, tenantId
  ├── 2. 校验租户状态（ACTIVE）
  ├── 3. 校验用户状态（ACTIVE）
  ├── 4. 构建 Report 实体（status = DRAFT, schema_version = 1）
  ├── 5. reportMapper.insert(report)
  ├── 6. workflowMapper.insert(workflow)  // 初始状态 DRAFT
  └── 7. 记录审计日志（operation = CREATE_REPORT）
        ↓
返回 Report + 当前 workflow 状态
```

**提交审查**：

```
POST /api/reports/{id}/submit
        ↓
ReportService.submitForReview()
  ├── 1. 查询报表 + 当前 workflow
  ├── 2. 校验当前状态为 DRAFT 或 REJECTED
  ├── 3. 校验操作者为 creator
  ├── 4. 校验 reviewerId 有效（≠ creator，状态 ACTIVE，同租户）
  ├── 5. 校验 reviewerId 角色为 USER 或 ADMIN（VIEWER 不可审查）
  ├── 6. 更新 workflow：status = PENDING_REVIEW, reviewed_by = reviewerId
  ├── 7. schema_version += 1（元数据变更）
  ├── 8. 插入新的 workflow 记录（保留历史轨迹）
  └── 9. 记录审计日志（operation = SUBMIT_REVIEW）
        ↓
返回更新后的报表 + workflow
```

**审批/驳回**：

```
PUT /api/reports/{id}/review
{ "action": "APPROVE", "comment": "数据准确，同意公开" }
        ↓
ReportService.reviewReport()
  ├── 1. 查询报表 + 当前 workflow
  ├── 2. 校验当前状态为 PENDING_REVIEW
  ├── 3. 校验操作者为 reviewed_by 指定的审查人（或 ADMIN）
  ├── 4. 更新 workflow：status = APPROVED / REJECTED
  ├── 5. 记录审批意见（review_comment）
  ├── 6. reviewed_at = NOW()
  ├── 7. 插入新的 workflow 记录
  └── 8. 记录审计日志（operation = APPROVE_REPORT / REJECT_REPORT）
        ↓
返回更新后的报表 + workflow
```

### 3.4 关键算法与策略

**状态机校验算法**：

```java
// ReportWorkflowService.java
private static final Map<ReportStatus, Set<ReportStatus>> ALLOWED_TRANSITIONS = Map.of(
    null,                           Set.of(DRAFT),               // 新建
    DRAFT,                          Set.of(PENDING_REVIEW),       // 提交
    PENDING_REVIEW,                 Set.of(APPROVED, REJECTED),   // 审批
    REJECTED,                       Set.of(DRAFT, PENDING_REVIEW),// 编辑/重新提交
    APPROVED,                       Set.of(PUBLISHED, REJECTED),  // 发布/下架
    PUBLISHED,                      Set.of(REJECTED)              // 下架
);

public void validateTransition(ReportStatus from, ReportStatus to) {
    if (!ALLOWED_TRANSITIONS.getOrDefault(from, Set.of()).contains(to)) {
        throw new BusinessException(ErrorCode.WORKFLOW_TRANSITION_DENIED,
            "不允许从 " + from + " 转换为 " + to);
    }
}
```

**乐观锁并发控制**：

```java
// ReportMapper.xml
<update id="update">
    UPDATE report_metadata
    SET name = #{name}, description = #{description},
        schema_version = schema_version + 1,
        updated_at = NOW()
    WHERE id = #{id}
      AND tenant_id = #{tenantId}
      AND updated_at = #{expectedUpdatedAt}   <!-- 乐观锁 -->
</update>

// 在 Service 层检查影响行数
int rows = reportMapper.update(report);
if (rows == 0) {
    throw new BusinessException(ErrorCode.CONCURRENT_MODIFICATION,
        "报表已被他人修改，请刷新后重试");
}
```

**版本控制策略**：

```
触发 schema_version 递增的操作：
  - 报表元数据编辑（name, description, template_config 变更）
  - 提交审查（报表内容冻结，版本号标记）
  - 模板配置变更

不触发版本递增的操作：
  - 状态流转本身（仅 workflow 状态变化）
  - 审查人指派
  - 查看/查询
```

> `schema_version` 存储在 `report_metadata` 表中，作为乐观锁的一部分。版本递增即 `schema_version = schema_version + 1`。

## 4. 权限模型

### 4.1 角色定义

权限控制三层叠加：

**第一层：JWT 系统角色**（来自 `users.role`，写入 Token）

| 角色   | 编码   | Token Claim         |
| ------ | ------ | ------------------- |
| 管理员 | ADMIN  | `"role": "ADMIN"`   |
| 普通用户 | USER  | `"role": "USER"`    |
| 访客   | VIEWER | `"role": "VIEWER"`  |

**第二层：工作流参与关系**（来自 `report_metadata.created_by` 和 `report_workflows.reviewed_by`）

| 关系     | 判定条件                     | 别名   |
| -------- | ---------------------------- | ------ |
| 制表人   | `report.createdBy == userId` | creator |
| 审查人   | `workflow.reviewedBy == userId` 且 `workflow.status == PENDING_REVIEW` | reviewer |
| 其他人   | 以上都不匹配                 | —      |

**第三层：报表状态可见性**（来自 `report_workflows.status`）

| 报表状态       | creator | reviewer | 同租户 USER | 同租户 VIEWER | ADMIN |
| -------------- | ------- | -------- | ----------- | ------------- | ----- |
| DRAFT           | 可读写   | —        | —           | —             | 可读写 |
| PENDING_REVIEW  | 只读     | 可审批   | —           | —             | 可读写 |
| APPROVED        | 可读     | 可读     | —           | —             | 可读写 |
| REJECTED        | 可读写   | 可读     | —           | —             | 可读写 |
| PUBLISHED       | 可读     | 可读     | 可读        | 可读          | 可读写 |

### 4.2 权限矩阵

按操作维度展开：

| 操作         | ADMIN              | USER（creator）     | USER（reviewer）      | USER（其他）         | VIEWER              |
| ------------ | ------------------ | ------------------- | --------------------- | -------------------- | ------------------- |
| 创建报表     | ✓                  | ✓                   | ✓                     | ✓                    | ✗                   |
| 编辑报表     | ✓（所有状态）      | ✓（仅 DRAFT/REJECTED） | ✗                   | ✗                    | ✗                   |
| 软删除报表   | ✓（所有状态）      | ✓（仅 DRAFT/REJECTED） | ✗                   | ✗                    | ✗                   |
| 提交审查     | ✓（任何报表）      | ✓（仅自己的报表）   | ✗                     | ✗                    | ✗                   |
| 审批/驳回     | ✓（任何 PENDING）  | ✗（除非是 reviewer） | ✓（仅自己被指派的）   | ✗                    | ✗                   |
| 公开发布     | ✓（任何 APPROVED） | ✓（仅自己的 APPROVED）| ✗                    | ✗                    | ✗                   |
| 下架         | ✓                  | ✗                   | ✗                     | ✗                    | ✗                   |
| 查看列表     | ✓（全部）          | ✓（自己 + PUBLISHED）| ✓（被指派 + PUBLISHED）| ✓（仅 PUBLISHED）   | ✓（仅 PUBLISHED）   |
| 查看详情     | ✓（全部）          | 按状态可见性         | 按状态可见性          | 按状态可见性         | ✓（仅 PUBLISHED）   |
| 导出报表     | ✓                  | ✓（仅 PUBLISHED）   | ✓（仅 PUBLISHED）     | ✓（仅 PUBLISHED）   | ✗                   |

### 4.3 权限校验流程

```
请求进入 Controller
        ↓
@PreAuthorize("hasRole('USER')")     ← 第一层：JWT 角色校验
        ↓
Controller 调用 Service
        ↓
Service 中调用 PermissionService
        ↓
ReportPermissionService.checkPermission(userId, report, action)
  ├── 1. 如果是 ADMIN → 直接通过
  ├── 2. 查询当前 workflow 状态
  ├── 3. 根据 action + status + 用户关系判定
  ├── 4. 判定不通过 → throw BusinessException(FORBIDDEN)
  └── 5. 判定通过 → 继续执行业务逻辑
        ↓
操作完成 → 记录审计日志
```

```java
// ReportPermissionService.java
public void checkPermission(Long userId, Report report, ReportAction action) {
    // 管理员豁免
    if (isAdmin(userId)) return;
    
    ReportStatus status = getCurrentStatus(report.getId());
    boolean isCreator = report.getCreatedBy().equals(userId);
    boolean isReviewer = isAssignedReviewer(report.getId(), userId);
    
    boolean allowed = switch (action) {
        case EDIT, DELETE -> isCreator
            && (status == DRAFT || status == REJECTED);
        case SUBMIT_REVIEW -> isCreator
            && (status == DRAFT || status == REJECTED);
        case REVIEW -> isReviewer && status == PENDING_REVIEW;
        case PUBLISH -> isCreator && status == APPROVED;
        case UNPUBLISH -> false;  // 仅 ADMIN
        case VIEW -> canView(status, isCreator, isReviewer);
        case EXPORT -> status == PUBLISHED;
    };
    
    if (!allowed) {
        throw new BusinessException(ErrorCode.FORBIDDEN);
    }
}
```

### 4.4 审查人指派

- 提交审查时由 creator 指定 `reviewerId`
- 审查人必须是同租户下状态为 ACTIVE 的用户
- 审查人角色不能是 VIEWER（VIEWER 无审查权限）
- **自审禁止**：`reviewerId ≠ creatorId`
- 管理员可以在 PENDING_REVIEW 状态下变更审查人（`PUT /api/reports/{id}/reviewer`）
- 审查人指派记录在 `report_workflows.reviewed_by`

## 5. 数据库设计

### 5.1 表结构变更

**变更 1：`report_metadata` 增加 `status`、`is_deleted`、`updated_by`**

```sql
-- 增加报表当前状态字段（冗余，避免每次查 workflow）
ALTER TABLE report_metadata ADD COLUMN status VARCHAR(32) NOT NULL DEFAULT 'DRAFT';
ALTER TABLE report_metadata ADD COLUMN is_deleted BOOLEAN NOT NULL DEFAULT FALSE;
ALTER TABLE report_metadata ADD COLUMN updated_by BIGINT;

COMMENT ON COLUMN report_metadata.status IS '当前状态：DRAFT/PENDING_REVIEW/APPROVED/REJECTED/PUBLISHED';
COMMENT ON COLUMN report_metadata.is_deleted IS '逻辑删除标记';
```

> `status` 字段作为 `report_workflows` 最新状态的冗余副本。每次 workflow 变更时同步更新。这避免了查询报表列表时必须 JOIN workflow 表的性能问题。

**变更 2：`report_workflows` 增加 CHECK 约束和 `updated_by`**

```sql
-- 约束状态值为合法枚举
ALTER TABLE report_workflows
    ADD CONSTRAINT chk_workflow_status
    CHECK (status IN ('DRAFT', 'PENDING_REVIEW', 'APPROVED', 'REJECTED', 'PUBLISHED'));

-- 增加操作人字段
ALTER TABLE report_workflows ADD COLUMN updated_by BIGINT;

COMMENT ON COLUMN report_workflows.updated_by IS '最后操作人（提交/审批/驳回者）';
```

**变更 3：`report_workflows` 增加外键约束（可选）**

```sql
-- 审查人外键（不强约束，仅用于数据完整性提示）
ALTER TABLE report_workflows
    ADD CONSTRAINT fk_workflow_reviewer
    FOREIGN KEY (reviewed_by) REFERENCES users(id)
    ON DELETE SET NULL;
```

### 5.2 索引设计

```sql
-- 按状态查询报表列表（高频）
CREATE INDEX idx_report_status ON report_metadata(tenant_id, status, created_at DESC);

-- 软删除过滤（高频，所有查询都加 is_deleted = false）
CREATE INDEX idx_report_not_deleted ON report_metadata(tenant_id, is_deleted, created_at DESC);

-- 按创建人查询（我的报表）
CREATE INDEX idx_report_created_by ON report_metadata(created_by, status);

-- 工作流按报表 ID 查最新状态（高频）
CREATE INDEX idx_workflow_report_status ON report_workflows(report_id, created_at DESC);

-- 按审查人查询待审查报表（待办列表）
CREATE INDEX idx_workflow_reviewer_status ON report_workflows(reviewed_by, status)
    WHERE status = 'PENDING_REVIEW';
```

### 5.3 数据迁移方案

**存量数据处理**：

```sql
-- Step 1: 为现有报表数据填充默认状态
UPDATE report_metadata SET status = 'DRAFT' WHERE status IS NULL;
UPDATE report_metadata SET is_deleted = FALSE WHERE is_deleted IS NULL;

-- Step 2: 为现有报表创建初始 workflow 记录（如果 report_workflows 为空）
INSERT INTO report_workflows (tenant_id, report_id, status, created_by, updated_by)
SELECT tenant_id, id, COALESCE(status, 'DRAFT'), created_by, created_by
FROM report_metadata
WHERE id NOT IN (SELECT report_id FROM report_workflows);

-- Step 3: 提交 Flyway 迁移脚本 V2__add_report_status.sql
```

> 存量的 `SELECT *` 查询需要改为显式列名 + `AND is_deleted = false` 过滤。

## 7. 异常处理

| 异常场景                 | HTTP 码 | 错误码 | 提示信息                           | 处理方式                           |
| ------------------------ | ------- | ------ | ---------------------------------- | ---------------------------------- |
| 报表不存在               | 404     | 2001   | 报表不存在                         | 记录审计日志                       |
| 报表已被软删除           | 404     | 2002   | 报表不存在或已删除                 | 不暴露软删除状态                   |
| 无权限访问               | 403     | 403    | 无权限                             | 记录审计日志                       |
| 非法的状态转换           | 400     | 2003   | 当前状态不允许此操作（from → to）   | 返回当前状态和期望的目标状态       |
| 审查人无效               | 400     | 2004   | 审查人不存在、已禁用或无权审查     | 提示排查审查人账号状态             |
| 不能自审                 | 400     | 2005   | 制表人和审查人不能是同一人         | 提示更换审查人                     |
| 并发修改冲突             | 409     | 2006   | 报表已被他人修改，请刷新后重试     | 返回当前最新数据                   |
| PENDING_REVIEW 状态编辑   | 400     | 2007   | 审查中的报表不可编辑，请先驳回     | 提示驳回后编辑                     |
| 非 creator 提交审查      | 403     | 403    | 无权限提交审查                     | 记录审计日志                       |

## 8. 性能考量

### 8.1 性能目标

| 场景                     | 指标      |
| ------------------------ | --------- |
| 报表创建                 | < 100ms   |
| 状态转换                 | < 50ms    |
| 列表查询（1000 条数据）  | < 200ms   |
| 并发状态变更             | 乐观锁无冲突 |
| workflow 历史查询        | < 50ms    |

### 8.2 性能优化点

1. **状态冗余**：`report_metadata.status` 避免每次列表查询都 JOIN `report_workflows`，以少量冗余换取查询性能
2. **覆盖索引**：`idx_report_status(tenant_id, status, created_at DESC)` 使列表查询走覆盖索引，无需回表
3. **workflow 历史查询**：`idx_workflow_report_status(report_id, created_at DESC)` 保证历史轨迹查询走索引
4. **乐观锁优于悲观锁**：状态变更场景冲突概率低，使用 `updated_at` 乐观锁避免数据库行锁
5. **待审查报表查询**：`idx_workflow_reviewer_status` 使用部分索引（WHERE status = 'PENDING_REVIEW'），大幅减少索引体积
6. **分页查询**：所有列表接口强制分页（默认 20 条，最大 100 条），防止全表扫描

**列表查询性能估算**：

```
SELECT * FROM report_metadata
WHERE tenant_id = ? AND is_deleted = false AND status = ?
ORDER BY created_at DESC LIMIT 20;

  → 使用 idx_report_not_deleted 或 idx_report_status
  → 索引扫描 ≈ 20 行 + 回表 20 次 ≈ 2ms ✅
```

## 9. 安全考量

### 9.1 权限控制

- 所有报表操作依赖 JWT Filter 注入的 `UserContext.getUserId()` 和 `TenantContext.getTenantId()`
- `createdBy` 字段由 `UserContext` 填充，不接受前端传入（防伪造 creator）
- `tenantId` 通过 MyBatis TenantInterceptor 自动注入（与 `report_metadata.tenant_id` 一致）
- 审查人指派时校验 `reviewer.tenantId == currentUser.tenantId`（跨租户审查禁止）
- 管理员通过 `@SkipTenantFilter` 可跨租户操作，但必须记录审计日志

### 9.2 数据安全

- 所有报表操作记录审计日志（create/update/delete/submit/review/publish/unpublish）
- 软删除保留数据，不物理删除（满足合规要求）
- workflow 历史记录不可修改（仅 INSERT，不 UPDATE 历史记录）
- `review_comment` 可能包含敏感信息，写入前校验长度（最长 1000 字）

### 9.3 防攻击

- **Creator 伪造**：`createdBy` 从 `UserContext` 读取，不从请求体获取
- **越权审批**：审批前校验 `workflow.reviewedBy == currentUserId || currentUser.role == ADMIN`
- **重复提交审查**：Service 层加 `@Transactional` + 唯一约束保护
- **并发状态修改**：乐观锁 + 数据库事务保证状态一致性
- **SQL 注入**：所有 SQL 通过 MyBatis `#{}` 预编译
- **批量操作**：不提供批量状态变更接口，防止误操作影响面扩大

## 12. 回滚方案

### 12.1 回滚触发条件

- 状态机存在致命 Bug（如 PUBLISHED 报表可被非 ADMIN 回退到 DRAFT）
- 权限校验大面积失效（普通用户可查看/编辑他人报表）
- 软删除逻辑导致查询性能大幅下降（列表查询 > 1s）
- workflow 历史表写入异常导致数据库压力过大

### 12.2 回滚步骤

1. 关闭功能开关（配置 `jinshu.report.workflow-enabled = false`，API 层对 workflow 接口返回 503）
2. 回滚 `api` 和 `common` 模块到上一版本
3. DDL 变更不回滚（新增字段不影响旧代码运行，无需回退表结构）
4. 通知用户：工作流功能临时维护

### 12.3 数据补救

- 回滚期间新创建的 `report_workflows` 记录：保留，不影响旧代码
- 新增字段 `report_metadata.status`：保留默认值 `DRAFT`，旧代码不读取该字段
- `report_metadata.is_deleted`：回滚后旧代码可能误查到软删除数据，需排查
