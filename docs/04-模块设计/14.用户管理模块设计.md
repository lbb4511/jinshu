## 1. 背景与目标

### 1.1 问题背景

`users` 表已在 DDL 中定义，JWT 认证授权设计已覆盖认证流程，但用户 CRUD、角色分配、状态管理等业务管理功能缺少模块级设计文档。

> **与 JWT 认证设计的边界**：`03.JWT认证授权设计.md` 负责"用户如何登录和获得 Token"，本模块负责"用户账户本身的 CRUD 管理"。

### 1.2 设计目标

| 指标     | 要求                                      |
| -------- | ----------------------------------------- |
| 密码安全 | BCrypt/PBKDF2 哈希，不可逆                |
| 角色管理 | 5 角色 RBAC，角色分配需租户管理员以上权限 |
| 用户隔离 | 用户仅属于一个租户，不可跨租户            |
| 配额限制 | 创建用户时校验租户用户数配额              |
| 审计覆盖 | 所有 CUD + 角色变更操作 100% 记录审计日志 |
| 登录安全 | 连续 5 次失败锁定 30 分钟                 |

### 1.3 非目标

- 不在此模块实现 JWT Token 签发/验证（由认证模块负责）
- 不实现 SSO / OAuth2 第三方登录（二期）
- 不实现用户行为分析/异常检测（二期 AI 模块）

## 2. 需求分析

### 2.1 功能需求

- [x] **创建用户**：设置用户名、密码、角色、邮箱（可选）
- [x] **编辑用户**：修改角色、邮箱、姓名，不支持修改用户名
- [x] **启用/禁用用户**：禁用后无法登录，已登录的 Token 加入黑名单
- [x] **重置密码**：管理员强制重置用户密码（需二次确认）
- [x] **修改密码**：用户自行修改密码（需旧密码验证）
- [x] **用户列表**：支持按用户名搜索、按角色筛选、分页
- [x] **用户详情**：含基本信息、角色、最后登录时间/IP
- [x] **登录失败锁定**：连续失败 5 次锁定 30 分钟
- [x] **角色管理**：分配/变更角色，含权限说明
- [x] **审计记录**：所有 CUD + 角色变更 100% 记录

### 2.2 用户故事

```
作为 租户管理员，我希望 创建新用户并分配"报表设计者"角色，
以便 团队成员可以设计报表。

作为 租户管理员，我希望 禁用离职员工的账号，
以便 确保系统安全。

作为 普通用户，我希望 修改自己的密码，
以便 定期更新密码保障安全。
```

### 2.3 边界场景

| 场景                   | 处理方式                                       |
| ---------------------- | ---------------------------------------------- |
| 用户名已存在(同租户)   | 返回错误"用户名已存在"                         |
| 创建用户达到配额上限   | 返回错误"用户数已达上限(N/N)"                  |
| 连续登录失败           | 5 次失败后锁定 30 分钟，记录安全事件到审计日志 |
| 管理员禁用自己         | 禁止操作，返回"不能禁用自己的账号"             |
| 删除最后一个租户管理员 | 禁止操作，返回"必须保留至少一个租户管理员"     |
| 角色变更影响当前权限   | 变更后强制用户重新登录                         |

## 3. 架构设计

### 3.1 整体流程

```
租户管理员
  │
  ▼
UserController  ── 参数校验、@AuditLog 自动采集
  │
  ▼
UserService  ── 业务逻辑、密码哈希、配额校验、角色管理
  │
  ├──► UserMapper  ── 数据库 CRUD
  ├──► PasswordService  ── BCrypt 哈希、密码强度校验
  ├──► QuotaService  ── 用户数配额校验
  ├──► TokenBlacklistService  ── 禁用用户时使 Token 失效
  └──► AuditLogger  ── 审计日志
```

### 3.2 模块划分

| 模块                | 职责                          | 所属模块 |
| ------------------- | ----------------------------- | -------- |
| UserController      | 参数校验、调用 Service        | `api/`   |
| UserService         | 用户 CRUD、角色管理、状态管理 | `api/`   |
| PasswordService     | BCrypt 哈希、密码强度校验     | `api/`   |
| LoginAttemptService | 登录失败计数、锁定判断        | `api/`   |
| UserMapper          | 数据库 CRUD                   | `api/`   |

## 4. 数据库设计

### 4.1 users 表

现有 DDL 已定义，模块设计补充说明：

| 字段                  | 类型            | 说明                                                                   |
| --------------------- | --------------- | ---------------------------------------------------------------------- |
| `id`                  | BIGSERIAL PK    | 主键                                                                   |
| `tenant_id`           | BIGINT NOT NULL | 租户 ID                                                                |
| `username`            | VARCHAR(100)    | 用户名（租户内唯一）                                                   |
| `password_hash`       | VARCHAR(255)    | BCrypt 哈希密码                                                        |
| `email`               | VARCHAR(200)    | 邮箱（可选）                                                           |
| `display_name`        | VARCHAR(100)    | 显示名称                                                               |
| `role`                | VARCHAR(50)     | 角色：SUPER_ADMIN / TENANT_ADMIN / REPORT_DESIGNER / REVIEWER / VIEWER |
| `status`              | VARCHAR(20)     | ACTIVE / DISABLED / LOCKED                                             |
| `login_fail_count`    | INTEGER         | 连续登录失败次数                                                       |
| `locked_until`        | TIMESTAMP       | 锁定到何时                                                             |
| `last_login_at`       | TIMESTAMP       | 最后登录时间                                                           |
| `last_login_ip`       | VARCHAR(64)     | 最后登录 IP                                                            |
| `password_updated_at` | TIMESTAMP       | 密码最后修改时间（用于密码过期策略）                                   |
| `created_at`          | TIMESTAMP       | 创建时间                                                               |
| `updated_at`          | TIMESTAMP       | 更新时间                                                               |

### 4.2 用户与角色映射

当前采用单角色设计（`role` 字段直接存储），理由：

- 系统角色层级清晰，一个用户同一时间只需一个角色
- 避免多角色权限合并的复杂判断

如需多角色支持（二期），可引入 `user_role` 关联表。

### 4.3 索引设计

```sql
CREATE UNIQUE INDEX uk_users_tenant_username ON users(tenant_id, username);
CREATE INDEX idx_users_tenant_id ON users(tenant_id);
CREATE INDEX idx_users_role ON users(role);
CREATE INDEX idx_users_status ON users(status);
```

## 5. API 接口设计

### 5.1 接口列表

| 方法  | 路径                             | 说明         | 权限           |
| ----- | -------------------------------- | ------------ | -------------- |
| POST  | `/api/users`                     | 创建用户     | 租户管理员以上 |
| GET   | `/api/users`                     | 用户列表     | 租户管理员以上 |
| GET   | `/api/users/{id}`                | 用户详情     | 租户管理员以上 |
| PUT   | `/api/users/{id}`                | 编辑用户     | 租户管理员以上 |
| PATCH | `/api/users/{id}/status`         | 变更状态     | 租户管理员以上 |
| POST  | `/api/users/{id}/reset-password` | 重置密码     | 租户管理员以上 |
| PUT   | `/api/users/password`            | 修改自己密码 | 登录用户       |
| GET   | `/api/users/me`                  | 当前用户信息 | 登录用户       |

### 5.2 请求/响应示例

**创建用户**：

```json
// POST /api/users
{
  "username": "zhangsan",
  "password": "Str0ng!P@ss",
  "displayName": "张三",
  "email": "zhangsan@example.com",
  "role": "REPORT_DESIGNER"
}

// 响应
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 2001,
    "username": "zhangsan",
    "displayName": "张三",
    "email": "zhangsan@example.com",
    "role": "REPORT_DESIGNER",
    "status": "ACTIVE",
    "createdAt": "2026-05-18T14:30:00+08:00"
  }
}
```

**修改自己密码**：

```json
// PUT /api/users/password
{
  "oldPassword": "OldP@ss1",
  "newPassword": "NewP@ss2"
}

// 响应
{
  "code": 0,
  "message": "密码修改成功，请重新登录"
}
```

## 6. 异常处理

| 场景                 | 错误码 | HTTP 状态码 | 处理方式                            |
| -------------------- | ------ | ----------- | ----------------------------------- |
| 用户名重复           | 40900  | 409         | 返回"用户名已存在"                  |
| 用户数达配额上限     | 42901  | 429         | 返回当前/上限数量                   |
| 用户不存在           | 40400  | 404         | 返回"用户不存在"                    |
| 无权操作（非本租户） | 40300  | 403         | 返回"无权操作"                      |
| 密码强度不足         | 40001  | 400         | 返回密码强度要求说明                |
| 旧密码错误           | 40100  | 401         | 返回"旧密码错误"                    |
| 账户被禁用           | 42300  | 423         | 返回"账户已被禁用"                  |
| 账户被锁定           | 42301  | 423         | 返回"账户已被锁定，请 N 分钟后重试" |
| 不能禁用/删除自己    | 40002  | 400         | 返回"不能操作自己的账号"            |
| 必须保留一个管理员   | 40003  | 400         | 返回"必须保留至少一个租户管理员"    |

## 7. 性能考量

| 考量点      | 策略                                           |
| ----------- | ---------------------------------------------- |
| BCrypt 哈希 | 工作因子 12，单次哈希 < 200ms                  |
| 用户列表    | 默认 20 条/页，COUNT 查询使用近似值            |
| 登录锁定    | Redis 计数器 + TTL，避免查库（高并发登录场景） |

## 8. 安全考量

### 8.1 密码安全

- 存储使用 BCrypt（工作因子 12），不可逆
- 密码最小长度 8 位，必须包含大写、小写、数字、特殊字符中至少三种
- 密码明文不出现在日志、API 响应、数据库中
- 密码修改后旧 Token 加入黑名单，强制重新登录
- 密码过期策略（可选，90 天提醒）

### 8.2 权限控制

| 操作         | 超管 | 租户管理员 | 报表设计者 | 审核者 | 查看者 |
| ------------ | ---- | ---------- | ---------- | ------ | ------ |
| 创建用户     | ✅   | ✅(本租户) | ❌         | ❌     | ❌     |
| 编辑用户     | ✅   | ✅(本租户) | ❌         | ❌     | ❌     |
| 禁用用户     | ✅   | ✅(本租户) | ❌         | ❌     | ❌     |
| 重置密码     | ✅   | ✅(本租户) | ❌         | ❌     | ❌     |
| 分配角色     | ✅   | ✅(本租户) | ❌         | ❌     | ❌     |
| 用户列表     | ✅   | ✅(本租户) | ❌         | ❌     | ❌     |
| 修改自己密码 | ✅   | ✅         | ✅         | ✅     | ✅     |

### 8.3 审计覆盖

用户管理操作属于 L2-L3 级别，需记录操作类型、目标用户、变更内容：

- 创建用户 → `CREATE / USER`
- 禁用用户 → `UPDATE / USER` (extra_info 含状态变化)
- 重置密码 → `CONFIG / USER` (不记录新密码，仅记录"密码已重置")
- 角色变更 → `GRANT / USER` (L3 级别，记录变更前后角色)

## 11. 回滚方案

用户管理为管理功能，回滚仅需下线接口。禁用/删除等操作的数据变更不可逆（设计如此），不建议回滚数据。
