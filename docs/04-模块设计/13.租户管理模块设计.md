## 1. 背景与目标

### 1.1 问题背景

租户是系统的顶层隔离单元。虽然 `tenant` 表已在 DDL 中定义，多租户架构设计文档已明确拦截器机制，但租户的**创建、配置、启停、配额管理**等业务操作缺乏模块级设计。

### 1.2 设计目标

| 指标         | 要求                                                   |
| ------------ | ------------------------------------------------------ |
| 租户管理入口 | 仅超管可操作（跨租户管理权限）                         |
| 配额管理     | 每个租户可配置最大用户数、最大存储配额、最大并发任务数 |
| 租户状态     | ACTIVE（正常）/ SUSPENDED（停用）/ ARCHIVED（归档）    |
| 资源限额     | 配额超限时拒绝操作并提示                               |
| 审计覆盖     | 所有租户管理操作 100% 记录审计日志                     |

### 1.3 非目标

- 不在此模块实现租户间的数据共享功能（二期）
- 不实现租户计费/账单功能
- 不实现租户自定义主题/品牌

## 2. 需求分析

### 2.1 功能需求

- [x] **创建租户**：设置租户名称、唯一编码、初始管理员账号
- [x] **编辑租户**：修改名称、描述、配额配置
- [x] **启用/停用租户**：停用后该租户所有用户无法登录
- [x] **归档租户**：长期不使用后归档，保留数据但不可操作
- [x] **租户列表**：支持按名称搜索、按状态筛选、分页
- [x] **租户详情**：含基本信息、配额使用情况、用户数/任务数统计
- [x] **配额管理**：配置最大用户数、最大存储(MB)、最大并发任务数
- [x] **审计记录**：所有操作 100% 记录到 audit_log（含变更前后内容）

### 2.2 用户故事

```
作为 超级管理员，我希望 创建一个新租户并为其分配初始管理员，
以便 新客户可以独立使用系统。

作为 超级管理员，我希望 在租户欠费或违规时暂停该租户，
以便 阻止其继续使用系统资源。

作为 租户管理员，我希望 查看本租户的配额使用情况，
以便 在接近限额时提前申请扩容。
```

### 2.3 边界场景

| 场景                 | 处理方式                                          |
| -------------------- | ------------------------------------------------- |
| 创建时租户编码重复   | 返回错误"租户编码已存在"，提示换一个              |
| 停用有活跃任务的租户 | 提示"该租户有 N 个运行中任务"，确认后停止所有任务 |
| 归档租户的数据处理   | 租户数据保留，但该租户的 Token 全部失效           |
| 配额超限时创建用户   | 返回错误"用户数已达上限(N/N)，请联系管理员扩容"   |
| 存储配额超限时导入   | 返回错误"存储配额不足，当前使用 X/Y MB"           |
| 并发任务数超限       | 新任务排队等待，超时后拒绝                        |

## 3. 架构设计

### 3.1 整体流程

```
超管
  │
  ▼
TenantController  ── @SkipTenantFilter（超管操作无租户上下文）
  │
  ▼
TenantService  ── 业务逻辑、配额校验
  │
  ├──► TenantMapper  ── 数据库 CRUD
  ├──► UserMapper  ── 创建初始管理员时调用
  ├──► QuotaService  ── 配额查询与校验
  └──► AuditLogger  ── 审计日志（L3 级别，记录变更前后内容）
```

### 3.2 模块划分

| 模块         | 职责                          | 所属 Gradle 模块 |
| ------------ | ----------------------------- | ---------------- |
| Controller   | 参数校验、调用 Service        | `api/`           |
| Service      | 租户 CRUD、配额校验、状态管理 | `api/`           |
| QuotaService | 配额查询、超限判断            | `api/`           |
| Mapper       | 数据库 CRUD                   | `api/`           |

## 4. 数据库设计

### 4.1 tenant 表

现有 DDL 已定义，模块设计补充说明：

| 字段           | 类型         | 说明                                |
| -------------- | ------------ | ----------------------------------- |
| `id`           | BIGSERIAL PK | 主键                                |
| `name`         | VARCHAR(200) | 租户名称                            |
| `code`         | VARCHAR(50)  | 租户编码（唯一，用于 API 路径标识） |
| `status`       | VARCHAR(20)  | ACTIVE / SUSPENDED / ARCHIVED       |
| `quota_config` | JSONB        | 配额配置 JSON                       |
| `description`  | TEXT         | 描述                                |
| `created_at`   | TIMESTAMP    | 创建时间                            |
| `updated_at`   | TIMESTAMP    | 更新时间                            |

**quota_config JSON 结构**：

```json
{
  "max_users": 100,
  "max_storage_mb": 10240,
  "max_concurrent_tasks": 5,
  "max_reports": 500,
  "max_datasources": 20,
  "features": {
    "pdf_export": true,
    "scheduled_import": true,
    "custom_watermark": false
  }
}
```

### 4.2 索引设计

现有 DDL 已包含索引：

```sql
CREATE UNIQUE INDEX uk_tenant_code ON tenant(code) WHERE deleted_at IS NULL;
CREATE INDEX idx_tenant_status ON tenant(status);
```

## 5. API 接口设计

### 5.1 接口列表

| 方法   | 路径                             | 说明       | 权限           |
| ------ | -------------------------------- | ---------- | -------------- |
| POST   | `/api/admin/tenants`             | 创建租户   | 超管           |
| GET    | `/api/admin/tenants`             | 租户列表   | 超管           |
| GET    | `/api/admin/tenants/{id}`        | 租户详情   | 超管           |
| PUT    | `/api/admin/tenants/{id}`        | 编辑租户   | 超管           |
| PATCH  | `/api/admin/tenants/{id}/status` | 变更状态   | 超管           |
| DELETE | `/api/admin/tenants/{id}`        | 归档租户   | 超管           |
| GET    | `/api/tenant/quota`              | 本租户配额 | 租户管理员以上 |

### 5.2 请求/响应示例

**创建租户**：

```json
// POST /api/admin/tenants
{
  "name": "张三科技",
  "code": "zhangsan-tech",
  "adminUsername": "admin",
  "adminPassword": "initialP@ss1",
  "quotaConfig": {
    "maxUsers": 100,
    "maxStorageMb": 10240,
    "maxConcurrentTasks": 5
  },
  "description": "张三科技有限公司"
}

// 响应
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 1001,
    "name": "张三科技",
    "code": "zhangsan-tech",
    "status": "ACTIVE",
    "adminUserId": 2001,
    "quotaConfig": { "maxUsers": 100, "maxStorageMb": 10240, "maxConcurrentTasks": 5 },
    "createdAt": "2026-05-18T14:30:00+08:00"
  }
}
```

**查看本租户配额**：

```json
// GET /api/tenant/quota
// 响应
{
  "code": 0,
  "message": "success",
  "data": {
    "quota": {
      "maxUsers": 100,
      "maxStorageMb": 10240,
      "maxConcurrentTasks": 5
    },
    "usage": {
      "users": 45,
      "storageMb": 3204,
      "concurrentTasks": 2,
      "reports": 87,
      "datasources": 5
    },
    "usagePercent": {
      "users": 45.0,
      "storageMb": 31.3,
      "concurrentTasks": 40.0
    }
  }
}
```

## 6. 异常处理

| 场景                   | 错误码 | HTTP 状态码 | 处理方式                   |
| ---------------------- | ------ | ----------- | -------------------------- |
| 租户编码重复           | 40900  | 409         | 返回"租户编码已存在"       |
| 租户不存在             | 40400  | 404         | 返回"租户不存在"           |
| 非超管操作             | 40300  | 403         | 返回"仅超级管理员可操作"   |
| 配额超限（创建用户时） | 42901  | 429         | 返回具体配额使用情况       |
| 租户已停用/归档时操作  | 42300  | 423         | 返回"租户已被停用/归档"    |
| 停用租户存在活跃任务   | 40902  | 409         | 返回活跃任务数量，要求确认 |
| 用户名已存在(同租户)   | 40900  | 409         | 返回"用户名已存在"         |

## 7. 性能考量

| 考量点       | 策略                                                    |
| ------------ | ------------------------------------------------------- |
| 配额查询     | 缓存租户配额（Redis，TTL 5 分钟），高频接口避免每次查库 |
| 使用量统计   | 用户数/报表数等统计数据异步计算，不实时查 count()       |
| 租户列表分页 | 默认 20 条/页，最大 100 条/页                           |

## 8. 安全考量

### 8.1 权限控制

租户管理接口仅超管可调用，使用 `@SkipTenantFilter` 标注，因为超管上下文不绑定任何租户。

### 8.2 审计覆盖

租户管理操作属于 L3 级别事件（严重），需记录变更前后内容到 `extra_info` JSONB：

```json
{
  "before": { "name": "旧名称", "status": "ACTIVE" },
  "after": { "name": "新名称", "status": "SUSPENDED" }
}
```
