## 1. 背景与目标

### 1.1 问题背景

当前文件存储相关的基础设施存在以下问题：

| 问题                                   | 严重度 | 影响                                       |
| -------------------------------------- | ------ | ------------------------------------------ |
| 配置存在但无代码消费                   | 🔴 高   | `jinshu.file.base-path` 等配置定义在 YAML 中，但无 Java 类读取 |
| Docker Compose 无共享存储卷            | 🔴 高   | 多容器（api/worker/batch）无法访问同一文件  |
| K8s PVC 使用 `ReadWriteOnce`           | 🟡 高   | 多 Pod 无法同时挂载，与设计文档要求的 `RWX` 矛盾 |
| 目录结构（upload/temp/output/archive）未创建 | 🟡 高 | 应用启动后无代码自动创建目录             |
| 文件下载端点未实现                     | 🟡 高   | 导出/PDF 模块设计引用 `/api/files/download` 但端点不存在 |
| 无文件清理机制                         | 🟡 高   | 临时文件可能无限增长，耗尽磁盘            |
| 无文件校验实现                         | 🟡 高   | Magic Number 校验、路径遍历防护均无代码   |

### 1.2 设计目标

| 指标             | 要求                                                         |
| ---------------- | ------------------------------------------------------------ |
| 目录结构         | 4 级分层：upload / temp / output / archive                   |
| 文件命名         | `{taskId}_{timestamp}_{segmentNo}.{ext}`，全局唯一            |
| 大文件上传       | ≤ 500MB，multipart/form-data 直传 + Magic Number 校验        |
| 下载安全         | 签名 JWT Token（有效期 1 小时），不直接暴露文件路径           |
| 临时文件清理     | 任务完成后保留 1 小时供下载，1 小时后自动清理                 |
| 并发安全         | `.tmp` 后缀写入 + rename 原子操作，防止读取不完整文件        |
| 路径遍历防护     | 强制校验输出路径前缀，禁止 `..` 注入                         |
| 存储容量         | 开发 10Gi，生产 50Gi（可在线扩容）                           |

### 1.3 非目标

- 不实现对象存储对接（MinIO / S3）（二期）
- 不实现文件分片上传（chunked upload）（当前 ≤ 500MB 直传可满足）
- 不实现文件版本管理（多版本保留）
- 不实现文件加密存储（透明加密由基础设施层负责）
- 不实现 CDN 加速（内网环境无需）

## 2. 需求分析

### 2.1 功能需求

- [x] **文件上传**：接收 multipart/form-data，校验文件类型、大小、Magic Number
- [x] **目录自动创建**：应用启动时创建 `upload/temp/output/archive` 四级目录
- [x] **文件下载**：通过签名 JWT Token 下载文件（1 小时有效期）
- [x] **临时文件清理**：定时任务清理超过 1 小时的 temp 文件和超过 24 小时的 output 文件
- [x] **并发写入保护**：`.tmp` 后缀写入，完成后 rename
- [x] **路径安全校验**：所有文件操作路径强制以 `base-path` 为根
- [x] **文件头校验**：Excel（`D0 CF 11 E0` / `50 4B 03 04`）、CSV（纯文本）、PDF（`25 50 44 46`）
- [x] **文件命名工具**：`FileNameUtil` 生成符合规范的唯一文件名

### 2.2 用户故事

```
作为 报表制表人，我希望 将 Excel 文件上传到系统用于数据导入，
以便 系统异步处理百万行数据而不阻塞我的操作。

作为 报表用户，我希望 导出的 Excel/PDF 文件通过安全链接下载，
以便 文件不会意外泄露给未授权的人。

作为 运维人员，我希望 临时文件能自动清理，
以便 服务器磁盘不会被历史文件占满。
```

### 2.3 边界场景

| 场景                         | 处理方式                                                     |
| ---------------------------- | ------------------------------------------------------------ |
| 上传文件超过 500MB           | 返回 413 Payload Too Large                                   |
| 上传文件类型不在白名单        | 返回 `FILE_TYPE_NOT_SUPPORTED`（4001），列出支持的类型       |
| 上传文件 Magic Number 不符   | 返回 `FILE_TYPE_NOT_SUPPORTED`（4001），提示"文件头校验失败" |
| 磁盘空间不足                 | 写入前检查 `File.getFreeSpace()`，不足返回 503                |
| 下载链接过期                 | JWT 验签失败返回 401，提示"下载链接已过期"                   |
| 下载文件已被清理              | 返回 404，"文件已过期或已被清理"                             |
| 并发写入同一文件             | `.tmp` 文件互不冲突，最终 rename 胜出（先到先得）            |
| 路径遍历攻击（`../../etc/passwd`） | `PathValidator` 拒绝所有含 `..` 或超出 `base-path` 的路径 |
| 零字节文件                   | 上传时校验，content-length = 0 直接拒绝                       |
| 同名文件覆盖                 | 文件名含时间戳 + UUID，理论上不会重复                        |

## 3. 架构设计

### 3.1 目录结构

```
{base-path}                             # /data/jinshu (K8s PVC 挂载点)
├── upload/                             # 用户上传的原始文件
│   └── {tenantId}/
│       └── {reportType}/
│           └── import_{taskId}_{timestamp}.xlsx
│
├── temp/                               # 任务执行中的临时文件
│   └── {taskId}/                       # 每个任务独立目录
│       ├── segment_001.tmp             # 写入中的分片文件
│       ├── segment_001.csv             # Write 完成的文件
│       └── segment_001.pdf
│
├── output/                             # 最终生成的可下载文件
│   └── {tenantId}/
│       └── {reportType}/
│           ├── export_{reportId}_{timestamp}.xlsx
│           ├── export_{reportId}_{timestamp}.zip
│           └── render_{reportId}_{timestamp}.pdf
│
└── archive/                            # 归档目录（长期保留）
    └── {yyyy-MM}/
        └── {original_file_name}_{uuid}.{ext}
```

**权限要求**：

| 目录    | 读权限            | 写权限            | 说明                     |
| ------- | ----------------- | ----------------- | ------------------------ |
| upload  | api, worker, batch | api               | 用户上传写入，Worker 读取 |
| temp    | api, worker       | worker, batch     | 任务执行中间产物         |
| output  | api               | worker, batch     | 最终产物，仅 API 可读    |
| archive | api               | CronJob 写入      | 归档文件，长期保存       |

### 3.2 文件生命周期

```
┌──────────────────────────────────────────────────────────────┐
│                    文件生命周期                               │
│                                                              │
│   [用户上传]                                                  │
│       ↓                                                      │
│   upload/{tenantId}/.../import_10086_20260517.xlsx           │
│       ↓                                                      │
│   Worker 读取 → 处理完成后 → 保留 7 天 → CronJob 清理        │
│                                                              │
│   [任务生成]                                                  │
│       ↓                                                      │
│   temp/{taskId}/segment_001.tmp  → 写入完成 → rename .csv    │
│       ↓                                                      │
│   [任务完成]                                                  │
│       ↓                                                      │
│   output/{tenantId}/.../export_2001_20260517.xlsx            │
│       ↓                                                      │
│   1 小时内可下载 → 1 小时后自动清理（或归档）                  │
│       ↓                                                      │
│   [可选归档] → archive/{yyyy-MM}/...                          │
│       ↓                                                      │
│   30 天后自动清理（归档文件）                                  │
└──────────────────────────────────────────────────────────────┘
```

**文件保留策略**：

| 目录    | 保留时长      | 清理方式              | 说明                           |
| ------- | ------------- | --------------------- | ------------------------------ |
| upload  | 7 天           | CronJob（每小时）     | 导入完成后的原始文件           |
| temp    | 1 小时（任务完成后） | CronJob（每 10 分钟） | 中间产物，任务结束即清理       |
| output  | 1 小时（任务完成后） | CronJob（每小时）     | 下载窗口，过期清理             |
| archive | 30 天          | CronJob（每天）       | 已归档文件保留 30 天           |

### 3.3 模块划分

```
com.jinshu.common                              # 公共模块

com.jinshu.api                                 # API 模块
├── config/
```

### 3.4 关键策略

**原子写入策略**：

```java
// FileStorageService.java
public Path writeTempFile(String taskId, String segmentName, InputStream data) {
    Path tempDir = Path.of(basePath, "temp", taskId);
    Files.createDirectories(tempDir);

    // 1. 先写入 .tmp
    Path tmpFile = tempDir.resolve(segmentName + ".tmp");
    Files.copy(data, tmpFile, StandardCopyOption.REPLACE_EXISTING);

    // 2. 写入完成后原子 rename
    Path finalFile = tempDir.resolve(segmentName);
    Files.move(tmpFile, finalFile, StandardCopyOption.ATOMIC_MOVE);

    return finalFile;
}
```

**路径安全检查**：

```java
// PathValidator.java
public Path validate(String... pathSegments) {
    Path resolved = basePath.resolve(Path.of("", pathSegments)).normalize();

    // 禁止路径遍历
    if (resolved.toString().contains("..")) {
        throw new BusinessException(ErrorCode.PARAM_ERROR, "非法文件路径");
    }

    // 强制在 basePath 范围内
    if (!resolved.startsWith(basePath)) {
        throw new BusinessException(ErrorCode.PARAM_ERROR, "文件路径超出允许范围");
    }

    return resolved;
}
```

**Scheduled 清理任务**：

```java
// TempFileCleanupTask.java (Worker 模块)
@Scheduled(fixedRate = 10 * 60 * 1000)  // 每 10 分钟
public void cleanTempFiles() {
    Path tempDir = Path.of(basePath, "temp");
    // 遍历 taskId 目录，删除 1 小时前修改的文件
    Files.walk(tempDir, 1)
        .filter(Files::isDirectory)
        .filter(this::isOlderThanOneHour)
        .forEach(this::deleteRecursively);
}
```

## 4. 文件命名规范

### 4.1 通用模板

```
{prefix}_{entityId}_{timestamp}_{segmentNo}.{ext}

prefix     = import | export | render | archive
entityId   = taskId | reportId
timestamp  = yyyyMMddHHmmss
segmentNo  = 001, 002, ... (可选，多分片时使用)
ext        = xlsx | csv | zip | pdf

[Writer]: com.jinshu.common.utils.FileNameUtil
```

### 4.2 各类文件命名

| 用途           | 命名示例                                    | 目录位置                         |
| -------------- | ------------------------------------------- | -------------------------------- |
| 上传原始文件   | `import_10086_20260517143025.xlsx`           | `upload/{tenantId}/excel/`       |
| 导出 Excel     | `export_2001_20260517150000.xlsx`            | `output/{tenantId}/export/`      |
| 导出 CSV       | `export_2001_20260517150000.csv`             | `output/{tenantId}/export/`      |
| 导出 CSV (ZIP) | `export_2001_20260517150000.zip`             | `output/{tenantId}/export/`      |
| PDF 渲染       | `render_2001_20260517153000.pdf`             | `output/{tenantId}/render/`      |
| PDF 分段       | `render_2001_20260517153000_001.tmp`         | `temp/{taskId}/`                 |
| PDF 分段完成   | `render_2001_20260517153000_001.pdf`         | `temp/{taskId}/`                 |
| 归档文件       | `Q1销售报表_a1b2c3d4.xlsx`                   | `archive/2026-05/`               |

```java
// FileNameUtil.java
public final class FileNameUtil {

    private static final DateTimeFormatter TS_FMT = DateTimeFormatter.ofPattern("yyyyMMddHHmmss");

    /** 上传文件名： import_{taskId}_{timestamp}.{ext} */
    public static String upload(Long taskId, String ext) {
        return "import_%d_%s.%s".formatted(taskId, now(), ext);
    }

    /** 导出文件名： export_{reportId}_{timestamp}.{ext} */
    public static String export(Long reportId, String ext) {
        return "export_%d_%s.%s".formatted(reportId, now(), ext);
    }

    /** 渲染文件名： render_{reportId}_{timestamp}.{ext} */
    public static String render(Long reportId, String ext) {
        return "render_%d_%s.%s".formatted(reportId, now(), ext);
    }

    /** 分段文件名： render_{reportId}_{timestamp}_{seq}.{ext} */
    public static String segment(Long reportId, int seq, String ext) {
        return "render_%d_%s_%03d.%s".formatted(reportId, now(), seq, ext);
    }

    /** 归档文件名： {originalName}_{shortUuid}.{ext} */
    public static String archive(String originalName) {
        String ext = originalName.substring(originalName.lastIndexOf('.'));
        return "%s_%s%s".formatted(
            originalName.substring(0, originalName.lastIndexOf('.')),
            UUID.randomUUID().toString().substring(0, 8), ext);
    }

    private static String now() {
        return TS_FMT.format(LocalDateTime.now());
    }

    private FileNameUtil() {}
}
```

## 7. 异常处理

| 异常场景                   | HTTP 码 | 错误码 | 提示信息                       | 处理方式                           |
| -------------------------- | ------- | ------ | ------------------------------ | ---------------------------------- |
| 文件大小超过限制           | 413     | 4002   | 文件大小超出限制（最大 500MB） | Spring multipart 配置自动拦截      |
| 文件类型不支持              | 400     | 4001   | 不支持的文件类型，仅支持 .xlsx/.csv | 返回支持的白名单列表           |
| Magic Number 不匹配         | 400     | 4001   | 文件头校验失败，文件可能被伪造 | 记录审计日志                       |
| 磁盘空间不足                | 503     | 4003   | 存储空间不足，请联系管理员     | 触发告警                           |
| 下载链接过期或无效          | 401     | 401    | 下载链接已过期                 | Token 验证失败                     |
| 文件已被清理                | 404     | 404    | 文件已过期或已被清理           | 提示重新执行导出/渲染               |
| 文件不存在（路径错误）      | 404     | 404    | 文件不存在                     | 记录 WARN 日志                     |
| 路径遍历攻击                | 400     | 400    | 参数错误                       | 不暴露具体安全细节                 |
| 零字节文件上传              | 400     | 400    | 文件内容为空                   | 拒绝，不创建 task                  |
| 目录创建失败（权限不足）    | 500     | 500    | 系统内部错误                   | 触发 P0 告警，检查 PVC 挂载权限    |

> 需在 `ErrorCode.java` 中新增 `STORAGE_FULL(4003, "存储空间不足")`。

## 8. 性能考量

### 8.1 性能目标

| 场景                 | 指标          |
| -------------------- | ------------- |
| 文件上传（500MB）     | < 30 秒（1Gbps 内网） |
| 文件下载（100MB）     | < 5 秒         |
| 目录统计（stats API） | < 500ms        |
| 临时文件清理单次      | < 1 秒         |

### 8.2 性能优化点

1. **零拷贝下载**：使用 `FileChannel.transferTo()` 或 Spring `Resource` 的 `InputStreamResource`，避免文件全量读入 JVM 堆内存
2. **流式上传**：`multipart/form-data` 流式接收，Spring Boot 默认写入临时文件，不占内存
3. **目录统计缓存**：stats API 的结果缓存在 Redis（TTL 5 分钟），避免每次遍历文件系统
4. **清理任务单线程**：`@Scheduled` 默认单线程执行，清理大量文件时不阻塞业务请求
5. **分段文件合并**：PDF 分段合并使用 PDFBox 的 `PDFMergerUtility` + `SetupTempFileOnly()` 模式，不将所有分片加载到内存

## 9. 安全考量

### 9.1 文件校验

| 校验项       | 实现方式                                                     | 拦截效果                               |
| ------------ | ------------------------------------------------------------ | -------------------------------------- |
| 文件后缀     | 白名单：`.xlsx`、`.xls`、`.csv`                              | 拒绝 `.exe` 等可执行文件伪装           |
| Magic Number | 读取文件前 4 字节比对已知魔数                                | 拒绝后缀伪造（如 `.exe` 改名 `.xlsx`） |
| 文件大小     | `spring.servlet.multipart.max-file-size = 500MB`             | 拒绝超大文件                           |
| 空文件       | `content-length == 0` 或解析后行数为 0                       | 拒绝空白文件                           |

**Magic Number 对照表**：

| 文件类型 | 后缀   | Magic Number (hex)     |
| -------- | ------ | ---------------------- |
| Excel 97 | .xls   | `D0 CF 11 E0 A1 B1 1A E1` |
| Excel 07+ | .xlsx | `50 4B 03 04`          |
| CSV      | .csv   | 纯文本（无 Magic Number，检查是否为有效 UTF-8 或 ASCII） |
| PDF      | .pdf   | `25 50 44 46`          |

### 9.2 路径安全

```java
// PathValidator.java 强制约束
1. 所有路径操作拼接到指定的 root 根路径
2. 禁止路径中包含 ".."（防遍历）
3. normalize() 后校验路径仍在 root 之下
4. 禁止符号链接（Files.isSymbolicLink()）
5. 外部传入的 path segment 只允许 [a-zA-Z0-9._-]+（拒绝特殊字符）
```

### 9.3 下载安全

- 文件不直接暴露 URL（如 `/output/1/export_2001.xlsx`），必须通过 `/api/files/download?token=xxx` 访问
- Token 使用 JWT 签名，包含：`filePath`、`userId`、`tenantId`、`exp`、`jti`
- Token jti 写入 Redis 黑名单（一次有效，防止重放下载）
- `Content-Disposition` 中文件名使用 `attachment; filename*=UTF-8''{encoded}` 防 XSS

## 12. 回滚方案

### 12.1 回滚触发条件

- 文件上传/下载大面积失败
- 路径校验误伤正常文件路径
- 清理任务误删有效文件
- 签名 Token 泄露导致未授权下载

### 12.2 回滚步骤

1. 回滚 `api` 和 `worker` 模块到上一版本
2. 手动清理错误的 `.tmp` 文件（如有）
3. 文件的原始存储路径不变，旧版本仍可访问
4. NOTICE：文件下载功能临时维护

### 12.3 数据补救

- 回滚期间上传的文件：保留在 `upload/` 目录，清理任务暂停
- 已下载的 Token：回滚后 Token 仍有效（JWT 本地验签）
- 无需数据库数据补救

## 13. 运维考量

### 13.1 部署环境差异

| 环境       | 存储类型        | 路径             | 持久化     |
| ---------- | --------------- | ---------------- | ---------- |
| 开发 (IDE) | 本地文件系统    | `./data/jinshu/` | 持久       |
| 开发 (Docker) | Docker Volume | `/data/jinshu/`  | 需手动配置 volume |
| 测试 (K3s) | RWX PVC         | `/data/jinshu/`  | 持久（PVC） |
| 生产 (K3s) | RWX PVC (50Gi)  | `/data/jinshu/`  | 持久 + 备份 |

**Docker Compose 需增加的共享卷**：

```yaml
# docker-compose.yml
services:
  api:
    volumes:
      - jinshu_data:/data/jinshu
  worker:
    volumes:
      - jinshu_data:/data/jinshu
  batch:
    volumes:
      - jinshu_data:/data/jinshu

volumes:
  jinshu_data:
    driver: local
```

### 13.2 存储容量规划

| 环境   | 配额   | 说明                           |
| ------ | ------ | ------------------------------ |
| 开发   | 10Gi   | 开发数据量小                   |
| 测试   | 20Gi   | 压测产生大量临时文件           |
| 生产   | 50Gi   | 支持在线扩容到 200Gi（K8s PVC expand） |

**容量估算（100 个任务/天）**：

```
upload:  100 × 500MB 原始文件         = 50GB  (7 天后清理)
temp:    100 × 20MB  中间产物          = 2GB   (1 小时后清理)
output:  100 × 10MB  最终文件          = 1GB   (1 小时后清理)
archive: 100 × 10MB  × 30 天          = 30GB  (30 天后清理)
--------------------------------------------------------------
峰值 ≈ 83GB（满载后趋于稳态约 35GB）
```

> 50Gi 配额对 MVP 足够。上线后根据实际使用量调整。

### 13.3 定时清理 CronJob

**K8s CronJob 作为兜底**（Java `@Scheduled` 之外的保障）：

```yaml
# k8s/cleanup-cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: jinshu-file-cleanup
spec:
  schedule: "0 * * * *"     # 每小时
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: cleanup
            image: busybox
            command:
            - /bin/sh
            - -c
            - |
              find /data/jinshu/temp -type f -mmin +60 -delete
              find /data/jinshu/temp -type d -empty -delete
              find /data/jinshu/output -type f -mmin +60 -delete
              find /data/jinshu/upload -type f -mtime +7 -delete
            volumeMounts:
            - name: data
              mountPath: /data/jinshu
          restartPolicy: OnFailure
          volumes:
          - name: data
            persistentVolumeClaim:
              claimName: jinshu-data
```

> `@Scheduled` 是主清理机制，CronJob 是兜底（防止 JVM 清理任务阻塞或异常）。
