## 1. 背景与目标

### 1.1 问题背景

报表系统的核心数据来源是用户上传的 Excel/CSV 文件。导入模块面临以下挑战：

| 挑战                   | 说明                                             |
| ---------------------- | ------------------------------------------------ |
| 数据量大               | 单文件可达 500MB、100 万行，全量加载必然 OOM     |
| 数据质量不可控         | 用户文件存在格式错误、类型错误、业务逻辑冲突     |
| 异步耗时               | 百万行导入需数分钟，同步等待用户体验不可接受     |
| 多租户并发             | 多租户同时导入大文件，必须公平分配资源           |
| 断点续传               | 服务中断后不能从头开始，浪费已处理数据           |
| 错误追踪               | 脏数据需要精确到行号的错误日志，方便用户修正     |

### 1.2 设计目标

| 指标             | 要求                                       |
| ---------------- | ------------------------------------------ |
| 最大支持行数     | 100 万行                                   |
| 单文件大小上限   | 500MB                                      |
| 导入速度         | > 2000 行/秒                               |
| 内存占用峰值     | < 512MB（100 万行导入时）                  |
| 三级校验         | 格式校验 → 类型校验 → 业务校验，准确率 100% |
| 错误率阈值       | > 30% 自动终止导入                         |
| 进度实时可查     | Redis 更新间隔 ≤ 5 秒                       |
| 失败重试         | 支持断点续传（基于分片记录）               |

### 1.3 非目标

- 不支持 `.xlsb`（二进制 Excel）、`.ods`（OpenOffice）格式
- 不支持从数据库直接导入（走 Data-Sync 模块）
- 不支持导入时自动创建报表元数据（需先通过报表 CRUD 创建）
- 不支持导入过程中修改表头映射（导入前完成映射配置）
- 不支持增量追加导入（每次导入覆盖该报表原数据）

## 2. 需求分析

### 2.1 功能需求

- [x] **文件上传**：multipart/form-data，白名单 `.xlsx`/`.xls`/`.csv`
- [x] **文件校验**：后缀白名单 + Magic Number 校验 + 文件大小限制
- [x] **异步导入**：上传后立即返回 taskId，通过 MQ 异步执行
- [x] **三级校验**：格式校验（必填列、列数）、类型校验（数字/日期/文本）、业务校验（唯一性/字典值/跨列逻辑）
- [x] **分片处理**：按行数自动分片，支持并发处理
- [x] **断点续传**：基于分片进度，服务重启后从断点继续
- [x] **错误日志**：脏数据精确到行号 + 错误原因，写入 `import_error_log`
- [x] **错误率控制**：错误率 > 30% 自动终止
- [x] **进度实时反馈**：Redis 存储进度，前端轮询
- [x] **完成通知**：导入成功/失败后更新任务状态，前端可查询

### 2.2 用户故事

```
作为 报表制表人，我希望 将一个 100 万行的 Excel 文件导入系统，
以便 快速将线下数据迁移到线上报表系统。

作为 报表制表人，我希望 导入失败时能看到每一行的具体错误原因，
以便 快速修正数据后重新导入。

作为 系统管理员，我希望 大文件导入不会影响其他租户的任务执行，
以便 保证系统的公平性和整体可用性。
```

### 2.3 边界场景

| 场景                       | 处理方式                                                     |
| -------------------------- | ------------------------------------------------------------ |
| 空文件（0 行）              | 直接标记 SUCCESS，不插入数据                                 |
| 全列空行                    | 校验阶段过滤，不计入有效行数                                 |
| 单文件超 500MB              | 返回 413，不创建任务                                         |
| 文件头 Magic Number 不匹配  | 返回 `FILE_TYPE_NOT_SUPPORTED`，提示"文件后缀与内容不匹配" |
| 导入到不存在的 reportId    | 返回 `REPORT_NOT_FOUND`                                     |
| 服务重启中断               | Worker 重启后消费 MQ → 读取已处理分片 → 跳过已完成分片继续 |
| 导入过程中文件被删除        | Worker 读取 upload 文件时捕获 IOException，标记 FAILED       |
| 数据库连接池耗尽            | Worker 重试 3 次，间隔递增（5s/30s/5min），全部失败进 DLQ  |
| 错误率超过 30%              | 立即终止导入，已插入数据库的数据**不回滚**（用户手动删除或重新导入覆盖） |
| 日期格式异常（如 2026-13-40）| 类型校验阶段捕获，记录错误行                                  |
| 长数字被 Excel 转为科学计数法 | 上传前提示用户将列格式设为"文本"；EasyExcel 流式读取时强制文本处理 |

## 3. 架构设计

### 3.1 整体流程

```
用户 POST /api/report/import (multipart file + reportId)
        ↓
ImportController.upload()
  ├── 1. 文件后缀白名单校验
  ├── 2. Magic Number 校验 (FileStorageService)
  ├── 3. 文件大小校验 (≤ 500MB)
  ├── 4. 保存文件到 upload/{tenantId}/excel/ (FileStorageService)
  ├── 5. 解析 Excel 表头，校验与报表 schema 的列映射
  ├── 6. 统计总行数（EasyExcel 快速扫描）
  ├── 7. 创建 task 记录 (type=IMPORT, status=PENDING)
  ├── 8. 计算优先级 → TaskPriorityCalculator
  ├── 9. 发送 ImportTaskMessage 到 jinshu.task.direct
  └── 10. 返回 taskId (HTTP 202)
        ↓
Worker 模块消费 MQ (jinshu.task.p0-p4)
        ↓
ImportTaskHandler.handle(message)
  ├── 1. 幂等检查 (Redis jinshu:mq:consumed:{msgId})
  ├── 2. 更新 task.status = PROCESSING
  ├── 3. 读取分片进度 (Redis)
  ├── 4. EasyExcel 流式逐行读取（从上次断点开始）
  │     ├── 每行执行三级校验
  │     │     ├── L1 格式校验（列数 ≠ 预期 → 拒绝）
  │     │     ├── L2 类型校验（数字/日期/文本 → 拒绝）
  │     │     └── L3 业务校验（唯一性/字典值/跨列逻辑 → 拒绝）
  │     ├── 校验通过 → 加入内存批次缓冲区
  │     ├── 校验失败 → 写入 import_error_log（行号 + 字段名 + 错误原因）
  │     └── 累计 1000 行 → MyBatis batchInsert → 清空缓冲
  ├── 5. 每片完成 → 更新 Redis 进度
  ├── 6. 错误率检查：错误行数 / 已处理行数 > 30% → 终止
  ├── 7. 全部分片完成 → task.status = SUCCESS
  └── 8. 记录审计日志 (IMPORT_REPORT)
```

### 3.2 三级校验引擎

| 层级 | 名称     | 校验内容                                                   | 失败处理                     |
| ---- | -------- | ---------------------------------------------------------- | ---------------------------- |
| L1   | 格式校验 | 列数匹配、必填列非空、日期格式合法（`yyyy-MM-dd` 等）     | 记录行号 + 列名 + 错误类型   |
| L2   | 类型校验 | 数字/日期/文本类型匹配，长数字强制文本处理（防科学计数法） | 记录行号 + 列名 + 期望类型   |
| L3   | 业务校验 | 唯一性约束（`UNIQUE(tenant_id, report_id, row_no)`）、字典值校验（状态/类型枚举）、跨列逻辑校验（结束时间 > 开始时间） | 记录行号 + 约束名称 + 实际值 |

```java
// ReportRowValidator.java
public class ReportRowValidator {

    public ValidationResult validate(Map<String, Object> row, int rowNo) {
        List<ValidationError> errors = new ArrayList<>();

        // L1: 格式校验
        for (ColumnSchema col : schema.getColumns()) {
            if (col.isRequired() && isEmpty(row.get(col.getName()))) {
                errors.add(new ValidationError(rowNo, col.getName(), "REQUIRED", "必填列为空"));
            }
            if (col.getType() == ColumnType.DATE && !isValidDate(row.get(col.getName()))) {
                errors.add(new ValidationError(rowNo, col.getName(), "FORMAT", "日期格式非法"));
            }
        }

        // L2: 类型校验
        for (ColumnSchema col : schema.getColumns()) {
            Object value = row.get(col.getName());
            if (value != null && !matchesType(value, col.getType())) {
                errors.add(new ValidationError(rowNo, col.getName(), "TYPE", "期望" + col.getType()));
            }
        }

        // L3: 业务校验
        for (BusinessRule rule : schema.getBusinessRules()) {
            if (!rule.validate(row)) {
                errors.add(new ValidationError(rowNo, rule.getColumn(), "BUSINESS", rule.getMessage()));
            }
        }

        return new ValidationResult(errors.isEmpty(), errors);
    }
}
```

## 6. 异常处理

| 异常场景                   | HTTP 码 | 错误码 | 提示信息                     | 处理方式                               |
| -------------------------- | ------- | ------ | ---------------------------- | -------------------------------------- |
| 文件类型不支持              | 400     | 4001   | 文件类型不支持，仅支持 .xlsx/.xls/.csv | 返回支持的白名单                   |
| Magic Number 校验失败       | 400     | 4001   | 文件后缀与内容不匹配          | 返回详细信息                           |
| 文件大小超限               | 413     | 4002   | 文件大小超出限制（最大 500MB） | Spring multipart 自动拦截             |
| 报表不存在                  | 404     | 2001   | 报表不存在                    | 不创建任务                             |
| 列映射不匹配                | 400     | 2008   | 文件表头与报表 schema 不匹配  | 返回差异详情（期望列 vs 实际列）      |
| 磁盘空间不足                | 503     | 4003   | 存储空间不足                  | 触发告警                               |
| 数据库写入失败（重试耗尽）  | —       | —      | —                             | 标记 FAILED，消息入 DLQ                |
| 错误率 > 30%               | —       | —      | —                             | 终止导入，已导入数据**不回滚**          |
| 服务重启中断               | —       | —      | —                             | 从分片断点继续                         |
| 文件内容全空（0 有效行）    | —       | —      | —                             | 标记 SUCCESS，不插入数据               |

> 需在 `ErrorCode.java` 新增 `COLUMN_MISMATCH(2008, "文件表头与报表 schema 不匹配")`。

## 7. 性能考量

### 7.1 性能目标

| 场景                   | 指标          |
| ---------------------- | ------------- |
| 10 万行导入（50MB）    | < 30 秒       |
| 100 万行导入（500MB）   | < 5 分钟      |
| 内存占用峰值           | < 512MB       |
| 单行校验延迟           | < 0.1ms       |
| 批量写入（1000 行）    | < 100ms       |
| 进度更新频率           | 每 1000 行或每 5 秒 |

### 7.2 防 OOM 设计

| 措施                     | 说明                                         |
| ------------------------ | -------------------------------------------- |
| EasyExcel 流式读取        | SAX 模式逐行读取，不将整个文件加载到内存     |
| 批量写入后 GC 提示        | 每 50000 行调用 `System.gc()`（生产环境关闭） |
| JVM 参数                 | `-Xmx512m -XX:+UseG1GC`                      |
| 并发控制                 | 同租户最大同时导入任务数 = 1                 |
| 内存缓冲区上限           | 1000 行 × 4 并发 ≈ 4MB                       |

**内存估算**：

```
单行数据 ≈ 1KB
批量缓冲 = 1000 行 ≈ 1MB
4 并发 × 1MB = 4MB
EasyExcel SAX 缓冲区 ≈ 10MB
JVM 基础开销 ≈ 100MB (Metaspace + Thread stacks)
-----------------------------------------
峰值预估 ≈ 120MB < 512MB ✅
```

### 7.3 性能优化点

1. **EasyExcel SAX 读取**：基于 SAX 事件驱动，逐行回调，不对文件全量加载
2. **MyBatis batchInsert**：单条 SQL 插入 1000 行，减少网络往返
3. **PostgreSQL COPY 协议**：超大批量场景可切换为 `COPY` 命令（比 INSERT 快 10x）
4. **错误日志异步写入**：错误日志不影响主导入流程，通过独立批量写入
5. **进度 Redis 批量更新**：每 1000 行或每 5 秒更新一次 Redis Hash，避免高频写入
6. **连接池隔离**：Batch 模块使用独立的 HikariCP 连接池（maxPoolSize=10），不影响 API 查询

## 8. 安全考量

### 8.1 文件安全

| 校验项       | 实现方式                           | 拦截效果                               |
| ------------ | ---------------------------------- | -------------------------------------- |
| 文件后缀     | 白名单：`.xlsx`、`.xls`、`.csv`  | 拒绝可执行文件伪装                     |
| Magic Number | 读取文件前 4-8 字节比对已知魔数    | 拒绝后缀伪造（`.exe` 改名 `.xlsx`）    |
| 文件大小     | `spring.servlet.multipart.max-file-size = 500MB` | 拒绝超大文件                 |
| 文件名       | 存储使用 `FileNameUtil` 生成唯一名  | 防路径注入                             |
| 文件存储     | 上传目录 `upload/{tenantId}/excel/` | 租户间文件隔离                         |

> Magic Number 校验实现见 `docs/04-模块设计/09.文件存储架构设计.md` §9.1。

### 8.2 权限控制

- 只有角色为 USER 或 ADMIN 的用户可以上传导入文件
- 用户只能导入到自己租户下的报表（`tenantId` 从 `TenantContext` 获取）
- VIEWER 角色禁止导入
- 导入操作记录审计日志（`operation = 'IMPORT_REPORT'`）

### 8.3 数据安全

- `report_id` 从请求中获取，但 `tenant_id` 由服务端 `TenantContext` 注入，禁止前端传入
- 导入的原始文件不删除，保留 7 天用于故障排查
- `import_error_log` 不直接暴露给前端用户（通过 API 查询，限本租户可见）
- CSV 公式注入防护：以 `=`/`+`/`-`/`@` 开头的单元格值前添加单引号，防止自动执行公式

## 11. 回滚方案

### 11.1 回滚触发条件

- 导入大面积失败（失败率 > 50%）
- 三级校验误拦截正常数据
- Batch 模块频繁 OOM
- 死信队列中导入消息堆积

### 11.2 回滚步骤

1. 设置 `jinshu.import.enabled = false`（ImportController 返回 503）
2. 停止 Batch 模块消费者的导入消息监听
3. 清空 MQ 中未消费的导入消息
4. 回滚 `api` 和 `batch` 模块到上一版本

### 11.3 数据补救

- 回滚期间已创建但未完成的 task：批量标记 FAILED
- 已导入的数据：保留在 `report_data` 中（不回滚数据）
- 上传文件：保留在 `upload/` 目录（7 天后自动清理）
