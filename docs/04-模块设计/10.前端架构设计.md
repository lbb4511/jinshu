## 1. 背景与目标

### 1.1 问题背景

前端项目已搭建骨架（React 19 + PrimeReact 10 + Vite 8），但存在以下问题：

| 问题             | 严重度  | 影响                                                       |
| ---------------- | ------- | ---------------------------------------------------------- |
| 无认证流程闭环   | 🔴 致命 | 登录页纯 UI 桩，无 API 调用，无 Token 存储，无路由守卫     |
| 无状态管理方案   | 🔴 致命 | 登录后用户信息无法全局共享，各组件各自管理状态             |
| 无布局架构       | 🟡 高   | 两个页面各自独立渲染，无导航栏/侧边栏/内容区统一壳         |
| 无路由守卫       | 🟡 高   | 未登录可直接访问 `/`（Dashboard）                          |
| 无错误处理体系   | 🟡 高   | Axios 响应拦截器仅 `console.error`，不处理 401 跳转        |
| Token 刷新未实现 | 🟡 高   | JWT 设计规定了 Access/Refresh Token，前端无刷新逻辑        |
| 类型定义未被消费 | 🟡 高   | 6 个 TypeScript 接口已定义但 0 个文件 import 它们          |
| 打印模式未实现   | 🟡 高   | PDF 模块依赖前端 `printMode=true` 隐藏交互元素，前端未支持 |
| 无环境变量管理   | 🟡 高   | 无 `.env`，API base URL 硬编码                             |

### 1.2 设计目标

| 指标              | 要求                                                      |
| ----------------- | --------------------------------------------------------- |
| 认证闭环          | 登录 → Token 存储 → 自动附加 → 过期刷新 → 登出清理        |
| 路由守卫          | 未登录强制跳转 `/login`，已登录访问 `/login` 重定向到 `/` |
| 状态管理          | React Context + `useReducer` 管理全局认证状态             |
| 布局统一          | AppLayout（顶部导航 + 侧边栏 + 内容区）                   |
| 错误处理          | 401/403 自动处理，全局错误 Toast                          |
| 打印模式          | URL 参数 `?printMode=true` 隐藏导航/侧边栏/按钮           |
| TypeScript 覆盖率 | 所有组件、Service、类型 100% TypeScript                   |
| 构建产物体积      | 首屏 JS < 300KB（gzip），总 bundle < 1.5MB                |

### 1.3 非目标

- 不实现 SSR（服务端渲染）—— 纯 SPA
- 不实现 PWA（离线缓存/Service Worker）
- 不实现国际化框架（二期）
- 不引入状态管理库（Redux/Zustand）—— React Context 足够 MVP 使用
- 不实现单元测试框架搭建（先完成功能实现，测试后补）

## 2. 需求分析

### 2.1 功能需求

- [x] **登录认证**：用户名 + 密码 → JWT Token → 存储 + 全局状态更新
- [x] **Token 管理**：Access Token 存内存，Refresh Token 存 localStorage，自动刷新
- [x] **路由守卫**：`ProtectedRoute` 包裹需要认证的路由，未登录跳转 `/login`
- [x] **应用布局**：顶部导航栏 + 侧边栏菜单 + 内容区（`<Outlet />`）
- [x] **API 层封装**：Axios 封装，自动注入 Token，自动刷新，统一错误处理
- [x] **打印模式**：`?printMode=true` 时隐藏导航/侧边栏/按钮/交互元素
- [x] **健康检查**：Dashboard 首页展示后端/数据库/Redis/RabbitMQ 状态
- [x] **错误边界**：`ErrorBoundary` 捕获渲染异常，显示友好错误页面
- [x] **404 页面**：未匹配路由显示 404 页面

### 2.2 用户故事

```
作为 用户，我希望 登录后系统记住我的身份直到 Token 过期，
以便 每次操作不需要重复输入密码。

作为 用户，我希望 在 Token 即将过期时系统自动刷新，
以便 不会在使用过程中突然被踢出。

作为 报表审查人，我希望 在打印模式下报表视图干净（无导航栏和按钮），
以便 生成的 PDF 与网页预览一致且格式正确。

作为 前端开发者，我希望 所有 API 调用有统一的错误处理，
以便 不需要在每个组件中重复写 try-catch。
```

### 2.3 边界场景

| 场景                       | 处理方式                                                                       |
| -------------------------- | ------------------------------------------------------------------------------ |
| 页面刷新后 Token 丢失      | 内存中 Access Token 丢失，从 localStorage 读取 Refresh Token，调用刷新接口恢复 |
| Access Token 过期 5 分钟内 | Axios 响应拦截器自动调用刷新接口，成功后重试原请求                             |
| Refresh Token 也过期       | 清除本地 Token，跳转 `/login`，显示"登录已过期，请重新登录"                    |
| 登录接口返回 401           | 显示"用户名或密码错误"，不清除已有 Token                                       |
| 登录接口返回 423（锁定）   | 显示"账号已锁定，请 N 分钟后重试"，不清除已有 Token                            |
| 多个 Tab 同时刷新 Token    | 每个 Tab 独立处理，Token 存在内存中，互不影响                                  |
| 网络断开                   | Axios 捕获 `Network Error`，显示 Toast "网络连接失败"                          |
| 后端 500                   | 显示 Toast "服务器繁忙，请稍后重试"                                            |
| 浏览器禁用 JavaScript      | `<noscript>` 标签提示"请启用 JavaScript"                                       |

### 3.5 路由守卫

```typescript
// ProtectedRoute.tsx
function ProtectedRoute({ roles }: { roles?: string[] }) {
  const { user, isAuthenticated, isLoading } = useAuth();

  // 正在从 localStorage 恢复 Token
  if (isLoading) return <LoadingSpinner />;

  // 未登录 → 跳转登录
  if (!isAuthenticated) return <Navigate to="/login" replace />;

  // 有角色要求 → 校验
  if (roles && !roles.includes(user!.role)) {
    return <Navigate to="/" replace />;
  }

  return <Outlet />;
}
```

**登录态路由重定向**：

```typescript
// Login.tsx 顶部
if (isAuthenticated) return <Navigate to="/" replace />;
```

## 4. 状态管理

### 4.1 方案选择

| 方案            | 优点                     | 缺点                   | 选择 |
| --------------- | ------------------------ | ---------------------- | ---- |
| React Context   | 零依赖，适合简单全局状态 | 不适合高频更新         | ✅   |
| useReducer      | 可预测的状态转换         | 样板代码               | ✅   |
| Redux / Zustand | 强大的中间件/DevTools    | 额外依赖，MVP 过度设计 | ❌   |

> MVP 阶段使用 `React Context + useReducer`。如果后续全局状态复杂度显著增长（如跨组件共享大量业务数据），可迁移到 Zustand。

### 4.2 AuthContext 设计

```typescript
// AuthContext.tsx
interface AuthState {
  user: UserInfo | null; // { id, username, tenantId, role }
  accessToken: string | null; // 存内存（不写 storage）
  isAuthenticated: boolean;
  isLoading: boolean; // 应用启动时恢复登录态
}

type AuthAction =
  | { type: "LOGIN_START" }
  | { type: "LOGIN_SUCCESS"; payload: { user: UserInfo; accessToken: string } }
  | { type: "LOGIN_FAILURE" }
  | { type: "TOKEN_REFRESHED"; payload: { accessToken: string } }
  | { type: "LOGOUT" }
  | { type: "RESTORE_COMPLETE" };

interface AuthContextValue extends AuthState {
  login: (username: string, password: string) => Promise<void>;
  logout: () => void;
  refreshToken: () => Promise<string | null>;
}
```

**启动时恢复登录态**：

```
App 挂载
  → AuthProvider 初始化，isLoading = true
  → 从 localStorage 读取 refreshToken
  → refreshToken 存在 → 调用 POST /api/auth/refresh
    → 成功 → dispatch LOGIN_SUCCESS（恢复 user + accessToken）
    → 失败 → dispatch RESTORE_COMPLETE（isAuthenticated = false）
  → refreshToken 不存在 → dispatch RESTORE_COMPLETE（isAuthenticated = false）
  → isLoading = false
  → 所有子组件渲染
```

### 4.3 数据流

```
Login.tsx
  → authService.login(username, password)
  → AuthContext.login()
     → dispatch LOGIN_SUCCESS({ user, accessToken })
     → localStorage.setItem('refreshToken', ...)
  → navigate('/') (React Router)

ProtectedRoute.tsx
  → useAuth().isAuthenticated
  → false → <Navigate to="/login" />
  → true → 正常渲染子路由

request.ts (Axios 拦截器)
  → 请求拦截：从 AuthContext 获取 accessToken → 注入 Header
  → 响应拦截：401 → 尝试 refreshToken → 刷新成功重试 → 失败则 logout
```

## 5. API 服务层设计

### 5.2 请求拦截链路

```
【请求拦截器】
  1. 从 tokenUtil.getAccessToken() 读取 Access Token
  2. 存在 → Authorization: Bearer <token>
  3. 不存在 → 不添加（登录接口无需 Token）

【响应拦截器 - 成功】
  1. 直接返回 response.data

【响应拦截器 - 失败】
  ├── 401 Unauthorized
  │     ├── 请求不是刷新接口本身
  │     ├── 尝试 refreshAccessToken()
  │     ├── 刷新成功 → 更新 accessToken
  │     │     → 重试原始请求（error.config.headers.Authorization 更新）
  │     └── 刷新失败 → clearTokens()
  │                   → window.location.href = '/login'
  ├── 403 Forbidden
  │     └── 显示 Toast "无权限执行此操作"
  ├── 423 Locked
  │     └── 显示 Toast "账号已锁定"
  ├── 5xx Server Error
  │     └── 显示 Toast "服务器繁忙，请稍后重试"
  └── Network Error
        └── 显示 Toast "网络连接失败，请检查网络"
```

**Token 管理工具**：

```typescript
// utils/token.ts
const ACCESS_TOKEN_KEY = "jinshu_access_token"; // 存 sessionStorage（关闭 Tab 即清除）
const REFRESH_TOKEN_KEY = "jinshu_refresh_token"; // 存 localStorage（持久化刷新）

export const tokenUtil = {
  getAccessToken: () => sessionStorage.getItem(ACCESS_TOKEN_KEY),
  setAccessToken: (token: string) =>
    sessionStorage.setItem(ACCESS_TOKEN_KEY, token),
  getRefreshToken: () => localStorage.getItem(REFRESH_TOKEN_KEY),
  setRefreshToken: (token: string) =>
    localStorage.setItem(REFRESH_TOKEN_KEY, token),
  clear: () => {
    sessionStorage.removeItem(ACCESS_TOKEN_KEY);
    localStorage.removeItem(REFRESH_TOKEN_KEY);
  },
};
```

> Access Token 存 `sessionStorage`（关闭浏览器窗口即清除），Refresh Token 存 `localStorage`（保持 7 天）。比纯 `localStorage` 方案更安全（Access Token 泄漏窗口更窄）。

### 6.4 错误边界

```typescript
class ErrorBoundary extends React.Component<Props, { hasError: boolean }> {
  static getDerivedStateFromError() {
    return { hasError: true };
  }

  render() {
    if (this.state.hasError) {
      return (
        <div className="error-boundary">
          <h2>页面出错了</h2>
          <p>请刷新页面重试，或联系管理员</p>
          <Button label="刷新页面" onClick={() => window.location.reload()} />
        </div>
      );
    }
    return this.props.children;
  }
}
```

## 9. 异常处理

| 异常场景                  | 用户感知                            | 处理方式                         |
| ------------------------- | ----------------------------------- | -------------------------------- |
| 401（Token 过期）         | 自动刷新后无感知（或跳转登录页）    | Axios 拦截器静默刷新             |
| 401（Refresh Token 过期） | Toast "登录已过期，请重新登录"      | 清除 Token → 跳转 `/login`       |
| 403（无权限）             | Toast "无权限执行此操作"            | 前端按钮也按权限隐藏（双重防护） |
| 404（资源不存在）         | Toast "资源不存在"                  | 或跳转 404 页面                  |
| 423（账号锁定）           | Toast "账号已锁定，请 N 分钟后重试" | 显示锁定倒计时                   |
| 500（服务器错误）         | Toast "服务器繁忙，请稍后重试"      | 记录到 console                   |
| 网络断开                  | Toast "网络连接失败"                | 定时重连检测                     |
| React 渲染异常            | ErrorBoundary 友好页面              | "页面出错了，请刷新"             |
| 路由不存在                | NotFound 页面                       | "您访问的页面不存在"             |

## 10. 性能考量

### 10.1 性能目标

| 指标                  | 目标    |
| --------------------- | ------- |
| 首屏加载时间 (FCP)    | < 1.5s  |
| 首屏 JS Bundle (gzip) | < 300KB |
| 总 Bundle (gzip)      | < 1.5MB |
| 路由切换时间          | < 100ms |
| Axios 请求拦截延迟    | < 1ms   |

### 10.2 性能优化点

1. **Code Splitting**：页面级 `React.lazy()` + `<Suspense>` 懒加载，首屏只加载 Login/Dashboard
2. **Vendor Chunk Split**：React、PrimeReact、ECharts 分包，利用浏览器缓存
3. **PrimeReact 按需导入**：每个组件独立 import（`import { Button } from 'primereact/button'`），Vite tree-shaking 自动排除未用组件
4. **ECharts 按需引入**：不使用 `import * as echarts`，使用 `import { BarChart, LineChart } from 'echarts/charts'`
5. **图表懒加载**：报表详情页的 ECharts 图表进入视口后再初始化
6. **Axios 缓存**：短期内重复的 GET 请求（如健康检查）复用上次结果（5 秒缓存）
7. **CSS 按需加载**：页面级 SCSS 通过 JS import 引入，Vite 自动代码分割

## 11. 安全考量

### 11.1 Token 安全

- Access Token 存 `sessionStorage`（关闭 Tab 后自动清除，降低 XSS 后持久化泄漏风险）
- Refresh Token 存 `localStorage`（浏览器安全策略内最安全的选择，MVP 阶段可接受）
- 生产环境建议升级为 httpOnly Cookie 方案（需后端配合 `Set-Cookie`）
- 打印模式路由（`/reports/:id/preview`）使用独立的内部 JWT Token，不暴露用户 Token

### 11.2 XSS 防护

- React 默认 JSX 输出转义（`<div>{userInput}</div>` 自动防 XSS）
- `dangerouslySetInnerHTML` 禁止使用（如有需要，必须先 DOMPurify 清洗）
- 所有 API 返回的 HTML 内容不入 DOM
- CSP Header 由 Nginx/K8s Ingress 层配置

## 14. 回滚方案

### 14.1 回滚触发条件

- 登录流程大面积失败
- 路由守卫导致白屏
- 构建产物大小异常导致加载超时

### 14.2 回滚步骤

1. Nginx 切换回旧版本 `dist/` 目录
2. 重新构建旧版本 Docker 镜像（如有）
3. 通知用户：前端更新已回退

### 14.3 数据补救

- 前端无持久数据，回滚不影响数据库
- `localStorage` 中的 Token 不受前端版本影响
