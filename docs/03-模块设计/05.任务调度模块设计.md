## 1. 背景与目标

### 1.1 问题背景

在多租户报表系统中，任务（导入、导出、PDF 渲染、数据同步）是异步执行的，由 Worker 集群通过消息队列消费处理。如果没有调度策略，会出现严重的资源抢占问题：

- **大任务阻塞**：租户 A 提交 100 万行导出任务，占用全部 Worker 线程 5 分钟，租户 B 的 100 行小导出排队等待 5 分钟
- **租户不公平**：高活跃租户持续提交任务，低活跃租户的任务永远得不到执行机会（饿死）
- **优先级混乱**：紧急的预览任务和后台批量任务混在同一个队列，紧急任务被批量任务拖慢
- **资源失控**：单个租户无限制地提交大任务，可能导致整个 Worker 集群内存/CPU 耗尽

### 1.2 设计目标

| 指标                        | 要求                          |
| --------------------------- | ----------------------------- |
| 小任务排队时间（<1 万行）   | < 3 秒（99% 百分位）          |
| 单租户最大资源占用          | ≤ 30% 总 Worker 资源          |
| 调度决策延迟                | < 10 毫秒                     |
| 大任务分片后对其他租户影响  | 最多额外等待 10 秒            |
| 租户配额动态调整            | 实时生效，无需重启            |
| 失败子任务重试              | 支持 3 次重试，支持自定义退避 |

### 1.3 非目标

- 不支持跨数据中心调度（当前仅单集群）
- 不支持任务执行中途暂停/恢复（仅支持取消和重试）
- 不支持 GPU 资源调度（当前无 GPU 任务）
- 首次版本不实现基于机器学习的智能优先级预测

## 2. 需求分析

### 2.1 功能需求

- [ ] **队列分级**：按任务类型和数据量划分 5 级优先级队列
- [ ] **租户并发限制**：每个租户同时执行的任务数不超过配额上限
- [ ] **时间片轮转调度**：按租户 Round-Robin 公平取任务，防止饿死
- [ ] **大任务分片**：超大任务（>10 万行导出、>50 页 PDF）自动拆分为子任务
- [ ] **子任务交错执行**：大任务的子任务之间插入其他租户的小任务
- [ ] **动态配额调整**：管理员可实时调整租户并发配额，即时生效
- [ ] **任务取消**：支持用户取消排队中或执行中的任务
- [ ] **死信队列**：重试 3 次后仍失败的任务进入死信队列，人工介入

### 2.2 用户故事

```
作为 普通用户，我希望 提交一个小导出后能在 3 秒内开始执行，
即使 系统中有其他租户的大任务在运行。

作为 系统管理员，我希望 某个租户提交大量任务时不会影响其他租户，
以便 保证系统的整体公平性和可用性。

作为 运营人员，我希望 VIP 租户能获得更高的并发配额，
以便 满足重要客户的 SLA 要求。
```

### 2.3 边界场景

| 场景                     | 处理方式                                                  |
| ------------------------ | --------------------------------------------------------- |
| 所有 Worker 挂掉         | MQ 消息持久化保留，Worker 恢复后自动继续消费              |
| 调度器本身崩溃           | 新 Worker 启动时自动选举 Leader，重建调度状态             |
| 租户配额配置为 0         | 该租户新任务直接拒绝，返回 429 "配额已耗尽"               |
| 子任务部分失败           | 仅重试失败的分片，不影响已完成分片，整体任务标记 PARTIAL  |
| 任务取消时正在执行       | 发送中断信号，Worker 捕获后优雅停止，清理临时文件         |
| 100+ 租户同时活跃        | 时间片轮转批次处理，每批取前 N 个租户，保证吞吐量         |
| 队列堆积 > 1000 条       | 触发扩容告警，HPA 自动增加 Worker Pod                     |

## 3. 架构设计

### 3.1 整体流程

```
任务提交（Import / Export / PDF / Sync）
        ↓
┌─────────────────────────────────────────────┐
│           TaskPriorityCalculator              │
│  根据任务类型 + 预估数据量计算优先级           │
│  P0(预览/小任务) → P4(后台批量)               │
└─────────────────────────────────────────────┘
        ↓
投递到对应优先级 RabbitMQ 队列
        ↓
┌─────────────────────────────────────────────┐
│          Tenant Fair Scheduler                │
│  1. 从各优先级队列按权重拉取消息               │
│  2. 检查租户并发配额（Redis 计数器）           │
│  3. 通过配额 → 分配给 Worker                  │
│  4. 超限 → 放回队列，取下一个租户              │
└─────────────────────────────────────────────┘
        ↓
Worker 线程池执行任务
        ↓
大任务判断（>10 万行 / >50 页）？
        ├─ 否 → 直接执行
        └─ 是 → 分片为子任务
                  ↓
          子任务投递到 MQ
                  ↓
          调度器按 Round-Robin 穿插执行
                  ↓
          全部子任务完成后合并结果
```

**队列分级设计**：

```
RabbitMQ Exchange: jinshu.task.direct
├── Queue: jinshu.task.p0  (优先级 10)  ← 预览、<1万行导出
├── Queue: jinshu.task.p1  (优先级 7)   ← 1-10万行导出、<50页PDF
├── Queue: jinshu.task.p2  (优先级 5)   ← >10万行导出、>50页PDF
├── Queue: jinshu.task.p3  (优先级 3)   ← 数据同步任务
└── Queue: jinshu.task.p4  (优先级 1)   ← 后台批量、定时任务
```

### 3.2 模块划分

```
com.jinshu.worker          # Worker 模块
├── scheduler/
│   ├── TaskPriorityCalculator.java    # 优先级计算
│   ├── TenantFairScheduler.java       # 公平调度器核心

│   ├── TaskShardingService.java       # 大任务分片服务
│   ├── TaskCancellationService.java   # 任务取消服务
│   └── WorkerLoadBalancer.java        # Worker 负载均衡
│
com.jinshu.api             # API 模块（管理接口）

│
com.jinshu.common          # 公共模块
```

### 3.3 关键算法与策略

**优先级计算算法**

```java
public int calculatePriority(TaskType type, int estimatedRows, int estimatedPages) {
    // 基础优先级由任务类型决定
    int basePriority = switch (type) {
        case PREVIEW -> 10;        // 预览最高
        case EXPORT -> 7;
        case PDF -> 7;
        case DATA_SYNC -> 3;
        case BATCH -> 1;
    };
    
    // 大数据量任务降低优先级（避免阻塞）
    if (estimatedRows > 100_000 || estimatedPages > 50) {
        basePriority -= 2;  // P0→P1, P1→P2
    }
    
    return Math.max(basePriority, 1);
}
```

**租户并发配额管理（Redis）**

```
Redis Key: jinshu:tenant:{tenantId}:running_tasks  (String, 计数器)
Redis Key: jinshu:tenant:{tenantId}:quota           (String, 配额上限)
Redis Key: jinshu:scheduler:tenant_order            (List, 租户轮转顺序)
```

配额检查逻辑：

```java
Long current = redisTemplate.opsForValue().increment("jinshu:tenant:" + tenantId + ":running_tasks");
Long quota = redisTemplate.opsForValue().get("jinshu:tenant:" + tenantId + ":quota");
if (current > quota) {
    redisTemplate.opsForValue().decrement("jinshu:tenant:" + tenantId + ":running_tasks");
    return false; // 配额超限，放回队列
}
// 任务完成后 decremented
```

**时间片轮转调度（Round-Robin）**

```
每轮调度周期 = 100ms
        ↓
从租户顺序列表中依次取一个租户
        ↓
检查该租户是否有待执行任务 && 未超配额
        ↓
有 → 分配任务给 Worker，租户移到列表末尾
无 → 跳过，取下一个租户
        ↓
一轮结束后，若仍有空闲 Worker，开启下一轮
```

**大任务分片策略**

| 任务类型 | 分片阈值   | 分片大小   | 最大子任务数 |
| -------- | ---------- | ---------- | ------------ |
| 导出     | > 10 万行  | 10 万行/片 | 10           |
| PDF      | > 50 页    | 50 页/片   | 10           |
| 导入     | > 10 万行  | 10 万行/片 | 10           |
| 数据同步 | > 100 万行 | 100 万行/片| 10           |

**子任务交错执行策略**

```
原队列: [租户A-子任务1] [租户A-子任务2] [租户A-子任务3] [租户B-小任务]
        ↓ 调度器重组
新队列: [租户A-子任务1] [租户B-小任务] [租户A-子任务2] [租户C-小任务] [租户A-子任务3]
```

> 每个子任务视为独立任务参与 Round-Robin，确保大任务的子任务之间插入其他租户任务。

## 4. 数据库设计

### 4.1 表结构变更

**复用现有 `task` 表**，新增 `parent_task_id` 字段用于关联子任务（大任务分片场景）。

```sql
-- 在 task 表上增加父任务关联（支持大任务分片）
ALTER TABLE task ADD COLUMN parent_task_id BIGINT DEFAULT NULL;
ALTER TABLE task ADD COLUMN shard_seq INTEGER DEFAULT NULL;  -- 分片序号
ALTER TABLE task ADD COLUMN shard_total INTEGER DEFAULT NULL; -- 总分片数

CREATE INDEX idx_task_parent ON task(parent_task_id);
```

**新增 task_log 表**（记录调度历史，用于审计和故障排查）：

```sql
CREATE TABLE task_log (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       BIGINT NOT NULL,
    task_id         BIGINT NOT NULL,
    operation       VARCHAR(32) NOT NULL,  -- SCHEDULED / STARTED / COMPLETED / FAILED / CANCELLED
    worker_node     VARCHAR(64),           -- 执行节点标识
    priority        INTEGER,
    shard_seq       INTEGER,
    message         TEXT,
    created_at      TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_task_log_task ON task_log(task_id, created_at DESC);
CREATE INDEX idx_task_log_tenant ON task_log(tenant_id, created_at DESC);
```

**租户配额配置**（复用 `tenant.quota_config` JSONB 字段）：

```json
{
  "max_concurrent_tasks": 3,      // 最大并发任务数
  "max_export_rows": 1000000,     // 单次导出最大行数
  "max_pdf_pages": 500,           // 单次 PDF 最大页数
  "daily_task_quota": 100,        // 每日任务配额
  "priority_boost": 0             // 优先级加成（VIP 租户可配置）
}
```

### 4.2 索引设计

```sql
-- task 表已有索引
CREATE INDEX idx_task_tenant_status ON task(tenant_id, status);
CREATE INDEX idx_task_priority ON task(priority DESC, created_at);

-- 新增索引
CREATE INDEX idx_task_parent ON task(parent_task_id);
CREATE INDEX idx_task_log_task ON task_log(task_id, created_at DESC);
```

### 4.3 数据迁移方案

- `task` 表新增字段允许 NULL，无需数据迁移
- `task_log` 表为新增表，无存量数据
- 租户配额配置通过管理接口动态写入 Redis，不强制要求立即更新数据库

## 6. 异常处理

| 异常场景               | 错误码 | 提示信息                     | 处理方式                                         |
| ---------------------- | ------ | ---------------------------- | ------------------------------------------------ |
| 调度器无可用 Worker    | —      | —                            | 任务保留在 MQ，触发 HPA 扩容告警                 |
| 租户配额超限           | 429    | 当前任务已满，请等待         | 放回队列，等待该租户任务完成后再调度             |
| 任务取消时已完成       | 400    | 任务已完成，无法取消         | 直接返回，不修改状态                             |
| 子任务全部失败         | 500    | 任务执行失败，请重新提交     | 父任务标记 FAILED，子任务保留日志                |
| 子任务部分成功         | —      | —                            | 父任务标记 PARTIAL，返回已完成分片数和失败分片数 |
| Redis 配额计数器丢失   | —      | —                            | 从数据库重建计数（查询该租户 PROCESSING 任务数） |
| 调度器 Leader 选举失败 | —      | —                            | 退化为简单轮询模式，所有 Worker 独立消费 MQ      |
| 死信队列堆积           | —      | —                            | 触发人工介入告警，管理员手动处理或批量重试       |

## 7. 性能考量

### 7.1 性能目标

| 场景                   | 指标       |
| ---------------------- | ---------- |
| 调度决策延迟           | < 10 ms    |
| 小任务排队时间（P0）   | < 3 秒     |
| 支持并发任务数         | ≥ 50       |
| 支持活跃租户数         | ≥ 100      |
| 大任务分片对其他租户影响 | 额外等待 < 10 秒 |
| Redis 配额检查 QPS     | ≥ 1000     |

### 7.2 性能优化点

1. **批量拉取**：Worker 每次从 MQ 批量拉取 10 条消息，减少网络往返
2. **本地缓存**：租户配额配置缓存到 Worker 本地 Caffeine，TTL 30 秒，减少 Redis 查询
3. **异步计数**：任务完成后异步更新 Redis 计数器，不阻塞下一个调度周期
4. **连接池复用**：RabbitMQ Connection/Channel 池化，避免频繁创建连接
5. **无锁调度**：Round-Robin 租户列表使用无锁队列（Disruptor），避免线程竞争

**调度器吞吐量估算**：

```
每轮调度周期 = 100ms
每轮可调度任务数 = min(活跃租户数, Worker 空闲数)
假设：100 活跃租户，20 个 Worker
每轮调度 20 个任务
每秒调度 = 20 × 10 = 200 任务/秒
远超预期峰值（50 并发）✅
```

## 8. 安全考量

### 8.1 权限控制

- 队列状态、调度器状态、租户配额调整：仅管理员角色可访问
- 任务取消：用户只能取消自己租户下的任务，管理员可取消任意任务
- 配额调整操作记录审计日志（`operation = 'QUOTA_CHANGE'`）

### 8.2 数据安全

- 调度决策不接触业务数据，仅操作任务元数据
- task_log 表记录完整调度历史，用于审计和故障追溯
- 租户配额配置变更必须通过管理接口，禁止直接修改 Redis

### 8.3 防攻击

- **配额耗尽攻击**：单租户每日任务配额限制（`daily_task_quota`），超限直接拒绝
- **优先级操纵**：优先级由系统根据任务类型和数据量自动计算，用户不可指定
- **取消风暴**：任务取消接口限制 10 次/分钟/用户，防止恶意批量取消
- **MQ 消息篡改**：所有 MQ 消息带 HMAC 签名，Worker 消费前校验签名

### 9.2 集成测试

- **公平性测试**：租户 A 持续提交大任务，租户 B 提交小任务，验证 B 的任务在 3 秒内得到执行

- **分片合并测试**：100 万行导出分 10 片执行，验证合并后文件完整、顺序正确

## 10. 上线方案

### 10.1 上线步骤

1. 部署 `worker` 模块更新包（新增调度器组件）
2. 执行 DDL：为 `task` 表添加 `parent_task_id`、`shard_seq`、`shard_total` 字段
3. 创建 `task_log` 表
4. 配置 RabbitMQ 优先级队列（`x-max-priority`）
5. 初始化 Redis 租户配额（默认 `max_concurrent_tasks = 2`）
6. 执行集成测试（公平性 + 并发压力）

### 10.2 灰度策略

- 首日调度器以"观察模式"运行：记录调度决策但不实际拦截，对比原先进先出模式
- 次日启用调度器，仅对新提交任务生效，已在队列中的任务不受影响
- 观察 48 小时，确认小任务排队时间和失败率达标后全量生效

### 10.3 监控告警

| 调度决策延迟 P99       | > 10 ms      | 企业微信        |

## 11. 回滚方案

### 11.1 回滚触发条件

- 小任务排队时间 > 30 秒（大面积阻塞）
- 调度器导致任务丢失或重复执行
- 租户配额配置错误导致大面积拒绝服务
- Worker 负载严重不均衡（部分满载、部分空闲）

### 11.2 回滚步骤

1. 关闭调度器开关（`scheduler.enabled = false`）
2. Worker 退化为简单 MQ 消费者模式（先进先出，无优先级、无配额限制）
3. 回滚 `worker` 模块到上一版本
4. 保留 Redis 配额数据，不删除
5. 通知管理员：调度功能临时回退，任务按提交顺序执行

### 11.3 数据补救

- 回滚期间正在执行的调度决策：记录到 task_log，人工核对是否有任务遗漏
- 已分片的子任务：若父任务未完成，标记 FAILED，通知用户重新提交
- 配额配置数据保留在 Redis，调度器恢复后自动生效
