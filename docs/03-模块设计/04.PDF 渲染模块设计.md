## 1. 背景与目标

### 1.1 问题背景

报表系统不仅需要支持数据导出（Excel/CSV），还需要将动态网页报表转换为高质量、可打印的 PDF 文件。PDF 渲染是资源密集型操作，涉及浏览器引擎、大内存消耗、多页排版、色彩空间转换等复杂问题。当前系统尚未实现 PDF 渲染功能，需从零设计。

核心挑战：

- **样式一致性**：动态网页（React + ECharts）转换为 PDF 时，CSS 和图表必须 100% 还原
- **内存风险**：Chrome Headless 渲染大报表时，单实例内存可能飙升至 1GB+，导致 Pod OOM
- **大页数处理**：100 页以上的复杂报表，单进程渲染耗时过长，容易阻塞
- **印刷级质量**：部分行业（金融、制造业）要求 PDF 支持 CMYK 色彩空间，确保专业打印无色差
- **服务稳定性**：Chrome 进程存在内存泄漏风险，长期运行后可能崩溃

### 1.2 设计目标

| 指标               | 要求                              |
| ------------------ | --------------------------------- |
| 简单 PDF（10 页）  | < 5 秒                            |
| 复杂 PDF（100 页） | < 30 秒                           |
| 单 Chrome 实例内存 | < 512MB                           |
| 最大支持页数       | 500 页（分段渲染）                |
| 样式还原度         | 与网页预览 100% 一致              |
| 色彩空间           | 默认 RGB，可选 CMYK（印刷级）     |
| 失败重试           | 自动重试 3 次，超次降级为兜底方案 |

### 1.3 非目标

- 不支持动态交互式 PDF（如表单填写、按钮点击）
- 不支持 IE 浏览器兼容（仅支持 Chromium 内核）
- 首次版本不实现 Browserless 网关服务（预留扩展接口）
- 不实现前端实时预览 PDF（先生成后下载）

## 2. 需求分析

### 2.1 功能需求

- [ ] **网页直出渲染**：Puppeteer 访问前端预览页，生成与网页 100% 一致的 PDF
- [ ] **分页控制**：通过 CSS `page-break-after` 精确控制分页位置，避免表格/图表被截断
- [ ] **分段渲染**：大报表（>50 页）自动拆分为多个子任务并行渲染，PDFBox 合并
- [ ] **双方案兜底**：Puppeteer 异常时自动降级为 Thymeleaf 纯后端模板渲染
- [ ] **CMYK 转换**：支持将 RGB PDF 转换为 CMYK 印刷级 PDF（Ghostscript）
- [ ] **水印注入**：支持在 PDF 中注入隐形溯源水印（8x8 点阵编码用户 ID+时间戳）
- [ ] **进度实时查询**：Redis 存储各分段进度，汇总后返回总进度
- [ ] **下载链接生成**：任务完成后生成带签名的临时下载链接

### 2.2 用户故事

```
作为 报表审查人，我希望 将审批通过的报表导出为 PDF，
以便 打印存档并提交给上级部门。

作为 财务人员，我希望 PDF 中的表格和图表与网页预览完全一致，
以便 打印后无需二次核对。

作为 制造业工程师，我希望 PDF 支持 CMYK 色彩空间，
以便 直接发送给印刷厂，确保打印成品无色差。
```

### 2.3 边界场景

| 场景                     | 处理方式                                                  |
| ------------------------ | --------------------------------------------------------- |
| 0 页报表                 | 生成 1 页空白 PDF（含表头），标记 SUCCESS                 |
| 500+ 页超大型报表        | 按每 50 页分片，最多 10 个子任务并行渲染                  |
| Chrome 崩溃/OOM          | 连接池自动销毁异常实例，任务重新投递 MQ，计数重试         |
| Puppeteer 启动失败       | 降级为 Thymeleaf 模板渲染，保证可用性                     |
| 分段合并时文件损坏       | 合并前校验每个分片 MD5，失败则重新渲染该分片              |
| 渲染超时（>60 秒）       | 强制终止 Chrome 进程，标记该分片 FAILED，触发重试         |
| 前端页面加载失败（5xx）  | Puppeteer 等待 10 秒后重试，失败则降级 Thymeleaf          |
| CMYK 转换失败            | 保留原始 RGB PDF，标记 warning，不影响主流程              |

## 3. 架构设计

### 3.1 整体流程

```
用户点击"生成 PDF"
        ↓
[同步] POST /api/pdf/submit
创建 task 记录，状态 = PENDING
发送消息到 RabbitMQ jinshu.render 队列
立即返回 taskId（HTTP 202 Accepted）
        ↓
[异步] Worker 模块消费消息，开始编排
        ↓
┌──────────────────────────────────────────────────────────┐
│  1. 查询报表页数/维度，决定分片策略                        │
│     - ≤ 50 页：单任务渲染                                 │
│     - > 50 页：按每 50 页拆分为子任务                      │
│  2. 每个子任务生成独立渲染参数（pageRanges）               │
│  3. 子任务投递到 MQ jinshu.render.segment 队列            │
└──────────────────────────────────────────────────────────┘
        ↓
[异步] Renderer 服务（Node + Puppeteer）消费子任务
        ↓
┌──────────────────────────────────────────────────────────┐
│  1. 从连接池获取 Chrome 实例                              │
│  2. 携带 JWT Token 访问前端预览页                          │
│     URL: /preview?reportId=xxx&printMode=true             │
│  3. 等待页面完全加载（含 ECharts 图表渲染完成）            │
│  4. 调用 printToPDF(pageRanges) 生成分段 PDF               │
│  5. 将分段 PDF 写入 PVC: /data/temp/{taskId}/              │
│  6. 释放 Chrome 实例回连接池                               │
└──────────────────────────────────────────────────────────┘
        ↓
所有子任务完成后，Worker 调用 PDFBox 合并分段 PDF
        ↓
[可选] Ghostscript 将 RGB PDF 转换为 CMYK PDF
        ↓
[可选] PDFBox 注入隐形水印
        ↓
更新 task 状态 = SUCCESS，记录 output_file_path
        ↓
Redis 推送完成通知 / WebSocket 通知前端
        ↓
用户调用 GET /api/pdf/{taskId}/download 获取签名下载链接
        ↓
24 小时后 CronJob 自动清理临时文件
```

**兜底流程（Puppeteer 失败时）**：

```
Puppeteer 渲染失败（3 次重试后）
        ↓
降级为 Thymeleaf 纯后端模板
        ↓
Java 读取报表数据 → Thymeleaf 填充 HTML 模板
        ↓
使用 OpenHTMLToPDF / Flying Saucer 生成 PDF
        ↓
无 CMYK 转换、无隐形水印、样式简化
        ↓
标记 task.warning = "DEGRADED_MODE"
```

### 3.3 关键算法与策略

**分片策略**

| 总页数   | 分片大小 | 子任务数 | 并发限制 |
| -------- | -------- | -------- | -------- |
| ≤ 50 页  | 不分片   | 1        | 1        |
| 51-100 页| 50 页/片 | 2        | 2        |
| 101-300 页| 50 页/片| 3-6      | 3        |
| 301-500 页| 50 页/片| 7-10     | 3        |
| > 500 页 | 拒绝，提示"请联系管理员" | — | — |

> 并发限制 3 是为了避免同时启动过多 Chrome 实例导致内存溢出。

**Puppeteer 连接池策略**

```javascript
const pool = {
  min: 2,           // 最少保持 2 个实例热备
  max: 10,          // 最多 10 个实例
  idleTimeout: 30000,  // 闲置 30 秒自动销毁
  maxUses: 50,      // 每个实例最多处理 50 个任务后强制重启（防内存泄漏）
  acquireTimeout: 10000  // 获取实例超时 10 秒
};
```

**Chrome 启动参数（内存限制）**

```javascript
const launchOptions = {
  headless: 'new',
  args: [
    '--no-sandbox',
    '--disable-setuid-sandbox',
    '--disable-dev-shm-usage',
    '--disable-accelerated-2d-canvas',
    '--no-first-run',
    '--no-zygote',
    '--single-process',
    '--disable-gpu',
    `--js-flags=--max-old-space-size=512`,  // JS 堆内存 512MB
    `--max-old-space-size=512`               // V8 内存上限
  ]
};
```

**printToPDF 参数优化**

```javascript
const pdfOptions = {
  format: 'A4',
  printBackground: true,        // 打印背景色和背景图
  preferCSSPageSize: true,      // 优先使用 CSS @page 定义的尺寸
  margin: { top: '1cm', bottom: '1cm', left: '1cm', right: '1cm' }
};
```

**分段合并校验**

```
1. 收集所有分片文件：segment_001.pdf, segment_002.pdf, ...
2. 校验每个分片：
   - 文件大小 > 0
   - 页数符合预期（PDFBox 读取页数）
   - MD5 校验（生成时计算，合并前校验）
3. 使用 PDFBox PDFMergerUtility 按顺序合并
4. 合并后校验：
   - 总页数 = 各分片页数之和
   - 目录/书签保持连续
```

## 7. 性能考量

### 7.1 性能目标

| 场景                  | 指标    |
| --------------------- | ------- |
| 10 页简单 PDF         | < 5 秒  |
| 100 页复杂 PDF        | < 30 秒 |
| 单 Chrome 实例内存    | < 512MB |
| 连接池最大实例数      | 10      |
| 单实例最大处理任务数  | 50 个后强制重启 |
| 分段合并耗时          | < 2 秒（10 个分片） |

### 7.2 性能优化点

1. **连接池热备**：保持 `min: 2` 个 Chrome 实例常驻内存，避免频繁启动（启动耗时 2-3 秒）
2. **分片并行**：多子任务通过 MQ 分发到多个 Worker 节点，利用多核并行渲染
3. **页面等待策略**：Puppeteer 等待 `networkidle2`（网络空闲 2 个连接），而非 `load`，加速 ECharts 图表渲染完成判定
4. **图片懒加载规避**：前端 `printMode=true` 时，强制所有图片同步加载，避免 Puppeteer 等待懒加载
5. **PDFBox 合并优化**：使用 `MemoryUsageSetting.setupTempFileOnly()`，避免大 PDF 合并时内存溢出
6. **资源限制**：K8s 层面为 Renderer Pod 设置 `resources.limits.memory = 2Gi`，`resources.requests.memory = 1Gi`

**内存估算（单 Chrome 实例）**：

```
Chrome 基础进程     ≈ 100MB
每页 DOM + CSS      ≈ 2MB
每页 ECharts 画布   ≈ 3MB
50 页报表           ≈ 250MB
+ JS 堆内存上限     = 512MB
------------------------
峰值预估 ≈ 450MB < 512MB ✅
```

## 8. 安全考量

### 8.1 权限控制

- 只有报表所属租户的用户可以生成该报表的 PDF（通过 `tenant_id` 过滤）
- 访客角色（GUEST）只能生成"已公开"状态报表的 PDF
- 下载链接使用 JWT 签名，包含 `taskId`、`userId`、`tenantId`、`exp`

### 8.2 数据安全

- 所有 PDF 生成操作记录审计日志（`operation = 'PDF_GENERATE'`）
- Puppeteer 访问的前端 URL 必须带 `printMode=true`，该模式下：
  - 隐藏导航栏、按钮、下拉菜单等交互元素
  - 隐藏敏感数据（如完整手机号、身份证号）
- 隐形水印编码 `userId + timestamp`，不存储明文，仅用于泄密溯源
- 下载链接有效期 1 小时

### 8.3 防攻击

- **URL 白名单**：Puppeteer 只允许访问系统内部前端域名，禁止访问外部链接（防 SSRF）
- **资源限制**：Chrome 启动参数限制 `--max-old-space-size=512`，防止恶意页面耗尽内存
- **渲染超时**：单任务 60 秒强制终止，防止死循环或恶意脚本阻塞
- **文件路径校验**：输出路径必须匹配 `/data/temp/{taskId}/` 或 `/data/output/{taskId}/`，禁止目录遍历

## 11. 回滚方案

### 11.1 回滚触发条件

- PDF 任务失败率 > 30%
- Renderer 频繁 OOM 或崩溃
- Puppeteer 大面积降级（>50%）
- 下载链接存在越权漏洞
- 合并后的 PDF 出现页码错乱或内容缺失

### 11.2 回滚步骤

1. 关闭 PDF 功能开关（`pdf.enabled = false`，API 层直接返回 503）
2. 回滚 `api`、`worker`、`renderer` 到上一版本
3. 清空 RabbitMQ `jinshu.render` 和 `jinshu.render.segment` 队列
4. Kill 所有 Renderer Pod，确保 Chrome 进程全部释放
5. 通知用户：PDF 生成功能临时维护

### 11.3 数据补救

- 回滚期间未完成的 `task` 记录：批量更新 `status = 'FAILED'`，`error_message = '系统维护，请重新提交'`
- 临时文件保留 7 天，不立即删除
- 已生成的 PDF 文件保留在 `/data/output/`，已获取下载链接的用户仍可下载
