## 1. 背景与目标

### 1.1 问题背景

锦书系统涉及 5 个后端子模块（api/batch/worker/data-sync/renderer）、消息队列、PostgreSQL、Redis 等多个组件，请求链路长（前端 → API → MQ → Worker → Renderer → PDF）。当前系统缺乏：

| 问题                     | 严重度 | 影响                                           |
| ------------------------ | ------ | ---------------------------------------------- |
| 无指标采集               | 🔴 致命 | 无法感知导入/导出/PDF 的成功率、耗时、QPS       |
| 无全链路追踪             | 🔴 致命 | 任务失败时无法定位故障环节（API 超时？Worker 堆积？Renderer OOM？） |
| 日志无结构化             | 🟡 高   | 多租户场景下无法按 tenantId/taskId 检索日志     |
| 无告警机制               | 🟡 高   | 队列积压、Pod OOM 等故障无法即时通知运维人员    |
| 前端报错不可见           | 🟡 高   | 用户白屏/JS 错误无人知晓，只能等用户反馈        |

### 1.2 设计目标

| 指标               | 要求                                                 |
| ------------------ | ---------------------------------------------------- |
| 指标覆盖           | 所有核心业务操作（导入/导出/PDF）+ 所有基础设施组件  |
| 全链路追踪         | TraceId 从前端到 Renderer 全链路透传，不中断         |
| 日志检索           | 支持按 traceId/tenantId/taskId/userId 实时检索       |
| 告警响应           | P0 故障（服务宕机）1 分钟内通知，P1（性能劣化）5 分钟 |
| 性能开销           | Metrics 采集 < 1% CPU，Tracing 采样 < 5% 请求延迟增加 |
| Dashboard          | Grafana 统一大盘，业务 + 系统指标一屏可览            |

### 1.3 非目标

- 不实现 APM（Application Performance Monitoring）商业产品集成（如 Datadog、Dynatrace）
- 不实现 profiling（火焰图、内存 dump 在线分析）—— 生产环境通过 Arthas 或 jmap 手动排查
- 不在 MVP 阶段实现日志存储长期归档策略（二期）

## 2. 总体架构

### 2.1 三大支柱


## 4. Tracing 全链路追踪

### 4.1 TraceId 生命周期

```
用户浏览器                      后端 API                     MQ                   Worker                 Renderer
─────────                      ────────                    ──                   ──────                 ────────
                               接收请求
点击按钮 ──── HTTP ──────►    提取/生成 TraceId
                              注入 UserContext
                                     │
                              执行业务逻辑
                                     │
                              发布 MQ 消息 ──────────►  消息 Header 中
                              携带 TraceId            携带 TraceId
                                                              │
                                                       消费消息
                                                       提取 TraceId
                                                       注入 MDC
                                                              │
                                                       执行导出任务
                                                              │
                                                       调用 Renderer ──────►  接收请求
                                                       HTTP Header             提取 TraceId
                                                       携带 TraceId            执行渲染
```

**TraceId 生成策略**：

- 前端发起请求时生成 UUID v4 格式的 TraceId（减少后端无 TraceId 的请求）
- 如请求未携带 `X-Trace-Id`，由 API 的 Filter 生成后补全
- TraceId 格式：32 位 hex（如 `a1b2c3d4e5f6789012345678abcdef01`）

### 4.2 TraceId 透传实现

**(1) HTTP 请求透传（前端 → API → Worker → Renderer）**

```java
// common/src/main/java/com/jinshu/common/context/TraceContext.java
public class TraceContext {
    private static final ThreadLocal<String> TRACE_ID = new ThreadLocal<>();

    public static void setTraceId(String traceId) {
        TRACE_ID.set(traceId);
        MDC.put("trace_id", traceId); // 同步到日志 MDC
    }

    public static String getTraceId() {
        String id = TRACE_ID.get();
        return id != null ? id : generateTraceId();
    }

    public static void clear() {
        TRACE_ID.remove();
        MDC.remove("trace_id");
    }

    private static String generateTraceId() {
        return UUID.randomUUID().toString().replace("-", "");
    }
}
```

```java
// common/src/main/java/com/jinshu/common/security/TraceFilter.java
@Component
@Order(Ordered.HIGHEST_PRECEDENCE + 10)
public class TraceFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws IOException, ServletException {
        String traceId = request.getHeader("X-Trace-Id");
        if (traceId == null || traceId.isEmpty()) {
            traceId = UUID.randomUUID().toString().replace("-", "");
        }

        TraceContext.setTraceId(traceId);
        response.setHeader("X-Trace-Id", traceId); // 返回给前端

        try {
            chain.doFilter(request, response);
        } finally {
            TraceContext.clear();
        }
    }
}
```

**(2) MQ 消息透传**

```java
// common/src/main/java/com/jinshu/common/mq/MqMessageHeader.java
public class MqMessageHeader {
    private String traceId;
    private String tenantId;    // 从 TenantContext 获取
    private String userId;      // 从 UserContext 获取

    // 在发送消息时自动注入：
    public static MqMessageHeader fromContext() {
        MqMessageHeader header = new MqMessageHeader();
        header.setTraceId(TraceContext.getTraceId());
        header.setTenantId(TenantContext.getTenantId() != null
            ? TenantContext.getTenantId().toString() : null);
        header.setUserId(UserContext.getUserId() != null
            ? UserContext.getUserId().toString() : null);
        return header;
    }
}
```

**(3) 服务间 HTTP 调用透传（RestTemplate 拦截器）**

```java
// common/src/main/java/com/jinshu/common/config/RestTemplateTraceInterceptor.java
public class RestTemplateTraceInterceptor implements ClientHttpRequestInterceptor {

    @Override
    public ClientHttpResponse intercept(HttpRequest request, byte[] body,
                                        ClientHttpRequestExecution execution) throws IOException {
        String traceId = TraceContext.getTraceId();
        if (traceId != null) {
            request.getHeaders().set("X-Trace-Id", traceId);
        }
        return execution.execute(request, body);
    }
}
```

### 4.4 前端 TraceId 展示

```typescript
// services/request.ts
const request = axios.create({ /* ... */ });

// 请求拦截器：生成并注入 TraceId
import { v4 as uuidv4 } from 'uuid';

request.interceptors.request.use((config) => {
  const traceId = uuidv4().replace(/-/g, '');
  config.headers['X-Trace-Id'] = traceId;
  return config;
});

// 响应拦截器：提取 TraceId（如果后端覆写了）
request.interceptors.response.use((response) => {
  const backendTraceId = response.headers['x-trace-id'];
  if (backendTraceId) {
    (window as any).__lastTraceId = backendTraceId;
  }
  return response;
});
```

```tsx
// components/common/TraceIdDisplay.tsx — 错误页/Footer 中展示
const TraceIdDisplay: React.FC = () => {
  const [traceId, setTraceId] = useState('');

  useEffect(() => {
    setTraceId((window as any).__lastTraceId || '');
  }, []);

  if (!traceId) return null;
  return (
    <span className="trace-id" style={{ fontSize: 11, color: '#999' }}>
      Trace ID: {traceId}
      <button onClick={() => navigator.clipboard.writeText(traceId)}>复制</button>
    </span>
  );
};
```

## 5. Logging 结构化日志

### 5.1 日志字段规范

所有业务日志输出为 JSON 格式，必须包含以下保留字段：

| 字段          | 类型   | 必填 | 说明                                   |
| ------------- | ------ | :--: | -------------------------------------- |
| `timestamp`   | String |  ✓   | ISO 8601 时间戳                        |
| `level`       | String |  ✓   | INFO/WARN/ERROR                        |
| `logger`      | String |  ✓   | Logger 名称                            |
| `message`     | String |  ✓   | 日志消息                               |
| `trace_id`    | String |  ✓   | 全链路追踪 ID（MDC 自动注入）          |
| `tenant_id`   | String |  ✓   | 租户 ID（TenantContext 注入）          |
| `user_id`     | String |      | 用户 ID（登录态下必填）                |
| `task_id`     | String |      | 异步任务 ID（Worker/Batch 必填）       |
| `service`     | String |  ✓   | 服务名（`spring.application.name`）    |
| `exception`   | String |      | 异常堆栈（ERROR 级别自动包含）         |

```json
  "logger": "com.jinshu.api.service.ReportService",
```

### 5.2 日志级别规范

| 级别  | 使用场景                                           | 示例                                       |
| ----- | -------------------------------------------------- | ------------------------------------------ |
| ERROR | 需要人工介入的故障（数据库连接失败、MQ 断连）      | `log.error("PostgreSQL 连接失败", e)`      |
| WARN  | 可自动恢复的异常（重试成功、数据校验失败跳过）     | `log.warn("导入行校验失败，跳过: {}", row)` |
| INFO  | 正常业务流程关键节点（任务开始/完成、状态变更）    | `log.info("PDF 生成完成, taskId={}", id)`  |
| DEBUG | 开发调试信息（SQL 参数、MQ 消息体）                | `log.debug("执行 SQL: {}", sql)`           |
| TRACE | 极细粒度（方法进入/退出、循环内日志）              | 不用于生产环境                             |

**禁止事项**：

- ❌ 禁止 `System.out.println()` 和 `e.printStackTrace()`
- ❌ 禁止在循环内打印 INFO 级别日志
- ❌ 禁止打印敏感数据（密码、Token、身份证号原文）
- ❌ 禁止打印大对象（> 1KB 的 JSON/XML）

### 5.3 Logback 配置

```xml
<!-- logback-spring.xml -->
<configuration>
    <!-- 控制台输出（JSON 格式，开发环境） -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder class="net.logstash.logback.encoder.LogstashEncoder">
            <includeMdcKeyName>trace_id</includeMdcKeyName>
            <includeMdcKeyName>tenant_id</includeMdcKeyName>
            <includeMdcKeyName>user_id</includeMdcKeyName>
            <includeMdcKeyName>task_id</includeMdcKeyName>
            <fieldNames>
                <timestamp>timestamp</timestamp>
                <version>[ignore]</version>
            </fieldNames>
        </encoder>
    </appender>

    <!-- 文件输出（JSON 格式，生产环境） -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>/var/log/jinshu/api.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>/var/log/jinshu/api.%d{yyyy-MM-dd}.log</fileNamePattern>
            <maxHistory>30</maxHistory>
        </rollingPolicy>
        <encoder class="net.logstash.logback.encoder.LogstashEncoder">
            <includeMdcKeyName>trace_id</includeMdcKeyName>
            <includeMdcKeyName>tenant_id</includeMdcKeyName>
            <includeMdcKeyName>user_id</includeMdcKeyName>
            <includeMdcKeyName>task_id</includeMdcKeyName>
            <fieldNames>
                <timestamp>timestamp</timestamp>
            </fieldNames>
        </encoder>
    </appender>

    <root level="INFO">
        <appender-ref ref="CONSOLE" />
        <appender-ref ref="FILE" />
    </root>

    <!-- 生产环境关闭 DEBUG 日志 -->
    <logger name="com.jinshu" level="${LOG_LEVEL:-INFO}" />
</configuration>
```

> MVP 阶段：仅使用 `CONSOLE` appender，由 K8s stdout 收集 + Promtail 推送到 Loki。

### 5.4 日志收集架构

```

     │
     ▼

Loki (日志存储 + 索引)
```

## 6. 告警规则

### 6.1 告警分级

| 级别 | 说明                           | 通知方式           | 响应时间 |
| :--: | ------------------------------ | ------------------ | :------: |
|  P0  | 服务不可用 / 数据丢失          | 企业微信 + 短信    | 5 分钟   |
|  P1  | 核心功能性能劣化               | 企业微信           | 30 分钟  |
|  P2  | 非核心指标异常 / 接近阈值      | 邮件               | 4 小时   |

### 6.2 告警规则定义

```yaml
# prometheus-rules.yml
groups:
  - name: jinshu_critical
    rules:
      # P0: 服务宕机
      - alert: ServiceDown
        expr: up{job=~"jinshu-.*"} == 0
        for: 1m
        labels:
          severity: P0
        annotations:
          summary: "服务 {{ $labels.job }} 已宕机"
          description: "{{ $labels.job }} 超过 1 分钟无响应"

      # P0: API 错误率过高
      - alert: HighErrorRate
        expr: rate(http_server_requests_seconds_count{status=~"5.."}[5m]) / rate(http_server_requests_seconds_count[5m]) > 0.05
        for: 2m
        labels:
          severity: P0
        annotations:
          summary: "API 5xx 错误率 > 5%"
          description: "当前错误率 {{ $value | humanizePercentage }}"

      # P1: API 响应慢
      - alert: SlowResponse
        expr: histogram_quantile(0.95, rate(http_server_requests_seconds_bucket[5m])) > 2
        for: 5m
        labels:
          severity: P1
        annotations:
          summary: "API P95 响应时间 > 2s"
          description: "当前 P95: {{ $value }}s"

      # P1: 数据库连接池耗尽
      - alert: DatabaseConnectionPoolExhausted
        expr: hikaricp_connections_pending > 5
        for: 1m
        labels:
          severity: P1
        annotations:
          summary: "数据库连接池耗尽"
          description: "{{ $value }} 个请求在等待连接"

      # P1: 任务队列积压
      - alert: TaskQueueBacklog
        expr: jinshu_task_queue_depth > 1000
        for: 5m
        labels:
          severity: P1
        annotations:
          summary: "任务队列积压 > 1000"
          description: "当前积压: {{ $value }}"

      # P1: PDF 成功率下降
      - alert: PdfSuccessRateDrop
        expr: |
          sum(rate(jinshu_pdf_generate_total{status="success"}[5m]))
          /
          sum(rate(jinshu_pdf_generate_total[5m])) < 0.95
        for: 5m
        labels:
          severity: P1
        annotations:
          summary: "PDF 生成成功率 < 95%"

      # P2: 磁盘空间不足
      - alert: LowDiskSpace
        expr: disk_free_bytes / disk_total_bytes < 0.2
        for: 10m
        labels:
          severity: P2
        annotations:
          summary: "磁盘使用率 > 80%"

      # P2: JVM 堆内存使用过高
      - alert: HighHeapUsage
        expr: jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} > 0.85
        for: 5m
        labels:
          severity: P2
        annotations:
          summary: "JVM 堆内存使用 > 85%"
```

## 7. 自定义指标埋点

### 7.1 业务指标代码示例

```java
// common/src/main/java/com/jinshu/common/metrics/BusinessMetrics.java
@Component
public class BusinessMetrics {

    private final MeterRegistry registry;

    // Counter: 导入次数
    private final Counter importCounter;
    // Histogram: 导入耗时
    private final Timer importTimer;

    public BusinessMetrics(MeterRegistry registry) {
        this.registry = registry;
        this.importCounter = Counter.builder("jinshu_report_import_total")
            .description("报表导入总次数")
            .register(registry);
        this.importTimer = Timer.builder("jinshu_report_import_duration_seconds")
            .description("报表导入耗时")
            .register(registry);
    }

    // 在 ImportService 中调用
    public void recordImport(String tenantId, boolean success, long durationMs) {
        importCounter
            .withTags("tenant_id", tenantId, "status", success ? "success" : "failed")
            .increment();
        importTimer.record(durationMs, TimeUnit.MILLISECONDS);
    }

    // Gauge: 活跃任务数
    public void setActiveTaskCount(String tenantId, String type, long count) {
        Gauge.builder("jinshu_task_active", () -> count)
            .tags("tenant_id", tenantId, "type", type)
            .register(registry);
    }
}
```

## 8. 异常处理与限流

- Prometheus 抓取端点（`/actuator/prometheus`）必须对 Prometheus ServiceMonitor 开放，不校验 JWT Token
- 日志不得输出密码、密钥、Token、身份证号原文（通过 Logback `<replace>` 或 MDC 过滤）
- Prometheus 端点限制内网访问（K8s NetworkPolicy 仅允许 monitoring namespace）
- 日志写入磁盘大小限制：单文件 ≤ 500MB，保留 ≤ 30 天（`maxHistory: 30`）
- 采样率：生产环境 Tracing 采样 10%，测试/预发环境 100%

## 9. 性能考量

| 组件           | 性能开销           | 优化措施                           |
| -------------- | ------------------ | ---------------------------------- |
| Micrometer     | < 0.5% CPU         | 零代码侵入，JVM 内置采集            |
| Prometheus 抓取| 每 15s 一次 HTTP   | 端点响应 < 10ms，无业务影响         |
| OpenTelemetry  | < 2% 延迟增加       | 采样率 10%，异步导出 Span           |
| JSON 日志编码  | < 5% 日志写入耗时  | LogstashEncoder 异步写入             |
| Promtail       | < 50MB 内存        | DaemonSet，不占应用资源             |

## 10. 安全考量

- `/actuator/prometheus` 端点走独立端口（`management.server.port: 8081`），不对外暴露
- Prometheus ServiceMonitor 通过 K8s RBAC 授权，仅 monitoring namespace 可访问
- Grafana 管理后台必须走 HTTPS + OAuth2 认证
- 日志中禁止输出 `Authorization` Header、密码、身份证号等敏感字段
- Alertmanager Webhook 使用 HTTPS + Token 校验

## 12. 上线方案

### 12.1 上线步骤

1. 引入 Micrometer + LogstashEncoder 依赖（`build.gradle.kts`）
2. 配置 `application.yml` 暴露 Prometheus 端点
3. 实现 `TraceFilter` + `TraceContext`
4. 配置 `logback-spring.xml` 输出 JSON 格式
5. MQ 消息体增加 `trace_id` Header
6. 部署 Prometheus Operator + Grafana（通过 Helm Chart）
7. 部署 Promtail DaemonSet + Loki（通过 Helm Chart）
8. 导入 Grafana Dashboard JSON
9. 配置 Alertmanager 告警规则
10. 验证：模拟故障 → 收到告警 → Grafana 大盘显示异常

### 12.2 灰度策略

- MVP 阶段先启用 Metrics + Logging，Tracing 延迟到二期
- Metrics 先采集不告警（静默模式），观察 1 周后逐步开启告警规则
- 告警通知先只发测试组，确认无误后扩大到全员

### 12.3 回滚方案

- Metrics 采集可随时关闭：`management.metrics.export.prometheus.enabled=false`
- Tracing 采样率设为 0 即可关闭（`management.tracing.sampling.probability=0`）
- JSON 日志可回退为文本格式（修改 Logback 配置，无需重启）
- 告警规则通过 Git 管理，随时回退到上一版本
