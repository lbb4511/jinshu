## 2. 用户与权限管理

### 2.1 角色体系

| 角色 | 权限范围 | 用途 |
|------|---------|------|
| `jinshu_admin` | 数据库超级用户 | DBA 日常运维 |
| `jinshu_app` | 业务表 CRUD + 序列使用 | 应用连接使用 |
| `jinshu_readonly` | 业务表 SELECT | 只读查询、监控 |
| `jinshu_backup` | `pg_dump` 权限 + COPY | 备份操作专用 |

创建角色：

```sql
-- DBA 角色
CREATE ROLE jinshu_admin WITH LOGIN SUPERUSER PASSWORD '强密码';

-- 应用角色（最小权限）
CREATE ROLE jinshu_app WITH LOGIN PASSWORD '应用密码';
GRANT CONNECT ON DATABASE jinshu TO jinshu_app;
GRANT USAGE ON SCHEMA public TO jinshu_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO jinshu_app;
GRANT USAGE ON ALL SEQUENCES IN SCHEMA public TO jinshu_app;

-- 只读角色
CREATE ROLE jinshu_readonly WITH LOGIN PASSWORD '只读密码';
GRANT CONNECT ON DATABASE jinshu TO jinshu_readonly;
GRANT USAGE ON SCHEMA public TO jinshu_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO jinshu_readonly;

-- 备份角色
CREATE ROLE jinshu_backup WITH LOGIN PASSWORD '备份密码';
GRANT CONNECT ON DATABASE jinshu TO jinshu_backup;
GRANT USAGE ON SCHEMA public TO jinshu_backup;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO jinshu_backup;
```

### 2.2 管理操作用户

```sql
-- 创建用户
CREATE USER "username" WITH PASSWORD '密码';

-- 授权
GRANT jinshu_app TO "username";

-- 撤销权限
REVOKE jinshu_app FROM "username";

-- 查看用户权限
\du
SELECT * FROM pg_roles;

-- 修改密码
ALTER USER "username" WITH PASSWORD '新密码';

-- 锁定用户
ALTER USER "username" WITH NOLOGIN;
```

## 3. 备份策略

### 3.1 物理备份 (pg_basebackup)

适合全量基础备份，用于 PITR：

```bash
# 创建基础备份
pg_basebackup -h localhost -U jinshu_backup -D /backup/base/$(date +%Y%m%d) \
  -Fp -Xs -P -v

# 压缩备份
pg_basebackup -h localhost -U jinshu_backup -D /backup/base/$(date +%Y%m%d) \
  -Ft -X s -z -P -v
```

### 3.2 逻辑备份 (pg_dump)

适合单库/单表备份，迁移和恢复更灵活：

```bash
# 全库备份
pg_dump -h localhost -U jinshu_backup -d jinshu -Fc -f /backup/jinshu_$(date +%Y%m%d).dump

# 压缩 SQL 格式（便于 grep 查看）
pg_dump -h localhost -U jinshu_backup -d jinshu -Fp | gzip > /backup/jinshu_$(date +%Y%m%d).sql.gz

# 单表备份
pg_dump -h localhost -U jinshu_backup -d jinshu -t audit_log -Fc \
  -f /backup/audit_log_$(date +%Y%m%d).dump

# 排除大表（report_data 可单独备份）
pg_dump -h localhost -U jinshu_backup -d jinshu -T report_data -Fc \
  -f /backup/jinshu_no_data_$(date +%Y%m%d).dump

# 多个并行任务加速 (pg_dump 4.0+)
pg_dump -h localhost -U jinshu_backup -d jinshu -j 4 -Fd \
  -f /backup/jinshu_dir_$(date +%Y%m%d)
```

### 3.3 WAL 归档

在 `postgresql.conf` 中启用 WAL 归档：

```ini
archive_mode = on
archive_command = 'test ! -f /archive/%f && cp %p /archive/%f'
archive_timeout = 300          # 每 5 分钟强制切换一次 WAL
wal_level = replica
max_wal_senders = 3
wal_keep_size = 1024           # 保留 1GB WAL
```

WAL 归档配合基础备份可实现任意时间点恢复 (PITR)。

### 3.4 备份保留策略

| 备份类型 | 频率 | 保留周期 | 存储位置 |
|---------|------|---------|---------|
| 物理全量备份 | 每周日 02:00 | 4 周 | 本地 + 远程 S3 |
| 逻辑全量备份 | 每日 03:00 | 30 天 | 本地 + 远程 S3 |
| WAL 归档 | 持续 | 7 天 | 本地 + 远程 S3 |

K3s CronJob 示例：

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: db-backup-daily
  namespace: jinshu
spec:
  schedule: "0 3 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: postgres:16-alpine
            command:
            - /bin/sh
            - -c
            - |
              pg_dump -h $DB_HOST -U $DB_USER -d $DB_NAME -Fc \
                -f /backup/jinshu_$(date +%Y%m%d_%H%M%S).dump
              # 删除 30 天前的备份
              find /backup -name "jinshu_*.dump" -mtime +30 -delete
            env:
            - name: DB_HOST
              value: jinshu-postgres.jinshu.svc.cluster.local
            - name: DB_USER
              value: jinshu_backup
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: jinshu-secret
                  key: db_password
            volumeMounts:
            - mountPath: /backup
              name: backup-volume
          restartPolicy: OnFailure
          volumes:
          - name: backup-volume
            persistentVolumeClaim:
              claimName: backup-pvc
```

### 3.5 备份验证

```bash
# 测试备份文件完整性
pg_restore -l /backup/jinshu_20260524.dump > /dev/null 2>&1 && echo "OK" || echo "CORRUPT"

# 定期在测试环境恢复验证（建议每月一次）
pg_restore -h test-db -U jinshu_admin -d jinshu_test --no-owner \
  /backup/jinshu_20260524.dump
```

## 4. 恢复操作

### 4.1 从逻辑备份恢复

```bash
# 恢复全库
pg_restore -h localhost -U jinshu_admin -d jinshu --clean --if-exists \
  /backup/jinshu_20260524.dump

# 恢复单表
pg_restore -h localhost -U jinshu_admin -d jinshu -t report_metadata \
  /backup/jinshu_20260524.dump

# 恢复到不同数据库
createdb -U jinshu_admin jinshu_restore
pg_restore -h localhost -U jinshu_admin -d jinshu_restore \
  /backup/jinshu_20260524.dump

# SQL 格式恢复
gunzip -c /backup/jinshu_20260524.sql.gz | psql -h localhost -U jinshu_admin -d jinshu
```

### 4.2 从物理备份+PITR 恢复

```bash
# 1. 停止 PostgreSQL
# 2. 清空数据目录
rm -rf /var/lib/postgresql/data/*

# 3. 恢复基础备份
tar -xzf /backup/base/20260524/base.tar.gz -C /var/lib/postgresql/data

# 4. 配置 recovery.conf (PG 12+ 使用 recovery.signal)
touch /var/lib/postgresql/data/recovery.signal

# 5. 配置恢复目标
cat >> /var/lib/postgresql/data/postgresql.conf <<EOF
restore_command = 'cp /archive/%f %p'
recovery_target_time = '2026-05-24 14:30:00+08'  # 恢复到误操作前的时刻
recovery_target_action = 'pause'                   # 恢复后暂停，确认后继续
EOF

# 6. 启动 PostgreSQL，自动开始恢复
pg_ctl start

# 7. 检查恢复进度
SELECT pg_is_in_recovery();
SELECT pg_last_wal_receive_lsn();
SELECT pg_last_xact_replay_timestamp();

# 8. 确认数据正确后，结束恢复
SELECT pg_wal_replay_resume();
```

### 4.3 单表恢复

从逻辑备份中提取单表：

```bash
# 从自定义格式备份中提取单表
pg_restore -h localhost -U jinshu_admin -d jinshu \
  -t report_metadata \
  --data-only \
  /backup/jinshu_20260524.dump

# 先导出到临时表再 rename
pg_restore -h localhost -U jinshu_admin -d jinshu \
  -t report_metadata \
  --data-only --table=jinshu.report_metadata_restore \
  /backup/jinshu_20260524.dump

# 确认数据后
BEGIN;
ALTER TABLE report_metadata RENAME TO report_metadata_corrupt;
ALTER TABLE report_metadata_restore RENAME TO report_metadata;
DROP TABLE report_metadata_corrupt;
COMMIT;
```

### 4.4 误操作数据找回

场景：误 UPDATE/DELETE 未提交或刚提交

```sql
-- 如果事务刚提交，使用 pg_waldump 分析 WAL
-- 需要 wal_level = replica 或 logical

-- 使用 flashback query 风格（通过 pg_dirtyread 扩展）
-- 安装 pg_dirtyread
CREATE EXTENSION IF NOT EXISTS pg_dirtyread;

-- 读取已删除但未被 VACUUM 回收的行
SELECT * FROM pg_dirtyread('report_metadata') AS t(
  id BIGINT, tenant_id BIGINT, name VARCHAR(255), ...
) WHERE id = 42;
```

> **预防建议**：所有 DDL 前自动执行 `CREATE TABLE {备份表} AS SELECT * FROM {原表}`。应用层已实现此兜底机制。

## 5. 监控与告警

### 5.1 关键监控指标

| 指标 | 告警阈值 | 严重级别 | 说明 |
|------|---------|---------|------|
| 连接数使用率 | > 80% | Warning | 接近 max_connections |
| 活跃连接数 | > 50 | Warning | 可能连接泄漏 |
| 复制延迟 | > 10s | Critical | 从库同步延迟 |
| 事务年龄 (age) | > 5 亿 | Critical | 防止事务 ID 回卷 |
| 长事务时间 | > 30min | Warning | 阻塞 VACUUM 和数据清理 |
| 慢查询 (超过 1s) | > 10/min | Warning | 需要索引优化 |
| 死锁次数 | > 0 | Critical | 立即排查 |
| 磁盘使用率 | > 85% | Warning | 清理或扩容 |
| WAL 生成速率 | 突增 2 倍 | Warning | 可能有大事务 |

### 5.2 连接数与连接池

```sql
-- 查看当前连接数
SELECT count(*) FROM pg_stat_activity;

-- 查看连接详情
SELECT pid, usename, application_name, client_addr, state, query_start, wait_event
FROM pg_stat_activity
ORDER BY query_start;

-- 查看连接数分布
SELECT usename, count(*) FROM pg_stat_activity GROUP BY usename;

-- 查看是否有闲置连接泄漏
SELECT pid, usename, state, query, age(now(), query_start) AS idle_duration
FROM pg_stat_activity
WHERE state = 'idle'
AND query_start < now() - interval '10 minutes';

-- 终止特定连接
SELECT pg_terminate_backend(pid) FROM pg_stat_activity
WHERE usename = 'jinshu_app' AND pid <> pg_backend_pid();
```

HikariCP 连接池配置（应用端）：

| 参数 | 值 | 说明 |
|------|-----|------|
| `maximum-pool-size` | 20 | 最大连接数 |
| `minimum-idle` | 5 | 最小空闲连接 |
| `connection-timeout` | 30000ms | 连接超时 |
| `idle-timeout` | 600000ms | 空闲超时 (10min) |
| `max-lifetime` | 1800000ms | 最大存活 (30min) |

### 5.3 慢查询分析

```sql
-- 查看当前运行中的慢查询
SELECT pid, now() - pg_stat_activity.query_start AS duration, query, state
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY duration DESC
LIMIT 10;

-- 启用慢查询日志 (postgresql.conf)
log_min_duration_statement = 1000  -- 记录超过 1 秒的查询

-- 查看 pg_stat_statements 统计（需安装扩展）
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 查询最耗时的 TOP 10
SELECT queryid, query, calls, total_exec_time / 1000 AS total_seconds,
       mean_exec_time, rows, shared_blks_hit::float / (shared_blks_hit + shared_blks_read) AS hit_ratio
FROM pg_stat_statements
WHERE query NOT LIKE '%pg_%'
ORDER BY total_exec_time DESC
LIMIT 10;

-- 查询调用最频繁的 TOP 10
SELECT queryid, query, calls, total_exec_time / calls AS avg_ms
FROM pg_stat_statements
ORDER BY calls DESC
LIMIT 10;
```

## 6. 性能调优

### 6.1 PostgreSQL 配置优化

以下为 4GB RAM 环境的推荐配置（K3s 单节点典型配置）：

```ini
# /var/lib/postgresql/data/postgresql.conf

# 内存配置
shared_buffers = 1GB           # 物理内存的 25%
effective_cache_size = 3GB     # 物理内存的 75%
work_mem = 16MB                # 排序/哈希内存（每个查询）
maintenance_work_mem = 256MB   # VACUUM/INDEX 维护内存
wal_buffers = 16MB             # WAL 缓冲区

# 查询优化
random_page_cost = 1.1         # SSD 环境
effective_io_concurrency = 200  # SSD 环境
default_statistics_target = 500 # 统计信息精度

# 并发与连接
max_connections = 100          # 应用层连接池 + 预留连接
max_worker_processes = 4       # 与 CPU 核数一致
max_parallel_workers = 4       # 与 CPU 核数一致
max_parallel_workers_per_gather = 2

# 检查点与 WAL
checkpoint_timeout = 15min     # 检查点间隔
checkpoint_completion_target = 0.9
min_wal_size = 2GB
max_wal_size = 8GB

# 自动清理 (autovacuum)
autovacuum = on
autovacuum_max_workers = 3
autovacuum_naptime = 1min
autovacuum_vacuum_threshold = 50
autovacuum_vacuum_scale_factor = 0.01
autovacuum_analyze_threshold = 50
autovacuum_analyze_scale_factor = 0.05
```

### 6.2 索引维护

```sql
-- 查看索引使用情况（未使用的索引浪费写入性能）
SELECT schemaname, tablename, indexname, idx_scan, idx_tup_read, idx_tup_fetch
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY tablename;

-- 查找重复索引
SELECT pg_size_pretty(sum(pg_relation_size(idx))::bigint) AS total_size,
       array_agg(idx) AS indexes
FROM (
  SELECT indrelid::regclass AS table_name,
         indexrelid::regclass AS idx,
         indkey::text AS cols
  FROM pg_index
) sub
GROUP BY table_name, cols
HAVING count(*) > 1;

-- 重建索引（避免索引膨胀）
REINDEX INDEX idx_task_tenant_status;
-- 或并行重建
REINDEX TABLE CONCURRENTLY task;

-- 创建部分索引（只索引活跃数据）
CREATE INDEX CONCURRENTLY idx_task_active
ON task(tenant_id, priority DESC, created_at)
WHERE status NOT IN ('COMPLETED', 'FAILED');
```

### 6.3 VACUUM 与 ANALYZE

```sql
-- 查看表膨胀情况
SELECT schemaname, relname, n_live_tup, n_dead_tup,
       round(n_dead_tup::float / (n_live_tup + n_dead_tup + 1) * 100, 2) AS dead_pct,
       last_autovacuum, last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;

-- 手动 VACUUM（大表使用 VACUUM ANALYZE）
VACUUM ANALYZE report_data;

-- 手动 FREEZE 防止事务 ID 回卷
VACUUM FREEZE task;

-- 查看需要紧急 FREEZE 的表
SELECT relname, age(relfrozenxid) AS xid_age
FROM pg_class
WHERE relkind = 'r'
ORDER BY xid_age DESC
LIMIT 10;
```

### 6.4 查询优化建议

1. **所有查询必须包含 `tenant_id` 过滤**——MyBatis 拦截器自动注入，底层索引以 `tenant_id` 为最左前缀。

2. **`report_data` 宽表查询**必须使用 `report_id` 过滤：

```sql
-- 高效查询
SELECT * FROM report_data WHERE tenant_id = ? AND report_id = ? ORDER BY row_no;

-- 低效查询（全表扫描 JSONB）
SELECT * FROM report_data WHERE data_json @> '{"status": "active"}';
-- 可通过 GIN 索引加速
CREATE INDEX idx_report_data_json_gin ON report_data USING GIN(data_json);
```

3. **审计日志查询**避免全表扫描：

```sql
-- 范围查询走索引
SELECT * FROM audit_log
WHERE tenant_id = ?
AND created_at >= '2026-05-01' AND created_at < '2026-05-02'
ORDER BY created_at DESC;
```

4. **定期 `ANALYZE`** 保持统计信息准确：

```sql
ANALYZE;
```

## 7. 分区与归档

### 7.1 分区策略

**audit_log 表按时间分区**（每月一个分区）：

```sql
-- 创建分区表
CREATE TABLE audit_log (
    id BIGSERIAL,
    tenant_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    username VARCHAR(64) NOT NULL,
    operation VARCHAR(32) NOT NULL,
    target_type VARCHAR(32),
```

### 7.2 历史数据归档

审计日志归档策略：

```bash
#!/bin/bash
# 归档 6 个月前的审计日志

ARCHIVE_MONTH=$(date -d "6 months ago" +%Y%m)
ARCHIVE_TABLE="audit_log_${ARCHIVE_MONTH}"

# 导出分区数据
pg_dump -h localhost -U jinshu_backup -d jinshu \
  -t ${ARCHIVE_TABLE} -Fc \
  -f /archive/${ARCHIVE_TABLE}.dump

# 确认导出成功
if pg_restore -l /archive/${ARCHIVE_TABLE}.dump > /dev/null 2>&1; then
  # 删除分区
  psql -h localhost -U jinshu_admin -d jinshu -c "
    DROP TABLE IF EXISTS ${ARCHIVE_TABLE};
  "
  echo "Archived and dropped ${ARCHIVE_TABLE}"
else
  echo "Backup validation failed for ${ARCHIVE_TABLE}, aborting"
  exit 1
fi
```

## 9. 常见故障处理

### 9.1 连接数耗尽

```sql
-- 查看连接数
SELECT count(*) FROM pg_stat_activity;

-- 紧急释放：保留超级用户连接，杀掉非关键连接
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE usename != 'jinshu_admin'
  AND pid <> pg_backend_pid()
  AND state = 'idle'
  AND query_start < now() - interval '5 minutes';

-- 临时扩大连接数（需重启生效的配置可临时用此方式）
-- 实际上 max_connections 需要重启，紧急方案是杀掉连接
ALTER SYSTEM SET max_connections TO 200;
-- 重启后生效
```

**预防措施**：
- 应用层 HikariCP 连接池上限配置为 20
- 监控连接使用率，超过 70% 告警
- 排查连接泄漏（检查 `state = 'idle'` 且长时间未关闭的连接）

### 9.2 磁盘空间满

```bash
# 1. 确认占用
df -h /var/lib/postgresql/data
du -sh /var/lib/postgresql/data/pg_wal/   # WAL 占用
du -sh /var/lib/postgresql/data/base/     # 数据占用

# 2. 查看大表
SELECT schemaname, relname, pg_size_pretty(pg_total_relation_size(relid)) AS total_size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 20;

# 3. 清理 WAL（应急 - 会丢失 PITR 能力）
SELECT pg_switch_wal();
-- 删除旧的 WAL 归档
find /archive -name "*.wal" -mtime +2 -delete

# 4. 清理垃圾数据
VACUUM FULL audit_log;   -- 注意：会锁表
REINDEX TABLE audit_log;

# 5. 归档/删除历史分区
DROP TABLE audit_log_202511;  -- 6 个月前数据
```

### 9.4 死锁

```sql
-- 查看当前锁等待
SELECT blocked_locks.pid AS blocked_pid,
       blocked_activity.usename AS blocked_user,
       blocked_locks.locktype AS blocked_lock_type,
       blocked_activity.query AS blocked_query,
       blocking_locks.pid AS blocking_pid,
       blocking_activity.usename AS blocking_user,
       blocking_activity.query AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks
  ON blocking_locks.locktype = blocked_locks.locktype
  AND blocking_locks.database = blocked_locks.database
  AND blocking_locks.relation = blocked_locks.relation
  AND blocking_locks.page = blocked_locks.page
  AND blocking_locks.tuple = blocked_locks.tuple
  AND blocking_locks.virtualxid = blocked_locks.virtualxid
  AND blocking_locks.transactionid = blocked_locks.transactionid
  AND blocking_locks.classid = blocked_locks.classid
  AND blocking_locks.objid = blocked_locks.objid
  AND blocking_locks.objsubid = blocked_locks.objsubid
  AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid;

-- 终止阻塞进程（最后手段）
SELECT pg_terminate_backend(blocking_pid);
```

## 10. DDL 变更流程

> 项目已实现 DDL 变更兜底：在 Web 界面执行改表结构前，Java 后端自动执行 `CREATE TABLE {备份表} AS SELECT * FROM {原表}`。

DDL 变更标准流程：

```
1. 开发提交设计文档 + SQL 脚本到 Git
    ↓
2. DBA 评审 SQL (Review)
    ├─ 索引是否合理？
    ├─ 是否锁表？(ALTER TABLE ADD COLUMN 在 PG16 不锁表)
    └─ 是否有迁移脚本？
    ↓
3. 测试环境执行
    │  psql -h test-db -U jinshu_admin -d jinshu -f change.sql
    ↓
4. 预发布环境验证
    ↓
5. 生产环境执行
    ├─ 先备份受影响表
    ├─ 低峰期执行 (业务低谷)
    └─ 观察 30 分钟后确认
```

DDL 最佳实践：

```sql
-- PostgreSQL 16 支持无锁 ADD COLUMN
ALTER TABLE report_metadata ADD COLUMN IF NOT EXISTS tags JSONB DEFAULT '[]';

-- 添加索引使用 CONCURRENTLY（不锁表）
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_report_metadata_tags
ON report_metadata USING GIN(tags);

-- 修改列默认值（不锁表）
ALTER TABLE task ALTER COLUMN priority SET DEFAULT 0;

-- 添加 NOT NULL 约束前，先填充数据
UPDATE task SET priority = 0 WHERE priority IS NULL;
ALTER TABLE task ALTER COLUMN priority SET NOT NULL;
```

## 附录：常用查询

```sql
-- 数据库大小
SELECT pg_database_size('jinshu') / 1024 / 1024 AS size_mb;

-- 各表大小
SELECT relname, pg_size_pretty(pg_total_relation_size(relid)) AS total
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC;

-- 缓存命中率
SELECT 'index hit rate' AS name,
       (sum(idx_blks_hit))::float / (sum(idx_blks_hit + idx_blks_read)) AS ratio
FROM pg_statio_user_indexes
UNION ALL
SELECT 'table hit rate',
       (sum(heap_blks_hit))::float / (sum(heap_blks_hit + heap_blks_read))
FROM pg_statio_user_tables;

-- 活跃查询
SELECT pid, age(now(), query_start) AS duration, state, query
FROM pg_stat_activity
WHERE state = 'active'
  AND query NOT LIKE '%pg_stat_activity%'
ORDER BY query_start;

-- 等待事件分布
SELECT wait_event_type, wait_event, count(*)
FROM pg_stat_activity
WHERE wait_event IS NOT NULL
GROUP BY wait_event_type, wait_event
ORDER BY count(*) DESC;

-- 复制状态
SELECT application_name, state, sync_state,
       write_lag, flush_lag, replay_lag
FROM pg_stat_replication;
```
