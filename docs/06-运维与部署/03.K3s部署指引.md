```
                       │ HTTPS

│               Ingress Controller (Nginx)                  │
│              jinshu.obtstar.lan (示例域名)                 │
├─────────┬────────────────────────┬──────────────────────┤
│ /api/*  │   / (静态资源)         │ /ws/* (WebSocket)    │
└────┬────┴──────────┬─────────────┴──────────┬───────────┘
     │               │                        │
┌────▼────┐   ┌──────▼──────┐   ┌─────────────▼───────┐
│ Backend │   │  Frontend   │   │   Worker (future)    │

│ :8080   │   │  :80 (Nginx)│   │  (HPA auto-scale)    │
└────┬────┘   └─────────────┘   └─────────────────────┘
     │
     ├────────────┬───────────────┬──────────────────┐
     │            │               │                  │
```

## 2. 环境要求

| 组件 | 最低配置 | 推荐配置 | 说明 |
|------|---------|---------|------|
| **K3s 节点** | 2 核 CPU, 4GB RAM, 20GB 磁盘 | 4 核 CPU, 8GB RAM, 50GB SSD | 单节点即可运行全部服务 |
| **K3s 版本** | v1.28+ | v1.30+ | 建议使用最新稳定版 |
| **Docker / containerd** | K3s 内置 containerd | - | 无需额外安装容器运行时 |
| **共享存储** | Longhorn 或 NFS | Longhorn (K3s 内置可选) | 生产环境推荐 Longhorn |
| **Ingress** | - | `traefik` (K3s 内置) 或 `nginx-ingress` | K3s 默认内置 Traefik |
| **操作系统** | Linux (Ubuntu 22.04+, Debian 12+, Rocky 9+) | Ubuntu 24.04 LTS | 内核 ≥ 5.10 |
| **网络** | 节点间 1Gbps | 10Gbps | 高可用场景需要 |

## 3. K3s 集群安装

### 3.1 单节点部署

适合开发、测试环境：

```bash
# 安装 K3s (内置 containerd + Traefik + Local Path Provisioner)
curl -sfL https://get.k3s.io | sh -s - \
  --write-kubeconfig-mode 644 \
  --disable=traefik \          # 可选：禁用内置 Traefik，改用 nginx-ingress
  --disable=local-storage \    # 可选：禁用内置 Local Path Provisioner
  --kubelet-arg="max-pods=50"

# 查看节点状态
kubectl get nodes

# 查看 kubeconfig 位置
export KUBECONFIG=/etc/rancher/k3s/k3s.yaml
```

### 3.2 高可用部署

生产环境推荐使用嵌入式 etcd 的高可用模式：

```bash
# 第一个 Server 节点
curl -sfL https://get.k3s.io | sh -s - server \
  --token=JINSHU_CLUSTER_SECRET \
  --cluster-init \
  --write-kubeconfig-mode 644 \
  --kubelet-arg="max-pods=50"

# 后续 Server 节点（加入集群）
curl -sfL https://get.k3s.io | sh -s - server \
  --token=JINSHU_CLUSTER_SECRET \
  --server=https://<FIRST_SERVER_IP>:6443 \
  --write-kubeconfig-mode 644

# Agent 节点（仅运行工作负载）
curl -sfL https://get.k3s.io | sh -s - agent \
  --token=JINSHU_CLUSTER_SECRET \
  --server=https://<SERVER_LB_IP>:6443
```

高可用架构要求：至少 3 个 Server 节点组成 etcd 集群 + 2 个以上 Agent 节点运行业务负载。

### 3.3 安装后验证

```bash
# 检查集群健康
kubectl cluster-info
kubectl get nodes -o wide
kubectl get pods -A

# 验证 CoreDNS
kubectl -n kube-system get pods -l k8s-app=kube-dns
```

## 4. 基础设施部署

### 4.1 PostgreSQL

使用 StatefulSet 部署，通过 `volumeClaimTemplates` 自动创建持久卷：

```bash
# 部署 PostgreSQL
kubectl apply -f k8s/infra.yaml

# 验证
kubectl -n jinshu get pods -l app=jinshu-postgres
kubectl -n jinshu logs -l app=jinshu-postgres --tail=20
```

关键配置项：
- 镜像：`postgres:16-alpine`
- 端口：5432
- 存储：10Gi (通过 volumeClaimTemplates 自动分配)
- 初始化 SQL：从 ConfigMap `jinshu-postgres-init` 挂载至 `/docker-entrypoint-initdb.d`
- 资源限制：request 512Mi/0.5CPU, limit 1Gi/1CPU

### 4.2 Redis

使用 Deployment + Service 部署：

```bash
# Redis 已在 infra.yaml 中定义，一起部署
kubectl -n jinshu get pods -l app=jinshu-redis
```

关键配置项：
- 镜像：`redis:7.4-alpine`
- 端口：6379
- 无密码（生产环境应启用 ACL 或密码认证）
- 资源限制：request 128Mi/0.1CPU, limit 256Mi/0.5CPU

### 4.3 RabbitMQ

使用 Deployment + Service 部署，暴露 5672 (AMQP) 和 15672 (管理界面)：

```bash
kubectl -n jinshu get pods -l app=jinshu-rabbitmq
```

关键配置项：
- 镜像：`rabbitmq:3.13-management-alpine`
- 端口：5672 (AMQP), 15672 (Web 管理)
- 默认用户：`jinshu` (通过 Secret 配置密码)
- 资源限制：request 256Mi/0.25CPU, limit 512Mi/0.5CPU

### 4.4 Ingress Controller

项目使用 nginx-ingress 处理南北向流量。如果 K3s 内置 Traefik 未禁用，需要额外部署：

```bash
# 安装 nginx-ingress
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.11.1/deploy/static/provider/baremetal/deploy.yaml

# 验证
kubectl -n ingress-nginx get pods
```

Ingress 配置 (`k8s/ingress.yaml`)：

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: jinshu
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "500m"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "120"
spec:
  ingressClassName: nginx
  rules:
  - host: jinshu.obtstar.lan
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: jinshu-frontend
            port: 80
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: jinshu-backend
            port: 8080
```

> **注意**：上传报表文件最大 500MB，需通过 annotation `proxy-body-size: 500m` 放开限制。

## 5. 应用服务部署

### 5.1 项目结构与镜像构建

```
jinshu/
├── backend/                  # Gradle Multi-Project
│   ├── Dockerfile            # 构建 api 模块镜像
│   └── ...                   # common / api / batch / worker / data-sync
├── frontend/                 # React + Vite
│   ├── Dockerfile            # 构建 Nginx 静态镜像
│   └── ...
└── k8s/                      # Kustomize 部署清单
    ├── kustomization.yaml
    ├── namespace.yaml
    ├── configmap.yaml
    ├── secret.yaml
    ├── deployment.yaml       # backend + frontend Deployment
    ├── service.yaml          # backend + frontend Service
    ├── infra.yaml            # PostgreSQL + Redis + RabbitMQ
    ├── ingress.yaml
    ├── pvc.yaml
    ├── rbac.yaml             # CI/CD 权限
    └── postgres-init-cm.yaml
```

> **构建方式**：后端使用 `backend/Dockerfile` 构建 `api` 模块的 Spring Boot 可执行 JAR；前端使用 `frontend/Dockerfile` 构建 Nginx 容器。`batch`、`worker`、`data-sync` 模块使用同一 Dockerfile（需修改 ENTRYPOINT），或各自独立构建。

### 5.2 使用 Kustomize 部署

```bash
# 一键部署全部资源
kubectl apply -k k8s/

# 或分步部署
kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/configmap.yaml
kubectl apply -f k8s/secret.yaml
kubectl apply -f k8s/postgres-init-cm.yaml
kubectl apply -f k8s/pvc.yaml
kubectl apply -f k8s/infra.yaml      # PostgreSQL + Redis + RabbitMQ
kubectl apply -f k8s/deployment.yaml # backend + frontend
kubectl apply -f k8s/service.yaml
kubectl apply -f k8s/ingress.yaml

# 检查部署状态
kubectl -n jinshu get all
kubectl -n jinshu get pods -w
```

### 5.3 环境变量配置

后端通过 ConfigMap + Secret 注入环境变量，Spring Boot 自动读取：

| 环境变量 | 来源 | 说明 |
|---------|------|------|
| `DB_HOST` | ConfigMap | PostgreSQL 集群内 DNS |
| `DB_PORT` | ConfigMap | 5432 |
| `DB_NAME` | ConfigMap | jinshu |
| `DB_USER` | ConfigMap | jinshu |
| `DB_PASSWORD` | Secret | 数据库密码 |
| `REDIS_HOST` | ConfigMap | Redis 集群内 DNS |
| `REDIS_PORT` | ConfigMap | 6379 |
| `REDIS_PASSWORD` | Secret | Redis 密码（可为空） |
| `RABBITMQ_HOST` | ConfigMap | RabbitMQ 集群内 DNS |
| `RABBITMQ_PORT` | ConfigMap | 5672 |
| `RABBITMQ_USER` | ConfigMap | jinshu |
| `RABBITMQ_PASSWORD` | Secret | RabbitMQ 密码 |
| `JINSHU_FILE_BASE_PATH` | ConfigMap | `/data/jinshu` |

ConfigMap 值参考 (`k8s/configmap.yaml`)：

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: jinshu-config
  namespace: jinshu
data:
  db_host: "jinshu-postgres.jinshu.svc.cluster.local"
  db_port: "5432"
  db_name: "jinshu"
  db_user: "jinshu"
  redis_host: "jinshu-redis.jinshu.svc.cluster.local"
  redis_port: "6379"
  rabbitmq_host: "jinshu-rabbitmq.jinshu.svc.cluster.local"
  rabbitmq_port: "5672"
  rabbitmq_user: "jinshu"
  jinshu_file_base_path: "/data/jinshu"
```

### 5.4 健康检查与探针

后端服务暴露三个健康检查端点：

| 端点 | 用途 | 探针类型 |
|------|------|---------|
| `/api/live` | 存活探针 (Liveness) | 容器是否存活，死锁时重启 |
| `/api/ready` | 就绪探针 (Readiness) | 服务是否可接收流量 |
| `/api/health` | 综合健康检查 | 外部监控用 |

Deployment 探针配置：

```yaml
livenessProbe:
  httpGet:
    path: /api/live
    port: 8080
  initialDelaySeconds: 60
  periodSeconds: 30
readinessProbe:
  httpGet:
    path: /api/ready
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
```

## 6. 存储配置

### 6.1 存储方案选型

| 场景 | 推荐方案 | 说明 |
|------|---------|------|
| **开发/测试** | K3s 内置 `local-path-provisioner` | 零配置，自动分配宿主机路径 |
| **生产 - 单节点** | `local-path-provisioner` + 定期备份 | 简单可靠，不需要分布式存储 |
| **生产 - 多节点** | Longhorn (推荐) 或 NFS | 分布式块存储，支持 RWX 和快照 |

### 6.2 PVC 配置

项目使用 PVC `jinshu-data` 作为共享文件存储：

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: jinshu-data
  namespace: jinshu
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
```

> **注意**：生产环境中如需多 Pod 同时读写，需使用 `ReadWriteMany` 模式的 PVC（Longhorn 支持 RWX 块设备，NFS 天然支持 RWX）。

### 6.3 文件目录规范

```
/data/jinshu/                 # PVC 挂载根目录
├── upload/                   # 用户上传原始文件
├── temp/{taskId}/            # 任务临时目录（各节点独占）
├── output/                   # 最终生成的 PDF/Excel 报表
└── archive/                  # 归档目录
```

- 文件命名规范：`{taskId}_{timestamp}_{segmentNo}.{ext}`
  - 例：`1001_20260512143025_003.pdf`
- 大文件写入时使用 `.tmp` 后缀，写入完成后 rename
- 任务完成后自动清理 `temp/{taskId}/`（兜底 24h CronJob 清理过期临时文件）

## 7. 网络与安全

### 7.1 Ingress 与 TLS

生产环境必须启用 HTTPS：

```bash
# 使用 cert-manager 自动签发 Let's Encrypt 证书
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/latest/download/cert-manager.yaml

# 创建 ClusterIssuer
cat <<EOF | kubectl apply -f -
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: admin@example.com
    privateKeySecretRef:
      name: letsencrypt-prod
    solvers:
    - http01:
        ingress:
          class: nginx
EOF

# 更新 Ingress 添加 TLS
ingress.yaml 需添加：
spec:
  tls:
  - hosts:
    - jinshu.obtstar.lan
    secretName: jinshu-tls
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
```

### 7.2 网络策略

限制 Pod 间通信，最小权限原则：

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-network-policy
  namespace: jinshu
spec:
  podSelector:
    matchLabels:
      app: jinshu-backend
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: jinshu-frontend
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: ingress-nginx
```

### 7.3 镜像仓库认证

私有镜像仓库需要配置拉取 Secret：

```bash
kubectl -n jinshu create secret docker-registry registry-cred \
  --docker-server=10.43.170.83:3000 \
  --docker-username=admin \
  --docker-password=admin123

# 在 Deployment 中引用
spec:
  template:
    spec:
      imagePullSecrets:
      - name: registry-cred
```

## 8. 水平扩缩容 (HPA)

Worker 模块（报表导入/导出/PDF 渲染）根据 MQ 积压数量自动扩缩容：

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: jinshu-worker
  namespace: jinshu
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: jinshu-worker
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
```

Backend 和 Frontend 通常无需 HPA，增加副本数即可应对峰值（2 副本提供基本 HA）。

## 9. 监控与日志

### 9.1 Prometheus + Grafana

使用 kube-prometheus-stack 快速搭建：

```bash
# 添加 Helm 仓库
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

# 安装监控栈
helm upgrade --install kube-prometheus prometheus-community/kube-prometheus-stack \
  --namespace monitoring --create-namespace \
  --set prometheus.prometheusSpec.serviceMonitorSelectorNilUsesHelmValues=false
```

应用服务暴露 Prometheus 指标端点（`/actuator/prometheus`），通过 ServiceMonitor 自动采集。

### 9.2 日志收集

使用 Loki + Promtail 收集集群日志：

```bash
helm repo add grafana https://grafana.github.io/helm-charts
helm upgrade --install loki grafana/loki-stack \
  --namespace logging --create-namespace \
  --set promtail.enabled=true \
  --set loki.persistence.enabled=true \
  --set loki.persistence.size=10Gi
```

后端日志输出为 JSON 格式，通过结构化字段可被 Loki 自动索引。

## 10. CI/CD 集成

### 10.1 Gitea + Gitea Actions

项目已有 CI/CD 基础设施（Gitea Runner + dind），部署脚本位于 `scripts/deploy.sh`：

```bash
# 手动触发部署（指定提交 SHA）
bash scripts/deploy.sh <git-commit-sha>
```

### 10.2 自动构建与部署流程

```
开发者 Push 代码至 main 分支
    ↓
Gitea Webhook 触发 Runner
    ↓
Runner 克隆代码到 WORK_DIR (/tmp/jinshu)
    ↓
在 dind (Docker in Docker) 容器中构建镜像
    ├─ backend/  → jinshu-backend:{SHA} + push to registry
    └─ frontend/ → jinshu-frontend:{SHA} + push to registry
    ↓
kubectl apply -f k8s/ (替换镜像 tag 为当前 SHA)
    ↓
kubectl rollout status 等待部署完成
    ↓
完成
```

CI 权限：Gitea Runner 的 ServiceAccount `default`（位于 `gitea` 命名空间）通过 RoleBinding `jinshu-ci-runner` 获得操作 `jinshu` 命名空间资源的权限。

## 11. 备份与恢复

| 资源 | 备份方式 | 频率 | 保留 |
|------|---------|------|------|
| PostgreSQL | pg_dump + 定时 CronJob | 每日全量 + WAL 归档 | 30 天 |
| PVC 文件 | rsync 到外部存储或 S3 | 每日增量 | 30 天 |
| K3s 集群配置 | `kubectl get all -A -o yaml` 导出 | 每次变更后 | 持续 |

> 详细备份恢复步骤见《数据库运维手册》。

配置备份 CronJob：

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: backup-pvc
```
