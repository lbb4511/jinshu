> 故障处理的第一原则：先恢复服务，再排查根因。
>
> **核心目标：** 减少故障影响时间，降低故障影响范围。

---

## 1. 故障分级

| 级别              | 定义                                     | 响应时间要求                  |
| ----------------- | ---------------------------------------- | ----------------------------- |
| **P0 灾难性故障** | 核心功能完全不可用，影响所有用户         | 立即响应，5分钟内必须有人介入 |
| **P1 严重故障**   | 核心功能部分不可用，或重要功能完全不可用 | 15分钟内响应                  |
| **P2 一般故障**   | 非核心功能不可用，或有 workaround        | 1小时内响应                   |
| **P3 轻微问题**   | 小bug、UI问题，不影响核心使用            | 下个工作日处理                |

### P0 示例

- 整个系统无法登录
- 所有报表都无法打开或导出
- 数据库挂了，所有请求都失败
- 大规模数据泄露或损坏

### P1 示例

- PDF生成功能完全不可用
- 某个租户的数据全部丢失
- 性能严重下降，响应时间超过10秒

### P2 示例

- 某个非核心报表展示有问题
- 导出的Excel格式有小瑕疵
- 某些边缘浏览器兼容性问题

## 2. 故障处理标准流程

### 2.1 第一步：发现与上报

**谁发现谁上报，第一时间上报。**

**上报内容必须包含：**

1. 故障现象描述（什么功能出了什么问题）
2. 影响范围（哪些用户/租户受影响）
3. 发生时间（什么时候开始的）
4. 严重程度预估（P0/P1/P2/P3）

**上报渠道：**

- P0/P1：电话/即时通讯 @ 所有核心维护者
- P2/P3：提Issue，标注对应标签

---

### 2.2 第二步：确认与定级

**15分钟内必须完成：**

1. 确认故障真实存在（不是误报）
2. 确认影响范围和严重程度
3. 正式定级（P0/P1/P2/P3）
4. 指定本次故障的"处理总指挥"

**处理总指挥的职责：**

- 协调整个故障处理过程
- 做关键决策（要不要回滚、要不要降级）
- 同步信息，避免多头指挥
- 记录整个处理过程

**重要原则：故障处理期间，所有人听总指挥的。**

---

### 2.3 第三步：止损与恢复

**这是最重要的一步，优先于一切根因分析。**

**按优先级尝试恢复手段：**

| 优先级 | 手段                     | 什么情况下用                         |
| ------ | ------------------------ | ------------------------------------ |
| 1      | **回滚到上一个稳定版本** | 刚发布新版本后出现故障               |
| 2      | **重启服务**             | 服务异常、内存泄漏、死锁             |
| 3      | **降级/熔断**            | 某个非核心功能拖垮整个系统，先关掉它 |
| 4      | **扩容**                 | 流量突增导致性能问题                 |
| 5      | **紧急修复发布**         | 没有其他办法，必须改代码             |

**黄金1小时原则：**
P0/P1故障，如果1小时内还找不到恢复办法，优先采用"能恢复但不完美"的方案，哪怕暂时牺牲一些功能，先把核心服务恢复了再说。

---

### 2.4 第四步：根因分析

**服务恢复后，再做根因分析。**

**5 Why 分析法：**

```
现象：PDF生成超时
  ↓
为什么？ → 因为服务器CPU 100%
  ↓
为什么？ → 因为某个大报表占用了所有CPU
  ↓
为什么？ → 因为任务调度没有限制，大任务直接执行
  ↓
为什么？ → 因为我们的公平调度器还没实现
  ↓
为什么？ → 因为排期的时候优先级放低了
```

**根因分析必须回答：**

1. 技术上的根本原因是什么？
2. 流程上的漏洞是什么？（为什么没测出来？为什么监控没告警？）
3. 人的因素有哪些？（为什么没考虑到？为什么没按流程来？）

---

### 2.5 第五步：复盘与改进

**故障解决后3个工作日内必须完成复盘。**

**复盘会必须讨论：**

1. 故障发生的时间线是怎样的？
2. 哪些地方处理得好？
3. 哪些地方处理得不好？
4. 有哪些经验教训？
5. 需要做哪些改进？（具体到人、具体到时间节点）

**复盘报告模板：**

| 项             | 内容                                                |
| -------------- | --------------------------------------------------- |
| 故障编号       | INC-YYYYMMDD-XXX                                    |
| 故障时间       | 从 YYYY-MM-DD HH:MM 到 YYYY-MM-DD HH:MM，持续XX分钟 |
| 故障级别       | P0/P1/P2/P3                                         |
| 影响范围       | XX个租户，XX个用户                                  |
| 故障现象       | 一句话描述                                          |
| 根本原因       | 技术根因 + 流程根因                                 |
| 处理过程时间线 | 关键时间点的关键动作                                |
| 做得好的地方   | ...                                                 |
| 做得不好的地方 | ...                                                 |
| 改进措施       | 具体的Action Item，每项都要有负责人和截止时间       |

## 3. 故障处理中的禁止事项

### ❌ 绝对禁止

1. **禁止在故障处理期间做根因分析** - 先恢复，再排查
2. **禁止多人同时操作生产环境** - 总指挥安排人操作，其他人观察
3. **禁止在生产环境调试** - 要调试拉到测试环境去
4. **禁止偷偷改东西不告诉别人** - 你做的任何操作都要告诉总指挥
5. **禁止互相指责、甩锅** - 故障处理完了再说，现在先解决问题

### ✅ 应该做

1. **记录所有操作** - 你敲的每一条命令，做的每一个变更，都要记下来
2. **及时同步状态** - 每隔15分钟跟大家同步一下当前进展
3. **大胆决策** - 该回滚就回滚，不要犹豫
4. **请求帮助** - 搞不定就叫人，不要硬撑

## 4. 发布回滚流程

### 4.1 回滚触发条件

出现以下情况之一，考虑回滚：

1. 发布后出现P0/P1级故障
2. 核心功能不可用
3. 数据出现不一致或损坏
4. 性能下降超过50%

### 4.2 谁可以决定回滚

- 发布负责人可以直接决定回滚
- 处理总指挥可以直接决定回滚
- 任何核心维护者都可以提议回滚，没人反对就执行

**重要原则：回滚不需要层层审批。觉得需要回就回，事后再解释。**

### 4.3 回滚后的操作

1. 确认服务恢复正常
2. 验证核心功能
3. 通知所有相关人员
4. 开始分析为什么新版本有问题

## 5. 故障信息同步

### 5.1 内部同步

- P0/P1：每隔30分钟同步一次进展
- P2：每2小时同步一次
- 重大进展随时同步

### 5.2 用户同步

**什么情况下需要通知用户：**

- P0/P1故障，持续超过30分钟
- 涉及用户数据丢失或损坏
- 需要用户配合做什么操作

**通知原则：**

- ✅ 诚实告知，不要隐瞒
- ✅ 说清楚影响范围和预计恢复时间
- ✅ 有进展及时更新
- ❌ 不要找借口
- ❌ 不要轻易说"是网络问题"、"是云服务商的问题"

---

## 6. 故障问责与改进

### 6.1 问责原则

我们对事不对人：

- ✅ 重点关注"流程哪里有漏洞"、"系统哪里有缺陷"
- ✅ 重点关注"怎么改进，避免下次再发生"
- ❌ 不重点关注"谁犯的错"、"谁来承担责任"

**例外：** 以下情况需要问责

- 故意违反流程导致的故障
- 反复犯同样的错误
- 故障后隐瞒、篡改信息

### 6.2 改进措施的跟踪

所有复盘会上提出的改进措施：

1. 必须有明确的负责人
2. 必须有明确的截止时间
3. 必须有人跟踪进度
4. 完成后要验证效果

---

## 7. 故障处理Checklist

### 故障发生时

- [ ] 第一时间上报，不要犹豫
- [ ] 指定处理总指挥
- [ ] 确认故障级别和影响范围
- [ ] 优先尝试恢复服务（回滚、重启）
- [ ] 记录所有操作和时间点
- [ ] 不要在故障期间做根因分析
- [ ] 及时同步进展

### 故障恢复后

- [ ] 确认所有功能恢复正常
- [ ] 通知受影响的用户
- [ ] 3个工作日内开复盘会
- [ ] 输出复盘报告
- [ ] 列出改进措施和负责人
- [ ] 跟踪改进措施的落地

## 8. 常见的反模式

### ❌ 最糟糕的做法

1. **"先看看是什么问题，先不急着回滚"**
   - 结果：看了1小时，问题还没找到，故障持续了1小时，最后还是回滚了，5分钟就好了
   - 正确做法：怀疑是发布导致的，先回滚，回滚完了慢慢查

2. **"我能搞定，不用叫人"**
   - 结果：一个人搞了2小时没搞定，实在没办法了才叫人，浪费了2小时
   - 正确做法：搞不定就叫人，越早越好

3. **好几个人同时在操作，谁也不知道别人在干嘛**
   - 结果：A在重启，B在改配置，C在杀进程，系统越来越乱
   - 正确做法：一个总指挥，一个人操作，其他人观察和出主意

> 📌 最后一句话：
>
> **故障不可怕，可怕的是同样的故障犯第二次。**
>
> 每一次故障都是给系统送上门的免费体检，好好利用它，让系统越来越健壮。
