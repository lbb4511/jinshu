## 1. 前言

本文档是本项目的"宪法"级文件，定义了从需求到上线的完整工作流程。

**所有参与者必须严格遵守本流程。**

## 2. 核心原则

### 2.1 设计先行原则

```
没有设计文档的代码，一行都不许写。

任何功能开发，必须先有设计文档，且设计文档通过评审，才能开始编码。
```

### 2.2 可追溯原则

```
任何代码变更，必须能追溯到对应的：
  - 需求文档
  - 设计文档
  - Issue / PR
```

### 2.3 双重保险原则

```
人类开发者有开发流程规范
AI 助手有 AI_CONTRACT 行为合约

两套体系同时生效，确保流程不乱。
```

## 4. 标准开发流程（10步）

任何功能从需求到上线，必须严格按以下10步执行：

### 第 1 步：需求确认

**输入**：用户需求、业务场景

**输出**：Issue 记录，明确需求边界

**检查项**：

- [ ] 需求边界清晰，没有模糊不清的地方
- [ ] 需求已经跟相关方确认

---

### 第 2 步：架构评审

**输入**：确认后的需求

**输出**：架构设计决策记录

**检查项**：

- [ ] 是否符合现有分层架构
- [ ] 是否符合多租户隔离要求
- [ ] 对性能的影响是否评估
- [ ] 安全风险是否评估

---

### 第 3 步：编写设计文档

**输入**：架构决策

**输出**：模块/功能设计文档

**必须使用模板**：`docs/02-流程与规范/01-个人+AI开发指南/01.设计文档模板.md`

**设计文档必须包含**：

- [ ] 数据库表结构设计
- [ ] API 接口设计
- [ ] 异常处理设计
- [ ] 安全设计
- [ ] 测试方案

---

### 第 4 步：设计文档评审

**输入**：设计文档初稿

**输出**：评审意见 + 评审通过结论

**评审必须至少 1 人通过**

**评审重点**：

- [ ] 设计是否符合架构规范
- [ ] 数据库设计是否合理
- [ ] API 设计是否合理
- [ ] 异常场景是否覆盖
- [ ] 安全是否有漏洞

---

### 第 5 步：编写测试设计

**输入**：评审通过的设计文档

**输出**：测试用例设计

**必须覆盖**：

- [ ] 正常流程用例
- [ ] 边界值用例
- [ ] 异常流程用例
- [ ] 权限校验用例
- [ ] 多租户隔离用例

---

### 第 6 步：准备测试数据

**输入**：测试设计

**输出**：`test-data/` 目录下的测试数据文件

**检查项**：

- [ ] 测试数据已经脱敏，无真实敏感信息
- [ ] 测试数据覆盖正常、异常、边界场景
- [ ] 大文件性能测试数据准备就绪

---

### 第 7 步：编码实现

**输入**：设计文档 + 测试设计

**输出**：业务代码 + 测试代码

**编码前自查清单**：

- [ ] 我已经完整阅读了设计文档
- [ ] 我理解数据库表结构
- [ ] 我理解 API 输入输出格式
- [ ] 我知道代码应该放哪个包
- [ ] 我已经阅读了命名规范
- [ ] 我知道多租户 SQL 怎么写
- [ ] 我知道异常怎么抛出

---

### 第 8 步：本地测试

**输入**：代码实现

**输出**：本地测试通过报告

**必须跑过**：

- [ ] 所有单元测试通过
- [ ] 所有集成测试通过
- [ ] 本地功能验证通过

---

### 第 9 步：提交 Code Review

**输入**：测试通过的代码

**输出**：CR 意见 + 通过结论

**PR 必须包含**：

- [ ] 关联的设计文档链接
- [ ] 关联的 Issue 链接
- [ ] 测试用例执行结果截图
- [ ] 代码变更说明

---

### 第 10 步：合并与上线

**输入**：CR 通过的代码

**输出**：线上功能可用

**上线前检查**：

- [ ] 数据库变更已经执行
- [ ] 配置项已经更新
- [ ] 监控告警已经配置
- [ ] 回滚方案已经准备好

## 5. 角色与职责

| 角色            | 职责                                           |
| --------------- | ---------------------------------------------- |
| **产品/需求方** | 提出需求，确认需求边界，验收功能               |
| **架构师**      | 架构评审，技术选型决策，设计文档评审           |
| **开发者**      | 编写设计文档，编码实现，编写测试               |
| **评审人**      | Code Review，设计评审，质量把关                |
| **DBA**         | 数据库设计评审，SQL 性能审核                   |
| **安全负责人**  | 安全设计评审，渗透测试                         |
| **AI 助手**     | 按设计文档实现代码，辅助编写测试，绝不凭空编造 |

## 6. 流程红线（违反即打回）

### 🚫 绝对禁止的 7 件事

| 序号 | 禁止行为                               | 后果                          |
| ---- | -------------------------------------- | ----------------------------- |
| 1    | 没有设计文档直接写代码                 | PR 直接打回，代码全部作废     |
| 2    | 设计文档没有评审通过就写代码           | PR 直接打回                   |
| 3    | 擅自修改数据库表结构而不更新设计文档   | 打回，必须先补设计文档        |
| 4    | 绕过 MyBatis 拦截器手写 tenant_id      | 安全红线，严肃处理            |
| 5    | 物理删除数据（审计日志甚至不允许软删） | 打回，必须改成软删            |
| 6    | 引入 GPL / 商业付费依赖                | 直接移除，重新评估            |
| 7    | AI 编造不存在的文件、类、方法          | 重新生成，反复违约则禁用该 AI |

## 7. AI 与人类的分工边界

| 任务     | AI 做什么                    | 人类做什么                 |
| -------- | ---------------------------- | -------------------------- |
| 需求分析 | 整理需求点，提边界问题       | 决策需求范围，确认优先级   |
| 设计文档 | 按模板起草，提供多个方案建议 | 决策采用哪个方案，评审设计 |
| 测试设计 | 按设计文档提取测试场景       | 确认测试覆盖的完整性       |
| 编码实现 | 按评审通过的设计文档编写代码 | 代码评审，验收功能         |
| 修复 Bug | 定位问题，编写修复代码       | 确认问题，验证修复         |
| 架构决策 | 提供选项，分析利弊           | 做最终决策                 |

> 🎯 **核心原则**：AI 负责实现，人类负责决策。

## 8. 如何判断流程是否走对了？

### 健康的项目应该是这样的：

✅ **每个 PR 都能找到对应的设计文档链接**

✅ **新来的开发者，只看文档就能上手，不需要到处问人**

✅ **代码有问题，只要对照设计文档就能判断是不是 Bug**

✅ **AI 写的代码质量稳定，不会出现"跟我们要的不一样"的情况**

✅ **上线前的 Checklist 都能逐项核对，心里有底**

## 9. 为什么要设计这么严格的流程？

### 对人类开发者：

1. 避免"边想边写边改"，浪费时间
2. 避免团队成员对同一个功能理解不一致
3. 降低知识传递成本，新人好接手
4. Code Review 有标准可依，不用凭感觉

### 对 AI 助手：

1. **避免幻觉**：有设计文档作为锚点，AI 不会凭空编造
2. **可预测性**：输出的代码结构固定，符合项目规范
3. **可验证性**：代码对不对，直接对照设计文档检查
4. **可积累性**：设计文档是项目的知识资产，越积累越丰富

### 最终目标：

```
  让"意外"降到最低
  让"质量"稳定可控
  让"效率"稳步提升
```
