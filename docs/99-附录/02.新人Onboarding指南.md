## 📅 第一周：了解项目

### Day 1：环境搭建，跑起来

**目标：** 把项目在本地跑起来，能正常登录、能正常使用核心功能。

**步骤：**

1. **拉代码**

   ```bash
   git clone <仓库地址>
   cd jinshu
   ```

2. **安装依赖**
   - JDK 21+
   - Node.js 22+
   - PostgreSQL 16
   - Docker（可选）

3. **初始化数据库**

   ```bash
   psql -U postgres -f sql/ddl/init_schema.sql
   ```

4. **启动后端**

   ```bash
   cd backend
   ./gradlew api:bootRun
   ```

5. **启动前端**

   ```bash
   cd frontend
   pnpm install
   pnpm dev
   ```

6. **验证**
   - 打开浏览器，访问 http://localhost:3000
   - 用测试账号登录：admin / admin123
   - 随便点点，看看各个功能是否正常

**遇到问题怎么办？**

- 先看 `README.md` 和常见问题文档
- 再在Issue里搜一搜有没有人遇到过同样的问题
- 还是解决不了，在群里问，记得附上：
  - 你的操作系统和版本
  - 错误截图或错误日志
  - 你已经试过什么方法

### Day 2：读文档，理解设计

**目标：** 理解项目是做什么的，整体架构是什么样的。

**必读文档（按顺序）：**

1. **`README.md`** - 项目简介、技术栈、快速开始
2. **`OPEN_CONSTRAINTS.md`** - 项目的核心约束和红线
3. **`docs/01-立项与架构/02.概要设计.md`** - 整体架构设计
4. **`docs/01-立项与架构/04.多租户架构设计.md`** - 多租户是怎么实现的（最重要！）
5. **`docs/01-立项与架构/06.数据库架构设计.md`** - 数据库表结构设计
6. **`docs/02-流程与规范/02-社区协作指南/01.CodeReview指南.md`** - Code Review的要求
7. **`AI_CONTRACT.md`** - AI助手的使用规范（如果你用AI辅助开发）

**读完后你应该能回答：**

- 这个项目是解决什么问题的？
- 整体技术栈是什么？
- 多租户隔离是怎么实现的？
- 核心模块有哪些？分别负责什么？
- 代码的目录结构是什么样的？
- 什么事情能做，什么事情不能做？

---

### Day 3-4：读代码，理解实现

**目标：** 看懂核心代码的实现。

**建议阅读顺序：**

1. **先看后端基础框架**
   - `JinshuApplication.java` - 启动类
   - `config/` 目录下的配置类
   - `security/` 目录下的安全拦截器
   - `common/` 目录下的工具类、异常处理、返回结果封装

2. **再看一个完整的功能示例**
   - 找一个最简单的Controller，顺着往下看：
     - Controller层：参数校验、权限校验
     - Service层：业务逻辑
     - DAO层：数据库操作
     - Entity层：数据结构
   - 理解数据是怎么流转的

3. **最后看核心机制**
   - 多租户SQL拦截器是怎么工作的？
   - 审计日志是怎么记录的？
   - 异常是怎么处理的？

**阅读代码小技巧：**

- 不要每一行都看懂，先看整体结构
- 画个时序图，帮助理解数据流转
- 看不懂的地方记下来，找人问
- 加个断点，Debug跑一遍，比读10遍代码都有用

---

### Day 5：第一个小贡献

**目标：** 提交第一个PR，熟悉整个流程。

**建议做什么：**

- 修复文档里的错别字、链接错误
- 给代码加个注释（你看不懂的地方，别人大概率也看不懂）
- 修复一个非常简单的小Bug
- 补充一个单元测试
- 优化一下错误提示信息

**步骤：**

1. 找一个标注 `good-first-issue` 的Issue，或者自己发现一个小问题
2. 在Issue下留言"我想做这个"，等维护者确认
3. 开一个新分支，开始开发
4. 写完后提交PR
5. 等待Code Review，根据意见修改
6. PR被合并 🎉

**第一个PR不要追求完美，能跑起来、符合规范就行。**

## 📖 常见问题

### Q: 我看不懂代码，怎么办？

**A:**

1. 先看对应的设计文档，理解为什么这么设计
2. 加断点Debug跑一遍，看数据是怎么变的
3. 找对应的代码提交记录，看看当时为什么这么写
4. 还是不懂就问，不要不好意思

### Q: 我应该问谁？

**A:**

- 特定模块的问题 → 找对应模块的Maintainer
- 环境问题 → 找运维/部署负责人
- 流程问题 → 找任何一个Core Maintainer
- 安全问题 → 找安全专员

### Q: 我可以用AI辅助开发吗？

**A:** 可以，但必须严格遵守 `AI_CONTRACT.md` 的要求。

- AI生成的代码你必须逐行审核
- 你要对AI生成的代码负全责
- 禁止编造不存在的功能和配置
- 不确定的地方就问，不要猜

### Q: 我担心我写的代码不好，不敢提交PR怎么办？

**A:** 没关系，所有人都是从新人过来的。

- Code Review就是帮你发现问题的
- 写得不好大家会告诉你怎么改，不会嘲笑你
- 多写几次就好了

## 💡 给新人的建议

1. **不要光看，动手做**
   看10遍代码，不如动手改一遍。哪怕只是改个变量名，加个注释。

2. **不要不懂装懂**
   看不懂就问，不丢人。不懂装懂，最后写出Bug，那才丢人。

3. **小步前进**
   不要一上来就想改核心架构，从简单的地方开始，一点点来。

4. **记录你的学习过程**
   你踩过的坑，以后的新人大概率也会踩。把它记下来，更新到文档里，帮助后来人。

5. **有问题先自己查，查不到再问**
   问问题的时候，说明你已经试过什么方法，查过什么文档。
