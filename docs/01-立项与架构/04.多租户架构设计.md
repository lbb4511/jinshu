## 1. 设计目标

| 目标     | 说明                               |
| -------- | ---------------------------------- |
| 数据隔离 | 租户间数据完全隔离，禁止跨租户访问 |
| 业务透明 | 业务开发无需感知租户逻辑，自动注入 |
| 性能影响 | SQL拦截额外开销 < 1ms              |
| 灵活度   | 支持特殊场景下跳过租户过滤         |

---

## 2. 租户隔离方案选型

| 方案        | 隔离级别 | 优点                 | 缺点               | 本项目选择        |
| ----------- | -------- | -------------------- | ------------------ | ----------------- |
| 独立数据库  | 高       | 完全隔离，物理级     | 成本高，扩缩容复杂 | ❌                |
| Schema 隔离 | 中       | 逻辑隔离，独立表空间 | 备份恢复复杂       | ❌                |
| 行级租户ID  | 低       | 成本低，扩缩容方便   | 需保证SQL不泄露    | ✅ **选择此方案** |

**选型结论**：基于行级 `tenant_id` + MyBatis 插件自动注入。

## 3. 核心架构设计

### 3.1 租户拦截器执行流程

```
HTTP 请求到达
    ↓
[认证过滤器] 解析 token，获取 tenantId + userId
    ↓
[TenantContext] 将 tenantId 存入 ThreadLocal
    ↓
[DAO 层执行 SQL]
    ↓
[TenantInterceptor] MyBatis 插件拦截
    ↓
检查方法是否标注 @SkipTenantFilter
    ├─ 是 → 直接执行原始 SQL
    └─ 否 → 自动拼接 WHERE tenant_id = ? 条件
    ↓
SQL 执行完成
    ↓
[请求结束] 清除 ThreadLocal
```

### 3.2 类设计

```
com.jinshu.security
├── TenantInterceptor.java       # MyBatis 租户拦截插件
├── SkipTenantFilter.java        # 跳过租户过滤注解
└── AuditLogAspect.java          # 审计日志切面

com.jinshu.common.context
├── TenantContext.java           # 租户上下文
└── UserContext.java             # 用户上下文
```

## 4. SQL 注入规则

### 4.1 支持的 SQL 类型

| SQL 类型 | 处理方式                                 |
| -------- | ---------------------------------------- |
| SELECT   | 注入 `tenant_id = ? AND`                 |
| UPDATE   | 注入 `tenant_id = ? AND`                 |
| DELETE   | 注入 `tenant_id = ? AND`                 |
| INSERT   | **不处理**（业务代码自行写入 tenant_id） |

### 4.2 SQL 改写示例

**原始 SQL**：

```sql
SELECT * FROM report WHERE id = ?
```

**改写后 SQL**：

```sql
SELECT * FROM report WHERE tenant_id = 1001 AND id = ?
```

## 5. 特殊场景处理

### 5.1 跨租户操作

**使用场景**：系统管理员查询所有租户数据

**解决方案**：Mapper 方法标注 `@SkipTenantFilter` 注解

```java
@SkipTenantFilter
@Select("SELECT * FROM tenant WHERE id = #{id}")
Tenant selectByIdForAdmin(Long id);
```

### 5.2 注意事项

1. `@SkipTenantFilter` 必须配合权限校验使用
2. Controller 层必须校验当前用户是否为系统管理员
3. 使用此注解必须在 Code Review 时重点审查

## 6. 数据库规范

### 6.1 所有业务表必须包含 tenant_id 字段

```sql
CREATE TABLE any_business_table (
    id          BIGSERIAL PRIMARY KEY,
    tenant_id   BIGINT NOT NULL,  -- 必须有，且为第一列索引字段
    -- 其他字段...
);

CREATE INDEX idx_table_tenant ON any_business_table(tenant_id);
```

### 6.2 禁止绕过拦截器的代码写法

```java
// ❌ 禁止：硬编码写死 tenant_id
@Select("SELECT * FROM report WHERE tenant_id = 1 AND id = #{id}")

// ✅ 正确：不写 tenant_id，拦截器自动注入
@Select("SELECT * FROM report WHERE id = #{id}")
```

## 7. 测试要点

| 测试场景                   | 预期结果                 |
| -------------------------- | ------------------------ |
| 普通用户查询               | SQL 自动拼接 tenant_id   |
| 无 tenantId 上下文         | 不拼接，直接执行原始 SQL |
| @SkipTenantFilter 标注方法 | 不拼接租户条件           |
| 嵌套子查询                 | 外层 WHERE 正确注入      |
| JOIN 查询                  | 主表 WHERE 正确注入      |

---

## 8. 边界说明

| 边界场景             | 处理方式                                            |
| -------------------- | --------------------------------------------------- |
| 复杂 SQL (多表 JOIN) | 仅在最外层 WHERE 注入，业务自行保证关联表的租户隔离 |
| 原生 JDBC            | 不经过 MyBatis，拦截器不生效，需自行处理            |
| 批量操作             | 拦截器正常生效                                      |
