### 决策 002：后端技术栈

| 项            | 内容                                                  |
| ------------- | ----------------------------------------------------- |
| 决策内容      | Spring Boot 3.x + MyBatis + PostgreSQL                |
| 决策日期      | 2026-05-12                                            |
| 决策人        | 项目初始团队                                          |
| 业务背景      | 企业级系统，需要稳定、成熟、招聘容易                  |
| ✅ 选中方案   | **Spring Boot 3.x + MyBatis**                         |
|               | 优点 1：生态成熟，社区活跃，问题容易搜到答案          |
|               | 优点 2：招聘容易，大部分 Java 开发者都会              |
|               | 优点 3：MyBatis 灵活可控，适合复杂报表 SQL            |
|               | 优点 4：PostgreSQL 对 JSON 支持好，适合报表的灵活字段 |
| ❌ 放弃的方案 | Spring Data JPA：复杂查询不如 MyBatis 灵活            |
|               | Go：团队 Java 背景更深厚，招聘 Go 更难                |
|               | MySQL：不如 PostgreSQL 的 JSON、CTE、窗口函数好用     |
| 潜在风险      | Spring Boot 3.x 要求 JDK 17，升级可能有坑             |
| 后续评估      | 不需要                                                |

### 决策 004：多租户架构方案

| 项            | 内容                                         |
| ------------- | -------------------------------------------- |
| 决策内容      | 逻辑隔离（共享数据库 + tenant_id 字段）      |
| 决策日期      | 2026-05-12                                   |
| 决策人        | 项目初始团队                                 |
| 业务背景      | 大部分租户是中小企业，数据量不大             |
| ✅ 选中方案   | **逻辑隔离（tenant_id 字段）**               |
|               | 优点 1：运维简单，一个数据库就行             |
|               | 优点 2：成本低，不需要为每个租户建库         |
|               | 优点 3：跨租户统计查询方便                   |
| ❌ 放弃的方案 | Schema 隔离：维护成本高，跨租户查询麻烦      |
|               | 数据库隔离：成本太高，适合超大型租户         |
| 潜在风险      | 某个租户的大查询可能影响其他租户             |
|               | 应对：公平调度 + 大任务分片 + 资源限流       |
| 后续评估      | 当出现单租户数据量超过 1000 万行时，重新评估 |

### 决策 005：PDF 渲染方案

| 项            | 内容                                                          |
| ------------- | ------------------------------------------------------------- |
| 决策内容      | Puppeteer (Chrome Headless) 渲染 HTML 转 PDF                  |
| 决策日期      | 2026-05-12                                                    |
| 决策人        | 项目初始团队                                                  |
| 业务背景      | 需要 100% 还原前端样式，支持复杂图表                          |
| ✅ 选中方案   | **Puppeteer + Chrome Headless**                               |
|               | 优点 1：样式 100% 还原，和浏览器看到的一样                    |
|               | 优点 2：支持 ECharts 等 JS 渲染的图表                         |
|               | 优点 3：前后端复用同一套组件代码                              |
| ❌ 放弃的方案 | iText / Apache PDFBox：需要手写 Java 代码画 PDF，维护成本太高 |
|               | Thymeleaf 模板：无法支持 JS 渲染的图表                        |
|               | wkhtmltopdf：WebKit 内核，和 Chrome 有差异，JS 支持不好       |
| 潜在风险      | Chrome 内存占用高，大 PDF 可能 OOM                            |
|               | 应对：限制单 PDF 页数 + 容器资源限制 + 超时销毁               |
| 后续评估      | 不需要                                                        |

## 决策流程

1. 提出选型需求，创建文档草稿
2. 调研至少 2-3 个备选方案
3. 列出各方案的优缺点、风险、成本
4. 团队讨论，达成共识
5. 记录到本文档
6. 按决策执行

> 重要的不是选了什么，而是知道为什么这么选，以及可能的风险。
