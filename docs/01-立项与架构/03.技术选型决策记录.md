| 项            | 内容                                         |
| ------------- | -------------------------------------------- |
| 决策内容      | 逻辑隔离（共享数据库 + tenant_id 字段）      |
| 决策日期      | 2026-05-12                                   |
| 决策人        | 项目初始团队                                 |
| 业务背景      | 大部分租户是中小企业，数据量不大             |

| ❌ 放弃的方案 | Schema 隔离：维护成本高，跨租户查询麻烦      |
|               | 数据库隔离：成本太高，适合超大型租户         |
| 潜在风险      | 某个租户的大查询可能影响其他租户             |
|               | 应对：公平调度 + 大任务分片 + 资源限流       |
| 后续评估      | 当出现单租户数据量超过 1000 万行时，重新评估 |

### 决策 006：Java 后端构建工具与工程结构

| 项            | 内容                                                                                                                                                                                                              |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 决策内容      | Gradle Kotlin DSL + Multi-Project（common / api / batch / worker / data-sync）                                                                                                                                    |
| 决策日期      | 2026-05-12                                                                                                                                                                                                        |
| 决策人        | 项目初始团队                                                                                                                                                                                                      |
| 业务背景      | `02.概要设计.md` §4.1 规划了 backend / worker / data-sync-service 等多个独立 K8s 服务，需要工程层面也能独立打包、独立伸缩；原始单 Maven 工程把 controller / batch / MQ 消费者全塞进一棵包树，与多服务部署目标不符 |
| ✅ 选中方案   | **Gradle Kotlin DSL + 5 子模块**                                                                                                                                                                                  |
|               | 优点 1：multi-project 配置比 Maven parent POM + `<module>` 列表简洁，`subprojects {}` 一处配公共依赖                                                                                                              |
|               | 优点 2：增量构建 / 并行构建快，5 子模块场景明显优于 Maven                                                                                                                                                         |
|               | 优点 3：Kotlin DSL 类型安全 + IDE 自动补全，减少 build 脚本写错的概率                                                                                                                                             |
|               | 优点 4：每个非 common 子模块可独立打 Spring Boot fat jar，各自打镜像、各自伸缩，与 §4.1 一一对应                                                                                                                  |
|               | 优点 5：`common` 作为内部 jar 被其它 4 个模块依赖，实体、租户上下文、异常等只写一份                                                                                                                               |
| ❌ 放弃的方案 | Maven 单工程：无法支撑多服务独立打包，与部署目标矛盾                                                                                                                                                              |
|               | Maven multi-module：配置冗长，继承体系复杂，增量构建弱                                                                                                                                                            |
|               | Gradle Groovy DSL：无类型检查，IDE 支持弱于 Kotlin DSL                                                                                                                                                            |
| 潜在风险      | 团队 Gradle 熟练度可能低于 Maven                                                                                                                                                                                  |
|               | 应对 1：统一用 Kotlin DSL，提供 `Makefile` / `scripts/` 包装常用命令（`make build` / `make run-api` 等）                                                                                                          |
|               | 应对 2：`gradle/libs.versions.toml` 集中版本号，降低脚本编辑频率                                                                                                                                                  |
|               | 应对 3：CI 锁定 Gradle Wrapper 版本，避免本地不一致                                                                                                                                                               |
| 后续评估      | 当子模块超过 8 个或单模块构建时间超过 5 分钟时，重新评估是否引入构建缓存（Gradle Build Cache / Develocity）                                                                                                       |

**模块依赖关系**：

```
api ──┐
batch ┤
worker┼──→ common
data-sync ┘
```

约束：

- `common` 不依赖任何业务子模块；不可独立启动
- 4 个业务子模块横向**禁止**直接依赖，跨模块协作走 HTTP REST 或 MQ
- 详细目录结构见 `05.整体架构分层设计.md` §4

### 决策 007：引入 Python 作为 AI/数据科学辅助语言

| 项            | 内容                                                                                                                              |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| 决策内容      | 引入 Python 3.12+ 作为辅助语言，用于 AI 报表生成、数据科学分析、运维脚本                                                          |
| 决策日期      | 2026-05-13                                                                                                                        |
| 决策人        | 项目团队                                                                                                                          |
| 业务背景      | 1) 智能报表生成需要调用 LLM；2) 高级数据分析（Pandas/NumPy/Scikit-learn）能力 Java 生态薄弱；3) 部分运维/数据导入脚本需要快速迭代 |
| ✅ 选中方案   | **Python 3.12+**，独立目录 `ai-services/`，使用 `uv` 或 `poetry` 管理依赖                                                         |
|               | 优点 1：AI/ML 生态最成熟（OpenAI SDK、LangChain、Transformers、Pandas）                                                           |
|               | 优点 2：脚本开发效率高，适合数据探索、原型验证                                                                                    |
|               | 优点 3：与 Java 通过 HTTP/gRPC 解耦，互不影响                                                                                     |
| ❌ 放弃的方案 | Java 直接调 LLM：缺少成熟 SDK，langchain4j 不如 Python 版成熟                                                                     |
|               | GraalPy 内嵌：性能与稳定性弱于 CPython，调试困难                                                                                  |
|               | Node.js：数据科学生态不如 Python                                                                                                  |
| 适用边界      | ✅ AI 报表生成、数据科学分析、运维/数据脚本                                                                                       |
|               | ❌ **禁止**用于业务核心（报表导入/导出、PDF 生成、审计日志）                                                                      |
|               | ❌ **禁止**直连业务数据库写操作，必须通过 Java API                                                                                |
| 潜在风险      | 1) 多语言增加运维复杂度；2) Python GIL 限制并发；3) 团队 Python 熟练度参差                                                        |
|               | 应对：① Python 模块仅作为独立服务/脚本 ② 使用 `ruff` + `mypy` 强制代码规范 ③ Docker 镜像统一基底                                  |
| 部署方式      | 待评估（候选：独立微服务 / Sidecar / Java 内嵌），见"待评估的决策"                                                                |
| 后续评估      | 上线 6 个月后评估 Python 模块占比，若 < 5% 业务量考虑下沉到脚本                                                                   |

### 决策 008：引入 Rust 作为高性能/系统级辅助语言

| 项            | 内容                                                                                                                       |
| ------------- | -------------------------------------------------------------------------------------------------------------------------- |
| 决策内容      | 引入 Rust 1.80+ 作为辅助语言，用于高性能 Excel 解析、WASM 前端组件、CLI 工具                                               |
| 决策日期      | 2026-05-13                                                                                                                 |
| 决策人        | 项目团队                                                                                                                   |
| 业务背景      | 1) 100 万行 Excel 解析在 Java 端可能成为瓶颈；2) 前端处理超大表格虚拟滚动需要高性能渲染；3) 需要分发给租户的命令行导入工具 |
| ✅ 选中方案   | **Rust 1.80+ (stable)**，独立目录 `native/`，使用 Cargo workspace 组织                                                     |
|               | 优点 1：零成本抽象 + 内存安全，无 GC 停顿，适合高性能数据处理                                                              |
|               | 优点 2：编译为 WASM 在浏览器运行，性能接近原生                                                                             |
|               | 优点 3：编译为单二进制 CLI，无运行时依赖，适合分发                                                                         |
|               | 优点 4：可编译为 `.so/.dll` 通过 JNI/Project Panama 给 Java 调用                                                           |
| ❌ 放弃的方案 | C/C++：内存安全风险高，缺少现代化包管理                                                                                    |
|               | Go：GC 影响延迟稳定性，WASM 体积偏大                                                                                       |
|               | Zig：生态不成熟，团队学习成本高                                                                                            |
| 适用边界      | ✅ 高性能 Excel 解析（calamine）、WASM 前端组件、租户 CLI 工具/SDK                                                         |
|               | ❌ **禁止**用于业务编排逻辑（仍由 Java 主导）                                                                              |
|               | ❌ **禁止**用于审计日志、租户拦截器等强业务约束模块                                                                        |
| 潜在风险      | 1) Rust 学习曲线陡峭；2) 与 Java JNI 调用增加复杂度；3) WASM 调试工具链不完善                                              |
|               | 应对：① 仅在性能瓶颈验证后引入 ② 优先 HTTP/gRPC 解耦，JNI 仅作为优化手段 ③ 提供 `cargo make` 统一构建脚本                  |
| 部署方式      | 待评估（候选：JNI 内嵌 / 独立微服务 / WASM 嵌入前端 / 独立 CLI 二进制），见"待评估的决策"                                  |
| 后续评估      | 首个 Rust 模块上线 3 个月后评估，对比 Java 实现的性能/维护成本差异                                                         |

## 决策流程

1. 提出选型需求，创建文档草稿
2. 调研至少 2-3 个备选方案
3. 列出各方案的优缺点、风险、成本
4. 团队讨论，达成共识
5. 记录到本文档
6. 按决策执行

> 重要的不是选了什么，而是知道为什么这么选，以及可能的风险。
