## 2. 计划概述

### 2.1 现状诊断

基于设计覆盖度扫描与代码审计，当前项目存在以下结构性问题：

| 问题类别 | 具体问题                                                                  | 严重度  |
| -------- | ------------------------------------------------------------------------- | ------- |
| 设计缺失 | P0 核心模块（导出、PDF、调度）无详细设计文档                              | 🔴 致命 |
| 设计缺失 | 运维与部署文档完成率仅 12%，无法投产                                      | 🔴 致命 |
| 架构缺陷 | 租户上下文注入链断裂：MyBatis 拦截器依赖 TenantContext，但 MVC 层无注入点 | 🔴 致命 |
| 架构缺陷 | Puppeteer 渲染与 API 服务混布于单体，大任务会阻塞全系统                   | 🔴 高   |
| 隐性缺失 | JWT 认证、Redis 缓存、RabbitMQ、前端架构等已配置但无设计策略              | 🟡 高   |
| 隐性缺失 | 无数据库迁移工具（Flyway/Liquibase），DDL 管理处于原始状态                | 🟡 高   |
| 文档脱节 | 任务调度器、数据加密、审计 AOP 等仅存在于设计伪代码，未落地               | 🟡 高   |

### 2.2 计划目标

1. **修复所有 P0 级架构缺陷**，确保多租户隔离、认证授权等基础安全机制可靠运行
2. **补齐全部 P0 模块设计文档**，使核心功能（导入、导出、PDF、调度）有明确设计依据
3. **完成最小可投产集（MVP）开发**，支持报表全生命周期 + 百万级导入导出 + PDF 生成
4. **建立可观测性与运维体系**，具备监控、告警、备份、恢复能力
5. **通过等保三级核心项自检**，审计日志、数据加密、权限分离达到合规基线

### 2.3 关键里程碑

| 里程碑                   | 日期（相对） | 标志                                               |
| ------------------------ | ------------ | -------------------------------------------------- |
| M1：架构基线修复完成     | 第 2 周末    | 多租户注入链、JWT、DB 迁移工具就绪，代码可安全运行 |
| M2：核心模块设计评审通过 | 第 4 周末    | 导出、PDF、调度三大模块设计文档通过评审            |
| M3：MVP 功能开发完成     | 第 8 周末    | 导入、导出、PDF、工作流、权限可在测试环境跑通      |
| M4：生产环境就绪         | 第 10 周末   | K3s 部署、监控、灾备、压测全部完成                 |
| M5：等保合规基线达标     | 第 12 周末   | 安全自检通过，具备上线条件                         |

## 10. 评审与跟踪机制

### 10.1 周会制度

| 会议       | 频率       | 参与人                   | 内容                             |
| ---------- | ---------- | ------------------------ | -------------------------------- |
| 站会       | 每日 15min | 全体开发                 | 昨日完成、今日计划、阻塞项       |
| 周会       | 每周五下午 | 全体 + 技术负责人        | 进度 Review、风险更新、下周调整  |
| 设计评审会 | 按需       | 架构师 + 安全 + 相关开发 | 设计文档评审，必须线下或视频会议 |
| 里程碑评审 | 每阶段末   | 全体 + 产品/需求方       | 里程碑交付物验收、Go/No-Go 决策  |

### 10.2 进度跟踪工具

- **文档状态**：在 `docs/README.md` 中维护实时完成率，每周更新
- **任务跟踪**：建议使用 GitHub Projects / Jira / Linear，每个任务关联：
  - 设计文档链接（如适用）
  - 对应 Issue / PR
  - 验收标准 Checklist
- **代码质量**：SonarQube 质量门禁 + 测试覆盖率 ≥ 80%（核心模块）

### 10.3 计划变更流程

1. 任何任务延期 > 2 天，负责人必须邮件/工单说明原因和影响
2. 阶段里程碑延期需经技术负责人审批，并同步调整后续阶段资源
3. 新增任务必须评估对关键路径的影响，必要时裁剪 P2 功能
