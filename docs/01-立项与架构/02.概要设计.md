## 1. 总体架构设计

系统采用分层微服务架构，部署于 K3s 集群中，整体划分为四大核心层级：

- **接入层 (Frontend)**：基于 React + PrimeReact 的单页应用（SPA），负责动态交互、报表预览及静态报表的展示。

- **业务服务层 (Backend API)**：基于 Java (Spring Boot) 的核心业务逻辑，负责报表全生命周期状态机流转、权限控制、元数据管理及任务调度。

- **数据处理与生成层 (Worker Cluster)**：
  - 批处理服务：基于 Java (Spring Batch) 处理 ETL 任务。
  - 渲染集群：基于 Chrome Headless 的无头浏览器渲染服务，负责静态 HTML 转 PDF。
  - 脚本执行器：独立隔离的脚本执行微服务，负责调用 PowerShell/BAT 进行数据同步。

- **基础设施与存储层 (Infrastructure)**：K3s 集群、PostgreSQL 数据库、消息队列（RabbitMQ/RocketMQ）、共享文件存储（NFS/RWX PVC）。

## 2. 核心模块详细设计

### 2.1 动静同构报表展示模块

| 项     | 说明                          |
| ------ | ----------------------------- |
| 技术栈 | React 18, PrimeReact, ECharts |

**核心设计**：

- **组件复用**：封装纯展示的报表组件，通过 `props` 接收数据，实现一套代码同时服务于动态网页与静态 HTML 渲染。

- **智能分页样式**：在 React 组件中全局引入打印专用 CSS，对表格行、图表容器强制设置 `page-break-inside: avoid`，从源头上杜绝内容被拦腰截断。

- **实时与静态双模式**：动态模式通过 API 拉取实时数据；静态模式通过 Node.js 预渲染或 Java 模板引擎注入数据生成纯 HTML。

### 2.2 报表全生命周期管理模块

- **状态机引擎**：在 PostgreSQL 中设计 `report_workflows` 表，通过 Java 代码实现严格的状态流转控制：

  ```
  创建 → 初始预览 → 待审查 → 驳回/通过 → 公开
  ```

- **元数据版本控制**：引入 `schema_version` 字段。每次表结构变更，版本号递增，确保历史报表能匹配正确的表结构元数据，防止因 DDL 变更导致旧报表渲染崩溃。

- **权限控制**：结合 Spring Security，根据用户角色（制表人、审查人、访客）动态控制前端按钮显隐与后端接口访问权限。

### 2.3 批处理与ETL模块

**数据导入方案选型对比**：

| 方案                                  | 技术实现                                                 | 安全性     | 可维护性   | 适用场景                     |
| ------------------------------------- | -------------------------------------------------------- | ---------- | ---------- | ---------------------------- |
| **方案一：JDBC 批量导入（推荐）**     | Java 原生 JDBC `COPY` API / `PreparedStatement` 批量插入 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | CSV/Excel 解析后直接内存导入 |
| **方案二：数据库链接服务**            | 部署独立的 Data Sync REST API 服务                       | ⭐⭐⭐⭐   | ⭐⭐⭐⭐   | 需要调用外部数据库           |
| **方案三：Airbyte / Fivetran 连接器** | 开源数据同步平台                                         | ⭐⭐⭐⭐   | ⭐⭐⭐⭐   | 复杂多源数据同步             |

**推荐方案：Java 原生批量导入**

- **文件上传与前置校验**：
  - Java 后端提供文件上传接口，原始数据文件存入共享存储（PVC）
  - 强校验：白名单 `.xlsx`/`.xls`/`.csv`，检查文件头 Magic Number（防后缀伪造）
  - 大小限制：单文件最大 500MB，超限立即拒绝

- **数据导入异步流程**：
  1. MQ 削峰：上传成功后立即返回，将任务推送到消息队列
  2. **三级强校验**（EasyExcel 流式读取时同步执行）：
     - 格式校验：列数匹配、必填列非空、日期格式合法
     - 类型校验：数字/日期/文本类型匹配，长数字强制文本处理（防科学计数法）
     - 业务校验：唯一性约束、字典值校验、跨列逻辑校验
  3. 错误处理：脏数据写入 `import_error_log` 表，错误率 >30% 自动终止
  4. 分批写入：每 1000 条一次 `batchUpdate` / PostgreSQL `COPY`，提交小事务
  5. 进度反馈：Redis 实时更新进度 `task:{taskId}:progress`，前端轮询展示

- **外部系统数据同步（如有跨库同步）**：
  - 配置独立的 **Data Sync Service**，提供 REST API
  - 通过 Spring RestTemplate / WebClient 调用
  - 支持数据源配置化管理，不硬编码连接信息

**任务调度**：引入消息队列，将耗时的数据同步与报表生成任务异步化。Java 消费者监听队列，触发 Spring Batch 任务进行数据清洗与转换。

### 2.4 数据导出模块

**导出前数据预估**：

- 用户选择筛选条件后，先调用预估接口 `GET /api/export/estimate?reportId=xxx&filters=...`
- 返回：`{ totalRows: 12560, estimatedFileSize: "3.2MB", suggestFormat: "EXCEL" }`
- 超过 100 万行自动提示用户改用 CSV 格式

**导出格式支持**：

| 格式              | 处理方式                                                       | 适用场景                    |
| ----------------- | -------------------------------------------------------------- | --------------------------- |
| **Excel (.xlsx)** | EasyExcel 流式写入 (SXSSF)，单 Sheet 上限 100 万行             | 带样式的正式报表，<100 万行 |
| **CSV**           | Java NIO 流式写入，**强制 UTF-8 with BOM** 编码，自动 ZIP 打包 | 超大数据量、纯数据导出      |

**核心流程**：

1. 用户确认数据量后，触发导出任务（异步）
2. MyBatis Cursor 流式查询：`FORWARD_ONLY + fetchSize = Integer.MIN_VALUE`，逐行拉取不占内存
3. EasyExcel / CSV Writer 边读边写入 PVC 临时文件
4. **CSV 自动打包**：CSV 文件生成后自动用 ZIP 压缩，减小下载体积
5. 进度写入 Redis，完成后 WebSocket 推送下载链接
6. 临时文件 24 小时后自动清理

**Excel 模板驱动模式（可选）**：

- 业务人员在 Excel 中画好标准样式（表头、合并单元格、字体、签字栏）
- 模板上传系统管理，单元格埋入占位符 `{reportName}`, `{dataList}`
- 导出时读取模板，EasyExcel 填充数据后输出，格式 100% 可控

### 2.5 高性能PDF生成与色彩管理模块

- **静态 HTML 生成方案选型**：

  | 方案                                   | 技术实现                                                | 组件复用率 | 稳定性     | 推荐度 |
  | -------------------------------------- | ------------------------------------------------------- | ---------- | ---------- | ------ |
  | **方案一：Puppeteer 访问渲染（首选）** | Puppeteer 直接访问前端 `/preview?reportId=xxx` 页面打印 | 100%       | ⭐⭐⭐⭐⭐ |
  | **方案二：Thymeleaf 模板（兜底）**     | Java 使用 Thymeleaf 模板 + 内联 CSS 生成 HTML           | 0%         | ⭐⭐⭐⭐⭐ |
  | **方案三：React SSR（谨慎）**          | Node.js 服务端渲染 React 组件为 HTML                    | 100%       | ⭐⭐       |

  **首选方案：Puppeteer 直出**
  - Puppeteer 携带 JWT Token 访问前端预览页：`https://frontend/preview?reportId={id}&printMode=true`
  - 前端 `printMode=true` 时隐藏导航栏、按钮等交互元素
  - Chrome 原生渲染，样式 100% 一致，无 SSR 适配成本

  **兜底方案：Thymeleaf 纯后端模板**
  - 核心报表组件（表格、图表）单独编写 HTML 模板
  - 内联 CSS，无外部依赖
  - 适合简单表格类报表，稳定性极高

- **Chrome Headless 原生分段打印方案**：

  | 方案                             | 说明                                                                                                                                                                   | 适用场景                   |
  | -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------- |
  | **方案一：CSS 分页控制（推荐）** | 利用 Chrome DevTools Protocol 的 `printToPDF` 原生支持 `pageRanges` 参数，在单个 HTML 中通过 CSS `page-break-after: always` 标记分页位置，一次性渲染后按页范围提取分段 | 报表结构清晰、分页位置固定 |
  | **方案二：流式分块渲染**         | 将大报表按业务维度（如按部门、月份）生成多个独立 HTML，通过单个 Chrome 实例串行渲染，复用浏览器上下文                                                                  | 分页维度与业务维度一致     |
  | **方案三：Browserless 服务**     | 部署 Browserless 作为渲染服务网关，提供连接池、并发控制、超时管理，Java 通过 REST API 调用                                                                             | 中大规模集群               |

  **关键优化点**：
  - 单个 Chrome 实例复用，避免频繁启动开销（启动耗时 2-3 秒）
  - Puppeteer 连接池配置：`min: 2, max: 10`，闲置超时 30 秒自动销毁
  - `printToPDF` 参数优化：`preferCSSPageSize: true`, `printBackground: true`

- **分段渲染与内存保护**：
  - Java 后端根据业务逻辑（如每 20 页或按 Tab 维度）将大任务拆分为多个子任务，通过 MQ 分发给渲染集群。
  - 渲染集群调用 Chrome Headless 生成独立的分段 PDF，最终由 **Apache PDFBox** 自动合并为单一 PDF 文件供前端下载，彻底规避内存溢出。

- **色彩空间转换（CMYK 支持）**：
  - 在生成完 RGB 模式的 PDF 后，Java Worker 调用 Ghostscript 命令行工具，将 RGB PDF 转换为 CMYK 印刷级 PDF，确保专业打印无色差。

## 3. 数据库设计（PostgreSQL）

### 3.1 业务数据存储方案选型

| 方案                         | 技术实现                                                | 表数量 | 运维复杂度 | 推荐度     |
| ---------------------------- | ------------------------------------------------------- | ------ | ---------- | ---------- |
| **方案一：宽表模式（推荐）** | 单张 `report_data` 大表，`report_id` + `jsonb` 字段存储 | 1 张   | 极低       | ⭐⭐⭐⭐⭐ |
| **方案二：按报表类型分表**   | 每张报表类型对应一张物理表，统一 `rpt_` 前缀            | N 张   | 中等       | ⭐⭐⭐     |
| **方案三：按报表实例分表**   | 每张报表实例一张物理表                                  | 大量   | 极高       | ❌ 不推荐  |

### 3.2 推荐方案：宽表模式

**表结构设计**：

```sql
CREATE TABLE report_data (
    id              BIGSERIAL PRIMARY KEY,
    report_id       BIGINT NOT NULL REFERENCES report_metadata(id),
    row_no          INTEGER NOT NULL,          -- 行号
    data_json       JSONB NOT NULL,            -- 业务数据
    created_at      TIMESTAMP DEFAULT NOW(),

    UNIQUE(report_id, row_no)
);

CREATE INDEX idx_report_data_report_id ON report_data(report_id);
CREATE INDEX idx_report_data_gin ON report_data USING GIN(data_json);
```

**优势**：

- 表数量固定，备份、迁移、索引优化简单
- JSONB 支持索引，查询性能优秀
- 天然支持表结构变更，无需 DDL 操作

**备选方案：动态表（如必须）**

如业务确需独立物理表，强制执行以下规范：

- 表名统一前缀：`rpt_{报表类型编码}_{版本号}`
- 所有动态表在 `report_metadata` 中登记映射关系
- DDL 操作前自动执行：`CREATE TABLE {备份表名} AS SELECT * FROM {原表}`
- 禁止在 Web 界面直接执行 DROP TABLE

### 3.3 核心元数据表

| 表名               | 说明                                                                                                                                  |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| `report_metadata`  | 报表元数据表，存储报表基本信息、关联的文件路径、`schema_version`（表结构版本号）、创建时间、动态表名（如使用）                        |
| `report_workflows` | 工作流状态表：`id` (PK), `report_id` (FK), `status` (VARCHAR), `created_by`, `created_at`, `updated_at`, `reviewed_by`, `reviewed_at` |
| 物化视图           | 针对复杂查询，建立物化视图并定时刷新，提升百页报表的数据加载速度                                                                      |

## 4. 部署与运维架构（K3s）

### 4.1 服务拆分与隔离

| 服务                | 说明                                                                                           | 对应代码模块                                                                                                   |
| ------------------- | ---------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| `frontend`          | Nginx 托管 React 静态资源                                                                      | `frontend/`（独立工程）                                                                                        |
| `backend`           | Spring Boot 应用，处理 HTTP 请求                                                               | `backend/api`（依赖 `common`）                                                                                 |
| `worker`            | Spring Batch 消费者 + MQ 消费者，负责 ETL 导入、导出编排、PDF 编排、文件打包                   | `backend/batch` + `backend/worker`（均依赖 `common`），打包为同一镜像或两个独立 Deployment，由运维侧按负载决策 |
| `renderer`          | Chrome Headless 渲染任务，配置严格的资源限制（Limit/Request）防止内存溢出                      | 非 Java 工程，Node + Puppeteer 独立目录                                                                        |
| `data-sync-service` | 独立数据同步服务，对外提供 REST API，负责跨库、跨系统数据同步。Java 原生实现，无需执行系统脚本 | `backend/data-sync`（依赖 `common`）                                                                           |
| `postgresql`        | 数据库持久化存储                                                                               | 基础设施（K8s StatefulSet）                                                                                    |
| `mq`                | 消息队列服务（RabbitMQ）                                                                       | 基础设施（K8s StatefulSet）                                                                                    |

**代码模块与运行时服务的映射约定**：

- Java 后端采用 Gradle multi-project，根目录 `backend/` 下 5 个子模块：`common`（被其它 4 个依赖，不独立部署）、`api`、`batch`、`worker`、`data-sync`。每个非 common 子模块都是独立可启动的 Spring Boot 应用，各自打镜像、各自独立伸缩，符合 §4.3 的 HPA 设计。
- `batch` 与 `worker` 在 K8s 层都归属 `worker` 服务名，可作为同一 Deployment 内的两个容器或两个独立 Deployment，由运维侧按负载决策，不影响代码层面的解耦。
- 构建工具与工程结构决策见 `03.技术选型决策记录.md` 决策 006；详细目录树见 `05.整体架构分层设计.md` §4。

### 4.2 共享存储

**存储规范**：

- 使用 K3s 的 `PersistentVolumeClaim` (RWX) 模式 PVC，挂载至 Backend、Worker、Renderer 容器
- 目录结构分层管理：

  ```
  /data/
  ├── upload/          # 用户上传的原始文件
  ├── temp/{taskId}/   # 任务临时目录，各节点独占
  ├── output/          # 最终生成的 PDF 报表
  └── archive/         # 归档目录
  ```

**并发控制**：

- **文件命名强制规范**：`{taskId}_{timestamp}_{segmentNo}.{ext}`
  - 例：`1001_20260512143025_003.pdf`
- 各 Worker 仅操作自己 `temp/{taskId}/` 目录下的文件
- 任务完成后自动清理临时目录（配置 24 小时兜底清理 CronJob）
- 大文件写入采用 `.tmp` 后缀，写入完成后 rename，避免读取到不完整文件

### 4.3 自动化运维

- 配置 K3s 的 CronJob 定期触发数据同步任务
- 配置 HPA（水平自动扩缩容）根据 MQ 积压情况自动扩展 Worker 与 Renderer 节点数量

## 5. 安全与容灾设计

- **DDL 变更兜底**：在 Web 界面执行任何改表结构（DDL）语句前，Java 后端自动执行 `CREATE TABLE ... AS SELECT *` 备份，留好后悔药。

- **数据同步安全**：数据同步统一通过 API 调用，不执行系统脚本。敏感数据配置加密存储，数据源连接信息通过 K8s Secret 管理。

- **打印色彩预览**：前端提供"印刷预览模式"，让 ECharts 等图表在开发阶段就使用更接近印刷色的 RGB 调色板，减少 CMYK 转换后的色差。

## 6. 多租户架构设计

### 6.1 租户隔离方案

| 隔离级别             | 方案                                                                                         | 适用场景                           |
| -------------------- | -------------------------------------------------------------------------------------------- | ---------------------------------- |
| **逻辑隔离（推荐）** | 所有租户共享数据库实例，所有业务表增加 `tenant_id` 字段，通过 MyBatis 拦截器自动注入过滤条件 | 绝大多数企业级场景，资源利用率最高 |
| **Schema 隔离**      | 每个租户独立 Schema                                                                          | 对数据隔离要求较高的场景           |
| **数据库隔离**       | 每个租户独立数据库实例                                                                       | 金融级强隔离需求                   |

### 6.2 核心设计

- **租户上下文传递**：通过 Spring MVC 拦截器 + ThreadLocal 存储租户上下文，贯穿请求全链路
- **SQL 自动注入**：MyBatis 拦截器自动为所有查询/更新语句追加 `tenant_id = ?` 条件，防止漏写
- **租户级资源配额**：Redis 实时统计租户资源使用量，超限自动拒绝新任务
- **平台管理员豁免**：超级管理员角色可绕过租户隔离，全局查询所有租户数据

### 6.3 关键表变更

所有业务表统一增加 `tenant_id` 字段：

```sql
-- 示例：报表元数据表
ALTER TABLE report_metadata ADD COLUMN tenant_id BIGINT NOT NULL DEFAULT 0;
CREATE INDEX idx_report_metadata_tenant ON report_metadata(tenant_id);
```

### 6.4 任务调度与防抢占机制

**问题场景**：某租户提交 100万行导出任务，占用 10 个 Worker 线程 5 分钟，导致其他租户的小任务全部排队等待。

**解决方案：多层级公平调度**

| 层级             | 机制                                                                                                                                                             | 效果                           |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------ |
| **队列分级**     | RabbitMQ 优先级队列，分 5 个优先级：<br>P0: 预览、小导出 (<1万行) <br>P1: 普通导出 (1-10万行) <br>P2: 大导出 (>10万行) <br>P3: 数据同步任务 <br>P4: 后台批量任务 | 小任务自动插队，秒级响应       |
| **租户并发限制** | 每个租户同时执行的任务数上限：`min(租户配额, 总并发数 / 活跃租户数)`                                                                                             | 单个租户最多占用 20-30% 资源   |
| **时间片轮转**   | 按租户 Round-Robin 取任务：先取租户A的1个，再取租户B的1个，再取租户C的1个...                                                                                     | 绝对公平，饿死概率为 0         |
| **大任务分片**   | 100万行导出拆分为 10个 10万行的子任务，中间穿插其他租户任务                                                                                                      | 大任务变成"流水线"，不连续阻塞 |

**实现架构**：

```
任务提交 → 预估数据量 → 计算优先级 → 写入对应优先级队列
                                                      ↓
                ┌─────────────────────────────────────┐
                │   Tenant Fair Scheduler (Java)     │
                │  ┌────┐  ┌────┐  ┌────┐            │
                │  │T-1 │  │T-2 │  │T-3 │  待调度池  │
                │  └────┘  └────┘  └────┘            │
                │     ↓       ↓       ↓              │
                │   Round-Robin 公平选择             │
                └──────────────────┬──────────────────┘
                                   ↓
                         Worker 线程池执行
```

**效果保证**：

- 99% 的小任务（<1万行）排队时间 < 3秒
- 极端场景下，大任务最多让其他租户的小任务多等 10 秒（不是 5 分钟）
- 租户配额支持动态调整：VIP 租户可配置更高并发上限

## 7. 审计与合规设计

### 7.1 审计日志架构

| 层级             | 记录内容                                 | 存储方案                             |
| ---------------- | ---------------------------------------- | ------------------------------------ |
| **用户行为审计** | 登录、登出、预览、下载、导出、报表增删改 | PostgreSQL `audit_log` 表 + 定期归档 |
| **敏感操作审计** | DDL变更、数据源配置、权限变更、用户管理  | 独立审计表 + 操作前快照 + 审批流程   |
| **系统操作审计** | 定时任务执行、系统配置变更、服务重启     | 结构化日志 + ELK 持久化              |

### 7.2 防篡改设计

- 审计日志表仅允许 INSERT，禁止 UPDATE/DELETE，通过数据库触发器强制约束
- 每条日志记录哈希校验，批量写入后生成 Merkle 树根哈希，防止单条篡改
- 审计日志保留期限 ≥ 6个月，满足等保三级要求
- 支持日志导出为 PDF 并加盖时间戳，用于合规审计

### 7.3 审批流程

敏感操作触发审批流程：

1. 普通用户提交 DDL 变更 / 数据源配置请求
2. 管理员收到待办通知
3. 审批通过后系统自动执行，拒绝则记录原因
4. 全流程留痕，支持回溯

## 8. 可观测性架构设计

### 8.1 Metrics 指标体系（Prometheus + Grafana）

**业务指标**：

- `report_import_total`：报表导入总量（按租户、状态打标）
- `report_export_total`：报表导出总量
- `pdf_generate_duration_seconds`：PDF 生成耗时分布

**系统指标**：

- JVM 堆内存、GC 次数、线程数
- PostgreSQL 连接池使用率、慢查询数
- Chrome Headless 实例数、内存使用率

### 8.2 Tracing 全链路追踪

- Spring Cloud Sleuth + OpenTelemetry 埋点
- TraceId 贯穿：前端请求 → 后端 API → MQ 消息 → Worker 任务 → 渲染服务
- 前端展示 TraceId，用户反馈问题时可一键复制

### 8.3 Logging 结构化日志

- 统一采用 JSON 格式日志，包含 `trace_id`, `tenant_id`, `user_id`, `task_id`
- ELK 日志收集，支持按租户、按任务、按用户检索
- 错误日志自动触发告警（企业微信/邮件）

## 9. 扩展功能模块设计

### 9.1 报表订阅与推送模块

**核心流程**：

1. 用户在报表页面配置订阅规则（Cron 表达式、推送方式、接收人）
2. 订阅规则存入 `report_subscriptions` 表
3. K8s CronJob 每分钟扫描到期订阅，触发报表生成任务
4. 生成完成后调用推送适配器：
   - 邮件推送：Spring Mail + 附件
   - Webhook：企业微信/钉钉机器人 API
   - 异常告警：同比/环比超过阈值时额外标注红色

### 9.2 水印与防泄露模块

| 水印类型         | 实现方式                                                                       |
| ---------------- | ------------------------------------------------------------------------------ |
| **前端预览水印** | React 组件生成全屏半透明浮水印 Canvas，覆盖于报表之上                          |
| **PDF 隐形水印** | PDFBox 在页面角落添加微小白点（肉眼不可见，可通过工具提取），编码用户ID+时间戳 |
| **数据脱敏**     | MyBatis 拦截器根据用户角色对敏感字段自动脱敏（138\*\*\*\*5678）                |

### 9.3 文件病毒扫描模块

- K3s 独立部署 ClamAV 开源杀毒引擎容器
- 文件上传完成后，后端异步调用 ClamAV REST API 扫描
- 扫描通过：文件移入正式目录，继续后续流程
- 扫描失败：文件移入隔离区，记录病毒特征，通知管理员

### 9.4 模板市场模块

- 模板分类：财务报表、销售报表、人力报表、运营报表
- 模板发布流程：用户报表 → 提交发布申请 → 管理员审核 → 上架模板市场
- 模板版本管理，支持模板升级后已使用该模板的报表平滑迁移
