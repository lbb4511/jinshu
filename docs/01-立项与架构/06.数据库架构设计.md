## 1. 技术选型决策

### 1.1 数据库选型：PostgreSQL 16

| 选型维度           | 说明                                                                      |
| ------------------ | ------------------------------------------------------------------------- |
| **JSON 支持**      | PostgreSQL 的 JSONB 类型性能优异，支持 GIN 索引，完美适配报表灵活字段需求 |
| **CTE 与窗口函数** | 报表统计需要复杂 SQL，PostgreSQL 支持最完善                               |
| **分区表**         | 原生支持范围分区、列表分区，审计日志与大表可按时间分区                    |
| **全文检索**       | 内置全文检索能力，无需额外引入 ES                                         |
| **开源协议**       | PostgreSQL License，宽松友好，无商业风险                                  |

### 1.2 为什么不选 MySQL

| 对比项    | MySQL 局限              |
| --------- | ----------------------- |
| JSON 类型 | 性能较差，索引支持弱    |
| 窗口函数  | 8.0+ 才支持，功能不完整 |
| 分区表    | 限制多，子表数量上限低  |
| 乐观锁    | 需要手动实现版本字段    |

## 2. 多租户隔离方案

### 2.1 隔离级别：行级隔离

所有业务表统一包含 `tenant_id` 字段：

```sql
tenant_id BIGINT NOT NULL
```

### 2.2 隔离实现

**应用层**：MyBatis 拦截器自动注入 `WHERE tenant_id = ?` 条件

**数据库层**：

- 所有索引以 `tenant_id` 作为最左前缀
- 按租户分区（未来可选）

### 2.3 租户上下文

```
HTTP 请求
    ↓
Spring MVC 拦截器提取 tenant_id
    ↓
ThreadLocal 存储租户上下文
    ↓
MyBatis 拦截器自动注入 SQL 条件
```

### 2.4 跨租户访问控制

- 普通用户：只能访问当前租户数据
- 超级管理员：标注 `@SkipTenantFilter` 的方法可跨租户访问
- 审计日志：所有跨租户操作必须记录审计日志

## 3. 命名规范（强制执行）

### 3.1 表名规范

| 规则                | 示例                                  |
| ------------------- | ------------------------------------- |
| 全小写 + 下划线     | `report_metadata`                     |
| 模块前缀            | `report_xxx`、`task_xxx`、`audit_xxx` |
| 禁止复数形式        | ❌ `reports`，✅ `report`             |
| 关联表用 `_x_` 连接 | `user_x_role`                         |

### 3.2 字段名规范

| 规则                     | 示例                        |
| ------------------------ | --------------------------- |
| 全小写 + 下划线          | `created_at`                |
| 布尔字段用 `is_` 前缀    | `is_deleted`、`is_vip`      |
| 时间字段用 `_at` 后缀    | `created_at`、`updated_at`  |
| 外键用 `_id` 后缀        | `tenant_id`、`user_id`      |
| JSON 字段用 `_json` 后缀 | `extra_json`、`config_json` |

### 3.3 索引命名规范

| 索引类型 | 命名规则               | 示例                        |
| -------- | ---------------------- | --------------------------- |
| 主键     | `pk_表名`              | `pk_report`                 |
| 唯一索引 | `uk_表名_字段名`       | `uk_user_username`          |
| 普通索引 | `idx_表名_字段名`      | `idx_report_tenant_created` |
| 联合索引 | `idx_表名_字段1_字段2` | `idx_task_tenant_status`    |

## 4. 通用字段规范（所有业务表必须包含）

```sql
-- 以下字段每张业务表必须有
id              BIGSERIAL PRIMARY KEY,      -- 主键
tenant_id       BIGINT NOT NULL,            -- 租户ID
created_at      TIMESTAMP DEFAULT NOW(),    -- 创建时间
created_by      BIGINT,                     -- 创建人ID
updated_at      TIMESTAMP DEFAULT NOW(),    -- 更新时间
updated_by      BIGINT,                     -- 更新人ID
is_deleted      BOOLEAN DEFAULT FALSE       -- 软删标记
```

> ⚠️ **重要**：不允许物理删除，所有删除都是软删。审计日志表甚至不允许软删。

## 5. Schema 划分方案

### 5.1 划分目标

逻辑分离而非物理分离 —— 所有 schema 在**同一个 PostgreSQL 实例 + 同一个 database** 内，
通过 schema 维度进行逻辑隔离，实现：

| 目标 | 说明 |
|------|------|
| 权限最小化 | 按 schema 分配数据库账号权限，精确控制访问边界 |
| 运维差异化 | 不同 schema 使用不同的备份频率、监控策略 |
| 领域清晰 | 内管审批域（sys/meta/workflow/audit）与对外报表域（data）天然分离 |
| 事务不跨越 | 跨 schema 的 JOIN 和事务仍然可用（同实例同 database） |

### 5.2 Schema 划分

```
jinshu_db (PostgreSQL 16)
├── sys schema          # 系统管理 —— 低频变更、高安全
├── meta schema         # 动态元数据 —— 低频读写、强一致
├── data schema         # 报表数据 —— 高频写入、高吞吐
├── workflow schema     # 审批流程 —— 中频事务
├── audit schema        # 审计日志 —— 仅追加写入 + 只读查询
└── task schema         # 任务调度 —— 中频读写
```

### 5.3 各 Schema 详情

#### 5.3.1 `sys` — 系统管理 Schema

| 表名 | 说明 | 访问模式 | 安全等级 |
|------|------|---------|---------|
| `tenant` | 租户表 | 低频读写 | 🔴 高 |
| `users` | 用户表 | 中频读写 | 🔴 高 |
| `role` | 角色表（待实现） | 低频读写 | 🔴 高 |
| `permission` | 权限表（待实现） | 低频读写 | 🔴 高 |
| `user_x_role` | 用户-角色关联（待实现） | 中频读写 | 🔴 高 |
| `role_x_permission` | 角色-权限关联（待实现） | 低频读写 | 🔴 高 |

**权限策略**：仅 `jinshu_app` 账号可读写，`jinshu_audit` 只读

#### 5.3.2 `meta` — 动态元数据 Schema

| 表名 | 说明 | 访问模式 | 安全等级 |
|------|------|---------|---------|
| `report_metadata` | 报表元数据（含 columns_meta JSONB） | 低频读写 | 🟡 中 |
| `data_source` | 数据源配置 | 低频读写 | 🔴 高 |
| `report_column_def` | 报表列定义（待实现） | 低频读写 | 🟡 中 |
| `report_schema` | 报表表结构定义（待实现） | 低频读写 | 🟡 中 |
| `sys_config` | 系统配置表（待实现） | 极低频 | 🟡 中 |
| `sys_dict` | 数据字典表（待实现） | 低频读写 | 🟢 低 |

**关键约束**：
- `meta` schema 是**内管审批域**的核心，与控制面数据。所有写操作必须经过审批流程
- `columns_meta` JSONB 定义了报表列结构，`data.report_data` 中的 `data_json` 与之对应
- `schema_version` 递增控制元数据版本，历史版本由 `schema_snapshot` 保留

#### 5.3.3 `data` — 报表数据 Schema

| 表名 | 说明 | 访问模式 | 安全等级 |
|------|------|---------|---------|
| `report_data` | 报表数据宽表 | **高频写入 + 高吞吐读取** | 🟢 低 |
| `report_data_yyyy_mm` | 按月份分区（未来） | 高频写入 | 🟢 低 |

**关键约束**：
- `data` schema 是**对外报表域**的核心，主要承载数据查询和导出
- 只有 `report_data` 一张核心表，JSONB 宽表结构
- 写入仅通过 `task` 中的导入/同步任务触发，不提供直接写 API
- 读取通过开放域 API 对外提供，性能要求最高
- 未来分库优先迁移此 schema（独立实例承载高吞吐）

#### 5.3.4 `workflow` — 审批流程 Schema

| 表名 | 说明 | 访问模式 | 安全等级 |
|------|------|---------|---------|
| `report_workflows` | 报表审批工作流 | 中频事务 | 🔴 高 |

**关键约束**：
- `workflow` schema 是**内管审批域**的核心审批链路
- 状态变更必须经过状态机校验（Java 层强制）
- 历史记录仅追加（INSERT），禁止修改
- 与 `meta.report_metadata` 通过 `report_id` 关联

#### 5.3.5 `audit` — 审计日志 Schema

| 表名 | 说明 | 访问模式 | 安全等级 |
|------|------|---------|---------|
| `audit_log` | 审计日志 | 仅追加写入 + 只读查询 | 🔴 高 |
| `audit_root_hash` | 审计哈希根 | 仅追加写入 + 只读查询 | 🔴 高 |

**关键约束**：
- `audit` schema 全库只允许 INSERT 和 SELECT，禁止 UPDATE/DELETE（触发器加固）
- 独立账号 `jinshu_audit` 仅对此 schema 拥有 SELECT 权限
- 按月份分区，永久保留
- 此 schema 的数据运维需要额外审批

#### 5.3.6 `task` — 任务调度 Schema

| 表名 | 说明 | 访问模式 | 安全等级 |
|------|------|---------|---------|
| `task` | 任务主表 | 中频读写 | 🟡 中 |
| `task_log` | 任务执行日志（待实现） | 中频写入 | 🟢 低 |

**关键约束**：
- `task` schema 是批处理和异步任务的控制面
- `task` 表状态驱动任务调度，与 `meta`、`data` 松耦合
- 导入任务写入 `data.report_data`，PDF 生成读取 `data.report_data`

### 5.4 Schema 间关系图

```
sys.tenant (1) ──< (*) sys.users
                     │
                     ├──< (*) meta.report_metadata
                     │         │
                     │         ├──< (*) data.report_data
                     │         └──< (*) workflow.report_workflows
                     │
                     ├──< (*) task.task
                     │
                     ├──< (*) meta.data_source
                     │
                     └──< (*) audit.audit_log
                               └── audit.audit_root_hash
```

### 5.5 schema search_path 配置

应用层通过 JDBC 连接配置默认 search_path，避免每次 SQL 都写 schema 前缀：

```yaml
# application.yml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/jinshu_db?currentSchema=sys,meta,data,workflow,audit,task
```

> 当所有 schema 都在 search_path 中时，`SELECT * FROM users` 会自动匹配到 `sys.users`；
> 但当不同 schema 有重名表时（理论上不会），必须使用 schema 前缀限定。
> DDL 脚本中**建议始终使用 schema 前缀**以避免歧义。

### 5.6 数据库账号权限矩阵

| 账号 | 可访问 Schema | 权限 |
|------|-------------|------|
| `jinshu_app` | sys, meta, data, workflow, task | SELECT, INSERT, UPDATE（禁止 DROP/TRUNCATE） |
| `jinshu_readonly` | data (对内管域只读) | SELECT |
| `jinshu_audit` | audit | SELECT 仅 |
| `jinshu_backup` | 全库 | SELECT 仅 |

## 8. 索引设计原则

### 8.1 必须建索引的场景

| 场景         | 要求       |
| ------------ | ---------- |
| 外键字段     | 必须建索引 |
| 查询条件字段 | 必须建索引 |
| 排序字段     | 必须建索引 |
| 分组字段     | 必须建索引 |

### 8.2 索引设计规范

1. **租户优先**：所有业务索引以 `tenant_id` 作为最左前缀
2. **区分度优先**：区分度高的字段放在前面
3. **控制数量**：单表索引不超过 5 个
4. **避免冗余**：已有 `(a, b)` 联合索引，就不要再单独建 `(a)`

### 8.3 反模式（禁止）

❌ 禁止对低区分度字段建索引（如 `status` 只有 2-3 个值）
❌ 禁止对小表建索引（少于 1000 行）
❌ 禁止对更新频繁的字段建索引（索引维护成本高）

## 8. 分区与归档策略

| 表名          | 分区策略               | 保留周期    |

| `audit_log`   | 按月范围分区           | 永久保留    |

```

        ↓
```

| 备份类型     | 频率          | 保留时间 |
| ------------ | ------------- | -------- |
| 全量备份     | 每日凌晨 2 点 | 30 天    |
| WAL 增量备份 | 每 5 分钟     | 7 天     |
| 关键表导出   | 每日          | 永久     |

## 9. 数据安全

### 9.1 敏感字段加密

| 加密级别 | 适用字段             | 实现方式                     |
| -------- | -------------------- | ---------------------------- |
| 高       | 密码、密钥、身份证号 | AES-256，密钥存在 K8s Secret |
| 中       | 手机号、邮箱         | 数据库透明加密（TDE）        |
| 低       | 普通业务数据         | 无需加密                     |

### 9.2 数据库账号权限最小化

| 账号            | 权限                   | 说明                           |
| --------------- | ---------------------- | ------------------------------ |
| `jinshu_app`    | SELECT, INSERT, UPDATE | 业务账号，禁止 DROP / TRUNCATE |
| `jinshu_audit`  | SELECT ON audit_log    | 审计员只读账号                 |
| `jinshu_backup` | SELECT 全库            | 备份专用账号                   |

### 9.3 审计

- 所有 DDL 操作必须记录
- 所有跨租户查询必须记录
- 所有高权限操作必须记录

## 10. 性能目标

| 指标                     | 要求    |
| ------------------------ | ------- |
| 单条简单查询             | < 10ms  |
| 100 万行表查询（带索引） | < 100ms |
| 批量插入 1000 行         | < 50ms  |
| 并发连接数               | ≥ 500   |

---

## 11. DDL 变更流程

**所有数据库变更必须严格遵循以下流程：**

```
1. 更新对应模块的设计文档
        ↓
2. 提交设计文档 CR，DBA 评审通过
        ↓
3. 编写 DDL 脚本（标注设计文档链接）
        ↓
4. 测试环境执行验证
        ↓
5. 生产环境执行
```

> ⚠️ **红线**：不允许先改数据库再补设计文档。代码评审发现 DDL 变更无对应设计文档，直接打回。

## 12. SQL 编写规范

详见待补充文档：`docs/07-附录/01.SQL编写规范.md`

关键要点：

- 禁止 `SELECT *`
- 禁止大事务（超过 1000 行）
- 禁止深度分页（`LIMIT 100000, 20`）
- 批量操作用 `COPY` 或 `UNION ALL`
- 深度分页用游标流式读取
